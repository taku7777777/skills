# エージェントセキュリティ・ガバナンススキル

Simon Willison(lethal trifecta)、Anthropic(sandboxing / how we contain Claude)、Meta(Agents Rule of Two)、OWASP Top 10 for Agentic Applications 2026、MCP Security Best Practices、Five Eyes「Careful Adoption of Agentic AI Services」の知見を統合した設計・審査プロセス。

## 大原則

1. **二層防御を基本形にする。** LLM/アプリ層の許可リスト(permission)は一次防御にすぎない。OSレベルのサンドボックス(プロセス隔離)+ネットワークegress許可リスト+認証情報のサンドボックス外保持を本丸とする。プロンプトによる防御は**単独では**境界に数えない。分類器・プローブ等のモデル層対策は検知・低減レイヤーとしては有効だが、適応攻撃への有効性は脅威モデルと評価条件に依存し、sandbox/egressの代替にはならない(時点依存の実測は `references/evidence.md`)。
2. **lethal trifecta / Rule of Two で考える。** lethal trifecta(Willison)は「①プライベートデータへのアクセス ②信頼できないコンテンツへの曝露 ③外部への送信能力」が揃うと**データ窃取**が可能になるというモデル。MetaのRule of Twoは第3属性を「**状態変更または**外部通信」に広げており、窃取だけでなく破壊・改竄も覆う。設計の基本形は**3属性のうち1つを構造的に断つ**こと。3つ揃わざるを得ないタスクは自律実行させず人間の確認を挟む(ただし承認疲労=大原則4により、HITLは構造的遮断を尽くした後の最後の手段)。
3. **エージェントへの指示は「入力」であり「命令」ではない。** エージェントが読むすべてのコンテンツ(Web、Issue、ドキュメント、ツール結果、skills)はプロンプトインジェクションの経路。信頼境界は「誰が書けるか」で引く。
4. **承認疲労は設計の失敗。** Anthropicが2026年に公開したClaude Codeテレメトリでは権限プロンプトの約93%が承認されており、反復確認だけでは注意を維持できない。これは全製品・全組織への普遍値ではないため、対象環境でも承認率と差し戻し率を測る。人間の確認は「本当に判断が必要なもの」だけに絞り、残りは決定論的に許可/遮断する。
5. **利便性を犠牲にした安全は持続しない。** 使いにくい安全策は迂回される。安全側に倒しつつ開発速度を保つ設計(細かい許可リストの整備、安全なデフォルト)こそが本題。

## ワークフロー

### ステップ1: ユースケースと資産の整理

- エージェントに何をさせるか(コード編集のみ / コマンド実行 / Web閲覧 / 外部サービス操作 / 物理デバイス制御)
- 到達可能な資産の分類: ソースコード、認証情報、顧客データ、社内文書、本番環境。**「エージェントが読めるもの」と「エージェントの出力が届く先」を列挙**する

### ステップ2: 脅威分析(lethal trifecta マッピング)

構成ごとに3要素の有無を表にする:

```markdown
| 構成要素 | ①機密アクセス | ②信頼できない入力 | ③外部送信 | 判定 |
|---|---|---|---|---|
| コーディング(Web検索あり) | ソース+`.env` | Web検索結果 | git push, API | 3つ揃い → 要遮断設計 |
```

3つ揃う構成には構造的遮断を入れる: 認証情報の分離(→ステップ3)、egress許可リスト、信頼できない入力の隔離処理。隔離処理のパターン: 信頼できない入力に触れる側のエージェントにはツール権限を与えず(Dual LLM)、親に戻すのは自由文ではなく**構造化データのみ**。計画を先に固定し、入力を読んだ後に変更させない(plan-then-execute)。理想形はCaMeL型の能力ベース制御(制御フローとデータフローの分離)。

加えて確認する攻撃面(2026年時点の実インシデント由来):

- **設定・拡張の供給網**: `.claude/settings.json`・hooks・skills・MCPサーバー自体が攻撃経路になる。サードパーティのskill/MCPは**コードレビューと同じ審査**を通し、取得元・バージョン・ハッシュを固定して更新時に再審査する(`references/checklists.md`)。市場件数・悪性率・CVE件数は急速に変わり、分母や「脆弱」「悪意」の定義も異なるため、本文へ固定せず `references/evidence.md` の出典・調査条件・アクセス日とセットで扱う
- **永続コンテキストの汚染**(OWASP ASI06): CLAUDE.md・エージェントのメモリファイル・自動更新される設定への注入は、単発のプロンプトインジェクションと異なり**セッションを跨いで残存する**。エージェント自身が書き換え可能なコンテキストファイルへの変更は差分レビュー必須とし、定期的に全文監査する
- **許可済みegress経路の悪用**: ドメインフロンティング、許可リストの解釈差異、プロキシ悪用。allowlistは「ドメイン単位で許可した先で何ができるか」まで見る
- **秘密情報の混入**: エージェントのログ・transcript・コミットへの認証情報混入
- **無人実行構成(CI・スケジュール実行・Issue/PRトリガー)**: 自動起動されるエージェントには「最後の手段」であるHITLが存在しない。公開リポジトリのIssue・PR本文・レビューコメントを読む構成では、②を「インターネット上の誰でも書ける入力」として数える(GitHub MCP経由の私有リポジトリ流出、CamoLeak等の実例)。無人構成はtrifectaの1属性を構造的に断てない限り承認しない

### ステップ3: 実行環境の設計(二層防御の実装)

1. **Permission層**: ツール許可リスト(読み取りは**ワークスペース内に限定して**広く、書き込み・実行系は狭く。機密パス — `~/.ssh`、`~/.aws`、keychain、シェル履歴、他プロジェクト — は**読み取りもdeny**)、破壊的操作の確認必須化、hooksによる決定論的遮断(例: `rm -rf`、本番向けコマンド、`--force` push)
2. **Sandbox層**: OSレベル隔離(macOS Seatbelt / Linux bubblewrap / コンテナ / microVM)。ファイルシステムはワークスペース限定の書き込み、プロセス分離
3. **ネットワーク層**: egress許可リスト(デフォルト拒否)。パッケージレジストリ・社内APIなど必要最小限のドメインのみ。**認証情報はサンドボックスの外に保持し、プロキシで注入**する(エージェントは生トークンを見ない)
4. **権限の時限性・スコープ**: 長寿命トークンを避ける。タスク単位のワークスペース分離(worktree等)で影響半径を限定
5. **多層の検証**: sandboxの実挙動は公式ドキュメントを信用せず**経験的に検証**する(実際に遮断されるか試すテストを持つ)。バイパスは再発している攻撃面であり、一度の確認で恒久安全とみなさない

### ステップ4: 組織導入のガバナンス

- **ロール別ポリシー**: 全員一律ではなく、リポジトリの機密度×メンバーの役割で権限テンプレートを分ける。managed settings で中央配布し、個人設定での緩和を制限
- **審査プロセス**: MCPサーバー・skills・hooksの導入申請と審査基準を明文化(チェックリスト参照)。バージョン固定と更新時の再審査
- **監査ログ**: 誰のエージェントが・何を実行し・何にアクセスしたかの記録と保持。インシデント時の追跡可能性
- **インシデント対応**: トークン一斉失効の手段、エージェント全停止の手段、報告フロー。「エージェントが変なことをした」の受付窓口
- **規制・ガイドライン対応**: 国内はAI事業者ガイドライン(エージェント追記あり)・業界ガイドライン、国際的にはEU AI Act・ISO/IEC 42001 との対応表を作り、「なぜこの構成で十分か」を説明可能にする
- **段階的導入**: 読み取り専用/低リスクリポジトリから開始 → 計測(承認率・インシデント・生産性)→ 拡大。最初から全権限を配らない

### ステップ5: 定期レビュー

- 許可リストの棚卸し(使われていない許可の削除、頻繁な手動承認の許可リスト化)
- 新しい攻撃手法のウォッチ(OWASP Agentic Top 10 の改訂、CVE、プラットフォームのセキュリティ更新)
- サンドボックス・permission実挙動の再検証(プラットフォーム更新のたび)
