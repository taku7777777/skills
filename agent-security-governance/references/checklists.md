# エージェントセキュリティ 審査チェックリスト集

## A. MCPサーバー導入審査

### 提供元・供給網

- [ ] 提供元の信頼性(公式ベンダー / 著名OSS / 個人)。メンテナンスの活発さ、既知の脆弱性報告
- [ ] インストール方式とバージョン固定(タグ/ハッシュ固定か。自動更新は再審査をすり抜ける)
- [ ] 依存パッケージの健全性(SCAスキャン。npm等の供給網ワーム事例を念頭に)

### 権限・データフロー

- [ ] 要求する認証情報のスコープは最小か(読み取りで足りるのに書き込み権限を要求していないか)
- [ ] Token Passthrough をしていないか(受け取ったトークンを別サービスへ横流しする設計はMCP仕様で禁止)
- [ ] Confused Deputy: サーバーが「ユーザーの権限」と「エージェントの指示」を混同する構造がないか
- [ ] ツールの説明文(description)にインジェクションがないか(ツール定義自体がプロンプトに入ることを悪用する攻撃)
- [ ] ツール定義が更新時に変わる場合の検知(rug pull: 審査後に悪意ある定義へ差し替え)
- [ ] 複数サーバー併用時のツール名衝突・シャドーイング(他サーバーの正規ツールの使い方を指示する説明文で挙動を書き換え誘導する攻撃)がないか
- [ ] リモート(HTTP)サーバーの場合: セッショントークンの寿命・バインド・TLS を確認。stdio(ローカル)かリモートかで審査基準を分ける

### trifecta判定

- [ ] このMCPで新たに「機密アクセス」「信頼できない入力」「外部送信」のどれが加わるか。既存構成と合わせて3つ揃わないか

## B. skills / hooks / 設定ファイル審査

- [ ] skillの全文を読む(SKILL.md+参照ファイル+同梱スクリプト)。「〜を無視して」「まず〜へ送信して」等の埋め込み指示がないか
- [ ] 同梱スクリプト・バイナリの内容確認(難読化されたコードは原則却下)
- [ ] 静的スキャナの通過を安全の根拠にしない(パッキングによるスキャナ回避が実証済み)。発行者の署名・実在性を確認できるか
- [ ] hooksが実行するコマンドの完全な把握(hooksは無確認で実行される=最も高権限)
- [ ] 外部からの取得物(`curl | sh` 型のインストール手順)を含まないか
- [ ] `.claude/settings.json` への変更提案は差分レビュー必須(permission緩和・hooks追加の混入)
- [ ] リポジトリに置かれた設定・skillは「そのリポジトリに書き込める全員」が書けることを前提に信頼度を決める

## C. エージェント実行環境の構成監査

### Permission層

- [ ] デフォルト拒否になっているか(許可リスト方式)
- [ ] 破壊的コマンド(削除系、force push、本番デプロイ、課金操作)が遮断or確認必須か
- [ ] 承認履歴の確認: 高頻度で手動承認されているものは許可リスト化 or 設計見直し(承認疲労の温床)
- [ ] permissionバイパスモード(`--dangerously-skip-permissions` 等)の利用条件が組織ポリシーで定義され、managed settingsで技術的に制限されているか(少なくともsandbox+egress制御なしでのバイパスは禁止)

### Sandbox層

- [ ] ファイル書き込みがワークスペースに限定されているか。`~/.ssh`、`~/.aws`、シェル設定(`.zshrc`等)、他プロジェクトへの書き込みが遮断されるか
- [ ] 機密パス(`~/.ssh`、`~/.aws`、keychain、シェル履歴、ブラウザプロファイル)の**読み取り**も遮断されるか(書き込み制限だけでは trifecta の①が残る)
- [ ] エージェントが書き換え可能なコンテキストファイル(CLAUDE.md、メモリ)への変更が差分レビューされるか(永続汚染の監査)
- [ ] **実挙動テスト**: 遮断されるはずの操作を実際に試して確認したか(ドキュメント信仰の禁止)。プラットフォーム更新後に再実施しているか

### ネットワーク層

- [ ] egressはデフォルト拒否+許可リストか
- [ ] 許可ドメインの棚卸し: そのドメインで「アップロード・書き込み」が可能か(gist、パステビン的機能、Camo型プロキシの悪用可能性)
- [ ] 認証情報がサンドボックス内に置かれていないか(プロキシ注入 or 環境分離)
- [ ] 許可ドメインからのリダイレクト先へ追従が許可されないか(リダイレクトは許可判定後に発生する)
- [ ] DNSトンネリング: DNS解決が素通しならクエリ名にデータを載せた窃取が可能。DNS解決自体がプロキシに集約・監視されているか

### 運用

- [ ] 監査ログの取得範囲と保持期間。個人を跨いだ集計が可能か
- [ ] トークン失効・全エージェント停止の手順が存在し、試したことがあるか
- [ ] インシデント報告の窓口と基準が周知されているか

## D. エージェント成果物の受け入れ(supply chainとしてのAI出力)

- [ ] エージェントが追加した依存パッケージの実在性・健全性確認(タイポスクワッティング、ハルシネーション由来の偽パッケージ)
- [ ] エージェントが生成した設定変更(CI、Dockerfile、IaC)は人間レビュー必須(HARDゲート)
- [ ] コミット・PR本文への機密情報混入チェック(自動スキャン)

## E. 組織導入の成熟度セルフアセスメント(簡易)

| レベル | 状態 |
|---|---|
| 0 | 個人が各自の判断で利用。ポリシーなし |
| 1 | 利用ガイドライン文書あり。技術的強制なし |
| 2 | permission層の統一設定を配布。監査ログ収集 |
| 3 | 二層防御(sandbox+egress制御)を標準化。MCP/skills審査プロセス運用 |
| 4 | ロール別ポリシー、定期監査、インシデント対応訓練、実挙動の継続検証 |

導入支援では現在地と次のレベルへの最短経路を提示する。
