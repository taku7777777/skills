# AI-Adaptable テスト戦略スキル

Kent Beck(TDD / Test Desiderata)、Michael Feathers(レガシーコード)、Gojko Adzic(Specification by Example)、Google Testing Blog・『Software Engineering at Google』(Test Sizes / flakiness / テストダブル)、testing in production(Charity Majors)の知見を、AIエージェント協働の文脈で再構成したテスト戦略プロセス。3層構造(**戦略層=本スキル** → 分析層 `test-analysis` → 設計層 `test-design`)の最上位。

## 大原則

1. **テストはAI時代の「実行可能な仕様」であり「報酬関数」。** エージェントはテストが通ることを目標に動く。テストが仕様を正しく表現していなければ、エージェントは「間違ったものを完璧に」作る。テスト品質への投資は委任可能範囲への投資そのもの。
2. **テストは騙せてはならない。** エージェントは悪意なくテストをすり抜ける最短経路を見つける(実装に合わせて期待値を書き換える、特殊ケースをハードコードする、テストをスキップする)。**テストの検出力を弱める変更 — 削除・スキップ・アサーションの緩和・期待値の書き換え・タイムアウト/リトライの拡大 — は人間承認ゲート(HARD)にする**(新規追加や期待を変えないリファクタはSOFT。一律HARDは承認をボトルネック化させ形骸化する — `harness-engineering` のゲート分類と整合)。この防御は現行モデルの癖への対症療法ではなく「検証を被検証者の善意と能力に依存させない」という監査の原則であり、モデルが賢くなっても外さない。
3. **振る舞いをテストし、実装をテストしない。** 内部構造に結合したテストはリファクタのたびに壊れ、AIにも人間にも「壊れたら直せばいい騒音」として学習される。public な振る舞い・契約に対してテストを書く。
4. **失敗メッセージは修正者への手紙。** テスト名は仕様文、失敗出力は期待と実際の差分が一目で分かる形に。エージェントの自己修正成功率はエラーメッセージの質で決まる。
5. **速いテストだけが使われる。** フィードバックが遅いテストは人間にもエージェントにも回避される。テストサイズを管理し、内側のループを秒単位に保つ。サイズは機械的に検証可能な制約で定義する(Small=単一プロセス・I/O/ネットワーク/sleep禁止、Medium=単一マシン・localhost可、Large=それ以上)。
6. **どのテストも全ての美点は持てない**(Beck の Test Desiderata)。テストが望ましく持つ性質(isolated / composable / deterministic / specific / behavioral / structure-insensitive / fast / writable / readable / automated / predictive)は互いにトレードオフし、全部を満たすテストは存在しない。配分の各層で「優先する性質と意図的に犠牲にする性質」を宣言する(単体=fast・deterministic優先、E2E=predictive〈全部通れば出荷可能と言える度合い〉のためにfastを犠牲、characterization=一時的にpredictiveを諦めて現挙動を固定)。「このテストはどの美点を捨てていて、それは意図か」がテスト追加・削除の共通言語になる。

## ワークフロー

### ステップ1: 現状分析とテスト戦略の決定

- 対象の性質を分類: コアドメインロジック / 統合・配線 / UI / インフラ
- **テスト配分の原則**: 配分の「形」(ピラミッド/トロフィー)は目的ではなく、決定性とフィードバック速度の制約下で忠実度を最大化した結果にすぎない。ドメインロジックが厚い対象はインフラ非依存の高速単体テストで網羅(DDDのドメイン層純粋性はテスト容易性でもある)。配線・クエリ・シリアライズ・ORマッピングが主体でドメインロジックが薄い対象は、モックで刻んだ単体では守れない — ローカルで本物を起動した狭い統合テスト(Testcontainers等でDB・キューを実起動、外部SaaSはfake。1統合点ずつ薄く=narrow integration test)を主力にする(Testing Trophy)。E2Eはクリティカルパスの少数精鋭。「統合=遅い・flaky」は共有環境依存と非決定性が真因であり、hermeticに保てば崩れる — 配分はTest Sizesの機械的制約(決定性・実行時間)で管理する
- **サービス境界は契約テストで固定**: 統合の正しさをE2Eで守らず、consumer-driven contract(Pact等)やOpenAPI/イベントスキーマ検証で「提供側と利用側の合意」をテスト化する。エージェントが片側だけ変更して契約を破る典型事故を、統合環境なしの高速テストで検出できる。個々の契約ケース(相互作用×provider state、スキーマ進化の互換ケース)の設計は `test-design`(技法カタログ§11)の領分 — 本スキルはツール選定とCI配線(ステップ5)を受け持つ
- **検証ポイントは事前テストに限らない(どこで検証するかの経済判断)**: すべてのリスクをマージ前テストで防ぐのは最適ではない。リスクごとに「マージ前テストで防止」「デプロイ時に検出(カナリア分析・SLOバーンレート監視・synthetic monitoring)」「フィーチャーフラグで影響半径を絞り即時復旧」のどれで守るかを配分する。どのリスクを本番検出に回すかの判断基準(事前再現コスト・本番データ/スケール依存・復旧容易性)の定義と適用は `test-analysis` ステップ4を正とする。`test-analysis` が「検証ポイント=production」と判定した条件はここで受け、監視条件・復旧手段の整備をアクションに含める(条件の定義は分析層、運用は本スキル)。本番検出を選べるのはプログレッシブデリバリと可観測性が前提なので、その整備状況(段階ロールアウト可否・メトリクス・アラート)も現状診断に含める
- **LLM機能は評価セット(eval)を別枠の回帰資産として配分に組み込む**: 何を測るか(入力コーパス×ルーブリック)の定義は `test-analysis`/`test-design` の領分。戦略層が持つのは運用 — 評価セットをコードと同じリポジトリでバージョン管理し、プロンプト・モデル・パラメータの変更を回帰ゲートのトリガーにする(モデル更新は依存更新ではなく挙動変更として扱う)。合否は単発一致でなく統計(合格率の閾値・ベースラインからの劣化)で判定し、コスト・レート制限があるため専用ジョブ・別予算で管理する。本番の失敗事例を評価セットへ還流する — evalは腐る資産で、living documentation(ステップ2)と同じ保守対象
- 既存テストの健康診断 — 数値はデータの取り方まで示す: 実行時間(ランナーのレポートから遅い順)/ flaky率(CIのリトライ通過記録を集計。無ければ同一スイートを数回実行して揺れたテストを列挙し、「flakyを計測する仕組みが無い」ことも診断に書く)/「実装の写経」率(モックの戻り値をassertするだけ — 無作為サンプリングで判定し件数と根拠を示す)/ カバレッジの空白(エラーパス・境界値 — 未実行分岐と例外処理のgrepを突合)

### ステップ2: 仕様のテスト化(Specification by Example)

新機能・仕様変更では、実装前に受け入れ基準を具体例で固定する:

- Given/When/Then 形式で**境界の具体例**を含める(`requirements-definition` スキルの成果物を引き継ぐ)
- ここで作るのは受け入れ合意のための**少数の鍵となる例(key examples)**であって網羅ではない(Adzicの仕様の洗練 — 例が多すぎる仕様は読まれず living documentation として機能しない)。境界値・組み合わせ・状態遷移の系統的なケース導出が要る規模なら `test-design` に委譲し、その出力を具体例セットに取り込む
- 具体例を作る過程で曖昧さを検出し、決まっていないケースはユーザーに確認
- この具体例セットが「エージェントに渡す完了の定義」になる — failing テストとして先に置けば、ハーネスの機能リストとして機能する(`harness-engineering` 参照)
- 具体例セットは実装後も**仕様書として維持する**(living documentation)。テストが通る限り最新であることが保証された、エージェントが参照して腐らない仕様になる
- **性質ベーステスト(property-based testing)の併用**: 具体例に加えて、成り立つべき性質を定義する(例: serialize→deserializeは恒等 / ソート結果は順序保証かつ要素集合が不変 / 割引後価格は0以上元価格以下)。具体例はハードコードで騙せるが、性質は特殊ケースの埋め込みでは通せず騙すコストが桁違いに上がる(ただし性質が弱いと退化実装でも通る — 例: serialize/deserializeが両方恒等関数ならround-trip性質は成立してしまう。具体例テストとの併用が係留になる)。性質・メタモルフィック関係の体系的な洗い出しは `test-analysis`、技法としての適用手順は `test-design`(技法カタログ§7)の領分 — 本スキルが決めるのは「具体例+性質の二層で仕様を固定する」方針とPBTツールのチーム標準化。ツール: fast-check(TS)/ Hypothesis(Python)/ jqwik(Java)。失敗時の最小反例(shrinking)はエージェントの自己修正入力としても優秀

### ステップ3: レガシー・リファクタ時の安全網構築(Feathers方式)

テストのないコードを変更する場合:

1. **Characterization Test**: 現在の挙動をそのまま記録するテストを書く(正しさではなく「今こうである」の固定)。大量の入出力ペアで固定するゴールデンテスト/スナップショット/承認テスト(Approval Tests)が有効。記録中に明らかなバグを見つけても期待値は**現挙動のまま**固定し、バグは疑いとして別途報告する(現挙動に依存する呼び出し元がありうる。修正は安全網完成後の独立した意思決定 — Feathers)。エージェントは「あるべき挙動」を期待値に書きがちで、それはcharacterizationの目的(現挙動からの差分検出)を壊す
2. **Seam(継ぎ目)の確保**: テスト不能な依存(DB、時刻、外部API)を切り離せる点を最小限の変更で作る。プロセス境界のゴールデンテストのようにseamなしで現挙動を固定できるなら1が先でよいが、**テストを書くために先にコードを変えざるを得ない場合(レガシーコードのジレンマ)は順序が逆**: コンパイラ・IDEで機械的に安全と言い切れる編集だけで先にseamを入れ、その後characterizationで固定する(Feathersの変更アルゴリズムでは依存の切断がテスト記述に先行する)
3. 安全網ができてから本来の変更に着手(`refactoring-strategy` スキルと連携)
4. ライブラリ脱却・大規模移行では「移行前後で同じ入力に同じ出力」を保証する**差分テスト**(両実装を並走させ結果比較)を検討

### ステップ4: AI協働のためのテスト整備

エージェントがテストを「読み・書き・実行」する前提で:

- **テストの自己完結性**: 1テストを読むだけで前提・操作・期待が分かる(共有fixtureの魔法を減らす、テストデータビルダーで意図を表す。DAMP=Descriptive And Meaningful Phrases: テストでは重複排除より自己記述性を優先する。ただしDAMPはDRYの置き換えではなく補完 — ビルダーのように意図を隠さない共通化は維持してよい)。**既存テストは事実上のプロンプト**でもある: エージェントは新規テストを書くとき周辺の既存テストを模倣するため、写経テスト・巨大fixtureが残る領域は悪例が量産の種になる。よく触る領域から手本を整備し、CLAUDE.mdに「テストの書き方は<代表ファイル>を参照」と明示する(ステップ1で測った写経率の修繕は変更頻度順に)
- **決定性**: 時刻・乱数・順序・並行性への依存を排除。標準手段は時刻のクロック注入・乱数のシード固定(seamの作り方はステップ3)。注入できない共有環境・E2Eに限り「現在時刻からの相対」データ等で代替する(データ設計の詳細は `test-design`)。flaky テストは即隔離(エージェントに「たまに落ちるのは無視していい」と学習させない)
- **失敗テストのトリアージ**(エージェントの既定動作として規定): テストが落ちたらまず実装を疑う。(1) 実装の欠陥 → 実装を直す(テストは触らない) (2) 意図的な仕様変更への追随 → 変更の根拠(要件・Issueのどの変更か)を差分に明記して人間承認へ。トレースを示せない期待値変更は認めない (3) テスト自体の欠陥(flaky・誤ったオラクル・環境依存)→ その場で直さず隔離し調査結果を添えて報告。「テストを通すためにテストを弱める」は選択肢に存在しないとCLAUDE.md等で明文化する。隔離したflakyの切り分け(単独実行とスイート実行の比較、実行順シャッフル、並列度変更、時刻・乱数シード固定)は委任しやすい定型タスク — 原因を共有状態/実行順依存/時間依存/並行性に分類させる
- **テストと実装の生成を分離する**: 同一エージェントが同一コンテキストで実装とテストを同時生成すると、実装の誤解をテストが共有し検証の独立性を失う(`impl-review` はこの場合テストの通過を割り引く)。仕様からテストを先に生成・レビューして固定してから実装に着手させる(ステップ2のfailingテスト先置きと同じ構図)。既存実装に後からテストを足す場合もコンテキストには実装差分ではなく仕様・テスト条件(`test-analysis` の成果物)を渡す — 実装を読ませて書かせたテストは写経になる(現挙動固定が目的のcharacterizationだけが例外)
- **実行導線の明確化**:「このファイルだけ実行」「失敗したものだけ再実行」がコマンド一発でできる。CLAUDE.md にテストコマンドと規約を記載
- **AIが書いたテストのレビュー観点**:
  - [ ] プロダクションコードを壊したら落ちるか — 実装側の差分だけを退避してredを実際に確認する(最安の検証)。テストと実装が同じ未コミット差分にあると素の `git stash` はテストごと退避するため `git stash push -- <実装ファイル>` でプロダクションコードのみ退避する(コミット済みなら対象をrevertした状態で)。恒常的にはミューテーションテストで測る
  - [ ] アサーションが意味を持つか(`toBeDefined()` だけ、実装と同じ計算式の再実装、になっていないか)
  - [ ] 実装の内部構造(privateメソッド、呼び出し回数)に不要に結合していないか
  - [ ] 境界値・エラーパスを含むか(生成テストで最も抜けやすい領域。含まれている「はず」で済ませず実物で確認する)
- **ミューテーションテストの活用**: テストスイート自体の検出力を定期計測する。「カバレッジが高いのに検出力が低い」はAI量産テストの典型的な負債

### ステップ5: 継続的な品質ゲート化(提案で終えず実装まで行う)

各ゲートの導入手順・しきい値の目安・形骸化パターンと導入の順序は `references/quality-gates-catalog.md` に従う(以下は要約 — 乖離したらカタログ優先):

- **テストを弱める全経路の検知**: 報酬ハックは削除だけでなく「弱体化」で起きる。CI上のdiff検査やClaude CodeのPreToolUseフックで検出し該当差分は人間レビュー必須にする(`harness-engineering` のHARDゲート = 自動マージ不可・人間承認必須)。テストコードの変更(削除、`skip|only`、アサーション緩和や期待値書き換え、タイムアウト/リトライ延長)はテストファイルのdiffに現れるので削除行検知で捕まる。**盲点はテストコードに現れない経路** — スナップショット/ゴールデンの一括再生成(`-u` 等は期待値を書き換えるがテストコードに触れない)、カバレッジ除外プラグマ(`istanbul ignore` 等)、テストコマンド定義・CI設定の変更(実行対象の絞り込み・ジョブ無効化) — をdiff検査に明示的に含める。正当な変更まで遮断せず、変更理由と仕様トレースを添えさせて承認フローに乗せる
- **カバレッジ**: 新規変更行に対してのみゲート化(diff-cover / patch coverage)。全体カバレッジの数値目標はゲーミングされる
- **flaky検出と隔離の出口管理**: CIのリトライ通過テストを記録・ラベル付けして隔離ジョブへ。隔離は入口であって処分ではない — オーナーと修復期限(例: 2週間)を付け、期限超過は「安全網の意図的な喪失」として削除可否を人間判断に上げる。復帰は反復実行(例: 連続100回green)で決定性を確認してから。隔離操作自体はスキップと同等の検出力低下なので上記の検知ゲートを通す
- **検出力の定点観測**: ミューテーションテスト(Stryker/PIT)は差分実行(変更コードの生存ミュータントのみ算出)をPRレビューに提示する形を基本にする — 全体スコアの閾値監視は計算コストが高いうえ行動につながらず、数値目標はカバレッジ同様ゲーミングされる(Google/Facebookの大規模運用の知見)。全量実行は傾向把握の定期ジョブに留め閾値ゲートにはしない
- **テスト実行時間の予算**: fitness functionとしてCIで監視(超過をfailに)
- **テスト選択(Test Impact Analysis)による二層CI**: スイートが実行時間予算を超えたら、PR時は変更差分から影響を受けるテストのみ選択実行し、全量はマージ後/夜間に回す(選択漏れは全量実行が安全網として拾う)。`test-analysis` の変更起点分析がここに接続する。マージ後全量が壊れた際の犯人特定(bisect)と差し戻し手順もセットで決める
- **本番検出のゲート化**: `test-analysis` が「検証ポイント=production」とした条件をデプロイパイプラインのゲートとして実装する — カナリア/段階ロールアウトのベースライン比較(逸脱で自動ロールバック)、SLOバーンレートアラート、クリティカルパスのsynthetic monitoring。「本番で検出する」と決めたのに監視が未実装の状態は、カバレッジの空白と同じ扱いで診断に載せる
- **契約テストのCI配線**: サービス境界の契約検証をconsumer/provider双方のパイプラインに組み込む(Pact broker + can-i-deploy 等)。契約を破る変更をマージ前に検出し、互換性未確認のデプロイを遮断する(個々の契約ケースの設計は `test-design` 技法カタログ§11)
- **ファジングの常時運用**: coverage-guidedファジング(ハーネス・オラクル設計は `test-design` 技法カタログ§9)のCI常時実行または夜間ジョブ化と、シードコーパスの資産管理。発見したクラッシュ再現入力は回帰テストへ昇格する

## アンチパターン早見表

| アンチパターン | 症状 | 対処 |
|---|---|---|
| 実装の写経 | モック設定とassertが鏡写し | 相互作用(呼び出し回数・引数)でなく状態・戻り値・副作用を検証(state testing > interaction testing — SWE at Google)。モックは境界(外部システム)のみ |
| テストのための実装公開 | private を public にしてテスト | public の振る舞い経由で検証。できなければ設計の凝集不足を疑う |
| 巨大セットアップ | 1テストに50行の準備 | テストデータビルダー、Object Mother。設計の結合過多のシグナルでもある |
| スナップショット濫用 | 差分の意味を誰も読まずに更新 | スナップショットは安定した出力形式のみ。「更新が儀式化したら削除」。一括再生成(`-u`)はステップ5の弱体化検知対象 |
| 検証なきカバレッジ | 数値は高いがバグを通す | ミューテーションテストで検出力を測る |
| モック地獄 | 境界内までstub/mockだらけで、テストが構造に固着 | real → fake → mock の優先で置換(SWE at Google)。よく使う境界には公式fake(インメモリ実装)を整備し、エージェントにもそれを使わせる |
| E2E偏重 | 遅い・flaky・原因特定不能 | 決定性を保てる下位レベル(単体・hermetic統合)へ再配分。E2Eはクリティカルパスのみ |
