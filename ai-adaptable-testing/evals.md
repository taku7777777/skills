# ai-adaptable-testing 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | うちのリポジトリ、単体と E2E のどちらに寄せるべきか誰も決めてなくて、テストの持ち方がバラバラ。方針を一緒に決めてほしい | テスト配分の立案(戦略層の中心用途) |
| F2 | Claude に実装を任せたら、通らないスナップショットを全部 `-u` で更新して green にしてた。こういうのを仕組みで防げないか | テスト弱体化検知・品質ゲート整備 |
| F3 | テストが1本もない月次集計バッチに手を入れることになった。壊してないと言い切れる状態を先に作りたい | レガシーコードへの安全網構築 |
| F4 | 新機能の受け入れ基準を、実装が始まる前に「実行して確認できる形」でチームに残しておきたい | 仕様のテスト化(Specification by Example) |
| F5 | LLM で問い合わせ文を分類する機能、モデルを更新するたびに挙動が変わる。回帰をどう守り続ければいい? | LLM機能の評価セット運用(回帰ゲート) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | この決済まわりの仕様、何をテストすべきか観点を洗い出して優先順位を付けてほしい | `test-analysis` が発火する(分析層=何を) |
| N2 | 会員登録フォームの入力チェック、どの値の組み合わせで試すべきかテストケースに落として | `test-design` が発火する(設計層=どう) |
| N3 | エージェントに実装を委任するときの承認フロー全体(コミット・デプロイ含む)を設計して | `harness-engineering` が発火する(テストに限らないゲート設計全般) |
| N4 | マージからデプロイまでのパイプラインを整備してリリースを自動化したい | `cicd-release` が発火する(パイプライン構築そのもの) |

## 検証シナリオ

### S1: SaaS のテスト戦略診断と配分見直し
- **タスク**: 「架空の会議室予約 SaaS(Rails + PostgreSQL + 外部カレンダー連携 API)。テストは約800件で CI に25分かかり、落ちたら再実行が慣習化。予約枠の重複判定などドメインロジックは Controller のテストで間接的にしか検証されていない。テスト戦略を診断して、直す順番を出してほしい」と依頼する。
- **スキルなしで予想される欠落**: 診断数値の測定方法の明示(flaky率をどう取るか、写経テスト率のサンプリング)、Test Sizes の機械的制約(Small/Medium/Large)による配分管理、外部カレンダー API 境界の契約テスト化、アクションと診断結果の対応付け。一般論の「テストピラミッドにしましょう」で終わる。
- **合格条件**:
  - [ ] 現状診断の各指標(実行時間 / flaky率 / 写経率 / カバレッジ空白)に測定値または測定方法が付いている
  - [ ] テスト配分が対象分類(コアドメイン/統合・配線など)× テストサイズの形で現状→目標として示される
  - [ ] 外部カレンダー連携の境界に契約テストまたは fake 整備への言及がある
  - [ ] 優先順位付きアクションの各項目に完了条件(機械判定できる形)がある
  - [ ] 配分の根拠が形(ピラミッド等)でなく決定性・速度の制約から説明されている(人間判定)

### S2: テストゼロのレガシーバッチへの安全網
- **タスク**: 「10年ものの給与控除額計算バッチ(Java、テストなし)。税率テーブルの改定対応で計算ロジックを変更する必要がある。着手前にやるべきことを具体的に示して。なお過去の出力を見ると、端数処理が仕様書と食い違っている疑いのある行が見つかっている」と依頼する。
- **スキルなしで予想される欠落**: characterization test で「バグの疑いがあっても期待値は現挙動のまま固定し、疑いは別途報告」という判断(あるべき値に直して固定しがち)。seam 導入と固定の順序判断(レガシーコードのジレンマ)。ゴールデンテストによる大量入出力固定という発想。
- **合格条件**:
  - [ ] 変更着手前に現挙動を固定するテスト(characterization / ゴールデン)の作成が計画に含まれる
  - [ ] 端数処理の疑いについて「期待値は現挙動のまま固定し、修正は別の意思決定として報告する」方針が明示される
  - [ ] テスト不能な依存(時刻・DB 等)がある場合の seam 導入手順(機械的に安全な編集で先に切る)への言及がある
  - [ ] 安全網の完成後に本来の変更(税率改定)へ進む順序になっている

### S3: エージェントの PR への途中介入(テスト弱体化)
- **タスク**: 「エージェントに在庫引当ロジックの修正を任せたら PR が来た。テストは全部通っている。diff を見ると、失敗していた2テストに `skip` が付き、1テストはタイムアウトが 5s→60s に伸び、アサーションが `toEqual` から `toBeDefined` に変わっている。このままマージしていい?」と、diff 抜粋を添えて聞く(「テスト戦略」という語は使わない)。
- **スキルなしで予想される欠落**: これをテスト検出力の弱体化(報酬ハック)として体系的に指摘すること。skip・タイムアウト延長・アサーション緩和を同じカテゴリの経路として列挙すること。再発防止として弱体化検知の HARD ゲート(diff 検査・フック)を実装まで提案すること。個別の diff 指摘だけで終わる。
- **合格条件**:
  - [ ] マージ不可の判定と、skip / タイムアウト延長 / アサーション緩和の3点すべてが弱体化として指摘される
  - [ ] 失敗テストのトリアージ手順(まず実装を疑う、トレースなき期待値変更は認めない)に沿った差し戻し指示がある
  - [ ] 再発防止として弱体化検知ゲート(CI の diff 検査またはフック)の整備が提案される
  - [ ] スナップショット一括再生成や CI 設定変更など「テストコードに現れない弱体化経路」にも言及がある
