# 品質ゲート実装カタログ(導入手順・運用目安・形骸化パターン)

SKILL.md ステップ5(継続的な品質ゲート化)の詳細展開。各ゲートを「目的 / 導入手順 / しきい値・運用の目安 / 失敗モード(形骸化のパターン)」で記述する。ゲートの HARD/SOFT/AUTO 分類とエスカレーション設計は `harness-engineering`(`references/gate-design-catalog.md`)を正とする — 本カタログはテスト側ゲートの実装詳細を受け持つ。

## 1. テスト弱体化検知(assertion削除・skip・golden更新の監視)

**目的**: 「テストを通すためにテストを弱める」全経路を機械検出し、人間承認(HARD)に乗せる。報酬ハックは削除だけでなく弱体化で起きる — 検出力の低下は静かに進行し、気づいたときには安全網が穴だらけになっている。

**導入手順**:
1. 検知対象の一覧を確定する。テストファイルのdiffに現れるもの(テスト削除、`skip`/`only`/`todo` の追加、アサーション行の削除・緩和、期待値の書き換え、タイムアウト/リトライの延長)と、**テストコードに現れない盲点**(スナップショット/ゴールデンの一括再生成、カバレッジ除外プラグマの追加、テストコマンド定義・CI設定の変更による実行対象の絞り込み)の両方を含める
2. CI上のdiff検査として実装する(例: テストファイルの削除行数・skipパターンのgrep、スナップショットファイルの変更数閾値、CI設定パスへの変更検出)。ローカルではClaude CodeのPreToolUseフックで同じ判定を早期警告として出す — **CIが強制、フックは補助**(フックはエージェント側の環境改変で無効化されうるため、単独では防御にならない)
3. 検出時のフローを設計する: 該当PRに自動ラベル+人間レビュー必須化(自動マージ不可)。遮断ではなく承認フローへの誘導 — 変更理由と仕様トレース(どの要件・Issueに基づくか)を差分に添えることを通過条件にする
4. 検知ルール自体をハーネス資産としてHARD管理下に置く(検知ルールを緩める変更も検知対象)

**しきい値・運用の目安**: 誤検知(正当な仕様変更への追随)が3割を超えるようなら、ルールを緩めるのではなく「トレース添付で即承認」の高速レーンを整える。スナップショット更新は「1PRあたりNファイル以上の一括更新」のみ引っ掛ける形から始めると騒音が少ない。

**失敗モード(形骸化)**:
- **承認の儀式化** — レビューアがトレースを読まずapproveし始めたら、検知は動いていても防御は死んでいる。承認時にトレースの妥当性を1行コメントで残す運用にして監査可能にする
- **検出パターンの陳腐化** — 新しいテストランナー・新しいスキップ手段の導入で検知をすり抜ける。テスト基盤の変更時に検知ルールの追随をチェックリスト化する
- **全部HARDにして詰まる** — 一律承認必須は承認のボトルネック化と形骸化を招く(SKILL.md 大原則2)。新規追加・期待を変えないリファクタはSOFTに残す

## 2. flakyテストの隔離と返済運用

**目的**: 非決定的なテストをブロッキングゲートから外しつつ、「隔離＝実質削除」への転落を防ぐ。flakyは人間以上にエージェントを害する — 失敗の誤帰属による無関係な「修正」、パスするまでのリトライループ、「たまに落ちるのは無視していい」という学習。

**導入手順**:
1. 検出: CIのリトライ通過テスト(1回目fail→再実行pass)を記録・集計する。リトライ機構がなければ、同一スイートの複数回実行で揺れたテストの列挙から始める(計測の仕組みがないこと自体を診断に書く — SKILL.md ステップ1)
2. 隔離: flaky判定されたテストを隔離ジョブ(非ブロッキング)へ移す。隔離操作は検出力低下なのでゲート1の検知対象に含め、隔離時に**オーナーと修復期限**(目安: 2週間)を必須メタデータにする
3. 返済: 切り分け(単独実行とスイート実行の比較、実行順シャッフル、並列度変更、時刻・乱数シード固定)は定型タスクとしてエージェントに委任しやすい — 原因を共有状態/実行順依存/時間依存/並行性に分類させる(SKILL.md ステップ4)
4. 復帰・出口: 修正後、反復実行(目安: 連続100回green)で決定性を確認してから本流へ戻す。期限超過は「安全網の意図的な喪失」として削除可否を人間判断に上げる — 黙って放置しない

**しきい値・運用の目安**: 隔離ジョブの滞留数を定点観測し、滞留が増加傾向なら新規流入(テストの書き方の問題)と返済速度(オーナー運用の問題)のどちらが原因かを切り分ける。ブロッキングスイートのflaky率は実質ゼロを維持する — 「ちょっとだけflaky」を許すとリトライ文化が戻ってくる。

**失敗モード(形骸化)**:
- **隔離が墓場になる** — 期限もオーナーもない隔離は削除の遠回り。期限超過の棚卸しを隔週で回す
- **リトライで蓋をする** — CIの自動リトライ回数を増やして「greenに見せる」のは隔離より悪い(検出もされない)。リトライ設定の変更をゲート1の検知対象に含める
- **復帰基準なしで戻す** — 「直った気がする」での復帰は再隔離の無限ループになる。反復実行の証跡を復帰の通過条件にする

## 3. ミューテーションテストの導入と運用

**目的**: テストスイート自体の検出力を測る。「カバレッジが高いのに検出力が低い」はAI量産テストの典型的な負債 — 実行はされるが、壊しても落ちないテストはカバレッジ計測では見えない。

**導入手順**:
1. ツール選定(Stryker〈JS/TS/C#〉、PIT〈JVM〉、mutmut等〈Python〉)し、まずコアドメインロジックの1モジュールでパイロット実行する(全量実行は計算コストが高く、初回から全体に掛けると遅さで死ぬ)
2. **差分実行をPRレビューに提示する形を基本にする**: 変更されたコードに対する生存ミュータント(殺されなかった変異)のみ算出し、PRコメントとして提示する。レビューアとエージェントの双方が「このテストは何を見逃すか」を具体的な変異で確認できる
3. 全量実行は傾向把握の定期ジョブ(週次〜月次)に留め、閾値ゲートにはしない
4. 生存ミュータントへの対応を運用に組み込む: 殺すテストを足す/そのコードが不要と判明したら消す/意図的に受容(equivalent mutant・低リスク)を記録する

**しきい値・運用の目安**: **全体スコアの閾値監視はしない** — 計算コストが高いうえ行動につながらず、数値目標はカバレッジ同様ゲーミングされる(Google/Facebookの大規模運用の知見 — SKILL.md ステップ5)。見るのは「PR差分の生存ミュータントに対処したか」と「コアドメインの傾向が悪化していないか」の2点。

**失敗モード(形骸化)**:
- **スコアがKPIになる** — スコアを上げるためだけの弱いテスト(変異は殺すが仕様は表現しない)が量産される。スコアは診断値であってゲート値ではない
- **equivalent mutantの山で信頼を失う** — 「対処不能な指摘ばかり」と感じられたら受容記録の仕組みが機能していない。受容済みを再提示しない設定を最初に入れる
- **遅すぎて誰も待たない** — 差分実行でもタイムアウトするならPRブロッキングにせず、非同期コメントに徹する

## 4. 写経テスト(実装をなぞるだけのテスト)の検出

**目的**: 「モックの戻り値をassertするだけ」「実装と同じ計算式の再実装」のような、実装が何であれ通るテストを検出・削減する。写経テストはバグを仕様として固定し、さらに**既存テストは事実上のプロンプト**であるため、エージェントが模倣して悪例を量産する種になる(SKILL.md ステップ4)。

**導入手順**:
1. 定義を運用可能な判定基準に落とす: (a) モック設定とアサーションが鏡写し(スタブした値をそのまま検証)、(b) 実装と同一の計算ロジックをテスト内に再実装して結果比較、(c) `toBeDefined()`/not-null程度しか主張しない、(d) 相互作用(呼び出し回数・引数)の検証だけで状態・戻り値・副作用を見ない
2. 計測は無作為サンプリングで行う: テストを無作為抽出し上記基準で判定、率と根拠(該当テストの引用)を診断に載せる(SKILL.md ステップ1の健康診断)。全数の機械判定は現実的でない — 機械化できるのは兆候の検出(アサーション密度、モック使用数、`toBeDefined`単独の検出)まで
3. 修繕は変更頻度順に行い、直した領域をCLAUDE.mdの「テストの書き方は<代表ファイル>を参照」に登録する — 良い手本を先に整備しないと、修繕してもエージェントが古い悪例から再生産する
4. 新規流入の防止はレビューゲートで行う: AIが書いたテストのレビュー観点(SKILL.md ステップ4 — 実装側だけ退避してredを確認する等)をSOFTゲートのレビューエージェントのルーブリックに載せる。恒常的な定量把握はゲート3(ミューテーション)が担う — 写経テストは変異を殺せないため生存ミュータントとして現れる

**しきい値・運用の目安**: サンプリング判定は四半期ごとの定点観測で十分(毎PRで測る性質のものではない)。率そのものより「よく触る領域に手本が整備されているか」を先行指標にする。

**失敗モード(形骸化)**:
- **兆候検出のlint化で満足する** — アサーション密度などの機械的兆候は簡単に満たせてしまい、写経の本質(実装への従属)は検出できない。機械検出は入口の絞りとして使い、判定はミューテーション結果とサンプリングレビューに置く
- **「消すと不安」で残る** — 何の変異も殺さないテストは削除が正しいが、削除はゲート1の検知に引っ掛かる。生存ミュータント証跡を添えて承認フローに乗せる、が正規の削除経路

## 5. LLM機能の評価セット運用

**目的**: LLM機能(生成・分類・RAG等)の品質を回帰資産として管理する。通常のテストと違い合否が確率的で、プロンプト・モデル・パラメータ・知識コーパスのどれが変わっても挙動が変わる — 「動いているように見える」を検証に置き換える。

**導入手順**:
1. 評価セット(入力コーパス×ルーブリック)の中身の定義は `test-analysis`/`test-design`(技法カタログ§7のLLM適用)の領分 — 本ゲートが受け持つのは運用配線
2. 評価セットをコードと同じリポジトリでバージョン管理し、**プロンプト変更・モデル更新・パラメータ変更・知識コーパス更新を回帰ゲートのトリガーにする**(モデル更新は依存更新ではなく挙動変更として扱う — SKILL.md ステップ1)
3. 合否は統計で判定する: 単発一致ではなく合格率の閾値とベースラインからの劣化幅(例: ベースライン比で合格率が規定ポイント低下したらfail)。非決定性があるためN回実行の集計で判定する
4. コスト・レート制限があるため専用CIジョブ・別予算で管理する(全PRで回さない — トリガー該当変更のみ)
5. 本番の失敗事例を評価セットに還流する定期運用(例: 隔週で失敗ログから代表例を追加)を置く — evalは腐る資産で、living documentationと同じ保守対象

**しきい値・運用の目安**: 閾値は機能のリスクで決める(ユーザー向け回答の正確性と内部要約では許容劣化が違う)。最初は「ベースラインから統計的に有意な劣化でフラグ(ブロックしない)」から始め、判定の安定を確認してからブロッキングに昇格する。LLM-as-judgeをルーブリック採点に使う場合、判定のブレを前提にブロックではなくフラグとして使う(`harness-engineering` のセンサーと同じ扱い)。

**失敗モード(形骸化)**:
- **評価セットが初期構築のまま腐る** — 本番の失敗が還流されないevalは「昔の問題は起きない」ことしか保証しない。還流の定期タスク化が対策
- **合格率100%が続く** — 難易度不足のサイン。全部通るevalは回帰を検出できない。境界的な例・過去の失敗例の比率を維持する
- **judgeのプロンプトを「通るように」調整する** — 評価側資産の変更はハーネス変更としてHARD(検証する側を被検証側から分離する原則)

## 6. testing in production(本番検出をテスト戦略に組み込む)

**目的**: 事前テストで防ぐには再現コストが高すぎるリスク(本番データ・スケール・実トラフィック依存)を、デプロイパイプラインの検出ゲートとして守る。「テストで防がない」は「守らない」ではない — 検出と復旧の速さで守る。

**導入手順**:
1. どのリスクを本番検出に回すかの判断は `test-analysis` ステップ4を正とする(事前再現コスト・本番データ/スケール依存・復旧容易性)。本ゲートは「検証ポイント=production」と判定された条件を受けて実装する
2. 前提の確認から始める: プログレッシブデリバリ(段階ロールアウト・フィーチャーフラグ)と可観測性(メトリクス・トレース・アラート)が無ければ本番検出は選べない — 整備状況を現状診断に含め、無ければ前提整備をアクションに積む
3. 実装する検出ゲート: カナリア/段階ロールアウトのベースライン比較(逸脱で自動ロールバック)、SLOバーンレートアラート、クリティカルパスのsynthetic monitoring。各条件に「何を監視するか」と「検出時の復旧手段(自動ロールバック/フラグOFF)」をペアで定義する
4. 「本番検出に回すと決めたのに監視が未実装」の状態をカバレッジの空白と同じ扱いで診断に載せ、棚卸しする

**しきい値・運用の目安**: カナリアの判定窓と比較指標(エラー率・レイテンシ・業務KPI)は機能ごとに宣言する。自動ロールバックの条件は保守的に始める(誤ロールバックのコストは低く、検出漏れのコストは高い)。synthetic monitoringはE2Eテストの本番版なのでクリティカルパスに絞る — 網羅しようとすると本番E2E偏重という新しいアンチパターンになる。

**失敗モード(形骸化)**:
- **「本番で気づけるから」が事前テストのサボりの言い訳になる** — 本番検出は分担であって代替ではない。`test-analysis` の判定基準を通っていないリスクを勝手に本番送りにしない
- **アラート疲れ** — 検出ゲートのアラートが騒がしいと本物が埋もれる。バーンレート方式(急激な消費のみ通知)を守り、通知したら必ず行動する運用を維持する
- **復旧手段の未接続** — 「検出はしたが戻せない」は検出の価値を半減させる。監視と復旧手段(ロールバック・フラグ)を必ずペアで実装する

## 導入の順序(全部を一度にやらない)

**ゲート1(弱体化検知)とゲート2(flaky隔離)が先。** この2つはエージェント協働の前提であり、他のゲートの信頼性(テストが信用できること・greenが意味を持つこと)を支える。次にゲート3(ミューテーション差分実行)で検出力の可視化を入れ、ゲート4(写経検出)はその計測結果を使って回す。ゲート5・6は該当機能(LLM機能・本番検出対象)が存在するときに導入する — 対象がないのに枠だけ作るのは過剰投資。
