# AI-Adaptable テスト戦略スキル 適用例

## 題材(架空シナリオ)

中規模のフードデリバリー基盤の注文サービス(Node.js/TypeScript + PostgreSQL + 外部決済SaaS)。5年もののコードベースで、テストは約1,200件あるがCIが20分かかり、たまに落ちるテストの再実行が習慣化している。最近AIエージェントに実装を委任し始めたところ、「テストは通るがレビューで見ると怪しい」PRが増えた。料金計算ロジックのリファクタを控えている。

## 入力(ユーザーの依頼と文脈)

「料金計算のリファクタ前にテストを固めたい。あとAIが書いたテストが信用できない。テスト戦略の診断と、直すべき順番を出してほしい」

## 各ステップでの判断

- **起動時の分岐**: 依頼は「戦略の立案・診断」+「レガシー変更の安全網」の複合 → ステップ1で診断し、料金計算はステップ3(Feathers方式)を適用。個別ケース設計が必要になった時点で `test-design` へ委譲
- **ステップ1(現状分析)**: 対象を分類 — 料金計算はコアドメインロジック(厚い)、注文APIは統合・配線が主体。健康診断はデータの取り方まで実施:
  - 実行時間: ランナーレポートから遅い順に集計。上位20件が実行時間の60%を占め、全てDB実起動の広い統合テスト
  - flaky率: CIにリトライ記録が無いため同一スイート5回実行で計測。揺れたのは17件 — 「flakyを計測する仕組みが無い」ことも診断に記載
  - 写経率: 無作為30件のサンプリングで判定。9件がモック戻り値の鏡写し
  - カバレッジ空白: 未実行分岐と例外処理のgrepを突合。決済失敗後の状態遷移が未検証と判明
- **ステップ2(仕様のテスト化)**: リファクタ対象の料金計算に現行仕様の曖昧点が3つ見つかり(深夜料金と割引の適用順など)、key examplesを作る過程でユーザーに確認。確定した具体例はGiven/When/Thenで固定し、性質(合計金額は0以上・明細の和と一致)をfast-checkで併設
- **ステップ3(安全網構築)**: 料金計算は入出力が閉じているためseamなしでcharacterizationが可能 — 本番ログから抽出した500入力の現挙動をゴールデンテストで固定。記録中に丸め誤差の疑いを1件発見したが、期待値は現挙動のまま固定し疑いは別途報告(Feathers)
- **ステップ4(AI協働整備)**: 写経テストの発生源が「既存テストの模倣」にあると判断し、よく触る注文ドメインに手本テストを整備してCLAUDE.mdに登録。失敗テストのトリアージ(テストを弱める選択肢は無い)を明文化
- **ステップ4(続き — AIテストの検収)**: 「信用できない」の裏取りとして、直近のAI生成テスト10件で実装側の差分のみ退避(`git stash push -- <実装ファイル>`)してredを確認 — 3件がgreenのまま通過(検出力ゼロ)で修繕リストに追加。恒常計測はミューテーション差分実行に引き継ぐ
- **ステップ5(品質ゲート化)**: `references/quality-gates-catalog.md` の導入順序に従い、弱体化検知とflaky隔離を先行実装。ミューテーションは料金計算モジュールの差分実行から。決済SaaS境界は契約テスト(Pact)をバックログに積む(本スコープでは配線まで)

## 成果物(戦略文書)

### 1. 現状診断

| 指標 | 測定値 | 測定方法 |
|---|---|---|
| 実行時間 | 全体20分。上位20件(全て広い統合テスト)で12分 | ランナーレポートの遅い順集計 |
| flaky率 | 17/1,200件(1.4%)。計測の仕組み自体が無い | 同一スイート5回実行の揺れ検出 |
| 写経テスト率 | 30%(9/30件。モック戻り値の鏡写し) | 無作為30件のサンプリング判定 |
| カバレッジ空白 | 料金計算のエラーパス(決済失敗後の状態)が未実行 | 未実行分岐と例外処理grepの突合 |
| 検出力 | 未計測(ミューテーション未導入) | — |

### 2. テスト配分(現状 → 目標)

| 対象分類 | 現状 | 目標 |
|---|---|---|
| 料金計算(コアドメイン) | 広い統合テストで間接的に検証 | Small単体で網羅+性質テスト。characterization 500件はリファクタ完了後に縮退 |
| 注文API(統合・配線) | モック単体が主体(写経の温床) | Testcontainersでの狭い統合テスト(Medium)を主力に再配分 |
| 決済SaaS境界 | E2Eで間接検証 | 契約テスト(Pact)で固定。E2Eはクリティカルパス2本のみ |
| 本番検出 | 監視は死活のみ | 注文成功率のSLOバーンレート監視を追加(`test-analysis` の判定を受けて) |

### 3. 優先順位付きアクション

| # | アクション | なぜ(診断との対応) | どこから(最初の一手) | 完了条件(機械判定) |
|---|---|---|---|---|
| 1 | 料金計算のcharacterization+key examples | リファクタの前提(安全網) | 本番ログから代表入力500件抽出 | ゴールデン500件green+曖昧点3件の仕様確定 |
| 2 | テスト弱体化検知+flaky隔離の導入 | skip再実行の習慣化・計測の欠如 | CI diff検査と隔離ジョブ(quality-gates-catalog §1・§2) | 検知CIが有効+隔離17件にオーナー/期限付与 |
| 3 | 写経テストの修繕と手本整備 | 写経率30%が量産の種 | 注文ドメインの手本テスト+CLAUDE.md登録 | 手本登録済み+修繕対象リスト化(変更頻度順) |
| 4 | 遅い統合テストの再配分 | 上位20件で12分 | 最遅5件を狭い統合に分解 | PR時CIを10分以内に |
| 5 | 決済境界の契約テスト導入 | E2E間接検証のみ(片側変更事故のリスク) | consumerが実際に使う相互作用の抽出 | Pact検証がprovider/consumer双方のCIでgreen |

### 4. 継続ゲートへの接続

実装するゲート(順に): 弱体化検知 → flaky隔離(返済運用込み) → ミューテーション差分実行(料金計算から) → 契約テストのCI配線。ゲート分類(HARD/SOFT/AUTO)とエスカレーション設計は `harness-engineering` に接続する。

個別テストケースの設計(料金計算のデシジョンテーブル×境界値、契約ケースの相互作用×provider state)は `test-design` へ委譲した。LLM機能・本番検出ゲートは対象が現状無いため導入を見送り、見送り判断を診断に明示した。
