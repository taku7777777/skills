# api-data-design 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | モバイルアプリ向けにコメント投稿まわりのバックエンドの入出力を決めたい。エラーの返し方も揃えたい | REST契約設計(「APIを設計して」と語彙が異なる) |
| F2 | 注文履歴の持ち方を考えてる。1テーブルに全部入れるか分けるか、制約はどこまで張るか相談したい | DBスキーマ設計 |
| F3 | レスポンスのこのフィールド名、いまさら変えたいんだけどクライアントが壊れないか不安。どう進めるのが正解? | 互換性判定+移行計画(破壊的変更の扱い) |
| F4 | もう誰も使ってないはずのカラムを消したい。稼働中のサービスで安全にやる段取りを組んで | expand-contract のマイグレーション計画(contract側) |
| F5 | 他チームがうちのDBを直接SELECTしていて、テーブルを変えるたびに壊れる。ちゃんとした窓口を用意したい | 境界の契約設計+データ所有権 |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 新規サービス全体をどうコンポーネント分割するか、認証・認可はどこに置くかを設計して | `arch-security-design` が発火する(システム分割・脅威モデリングの領分) |
| N2 | 注文と在庫の集約境界と不変条件を整理して、ドメインモデルを設計してほしい | `ddd-design` が発火する(内部のドメインモデル。境界に公開する契約ではない) |
| N3 | このスキーマ変更、どの順でパイプラインに載せてデプロイすれば安全?リリース工程を組んで | `cicd-release` が発火する(マイグレーション計画は本スキル、デプロイ順序・リリース工程は cicd-release) |
| N4 | このAPIクライアントのリトライ処理のバグを直して | スキルなし(または `systematic-debugging`)。API という語に釣られない |

## 検証シナリオ

### S1: 社内他チーム向けクーポンAPIとテーブルの新規設計
- **タスク**: 「架空のECプラットフォームで、プロモーションチームが管理するクーポンを、注文チーム(決済時の適用検証)とアプリチーム(ユーザーへの一覧表示)が使えるようにAPIを新設したい。要件: クーポンの発行(プロモチームの管理画面から)、有効なクーポンの一覧取得(ユーザー単位、ピーク200 req/s)、決済時の適用(同時に同じクーポンが二重適用されない保証が必要。注文のリトライあり)。既存DBはない。API契約とテーブル設計をまとめて」
- **スキルなしで予想される欠落**: エンドポイントとカラムの羅列だけで、利用者・ユースケース整理を飛ばす。エラー形式(RFC 9457等)・ページングをAPI全体の規約として先に決めず個別実装に任せる。リトライに対する冪等性(Idempotency-Key等)の欠落。一意制約・CHECKなどDB宣言による担保の欠落。OpenAPI等の機械可読な契約を正とする方針の欠落。
- **合格条件**:
  - [ ] 利用者(プロモ/注文/アプリの各チーム)ごとのユースケースと公開範囲の整理が契約より前にある
  - [ ] エラー形式とページングがAPI全体の規約として一貫して定義されている
  - [ ] 適用操作にリトライ・二重適用対策としての冪等性設計(冪等キー等)が含まれる
  - [ ] 二重適用防止がアプリロジックだけでなくDB制約(一意制約等)で宣言されている
  - [ ] OpenAPI(または同等の機械可読定義)を契約の正とする方針が明記されている

### S2: 変更一覧の互換性一括判定
- **タスク**: 「パートナー3社が使っている配送ステータス照会API(JSON/REST)に対して、次の変更案が出ている。それぞれ出してよいか判定して: ①レスポンスに `estimated_arrival` フィールドを追加 ②ステータスenumに新値 `CUSTOMS_HOLD` を追加 ③リクエストの `carrier_code` を任意から必須に変更 ④`weight` の型を整数(グラム)から小数(キログラム)に変更 ⑤誰も使っていないはずの `legacy_ref` フィールドをレスポンスから削除 ⑥404時のエラーボディを独自形式からRFC 9457形式に変更」
- **スキルなしで予想される欠落**: 「①は安全、③④は危険」程度の粗い直感判定。enum値追加(②)が消費側の実装次第で破壊的になりうるという判定の欠落。「誰も使っていないはず」(⑤)を検証なしに受け入れる。エラーボディの形式変更(⑥)もパースしている消費者には破壊的という視点の欠落。破壊的と判定したものに「禁止」でなく移行計画をセットで付ける対応の欠落。
- **合格条件**:
  - [ ] 6件すべてに互換/破壊(または条件つき)の判定が表形式などで個別に付いている
  - [ ] ②のenum値追加について、未知の値を受けた消費者側の挙動に依存する(無条件に安全ではない)ことが指摘されている
  - [ ] ⑤について「使っていないはず」を実測(アクセスログ・consumer contract等)で確認してから進める条件が付いている
  - [ ] 破壊的と判定された変更に、拒否ではなく移行計画(expand-contract、告知と期限等)の方向づけがある
  - [ ] パートナー公開(partner)という公開範囲が変更コストの判断に反映されている(人間判定)

### S3: 実装済みAPIの公開直前レビュー(途中介入)
- **タスク**: 「もう実装は終わってるんだけど、来週パートナー公開する前に一応見てもらえる?」と依頼し、次の資料を渡す — レスポンス例: `{"tbl_seminar_appl": {"appl_id": 10234, "usr_mail_addr": "a@example.com", "del_flg": 0, "created_by_batch": true}}`、エンドポイント: `GET /getApplList?page=3`(別エンドポイントは `POST /seminar/apply` で成功時 `{"result": "OK"}`、失敗時 `{"error_code": 500}` を200で返す)、DDL: `tbl_seminar_appl` テーブルをそのままシリアライズしている。
- **スキルなしで予想される欠落**: 命名の指摘(スネークケース・略語)など表面的なレビューで終わる。内部テーブルがそのまま契約に露出している構造問題(実装の変更が全利用者の変更になる)の指摘漏れ。エラー形式・成功形式の不統一と「失敗を200で返す」問題の欠落。「公開後は一方通行、公開前が直せる最後の機会」という緊急度づけの欠落。
- **合格条件**:
  - [ ] 内部テーブル(`tbl_seminar_appl`・`del_flg` 等)が契約に露出している問題が、個別の命名でなく「実装の露出」として構造的に指摘されている
  - [ ] エラーを200+独自ボディで返している点が指摘され、統一エラー形式(RFC 9457等)への是正が提案されている
  - [ ] `GET /getApplList` のような動詞パスがリソース指向へ是正されている
  - [ ] 「パートナー公開後は破壊的変更のコストが桁で増えるため、公開前のいま直すべき」という優先度づけの説明がある
  - [ ] 「実装済み」を理由に契約の問題を容認せず、公開スケジュールとの現実的な折り合い(最低限直すもの/公開後に追加のみで直せるもの)を区別している(人間判定)
