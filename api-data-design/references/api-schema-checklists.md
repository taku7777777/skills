# API・スキーマ設計チェックリストと互換性判定表

SKILL.md のステップ3(契約の詳細設計)・ステップ4(データモデリング)・ステップ5(互換性方針)・ステップ6(マイグレーション計画)から参照する実務資料。チェックリストは「全項目を埋める」のではなく「決めていない項目がないか」の検査に使う — 決めない判断も判断として記録する。

## 1. REST API設計チェックリスト

### 1.1 命名・リソース設計

- [ ] リソース名は複数形の名詞(`/reservations`)。動詞をパスに入れない(標準メソッドで表せない操作のみ `:cancel` 形式のカスタムメソッド)
- [ ] 名前はユビキタス言語(用語集)と一致している。同じ概念に2つの名前、2つの概念に1つの名前がない
- [ ] JSONフィールドの命名規約(snake_case / camelCase)がAPI全体で統一され、規約文書に明記されている
- [ ] 階層は所有関係のみ(`/studios/{id}/lessons`)。3階層を超えるネストは検索条件(クエリパラメータ)への変換を検討
- [ ] IDは内部実装を不必要に漏らさない。連番の露出は総数・成長率の推測や列挙を容易にするため、外部公開にはUUID/ULID等を検討する。ただし不透明なIDは防御の補助であり、リソース単位の認可を代替しない
- [ ] 日時はISO 8601・タイムゾーン明示(UTC推奨)。金額は通貨と最小単位を明示(浮動小数点で金額を運ばない)

### 1.2 HTTPメソッドとステータスコード

- [ ] GET=安全(副作用なし)、PUT=全置換で冪等、PATCH=部分更新、DELETE=冪等。この意味論を破るエンドポイントがない(GETで状態変更は厳禁)
- [ ] 201(作成、Locationヘッダつき)と200/204(更新・削除)を使い分けている
- [ ] 400(形式不正)と422(形式は正しいが業務的に処理不能)の使い分けをAPI規約として決めている
- [ ] 401(未認証)と403(認証済みだが権限なし)を混同していない。**存在を隠すべきリソースへの403は404に倒す**(他テナントのIDを試すと403が返る設計は、存在の有無を漏らす)
- [ ] 409(競合: 一意制約違反・楽観ロック失敗)と412(前提条件失敗: If-Match)の使い所を決めている
- [ ] 429(レートリミット)に Retry-After を返す

### 1.3 ページング・フィルタ・ソート

- [ ] 件数が利用者・設定・時間経過で増えうる一覧APIはページングする。列挙値・固定設定など上限が小さく契約で保証できる場合は、非ページングを選んだ理由と上限を記録する
- [ ] 方式をユースケースで選んでいる: オフセット(実装容易・任意ページへ移動可・深いページで性能劣化・更新中のズレ)vs カーソル(安定・大規模/高更新頻度向き・任意ページへ飛べない)。更新頻度、深いページの必要性、整合性、キャッシュ性から決め、「迷ったら」の一律既定にしない
- [ ] ページサイズに既定値と上限がある(上限なしはDoSの入口)
- [ ] レスポンスに次ページの手がかり(`next_cursor` 等)を含む。総件数の提供はコストを理解した上で判断(大規模では COUNT が最も重いクエリになる)
- [ ] フィルタ・ソートは**利用者のユースケースにあるものだけ**公開する。全フィールド検索可能にすると、全フィールドにインデックスと互換性保証の義務を負う
- [ ] ソート順は明示指定がなくても決定的にする(タイブレークに主キーを含める — ページング中の重複・欠落を防ぐ)

### 1.4 エラー形式(RFC 9457 Problem Details)

- [ ] エラーレスポンスは RFC 9457(`application/problem+json`)に統一: `type`(エラー種別のURI)、`title`、`status`、`detail`、`instance`
- [ ] `type` のURIをエラーカタログとして一覧化し、利用者が機械的に分岐できる(HTTPステータスだけでは業務エラーを区別できない)
- [ ] バリデーションエラーはフィールド単位の詳細(拡張メンバ、例: `errors: [{field, reason}]`)を返す
- [ ] エラーメッセージに内部情報(スタックトレース、SQL、内部ホスト名)を漏らさない
- [ ] エラー契約も互換性保証の対象と明記する(利用者は `type` で分岐する。`type` の変更は破壊的変更)

### 1.5 冪等性

- [ ] 全ての再試行経路を洗い出している(クライアント再送、ゲートウェイのリトライ、イベントの再配信)。**at-least-once の世界では「二重に届く」が正常系**
- [ ] POSTによる作成系には冪等性キー(`Idempotency-Key` ヘッダ等)を設計: 同一キー+同一ペイロード→初回と同じレスポンスを返す / 同一キー+異なるペイロード→エラー(422等)
- [ ] キーの保存期間(TTL)とスコープ(利用者単位)を決めている
- [ ] 冪等性の実装はDB制約(一意制約)まで到達している(アプリ内のチェックだけでは同時実行ですり抜ける — 同時実行の検証は `test-design` §13)
- [ ] 更新の競合制御を決めている: 楽観ロック(ETag + If-Match / versionカラム)を既定とし、失敗時は409/412

### 1.6 認可の粒度

- [ ] 認可の判定単位を決めている: エンドポイント単位(操作できるか)+**リソース単位(そのIDのデータに触れるか)**。後者の欠落が IDOR/BOLA(他人のIDを指定するだけで読める)を生む
- [ ] 一覧APIは認可フィルタをクエリ条件に織り込む(取得後にアプリで間引く方式は、件数とページングを壊し、漏れやすい)
- [ ] フィールド単位の可視性(管理者のみ見えるフィールド等)が必要なら、レスポンス分岐ではなく**別リソース/別エンドポイントへの分離**を先に検討(条件付きフィールドは契約を曖昧にする)
- [ ] マルチテナントの分離条件が全クエリに強制される仕組みがある(認可モデル自体の設計は `arch-security-design` の領分)

## 2. スキーマ設計チェックリスト

### 2.1 型と制約

- [ ] 主キー方針を決めている。変更可能・複合・外部制度依存の業務キーでは代理キーを優先し、自然キーは一意制約として別途持つ。安定性が契約で保証される小さな参照表などで自然キーを選ぶ場合は、変更時の移行コストを記録する
- [ ] 金額は DECIMAL(浮動小数点は不可)、日時はタイムゾーン付き型(またはUTC固定の規約)、真偽は boolean 型(フラグ文字列 '0'/'1' を使わない)
- [ ] 文字列長の上限に根拠がある(「とりあえず255」を業務上の根拠に置き換える)。無制限テキストは意図的な選択として記録
- [ ] 外部キー制約を張っている。張らない場合(パーティション・シャーディング等)は理由と整合性の代替検証手段を記録
- [ ] 値の種類が閉じている列は CHECK 制約または参照テーブルで制限(アプリ定数だけに頼らない)
- [ ] ステータス列は取りうる値と遷移を文書化(状態遷移の設計は `ddd-design`、遷移テストは `test-design` §3)

### 2.2 NULLの規律

- [ ] NOT NULL を既定とし、NULL許容には「未入力」「非該当」「不明」のどの意味かを列コメントで明記(NULLの意味が複数あるのは設計の穴)
- [ ] 「後で必ず入る」列は expand-contract の expand 段階の一時的NULL許容か、恒久的な仕様かを区別している
- [ ] NULL と空文字・ゼロを混在させない(検索・一意制約・集計の挙動が割れる)

### 2.3 監査列

- [ ] 作成・更新時刻が監査、同期、競合制御、運用調査に必要なテーブルへ `created_at` / `updated_at` を持たせる。更新されない参照表・一時表・派生表まで機械的に追加せず、省略理由を記録する。更新者・作成者の記録要否は業務要件から判断
- [ ] 「誰がいつ何を変えたか」の要件がある場合、監査列では足りない — 履歴テーブル/イベントログを別途設計(監査ログ全体の設計は `arch-security-design`)
- [ ] 監査列をアプリ規約でなくDB既定値・トリガーで強制するか決めている(経路が増えても漏れない方)

### 2.4 論理削除の判断

論理削除(`deleted_at`)を既定にしない。以下で判断する:

- [ ] 削除後も参照される業務要件(取消済み注文の履歴表示等)があるか → あるなら論理削除ではなく**状態(cancelled等)としてモデル化**する方が正確ではないか
- [ ] 復元要件が本当にあるか(「念のため」はバックアップの仕事)
- [ ] 論理削除を選ぶ場合のコストを引き受けたか: 全クエリに `deleted_at IS NULL`(ビュー/ORMスコープで強制)、一意制約との併存(部分インデックス等)、個人情報の削除義務(論理削除では消えない)との整合
- [ ] 物理削除を選ぶ場合、外部キーのカスケード挙動と、参照整合性を壊さない削除順序を確認したか

## 3. 互換性判定表

契約変更のレビューで機械的に使う。「グレー」は利用者との契約(未知の値の扱い)次第 — 判定できない時点で利用者と合意してから進める。

### 3.1 API(リクエスト/レスポンス)

| 変更 | 判定 | 備考 |
|---|---|---|
| 新しいエンドポイントの追加 | 非破壊 | |
| レスポンスへの任意フィールド追加 | 非破壊 | 利用者側が tolerant reader(未知フィールド無視)であることが契約の前提 |
| リクエストへの任意パラメータ追加 | 非破壊 | 既定値が従来挙動と同じであること |
| リクエストへの必須フィールド追加 | **破壊的** | |
| フィールドの削除・リネーム | **破壊的** | リネーム=削除+追加。expand-contract で |
| フィールドの型変更 | **破壊的** | 文字列→数値、単一→配列など全て |
| 任意→必須への変更 | **破壊的** | 必須→任意は非破壊 |
| バリデーションの厳格化(受理していた値の拒否) | **破壊的** | 緩和は原則非破壊 |
| **フィールドの意味・単位・タイムゾーンの変更** | **破壊的** | 形が同じまま意味が変わる、最も検出しにくい破壊 |
| enum への値追加(レスポンス) | グレー | 利用者が未知値を安全に処理する契約があれば非破壊。なければ破壊扱い |
| デフォルト値・暗黙の挙動の変更(ソート順・件数上限) | グレー | 観測されている挙動は事実上の契約。破壊側に倒して扱う |
| エラーの `type`・ステータスコードの変更 | **破壊的** | エラー契約も契約 |
| URLパス・パラメータ名の変更 | **破壊的** | |
| 認可の厳格化(返していたデータを返さなくする) | **破壊的**(意図的でも) | セキュリティ修正として正当でも、移行連絡は必要 |
| レートリミットの引き下げ | グレー | SLAに準じて破壊側で扱う |

### 3.2 イベントスキーマ

| 変更 | 判定 | 備考 |
|---|---|---|
| 任意フィールドの追加 | 非破壊 | 購読者は未知フィールドを無視する契約 |
| フィールドの削除・リネーム・型変更 | **破壊的** | 保存済みイベントの再生も壊す(APIより影響が長い) |
| 必須フィールドの追加 | **破壊的** | 旧イベントを読む側に「無い」ケースが必ず来る |
| イベント名・トピックの変更 | **破壊的** | |
| 発行順序・重複配信特性の変更 | **破壊的** | 配信保証は契約の一部 |

破壊的変更が必要な場合の選択肢: バージョン付き新イベント(v2)の並行発行 → 購読者移行 → v1停止(expand-contract)/ 読み取り時変換(upcaster)。スキーマレジストリがあれば互換性モード(BACKWARD/FORWARD)をCIで強制する。

### 3.3 DBスキーマ(ローリングデプロイ中の新旧コード共存)

同一アプリでも、デプロイ中は旧コードと新スキーマが共存する。この間の互換性:

| 変更 | 判定 | 備考 |
|---|---|---|
| NULL許容(またはデフォルト付き)カラムの追加 | 非破壊 | `SELECT *` 依存のコードがないこと |
| テーブル追加・インデックス追加 | 非破壊 | 大テーブルへのインデックス追加はロック方式を確認(online DDL) |
| カラムの削除・リネーム・型変更 | **破壊的** | 必ず expand-contract(§4.2) |
| NOT NULL 制約の追加 | **破壊的** | 旧コードがNULLを書く間は張れない。contract 段階の最後に |
| デフォルト値・トリガーの変更 | グレー | 旧コードの書き込み経路への影響を確認 |

## 4. expand-contract 手順書

原則: **各段階は独立にデプロイでき、独立にロールバックできる。** 前の段階の完了を計測で確認してから次へ進む。1つのデプロイに expand と contract を同居させない。

### 4.1 API版(フィールド/エンドポイントの置き換え)

1. **Expand**: 新フィールド/新エンドポイントを追加する。旧はそのまま動かし、サーバーは両方に応答する(二重提供)。この時点で契約(OpenAPI)に新旧両方と移行期限を明記
2. **Migrate**: 利用者を新へ移行する。旧に Deprecation の告知(Sunset ヘッダ〔RFC 8594〕・変更ログ・利用者への直接連絡)を付け、**旧の呼び出し元をアクセスログで計測**する(誰が・どれだけ残っているか)
3. **Contract**: 実施条件(例: 旧呼び出しゼロを14日間観測)を満たしたら旧を削除する。条件未達で期限が来たら、削除ではなく残存利用者との調整に切り替える(黙って消さない)

### 4.2 DB版(カラム移行の例: 旧カラム → 新カラム/新テーブル)

1. **Expand**: 新カラム/新テーブルを追加する(NULL許容またはデフォルト付き — 旧コードを壊さない)。デプロイ後、アプリは**旧に書き、旧を読む**(まだ挙動不変)
2. **二重書き込み**: アプリを「旧と新の両方に書く」に変更してデプロイ(トリガーで代替してもよい。どちらを正とするかを明記)
3. **バックフィル**: 既存データを旧→新へコピーする。チャンク分割(ロック・レプリケーション遅延対策)+ 冪等な再実行 + **突合検証**(件数・値の一致。`test-design` §12 と同型)
4. **読み取り切替**: 読み取りを新へ向ける(可能ならフィーチャーフラグで段階切替)。問題があればフラグで旧読みに即時ロールバック
5. **書き込み縮退**: 旧への書き込みを停止する(新のみに書く)。ここが実質の不可逆点 — 直前に突合を再実行
6. **Contract**: 監視期間の後、旧カラムを削除し、新カラムに NOT NULL 等の本来の制約を張る

### 4.3 共通の注意

- 各段階を「変更内容 / デプロイ単位 / 完了の計測 / ロールバック手段」の4列で段取り表にする(SKILL.md 成果物6章)
- ロールバック不能な操作(カラム削除、縮小系の型変更)は contract 段階だけに集める
- マイグレーションスクリプトはコードと同じリポジトリでバージョン管理し、適用順序を機械化する(実行パイプラインの設計は `cicd-release`)
