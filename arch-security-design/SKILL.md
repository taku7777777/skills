# アーキテクチャ・セキュリティ設計スキル

Simon Brown(C4モデル)・Michael Nygard(ADR / Release It!)・SEI(品質特性シナリオ)・Neal Ford & Mark Richards(トレードオフ分析・fitness functions)・Gregor Hohpe(メッセージングパターン)・Adam Shostack(脅威モデリング)・OWASP・AWS Well-Architected の知見を統合した設計プロセス。

## 大原則

1. **アーキテクチャとは「変更しにくい決定」のこと。** 変更容易なものは軽く決め、変更困難なもの(データモデル、システム分割、技術基盤、認証方式)に設計労力を集中する。
2. **品質特性が構造を決める。** 機能はどんな構造でも実現できる。構造を決めるのは性能・可用性・セキュリティ・変更容易性などの品質要求。だから**品質要求を数値で確定させずに構造を決めない**。数値が出ない場合はドメイン相場から仮値を提案して合意を取り、仮定として設計書のリスク欄に記録して進む(設計を止めない)。
3. **トレードオフのないアーキテクチャ判断は存在しない**(Ford & Richards, アーキテクチャ第一法則)。「何を得て何を捨てたか」を明示しない設計書は設計書ではない。
4. **セキュリティは後付けできない。** 脅威モデリングは設計と同時に行う(Shift Left)。
5. **シンプル側に倒す。** 分散システム・マイクロサービス・イベント駆動は明確な必要性が示されるまで採用しない。「退屈な技術」(Boring Technology)を選ぶことはエンジニアリングの成熟。

## 設計ワークフロー

### ステップ0: スコープ判定

依頼の規模でステップを取捨する(省略は成果物に明示):

- **新規システム / 主要サブシステム** → 全ステップ
- **既存システムへの機能追加** → ステップ1・4(差分の信頼境界のみ。参照できる既存構成図がない場合は、影響範囲に限定した簡易コンテナ図を先に描いてから境界を書き込む)・7 + 影響を受ける既存ADRの更新
- **技術選定単体** → ステップ2(関連シナリオのみ)・6

### ステップ1: 前提の確定

以下が不明なら設計を始める前にユーザーに確認する(`requirements-definition` スキルの領分が広く欠けている場合はそちらを先に提案):

- ビジネス目標と制約(納期、予算、チーム規模・スキル)
- 想定規模: ユーザー数、RPS、データ量、成長率(現在と1〜3年後)
- 既存資産: 現行システム、使用中の技術スタック、組織の運用能力
- **品質特性の優先順位**: 全部「高」はあり得ない。上位3つを選ばせる

### ステップ2: 品質特性シナリオ化(SEI方式)

「高可用性」のような形容詞を、検証可能なシナリオに変換する。SEIの6部構成(刺激源/刺激/対象物/環境/応答/**応答測定**)で書く。特に応答測定を独立させないとシナリオは検証可能性を失う:

```
刺激源: AZ単位のインフラ障害
刺激: 東京リージョンの1AZが停止した
対象物: 注文処理サービスとプライマリDB
環境: ピークトラフィック時(平常の3倍)
応答: 残存AZへ自動フェイルオーバーする
応答測定: RTO 5分以内、RPO 1分以内、期間中エラー率1%未満
```

主要な品質特性(性能・可用性・拡張性・セキュリティ・変更容易性・運用性)それぞれについて、少なくとも代表シナリオを1つ作る(ID付与: QAS-1, QAS-2…)。これが後の設計判断とレビューの基準になる。

- 各シナリオには可能な限り**自動検証手段(fitness function)**を対応づける: p99レイテンシのCIゲート、依存方向のアーキテクチャテスト(ArchUnit等)、フェイルオーバーのカオス実験。自動化できないものは「設計レビューで検証」と明示する
- 非決定コンポーネント(LLM等)を含む場合、その品質の応答測定は単発の正誤ではなく**評価セットに対する統計値**で書く(例: 「代表タスク200件でタスク成功率95%以上、方針違反出力0件」)。対応するfitness functionはevalスイートのCIゲートであり、モデル・プロンプトの変更ごとに実行する
- シナリオ化と別に**ストレッサー分析**(residuality theory, Barry O'Reilly)を行う: 「主要ベンダーの買収」「データの国内保持義務化」「トラフィック100倍/10分の1」など予測不能なストレスを10個程度列挙し、各ストレスで設計のどこが壊れるかを確認する。既知のシナリオ駆動では拾えない「想定外への耐性」を、多くのストレスを生き残る構造の選択で確保する

### ステップ3: 構造設計(C4モデルで段階的に)

1. **システムコンテキスト図(C4 Level 1)**: システム、ユーザー、外部システムとデータフロー
2. **コンテナ図(C4 Level 2)**: デプロイ単位(アプリ、DB、キュー、キャッシュ)とプロトコル
3. 必要に応じてコンポーネント図(Level 3)

図はMermaidで描く(C4記法のレンダリングが不安定な環境ではflowchart+subgraphで信頼境界を表現)。同時に決めること:

- **システム分割**: モノリス/モジュラーモノリス/マイクロサービス。**デフォルトはモジュラーモノリス**。分割の根拠は「独立したデプロイ・スケール・チームの必要性」であり「将来のため」ではない。分割はチーム構造と整合させる(逆Conway戦略 — 1コンテナを2チームが共有する構図は分割失敗のサイン)
- **データ管理**: DB選定(まずRDBを検討し、外れる理由がある場合のみNoSQL)、主要エンティティの所有権とライフサイクル、整合性モデル(強整合 vs 結果整合の箇所を明示)、スキーマ進化の方針、OLTPと分析系の分離(CDC/ETL)、データ保持・削除ポリシー
- **通信**: 同期(REST/gRPC)vs 非同期(イベント/キュー)。非同期にする箇所はメッセージングパターンの語彙(Hohpe: 配信保証・順序・重複・デッドレター)で挙動を明記し、at-least-once前提の冪等設計とする
- **状態管理**: セッション、キャッシュ戦略とその無効化
- **AI/エージェントコンポーネント**(該当する場合): LLM呼び出し・エージェントループ・ツール実行環境をそれぞれ独立したコンテナとしてC4図に描き分ける。エージェントの**実行境界**(読めるデータ、呼べるツール、出力が届く先、ネットワークegress)は後付けの防御ではなく構造の一部としてここで決め、ステップ4の信頼境界として引き継ぐ。モデル・プロバイダは差し替え可能な依存として抽象化する(モデル世代の交代は例外事象ではなく前提)。境界内部の防御実装(permission/sandbox/egress)の詳細設計は `agent-security-governance` スキルへ

### ステップ4: セキュリティ設計(脅威モデリング)

`references/security-design.md` の詳細ガイドに従う。Shostackの4つの質問で進める:

1. **何を作っているか** → ステップ3の構造図に**信頼境界**(インターネット/内部、テナント間、権限レベル間)を書き込む
2. **何がまずくなりうるか** → 信頼境界を越えるデータフローごとに**STRIDE**で脅威を列挙(なりすまし/改ざん/否認/情報漏洩/DoS/権限昇格)
3. **どう対処するか** → 脅威ごとに 緩和/排除/転嫁/受容 を決定し、脅威モデル表にまとめる
4. **十分か** → 最新版OWASP ASVSの該当レベルと突き合わせ(目安 L1: 全Webアプリ / L2: 機密データを扱う / L3: 高価値ターゲット)

同時に基本設計として必ず決めること: 認証方式、認可モデル(RBAC/ABAC、マルチテナント分離)、データ分類と暗号化(転送時・保存時)、シークレット管理、監査ログ。

### ステップ5: 横断的関心事・運用設計

- **障害設計**(Nygard『Release It!』): タイムアウト、リトライ+バックオフ+ジッター、サーキットブレーカー、バルクヘッド。**依存先は必ず落ちる前提**(Vogels: "Everything fails, all the time")で各依存の障害時挙動(全停止/機能縮退/キューイング)を決める
- **非決定コンポーネントの信頼性**(LLM等がある場合): 従来の障害モード(プロバイダ障害・レートリミット・レイテンシ長尾)に加え、**呼び出しは成功するがもっともらしく誤る「意味的失敗」**を障害モードとして扱う。出力のスキーマ検証・ビジネスルール検証を境界に置き、evalスイートで品質回帰を検知し(モデル・プロンプトの変更はコードデプロイと同格の変更として扱う)、品質劣化・プロバイダ障害それぞれの縮退経路(別モデルへのルーティング/キャッシュ済み応答/非AI経路への切替)を決める
- **可観測性**: 構造化ログ、メトリクス、分散トレーシング、相関ID。SLI/SLOの定義
- **デプロイ・リリース**: CI/CD、フィーチャーフラグ、カナリア/ブルーグリーン、DBマイグレーション戦略(expand-contract)
- **キャパシティ**: ボトルネックの見積もり(封筒裏計算でよいので数字を出す)
- **コスト**: 主要コンポーネント別の月額概算、スケール時のコスト曲線(線形か階段か)、コストが暴走しうる箇所(egress、LLM API、ログ量)と上限アラート。LLMが主要コンポーネントの場合、コストは事後最適化ではなく**構造を決める設計ドライバ**として扱う: タスク難易度に応じたモデル階層ルーティング、プロンプトキャッシュ/セマンティックキャッシュ、コンテキストに何を見せるかの設計(トークン量=コスト)、タスク単価(円/リクエスト)の目標値を決めてfitness functionとして継続追跡する
- **段階的移行**: 既存システムがある場合、移行戦略(Strangler Fig / 並行稼働 / カットオーバー基準 / ロールバック計画)をここで設計する(成果物9章に対応。詳細な実行計画は `refactoring-strategy` スキル)

### ステップ6: 代替案とADR

主要な決定(2〜5個程度)それぞれについて:

- **検討した代替案を最低2つ**挙げ、品質特性シナリオに照らして比較表を作る
- 採用案を **ADR**(`references/adr-template.md`)として記録する。「なぜこれを選んだか」だけでなく「**なぜ他を選ばなかったか**」「どうなったらこの決定を見直すか」を書く

### ステップ7: セルフレビュー

設計書提出前に自問する:

- [ ] 各設計判断は品質特性シナリオまで遡れるか(「なんとなくベストプラクティス」になっていないか)
- [ ] 単一障害点はどこか。それは意図的な受容か
- [ ] 最初に落ちるボトルネックはどこで、それは想定規模の何倍まで耐えるか
- [ ] 信頼境界を越える全データフローに認証・認可・検証があるか
- [ ] 「この設計で一番不安な点」を1つ挙げられるか → それを設計書のリスク欄に正直に書く
- [ ] チームのスキル・運用能力で維持できるか(2ページャーの運用手順で回るか)

## 成果物フォーマット

設計書は以下の構成で出力する(規模に応じて省略可、ただし省略は明示):

```markdown
# <システム/機能名> 設計書
## 1. 背景と目標(3行以内)
## 2. 品質特性の優先順位とシナリオ(ストレッサー分析: ストレス一覧×壊れる箇所×構造で吸収する/受容するの判断を含む)
## 3. システム構成(C4図 + 説明)
## 4. データ設計(主要エンティティ、所有権、整合性)
## 5. セキュリティ設計(信頼境界図、脅威モデル表、認証認可)
## 6. 障害・運用設計
## 7. 主要な設計判断(ADR形式、代替案比較つき)
## 8. リスクと未解決事項(正直に)
## 9. 段階的移行/実装計画
```
