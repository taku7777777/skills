# arch-security-design 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの references/eval-template.md を参照。

追加の実行可能なraw fixtureとhidden rubricは [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離し、rubricをexecutorへ見せない。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 複数店舗で使ってもらう予約管理のWebサービスを新しく立ち上げる。全体をどう組み立てるか壁打ちしたい | 新規システムの構造設計(「アーキテクチャ」の語を使わない言い回し) |
| F2 | この機能で個人情報を預かることになった。作りの段階で漏洩につながる穴がないか洗い出しておきたい | 脅威モデリング(Shift Left) |
| F3 | 非同期処理の基盤、マネージドのキューにするか自前で立てるか決めたい。あとで理由を聞かれても答えられる形で | 技術選定単体 + ADRによる決定の記録 |
| F4 | 「落ちないようにしといて」としか言われてない。稼働率の目標ってどう決めて、構成にどう反映すればいい? | 品質特性の数値化とシナリオ化 |
| F5 | 依存してる外部の決済サービスが止まったとき、うちのシステムがどう振る舞うべきか今のうちに決めておきたい | 障害設計(依存先は必ず落ちる前提) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | この機能のRESTエンドポイントとリクエスト/レスポンスのJSONスキーマを設計して | `api-data-design` が発火する(API・スキーマの具体設計の領分) |
| N2 | 社内で動かすコーディングエージェントに、どこまで権限を与えてどうサンドボックス化するか設計したい | `agent-security-governance` が発火する(エージェント自体の安全) |
| N3 | 実装済みのこのコードにSQLインジェクションや認可漏れがないかチェックして | `impl-review` が発火する(設計ではなく実装済みコードの検査) |

## 検証シナリオ

### S1: マルチテナントSaaSの新規設計(中心ユースケース)
- **タスク**: 「美容室・整体院など小規模店舗向けのオンライン予約SaaSを新規開発する。店舗(テナント)ごとに顧客情報と予約を管理し、想定は初年度100テナント・同時100ユーザー程度、開発チームは4人。セキュリティも含めて設計して」と依頼する。
- **スキルなしで予想される欠落**: 品質特性を形容詞のまま進める(SEI 6部構成、特に応答測定の欠落)。信頼境界の図示とSTRIDEによる系統的な脅威列挙の欠落(一般的なセキュリティ注意点の羅列に留まる)。設計判断の代替案比較(ADR)の欠落。規模を無視したマイクロサービス提案。
- **合格条件**:
  - [ ] 品質特性シナリオが刺激源/刺激/対象物/環境/応答/応答測定の6部構成で書かれ、応答測定が数値である
  - [ ] C4のコンテキスト図・コンテナ図に相当する構成図があり、テナント間を含む信頼境界が示されている
  - [ ] 信頼境界を越えるデータフローに対するSTRIDE分類つきの脅威一覧と対処(緩和/排除/転嫁/受容)の表がある
  - [ ] 主要な設計判断に代替案が2つ以上比較され、「なぜ他を選ばなかったか」が書かれている
  - [ ] システム分割の判断に根拠が明示され、チーム規模に対して過剰な分散構成になっていない(人間判定)

### S2: LLMコンポーネントを含む機能追加(境界・例外)
- **タスク**: 「既存の問い合わせ管理システム(モジュラーモノリス)に、受信メールをLLMで要約して担当部署へ自動振り分けする機能を追加したい。月2万件処理する。設計して」と依頼する。
- **スキルなしで予想される欠落**: LLMの障害モードをプロバイダ障害・タイムアウトだけで考える(呼び出しは成功するがもっともらしく誤る「意味的失敗」の欠落)。品質の応答測定を単発の正誤で書く(評価セットに対する統計値の欠落)。品質劣化時の縮退経路の欠落。LLMコストを設計ドライバとして扱わない。
- **合格条件**:
  - [ ] 「呼び出しは成功するが内容が誤っている」失敗が障害モードとして明示され、出力のスキーマ検証・ビジネスルール検証を境界に置く設計がある
  - [ ] 振り分け品質の目標が評価セットに対する統計値(例: 代表メールN件で正解率X%)で定義され、モデル・プロンプト変更時の回帰検知(evalのCIゲート等)に言及がある
  - [ ] 品質劣化・プロバイダ障害それぞれの縮退経路(人手振り分けへのフォールバック等)が定義されている
  - [ ] LLM呼び出しコスト(トークン量・タスク単価の目標や上限アラート)への言及がある

### S3: 持ち込まれた構成案への途中介入(変化球)
- **タスク**: 「もう構成は決めてあるので、このまま細部を詰めるのを手伝って」として、月間アクティブユーザー500人規模の社内向け備品管理システムに対し、5つのマイクロサービス+イベントブローカー+サービスごとの個別DBという構成メモを渡す(品質要求の数値はどこにも書かれていない)。
- **スキルなしで予想される欠落**: 持ち込まれた構成を所与として細部だけ詰める(規模と構成の不釣り合いへの疑義の欠落)。品質要求の数値が未確定のまま構造を確定させることへの指摘の欠落。「何を得て何を捨てるか」のトレードオフ明示の欠落。
- **合格条件**:
  - [ ] 想定規模に対する分散構成の過剰さが指摘され、よりシンプルな代替(モジュラーモノリス等)が比較として提示されている
  - [ ] 品質特性の数値(規模・可用性等)が未確定であることが指摘され、確認の質問または仮値の提案と記録がある
  - [ ] 現構成案と代替案について、得るもの・捨てるもの(運用負荷、整合性、変更容易性等)の比較がある
  - [ ] ユーザーの決定を頭ごなしに否定せず、判断材料を示した上で選択を委ねる形になっている(人間判定)
