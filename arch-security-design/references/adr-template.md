# ADR(Architecture Decision Record)テンプレート

Michael Nygard 形式をベースに、代替案比較と見直し条件を追加した拡張版。
1決定 = 1ADR。ファイル名は `adr-NNNN-<決定の短い名前>.md`。

```markdown
# ADR-NNNN: <決定内容を能動態の一文で(例: 注文イベントの配信にSQSを採用する)>

- ステータス: Proposed | Accepted | Rejected | Deprecated | Superseded by ADR-XXXX
- 日付: YYYY-MM-DD
- 決定者: <名前/ロール>

## コンテキスト

<この決定が必要になった背景。関連する品質特性シナリオ(QAS-N のIDで参照)・制約(チームスキル、コスト、納期)を含める。
ここは「事実」を書く場所。意見や結論を書かない。>

## 決定

<採用する案を明確に。「〜を検討する」ではなく「〜する」と断定する。>

## 検討した代替案

| 観点 | 案A: <採用案> | 案B | 案C |
|---|---|---|---|
| <品質特性1(例: スループット)> | | | |
| <品質特性2(例: 運用負荷)> | | | |
| コスト | | | |
| チームの習熟 | | | |
| 撤退容易性 | | | |

### 案B を選ばなかった理由
<1-3文。「劣っているから」ではなく具体的なトレードオフで>

### 案C を選ばなかった理由
<同上>

## 帰結(Consequences)

### 得られるもの
- <ポジティブな帰結>

### 引き受けるもの
- <ネガティブな帰結・新たに発生する制約や作業。ゼロはあり得ない — 空欄なら分析不足>

## 見直し条件

<この決定を再検討すべきトリガーを具体的に。
例: 「イベント量が10k/秒を超えたら」「チームが3人を下回ったら」「XXのマネージドサービスがGAしたら」>
```

## 運用ルール

- 比較表のセルは◯△×ではなく**事実・数値で埋める**(例: 「p99 40ms(ベンチ済)」「チーム経験者0名」)。測っていないなら「未計測・推定」と書く。
- ADRは**追記のみ**(immutable)。決定が変わったら新しいADRを書き、古い方を Superseded にする。履歴が組織の学習資産になる。
- 「決めなかったこと」も重要な決定なら記録する(例: 「現時点ではマルチリージョン対応をしない」)。
- レビュー時は「コンテキストに書かれた事実だけから、この決定に至れるか」を確認する。至れないなら暗黙の前提が漏れている。
