## 脅威モデリングの実施手順

### 1. 対象の分解

- C4コンテナ図に**信頼境界**を点線で描き込む。典型的な境界:
  - インターネット ↔ エッジ(WAF/LB)
  - エッジ ↔ アプリケーション
  - アプリケーション ↔ データストア
  - テナントA ↔ テナントB(マルチテナント)
  - 一般ユーザー権限 ↔ 管理者権限
  - 自社システム ↔ サードパーティ(外部API、CDN、サプライチェーン)
  - LLM/エージェント ↔ ツール・実行環境(モデルが読む外部コンテンツは攻撃者入力として扱い、モデル出力が届く先への到達はすべて信頼境界越えとみなす)
- 資産を列挙し**データ分類**する: 公開 / 内部 / 機密 / 規制対象(個人情報、決済情報、認証情報)

### 2. STRIDE による脅威列挙

信頼境界を越える各データフローについて:

| 脅威 | 問い | 代表的な対策 |
|---|---|---|
| **S**poofing(なりすまし) | 通信相手は本物か? | 認証(OIDC/mTLS)、署名検証、anti-CSRF |
| **T**ampering(改ざん) | データは途中・保存中に書き換えられないか? | TLS、署名、整合性チェック、WORM監査ログ |
| **R**epudiation(否認) | 「やってない」と言われたら証明できるか? | 監査ログ(改ざん耐性つき)、タイムスタンプ |
| **I**nformation Disclosure(情報漏洩) | 見えてはいけない人に見えないか? | 暗号化、認可、最小権限、エラーメッセージ抑制 |
| **D**enial of Service | 落とされないか? 枯渇しないか? | レートリミット、キュー、オートスケール、リソース上限 |
| **E**levation of Privilege(権限昇格) | 与えた以上の権限を取れないか? | 認可の多層化、サンドボックス、入力検証 |

### 3. 脅威モデル表(成果物)

```markdown
| # | 脅威(STRIDE) | 対象フロー | 攻撃シナリオ | 影響度 | 発生可能性 | 対応 | 対策内容 |
|---|---|---|---|---|---|---|---|
| 1 | E/I | ユーザー→API | IDORで他ユーザーの注文閲覧(認可バイパスが原因、漏洩は結果) | 高 | 高 | 緩和 | 全クエリにテナントID条件を強制(リポジトリ層) |
| 2 | D | 公開API | 認証前エンドポイントへの大量リクエスト | 中 | 高 | 緩和 | IPベースレートリミット + CAPTCHA |
| 3 | T | Webhook受信 | 偽装Webhookによる状態改ざん | 高 | 中 | 排除 | HMAC署名検証必須化 |
```

対応は 緩和 / 排除(機能・データ自体をなくす)/ 転嫁(保険・外部サービス)/ 受容(理由と承認者を明記)の4択。

## 設計時に必ず決めるチェックリスト

### 認証

- [ ] 方式: OIDC / SAML / 独自(独自は原則避け、IdP・マネージドサービスを使う)
- [ ] セッション管理: トークン形式(JWTなら失効戦略必須)、有効期限、リフレッシュ、同時セッション
- [ ] MFA: **フィッシング耐性のある方式(パスキー/FIDO2)を第一候補**とする(NIST SP 800-63B)。SMS OTPは避ける。管理者はフィッシング耐性MFA必須
- [ ] パスワード保存: Argon2id / bcrypt(自作ハッシュ禁止)
- [ ] M2M認証: サービス間はmTLSまたは短命トークン。長寿命APIキーを避ける

### 認可

- [ ] モデル: RBAC / ABAC / ReBAC。権限マトリクス(ロール × 操作)を表で作る
- [ ] **強制ポイントを一元化**: 認可チェックをUI・コントローラ任せにせず、単一のミドルウェア/レイヤーで強制
- [ ] オブジェクトレベル認可(IDOR対策): 「認証済みか」だけでなく「そのリソースの所有者/許可者か」
- [ ] マルチテナント分離: アプリ層フィルタのみ / スキーマ分離 / DB分離 — データ分類に応じて選択し、**クエリレベルで強制**(RLS等)
- [ ] デフォルト拒否(fail-close)。新エンドポイントが自動的に保護される構造か

### データ保護

- [ ] 転送時: TLS 1.2+ 必須。内部通信も暗号化し、**サービス間もアイデンティティベースで認証・認可する**(ゼロトラスト NIST SP 800-207 — ネットワーク位置に暗黙の信頼を置かない。暗号化はその一部にすぎない)
- [ ] 暗号アジリティ: 保護期間が長い機密データ、将来復号されても影響が残るデータ、規制・顧客要件がある場合はアルゴリズム差し替え可能性とPQC移行("harvest now, decrypt later" 対策)を検討する。短命・低機密のデータへ一律にPQC計画を要求せず、保護期間×攻撃者能力×移行リードタイムで判断する
- [ ] 保存時: ディスク暗号化 + 機密フィールドのアプリケーションレベル暗号化の要否
- [ ] 鍵管理: KMS使用、ローテーション方針
- [ ] シークレット: Vault/クラウドシークレットマネージャ。環境変数直書き・リポジトリ混入の防止(pre-commit scan)
- [ ] 個人情報: 保持期間、削除要求への対応(GDPR/個人情報保護法)、ログへの混入防止(マスキング)。プライバシー脅威(識別可能性・リンク可能性等)はSTRIDEでは拾えないため、個人情報を扱うシステムではLINDDUNを参照
- [ ] バックアップ: 暗号化、リストア試験、ランサムウェア対策(イミュータブルバックアップ)

### アプリケーション防御

- [ ] 入力検証: 信頼境界で全入力をスキーマ検証(許可リスト方式)
- [ ] 出力エンコード: コンテキスト別エスケープ、CSP
- [ ] SSRF対策: ユーザー指定URLのfetchは許可リスト + メタデータエンドポイント遮断
- [ ] ファイルアップロード: 型検証、サイズ制限、別ドメイン配信、マルウェアスキャン
- [ ] レートリミット: 認証エンドポイント(ブルートフォース)、コスト高API、テナント別クォータ
- [ ] セキュリティヘッダ: CSP, HSTS, X-Content-Type-Options, Frame-Options

### サプライチェーン・運用

- [ ] 依存関係: SCA(Dependabot等)、lockファイル、既知脆弱性のSLA(Critical: 48h以内等)
- [ ] SBOM生成(CycloneDX/SPDX)と保管。アーティファクト署名と来歴検証(Sigstore/SLSA)。ビルド環境の隔離
- [ ] コンテナ/インフラ: 最小ベースイメージ、非rootユーザー実行、IaCスキャン
- [ ] CI/CD: シークレットのスコープ最小化、ブランチ保護、ビルドの再現性
- [ ] 監査ログ: 認証イベント、権限変更、機密データアクセス、管理操作。**ログ自体の改ざん防止**
- [ ] インシデント対応: 検知(アラート基準)、トークン一斉失効の手段、連絡フロー

### AI/LLM機能がある場合(OWASP LLM Top 10)

- [ ] プロンプトインジェクション: 外部コンテンツ(ユーザー入力、取得したWeb/文書)を信頼しない。ツール実行は最小権限+確認
- [ ] 出力の無害化: LLM出力をそのままHTML/SQL/シェルに流さない
- [ ] 機密データ: プロンプトへの混入制御、学習利用の可否
- [ ] 過剰な代理権限(Excessive Agency): 破壊的操作には人間の承認を挟む
- [ ] エージェントの実行境界: 「読めるもの」「実行できるもの」「送信できる先」を列挙し、機密アクセス・信頼できない入力・外部送信の3つが揃う構成は構造的に1つを断つ(実行環境の詳細設計は `agent-security-governance` スキル)
- [ ] RAG/ベクトルストア: 埋め込み・チャンクにも元文書と同じ認可・テナント分離を強制(検索経由の越境漏洩)。取り込みパイプラインの汚染(データポイズニング)を脅威モデルに含める
- [ ] 資源枯渇(Unbounded Consumption): トークン・呼び出し回数のリクエスト別/テナント別上限。コスト暴走はDoSの一形態として脅威モデル表に載せる

## リスク評価の目安

影響度 × 発生可能性で優先度を決める。迷ったら:

- **認証・認可・お金・個人情報**に関わる脅威は原則「高」
- 攻撃に内部アクセスが必要なら1段階下げる、認証不要でインターネットから可能なら1段階上げる
- 対応しない(受容)場合は必ず理由と承認をADRに残す
