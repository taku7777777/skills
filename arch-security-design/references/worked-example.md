# arch-security-design 適用例

題材: 架空の中堅企業向け経費精算SaaS「ケイヒフロー」(TypeScriptのモジュラーモノリス + PostgreSQL)への「多段階承認ワークフロー」追加。`requirements-definition` の要件定義書(US-01〜03 / FR-01〜02 / NFR-01〜02)が入力になっている前提だが、本例は単体でも読める。

## 入力(ユーザーの依頼と文脈)

> 「多段階承認ワークフローの要件定義ができた。セキュリティも含めて設計してほしい」

既存システムの全体構成図(C4)は存在しない。

## 各ステップでの判断

- **ステップ0(スコープ判定)**: 既存システムへの機能追加 → ステップ1・4・7に絞る。既存構成図がないため、影響範囲に限定した簡易コンテナ図を先に描いてから信頼境界を書き込む。省略したステップ: 2は要件のNFRから代表シナリオ2本のみ作成、3は簡易図で代替、5は既存の障害設計を継承しメール通知依存の障害挙動のみ追記、6はADR 1件のみ(いずれも成果物に明示)。
- **ステップ1(前提)**: 要件定義から引き継ぎ済み — 品質特性の優先順位はセキュリティ > 監査性 > 性能。規模は同時50ユーザー・申請月2,000件で、構造を変えるほどの負荷ではないと判断(新規コンテナは追加しない)。
- **ステップ2(品質特性シナリオ)**: 「職務分掌」を検証可能なQAS-1に変換。fitness functionとして認可の統合テストをCIゲート化。ストレッサー分析は機能追加の規模に対し過剰なため省略(理由: 構造決定を伴わない)。
- **ステップ4(脅威モデリング)**: 差分の信頼境界は (a) ロール間(申請者/上長/部門長)、(b) アプリ→メール送信サービス。境界を越えるフローをSTRIDEで列挙 → 最重要は権限昇格(自己承認・他人のステップの承認)と改ざん(承認開始後の金額編集で閾値を回避)。
- **ステップ6(ADR)**: 「承認ルートをいつ決定するか」が唯一の変更しにくい決定と判断し、ADR-01として記録。
- **ステップ7(セルフレビュー)**: 一番不安な点 =「組織マスタの上長情報が古い場合の誤ルート」→ リスク欄に正直に記載。

## 成果物

```markdown
# 多段階承認ワークフロー 設計書

## 1. 背景と目標
内部統制監査の指摘(高額経費の単独承認)に対応するため、金額閾値による2段階承認を追加する。既存モジュラーモノリスへの機能追加であり、新規コンテナは設けない。

## 2. 品質特性の優先順位とシナリオ
優先順位: セキュリティ > 監査性 > 性能(要件定義より引き継ぎ)。ストレッサー分析は省略(構造決定を伴わない機能追加のため)。
- QAS-1(セキュリティ): 刺激源=認証済みの申請者本人 / 刺激=自分の申請の承認APIを直接呼ぶ / 対象物=承認ユースケース / 環境=通常運用時 / 応答=拒否し監査ログに記録 / 応答測定=拒否率100%、記録漏れ0件。fitness function: 認可統合テストをCIゲート化
- QAS-2(監査性): 刺激=監査人が任意の承認済み申請の承認履歴を要求 / 応答=操作者・日時・操作の完全な履歴を提示 / 応答測定=7年分、欠損0件

## 3. システム構成(影響範囲の簡易コンテナ図)
- Webアプリ(SPA)→ APIアプリ(モジュラーモノリス内の経費精算モジュール)→ PostgreSQL
- APIアプリ → メール送信サービス(外部SaaS): エスカレーション・承認依頼通知
- 通知は非同期(ジョブキュー経由、at-least-once前提で通知の重複は許容)

## 4. データ設計
- 承認ルートは申請時に確定しExpenseReportに保持(ADR-01)。監査ログは追記専用テーブル(UPDATE/DELETE権限を付与しない)。エンティティ詳細は `ddd-design` の集約設計へ

## 5. セキュリティ設計
信頼境界: ロール間(申請者/承認者)、アプリ→メール送信サービス。脅威モデル表(抜粋):
| 脅威(STRIDE) | 内容 | 対処 |
|---|---|---|
| E: 権限昇格 | 申請者が自分の申請を承認/担当外ステップを承認 | 緩和: 承認時に担当承認者IDを検証(ドメイン層の不変条件 + 認可テスト) |
| T: 改ざん | 承認開始後に金額を編集し閾値を回避 | 排除: 提出後の申請は編集不可(差し戻しで再申請) |
| T: 改ざん | 承認ルートの直接書き換え | 緩和: ルートは申請時にサーバ側で決定、APIから変更不可 |
| R: 否認 | 「承認していない」という主張 | 緩和: 追記専用監査ログ(FR-02) |
| I: 情報漏洩 | 承認権限のない社員が他人の明細を閲覧 | 緩和: 一覧・詳細APIに申請者/承認ルート当事者のみの認可 |
認証は既存セッション方式を継承。認可は既存RBACに「承認ステップ担当者」の文脈条件を追加。ASVS L2の該当項目と突き合わせ済み。

## 6. 障害・運用設計
- メール送信サービス停止時: 承認操作は継続(通知のみ縮退)、ジョブキューに退避しリトライ+バックオフ。エスカレーション判定は日次バッチなので障害時は翌日実行で許容
- 可観測性: 承認操作の監査ログとは別に、滞留日数メトリクスをダッシュボード化(要件のリードタイム測定に対応)。他は既存の運用設計を継承(略)

## 7. 主要な設計判断(ADR-01: 承認ルートの決定タイミング)
- 決定: 申請提出時に閾値を評価し、ルートを申請にスナップショットとして固定する
- 代替案A(承認のたびに都度評価): 組織変更に自動追随するが、承認途中でルートが変わり監査証跡が説明困難になるため不採用
- 代替案B(夜間バッチで再評価): 複雑さに見合う要求がなく不採用
- 見直し条件: 組織変更頻度が高く誤ルートが月数件を超えたら再検討

## 8. リスクと未解決事項
- 組織マスタの上長情報が古い場合、誤った承認者にルートされる(スナップショット方式は自動追随しない)。運用で組織変更時の棚卸しを追加提案
- TBD-01(申請者=部門長のときの2段階目)は要件side未決。設計はルートを可変長にして両案を吸収可能にした

## 9. 段階的移行/実装計画
既存の1段階承認データはルート長1の申請として読み替え可能(スキーマはexpand-contractで移行)。詳細は(略)
```
