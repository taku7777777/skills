# CI/CD・リリース戦略スキル

Jez Humble & David Farley『継続的デリバリー』(デプロイメントパイプライン、リリースの脱イベント化)、Nicole Forsgren らの『Accelerate』とDORA研究(4キーメトリクス、速度と安定性は両立するという実証)、trunk-based development(Paul Hammant)、Pete Hodgson のフィーチャーフラグ論考(martinfowler.com)、progressive delivery(カナリア・段階的ロールアウト)、Google SRE(リリースエンジニアリング)の知見を統合した、リリースプロセスの設計・改善プロセス。

領分の境界: テストの中身と品質ゲートの詳細は `ai-adaptable-testing`、本番の監視・SLO設計は `ops-observability`、DBスキーマ・APIの互換性設計は `api-data-design` の領分。本スキルはそれらをパイプラインとリリース手順に配線する。

## 大原則

1. **デプロイとリリースは別の行為である。** デプロイはコードを本番に置く技術的行為、リリースは機能をユーザーに見せるビジネス判断。フィーチャーフラグで両者を分離すれば、デプロイは低リスクの日常作業になり、リリースのタイミングは技術の都合から解放される(Humble & Farley の「リリースの脱イベント化」の現代形)。
2. **小さく頻繁に。** バッチサイズの縮小は、変更ごとのリスク・原因特定コスト・復旧時間を下げやすい。DORA研究では高いデプロイ性能と安定性の両立が観測されているが、単純な「頻度を上げれば障害が減る」という因果則として扱わない。自動テスト、変更分割、観測性、復旧能力を伴わない頻度向上は危険である。自チームの計測でボトルネックと結果を確認する。
3. **パイプラインを唯一の本番経路にする。** 手動デプロイ・SSHでの直接修正・緊急バイパスは、検証の抜け道であり事故と構成ドリフトの温床。緊急時こそパイプラインを通す — だから通常時からパイプラインを速く保つ(Google SREのリリースエンジニアリングも自動化とポリシー強制を柱とする)。
4. **ロールバック可能性は後付けできない。** 障害時に「戻せるか」は、デプロイ戦略・DBスキーマ互換性・フラグ設計で事前に決まっている。戻せない変更(破壊的マイグレーション、非互換API)は expand-contract 等で「どの時点でも戻れる形」に分解してから出す。
5. **フィーチャーフラグは負債である。** フラグは条件分岐の増殖・テスト組み合わせの爆発・死んだコードの温床。導入時に分類(release/experiment/ops/permission)・オーナー・除去条件を決め、除去までがライフサイクル。「とりあえずフラグで」の無期限残置を許さない。
6. **エージェント時代の補正: パイプラインの検出力が委任可能範囲を決める。** エージェントが書くコードが増えるほど、「パイプラインを通ったら何が保証されるか」が人間レビューを軽量化できる範囲を直接規定する(検証ゲートの設計は `harness-engineering` と接続)。またエージェントによるデプロイ・フラグ操作・ロールバック実行は本番挙動の変更そのものであり、人間の承認ゲートと監査ログを必須にする。

## ワークフロー

### ステップ1: 現状診断

- DORA 4キーメトリクス(デプロイ頻度 / 変更リードタイム / 変更障害率 / 復旧時間)を測定する(測定方法は [references/deployment-patterns.md](references/deployment-patterns.md) §4)。測定できない指標があること自体が診断結果 — 「計測の仕組みが無い」ことも明記する
- リリース1回を解剖する: コミットから本番までの全工程を書き出し、各工程の所要時間・手動/自動・待ち時間を可視化する。最大の待ちが最初の改善対象
- 痛点を具体化する: リリースが「怖い」の中身(戻せない / 検証が手動 / 特定の人しかできない / 障害の検知が遅い)を分解し、後続ステップに対応づける
- リリース承認プロセスの実効性も診断する: DORAの研究では、変更内容を知らない外部者・承認会議による承認は安定性を改善せず速度だけを落とす。承認は自動ゲート+コードに近い者のレビューへ寄せる

### ステップ2: パイプライン設計

- ステージ構成を設計する(テンプレートは [references/deployment-patterns.md](references/deployment-patterns.md) §3): コミットステージ(10分以内が目安)→ 自動受け入れ → 追加検証 → 本番展開
- **成果物は一度だけビルドし、同一バイナリを環境間で昇格させる。** ステージごとの再ビルドは「テストしたものと出すものが違う」事故の源。環境差は設定で注入する
- 各ステージに時間予算を設定し、超過をCIで検知する(遅いパイプラインは迂回され、大原則3が崩れる)
- trunk-based development を前提にする: 長寿命ブランチは統合の遅延=バッチサイズの拡大。未完成機能はブランチではなくフラグで隠す(ステップ4)
- どのステージに何のテストを置くかは配線として本スキルが決め、テストの中身・配分は `ai-adaptable-testing` に委譲する

### ステップ3: デプロイ戦略の選択

- 判断表([references/deployment-patterns.md](references/deployment-patterns.md) §1)で選ぶ: トラフィック規模・状態の有無・インフラコスト・可観測性の成熟度が主変数。既定はローリング、即時切替が要るならブルーグリーン、本番でしか検証できないリスクがあるならカナリア、応答に影響させず検証したいならシャドー
- **どの戦略でも新旧バージョンの共存(N/N-1互換)が前提。** API・DBスキーマ・キャッシュ・セッションの互換性を確認する
- カナリアは可観測性が前提: 比較するメトリクスと判定基準を先に決める(メトリクス・SLOの設計は `ops-observability`)
- 戦略は段階導入でよい: いきなり理想形を狙わず、ローリング+自動スモークから始めて可観測性・トラフィック制御の成熟に合わせてカナリアへ進める

### ステップ4: フィーチャーフラグ運用設計

- フラグを4分類(release/experiment/ops/permission — [references/deployment-patterns.md](references/deployment-patterns.md) §2)し、分類ごとに寿命・実装方式・除去条件を決める
- 導入時にオーナーと除去条件を台帳に登録し、期限切れをCIで検知する。フラグ除去は機能リリースの完了条件に含める
- 判定点はコードに散らさず1箇所に集約する。テストは全組み合わせではなく「現在の本番構成」と「次に出す構成」を対象にする
- エージェントによるフラグ操作(特にops toggle)は本番挙動の変更 — 承認ゲートと監査ログの対象にする(大原則6)

### ステップ5: ロールバックとガードレール

- 変更ごとに最速の戻し方を事前に決める: ロールバック(前バージョン再デプロイ)/ ロールフォワード(修正を最速で通す)/ フラグoff。「どれで戻すか」を障害時に考え始めるのが最悪
- 自動ロールバック条件を定義する: カナリア分析のベースライン比較、エラー率・レイテンシ・SLOバーンレートの閾値(閾値の設計は `ops-observability` と接続)
- DBマイグレーションとの順序を規約化する([references/deployment-patterns.md](references/deployment-patterns.md) §5): スキーマ変更はアプリデプロイと分離、expand-contract、ロールバックはコードのみ戻す
- 緊急修正もパイプライン経由にする(大原則3)。バイパスしたくなる事態はパイプラインの速度不足のシグナルとして扱う
- ロールバックは訓練する: 一度も使ったことのない切り戻し手順は障害時に機能しない。定期的にリハーサル(本番前環境での実施、ゲームデー)で検証する
- エージェントにデプロイ・ロールバックを委任する場合は、実行前の人間承認・全操作の監査ログ・影響半径の制限(1環境・1サービス単位)をセットで設計する

### ステップ6: 継続改善

- DORA 4キーをダッシュボード化し、定期レビューで推移を見る。目的はチーム間比較ではなく自チームのボトルネック発見 — 数値の目標化はゲーミングを招く
- 変更障害が起きたら「どのステージで検出できたはずか」を振り返り、テスト・ゲートの追加として `ai-adaptable-testing` / `harness-engineering` に還流する
- フラグ台帳の棚卸しと時間予算の超過確認を定期タスク化する
- 改善は1ボトルネックずつ。ステップ1の診断に戻り、次の最大の待ち・痛点を狙う

## 成果物フォーマット(戦略文書)

```markdown
# <対象> CI/CD・リリース戦略
## 1. 現状診断(DORA 4キーの測定値と測定方法 / リリース解剖の所要時間内訳 / 痛点)
## 2. パイプライン設計(ステージ構成 / 各ステージの検証内容と時間予算 / 成果物の昇格経路)
## 3. デプロイ戦略(採用戦略と判断根拠 / N/N-1互換の確認項目 / トラフィック制御の仕組み)
## 4. フィーチャーフラグ運用(採用する分類 / 台帳の場所 / 除去ルールと期限検知)
## 5. ロールバック設計(変更種別ごとの戻し方 / 自動ロールバック条件 / DBマイグレーション順序の規約)
## 6. 改善サイクル(監視するメトリクス / レビュー周期 / 次のボトルネック候補)
```

小規模な対象では 1・3・5 の軽量版でよい(省略を明示)。実例: [references/worked-example.md](references/worked-example.md)
