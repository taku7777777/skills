# cicd-release 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | リリースのたびに全員が緊張して金曜は誰も出したがらない。もっと気軽に本番に出せる状態にしたい | リリースプロセス改善(頻度と安定性、「デプロイ頻度を上げたい」と語彙が異なる) |
| F2 | 新機能をまず社内メンバーだけに見せて、問題なければ徐々に一般ユーザーへ広げたい | 段階的ロールアウト+フラグによるデプロイ/リリース分離 |
| F3 | この前の障害、切り戻すのに1時間半かかった。次はワンアクションで戻せるようにしておきたい | ロールバック設計 |
| F4 | CIが40分かかるせいでみんな「あとでまとめてマージ」し始めてる。この状況を立て直したい | パイプライン診断・改善(時間予算、バッチサイズ) |
| F5 | mainにマージされてから本番に届くまで平均5日かかってる。この流れ全体を見直してほしい | リードタイム診断=デプロイメントパイプライン設計 |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 昨日デプロイした直後から500エラーが増えてる。何が原因か調べてほしい | `systematic-debugging` が発火する(デプロイ絡みでも原因調査は調査スキルの領分) |
| N2 | リリース後の異常にすぐ気づけるように、SLOと通知の仕組みを整えたい | `ops-observability` が発火する(検知・監視の整備。カナリアの判定基準の入力になる側) |
| N3 | 受け入れテストに何を書くべきか、ケースを設計してほしい | `ai-adaptable-testing` / `test-design` が発火する(テストの中身はパイプラインの配線と別) |
| N4 | エージェントにPR作成からマージまで任せて安全なように、検証ゲートを設計して | `harness-engineering` が発火する(委任ゲートの設計) |

## 検証シナリオ

### S1: 求人マッチングサービスのリリース改善戦略
- **タスク**: 「架空の求人マッチングWebサービス(エンジニア6名、ECS上のモノリス+RDB)のリリースを改善したい。現状: リリースは月1回・火曜深夜にメンテ画面を出して手動デプロイ(手順書3ページ)、リリース用ブランチを2週間前に切ってQAチームが手動検証、直近半年で6回中2回リリース起因の障害、戻すときは前のイメージを手で再デプロイ(過去実績90分)。『週1回、できれば日中に出したい』という要望に応える戦略文書を作って」
- **スキルなしで予想される欠落**: CI/CDツールの一般論の羅列。DORA 4キーメトリクスによる現状診断と「測れていないこと自体が診断結果」という扱いの欠落。デプロイとリリースの分離(フラグ)により日中デプロイを可能にする発想の欠落。リリースブランチ廃止(trunk-based)・一度だけビルドして昇格、の原則の欠落。「怖いから月1」を頻度を上げる方向で解く(バッチサイズ縮小)論理の欠落。
- **合格条件**:
  - [ ] DORA 4キーメトリクス(デプロイ頻度/リードタイム/変更障害率/復旧時間)による現状診断が含まれる
  - [ ] デプロイ(本番に置く)とリリース(ユーザーに見せる)を分離する方針とフィーチャーフラグの運用(分類・除去条件のいずれか)への言及がある
  - [ ] 長寿命のリリースブランチをやめる方向(trunk-based化・バッチサイズ縮小)が根拠つきで提案されている
  - [ ] 成果物を一度だけビルドし同一イメージを環境間で昇格させる方針が含まれる
  - [ ] 変更種別ごとの戻し方(ロールバック/ロールフォワード/フラグoff)の事前定義が含まれる

### S2: カラム分割を伴う機能のノーダウンタイムリリース
- **タスク**: 「会員管理サービスで、`users.address`(1カラムの自由記述)を `postal_code` / `prefecture` / `city` / `street` の4カラムに分割する機能をリリースしたい。テーブルは約800万行、サービスは24時間稼働でメンテ画面は出せない。アプリはローリングデプロイ(新旧Podが数分共存)。マイグレーションSQLとアプリ変更をどの順番でどう出すか、失敗したときの戻し方まで含めてリリース計画を立てて」
- **スキルなしで予想される欠落**: 「マイグレーションを流してからデプロイ」の一手順のみで、新旧共存(N/N-1互換)を考慮しない計画。expand-contract への分解(追加→二重書き込み→バックフィル→切替→削除)の欠落。「ロールバック時はコードのみ戻しスキーマは戻さない」原則の欠落。大テーブルALTERのロック挙動・バックフィルのチャンク分割への言及漏れ。
- **合格条件**:
  - [ ] 計画が expand-contract(追加→移行→削除)の複数段階に分解され、各段階が独立にデプロイ可能である
  - [ ] ローリング中の新旧バージョン共存(N/N-1互換)を前提に、旧コードが動き続けても壊れない順序になっている
  - [ ] ロールバック手順で「コードのみ戻し、適用済みスキーマ変更は戻さない(前方互換な expand を保つ)」に相当する方針が明示されている
  - [ ] 800万行のバックフィルについてロック・負荷への配慮(チャンク分割等)がある
  - [ ] contract(旧カラム削除)の実施条件が観測に基づいて定義されている(例: 旧カラム参照ゼロを一定期間確認)

### S3: 緊急修正でパイプラインを飛ばしたいという相談(途中介入)
- **タスク**: 「認証ライブラリの脆弱性修正を今すぐ本番に入れたい。うちのパイプラインはフルで50分かかるから、今回だけCIを飛ばして本番サーバーに直接パッチを当てようと思う。手順を手伝って」と依頼する。
- **スキルなしで予想される欠落**: 依頼どおり直接パッチの手順を案内してしまう。「緊急時こそパイプラインを通す」(バイパスは検証の抜け道・構成ドリフトの温床)の原則の欠落。バイパス衝動を「パイプラインが遅すぎる」というシグナルとして扱い、恒常的な高速化につなげる視点の欠落。
- **合格条件**:
  - [ ] 本番サーバーへの直接パッチをそのまま実行する手順を提供せず、リスク(未検証コード・構成ドリフト・再デプロイで消える)を指摘している
  - [ ] パイプライン経由で最速で出す現実的な代替(不要ステージの一時的短縮の是非も含む)を提示している
  - [ ] 事後の恒久対応として、パイプラインの高速化(時間予算・ボトルネック特定)を提案している
  - [ ] 緊急変更にも記録・レビュー(事後レビュー含む)を残す段取りがある
  - [ ] 提案全体が緊急性(脆弱性対応の速さ)を犠牲にしない現実的なものになっている(人間判定)
