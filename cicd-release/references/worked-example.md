# cicd-release 適用例

## 題材

架空のB2B SaaS「備品・在庫管理サービス」。エンジニア8名の単一チーム、モノリシックなWebアプリ(Kubernetes上)+ PostgreSQL。現状は隔週水曜夜の定期リリース: 前々日にコードフリーズ、前日に手動リグレッション半日、当日は手順書ベースの作業2時間。月1回程度リリース起因の障害があり、切り戻しは「前タグの再デプロイ+手動確認」で平均3時間かかる。

## 入力

「リリースを週次以上にしたい。ただ、いまでも月1回は障害が出ているので、頻度を上げるのが怖い」という依頼。

## 各ステップでの判断

### ステップ1: 現状診断

- DORA計測: デプロイ頻度=月2回強 / リードタイム中央値=9日(フリーズ待ちが大半)/ 変更障害率=約25%(直近12回中3回)/ 復旧時間=平均3時間。障害率と復旧時間はインシデント記録とデプロイ履歴の突合を今回初めて実施 — 恒常的な紐付け運用の欠如も改善項目に載せる
- リリース解剖: 最大の待ちは「フリーズ〜リリース」の2日、次いで手動リグレッション半日。障害3件の内訳は、マイグレーションとコードの非互換が2件、手動検証の漏れが1件
- 診断: バッチが大きい(2週間分の変更)ことが障害率と復旧困難の主因。「怖いから慎重に=まとめて出す」が悪循環を作っている(大原則2)

### ステップ2: パイプライン設計

- コミットステージ: 現状12分 → テスト並列化で8分に短縮(予算10分をCIで監視、超過でfail)
- 手動リグレッションの主要10シナリオを自動受け入れテスト化(テストの設計は `ai-adaptable-testing` に委譲)。25分で完走し、リリース前日の半日作業を廃止
- コンテナイメージはコミットステージで一度だけビルドし、同一イメージをタグで昇格
- コードフリーズを廃止し trunk-based に移行。未完成機能はブランチではなく release flag で隠す(ステップ4)

### ステップ3: デプロイ戦略の選択

- 判断表(deployment-patterns.md §1)を適用: Kubernetes上でローリングは即導入可、メトリクス基盤(Prometheus)あり、トラフィックは中規模
- 決定: 第1段階はローリング+readinessプローブ+デプロイ直後の自動スモーク。カナリア(5%→25%→100%)はトラフィック分割の仕組みを整備した後の第2段階として計画
- ブルーグリーンは2面分のコストとDB共有の複雑さが見合わず不採用。シャドーは実装リプレース案件がないため対象外
- N/N-1互換の確認をリリースチェックからPRレビュー観点へ前倒し

### ステップ4: フィーチャーフラグ運用設計

- まず release flag のみ導入(experiment/permission は需要が出てから)。台帳はリポジトリ内のYAMLで管理し、オーナー・導入日・除去条件を必須項目にする
- 除去条件: 全ユーザー公開から14日以内に除去PR。導入から30日超過でCI警告
- 判定は toggle router 1箇所に集約し、散在するif文を禁止

### ステップ5: ロールバックとガードレール

- 自動ロールバック条件: デプロイ後15分間、5xxエラー率がベースラインの2倍超、または p99 レイテンシが閾値超過で前バージョンへ自動ロールバック(閾値の設計は `ops-observability` と接続)
- マイグレーション起因の障害2件への再発防止: スキーマPRとアプリPRを分離し、expand-contract を規約化。「ロールバックはコードのみ、スキーマは戻さない」を明文化
- 緊急修正もパイプライン経由に一本化(コミットステージ+スモークで最短25分)。本番へのSSH直接修正の権限を回収
- エージェントによるデプロイ実行は人間承認ゲートを必須にし、デプロイ・フラグ操作の全履歴を監査ログへ記録

### ステップ6: 継続改善

- 4キーのダッシュボードを整備し月次レビュー。3ヶ月後の目標: 週2回デプロイ / 変更障害率15%以下 / 復旧30分以内(フラグoffと自動ロールバックで)
- 変更障害が起きるたびに「どのステージで検出できたはずか」を振り返り、テスト・ゲート追加に還流する

## 成果物(抜粋)

```markdown
# 在庫管理サービス CI/CD・リリース戦略
## 1. 現状診断
- デプロイ頻度: 月2回 / リードタイム中央値: 9日 / 変更障害率: 25% / 復旧: 3時間
- 痛点: バッチサイズ(2週間分)、手動リグレッション半日、マイグレーション非互換の障害2件
## 2. パイプライン設計
- コミットステージ8分(予算10分・超過fail)→ 自動受け入れ25分 → 本番展開
- イメージはコミットステージで一度だけビルドし、同一物をタグで昇格
## 3. デプロイ戦略
- ローリング(readinessプローブ+自動スモーク)。第2段階でカナリア(5%→25%→100%)
## 4. フィーチャーフラグ運用
- release flagのみ。台帳をリポジトリ管理、公開後14日で除去、30日超過でCI警告
## 5. ロールバック設計
- 自動条件: 15分間5xx率2倍超 or p99閾値超過。スキーマはexpand-contract、戻すのはコードのみ
## 6. 改善サイクル
- 4キー月次レビュー。目標: 週2デプロイ / 障害率15%以下 / 復旧30分以内
```
