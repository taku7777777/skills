# DDD 設計・実装スキル

Eric Evans『Domain-Driven Design』、Vaughn Vernon『Implementing DDD』、Scott Wlaschin『Domain Modeling Made Functional』、Alberto Brandolini(EventStorming)の知見を統合した設計・実装プロセス。

## 大原則

1. **コードより先にドメインを理解する。** 技術の話を始める前に、ビジネスの問題・用語・ルールを言語化する。
2. **ユビキタス言語を貫く。** ドメインエキスパートの言葉 = 設計書の言葉 = クラス名・メソッド名。翻訳が発生したら設計の危険信号。
3. **複雑さに見合う道具を選ぶ。** CRUDで十分な箇所に戦術的パターンを持ち込まない。DDDの価値はコアドメインの複雑さに投資することにある。
4. **不正な状態を表現できない型を作る**(Make illegal states unrepresentable)。バリデーションで守るのではなく、型で守る。

## ワークフロー

ユーザーから設計・実装の依頼を受けたら、以下の順で進める。ドメインが自明な小変更ならフェーズ1-2を口頭確認レベルに圧縮してよいが、**スキップ判断は明示する**。

### フェーズ0: 既存プロジェクトの調査

既存コードベースがある場合、まず既存のドメインモデル・パッケージ構造・命名・永続化方式・トランザクション慣行を調査し、すでに使われている用語をフェーズ1の用語集の初期値にする。以降の設計は既存構造との整合の上に置く(乖離が必要なら理由を明示)。

### フェーズ1: ドメイン理解(EventStorming的整理)

コードを書く前に、以下を箇条書きで整理してユーザーに提示する:

- **ドメインイベント**: この業務で「起きること」を過去形で列挙(例: 注文が確定された、在庫が引き当てられた)
- **コマンド**: イベントを引き起こす操作(例: 注文を確定する)
- **アクター**: 誰が/何がコマンドを発行するか
- **ビジネスルール(不変条件)**: 常に守られるべき制約(例: 注文合計は明細の合計と一致する)
- **ポリシー**: 「〜が起きたら〜する」という反応ルール
- **リードモデル**: アクターがコマンド発行の判断に使う情報(フェーズ5のCQRS読み取りモデルの種になる)
- **外部システム**: コマンド/イベントの発生源・通知先となる自コントロール外のシステム
- **ホットスポット**: 理解が割れている点・未解決の疑問。質問はここから発行し、解消するまで消さない
- **用語集(ユビキタス言語)**: ドメイン用語 → 定義 → コード上の名前。曖昧な用語・同音異義語(同じ「商品」がコンテキストで意味が違う等)を特定する

不明点はここで質問する。**ドメインの理解が曖昧なまま実装に進まない。**

### フェーズ2: 戦略的設計

- **サブドメイン分類**: Core(競争優位の源泉・複雑さに投資)/ Supporting(必要だが差別化しない)/ Generic(買ってくる・ライブラリで済ます)を判定。DDDの重い設計はCoreにだけ適用する。
- **境界づけられたコンテキスト**: モデルの一貫性が保てる境界を引く。判断基準は「同じ用語が同じ意味を持つ範囲」「チーム・デプロイの境界」「モデルの変更理由が揃う範囲」— 言語的・モデル的な境界であり、トランザクション整合性は集約の境界基準であってBCの基準ではない(強い整合性が必要な集約群が複数コンテキストに跨って見えたら、境界の引き方を疑う)。AI協働の観点では、BCは人間チームだけでなく**エージェントの作業境界・情報隠蔽の単位**でもある。1つのBC内のタスクが、そのBCの用語集・不変条件・モジュール別CLAUDE.mdだけで完結し、他BCの内部モデルを読み込まずに済む構造は、コンテキスト効率と誤用防止の両方に効く(文書の配置規約は `docs-for-ai` スキル)。BCを跨ぐ変更は必ずPublished Language(公開契約)経由とし、公開契約の変更は人間承認ゲートの対象に対応づける(ゲート設計は `harness-engineering` スキル)。
- **コンテキストマップ**: 複数コンテキストが関わる場合、関係パターンを明示する(Partnership / Shared Kernel / Customer-Supplier / Conformist / **Anticorruption Layer** / Open Host Service / Published Language / **Separate Ways**=統合コスト>価値なら統合しない判断)。外部システム・レガシーとの統合には原則ACLを置く。

### フェーズ3: 戦術的設計

`references/tactical-patterns.md` の詳細ガイドに従い、以下を設計する:

1. **値オブジェクト(Value Object)を不変条件のある概念から抽出する。** 金額、数量、メールアドレス、期間などでも、独自の検証・単位・等価性・演算規則がある、複数箇所で意味が揺れる、取り違えの影響が大きい場合にVOを優先する。不変・等価性は値で判定し、不正値を生成境界で拒否する。単純な入出力DTOの末端値まで機械的にクラス化せず、型エイリアスやスキーマ検証で十分な場合はその判断を記録する。
2. **エンティティ**: 同一性(ID)で追跡が必要なものだけ。属性はできる限りVOで構成する。
3. **集約(Aggregate)**: Vaughn Vernonの4原則で境界を決める:
   - ① **真の不変条件でトランザクション境界を決める**(1トランザクション = 1集約の更新)
   - ② **集約は小さく設計する**(迷ったら分ける。巨大集約は競合とロックの温床)
   - ③ **他の集約はIDでのみ参照する**(オブジェクト参照を持たない)
   - ④ **集約間の整合性は結果整合性(ドメインイベント)で保つ**
4. **ドメインイベント**: 集約の状態変更で意味のあるものはイベントとして発行。集約間連携・監査ログの起点にする。**コンテキスト内部のドメインイベントと外部公開用の統合イベントは区別する**: 他コンテキストへは内部イベントをそのまま流さず、Published Languageとして設計したスキーマバージョン付きの統合イベントへ変換する(内部モデルの変更が公開契約を壊さないため)。
5. **長期プロセス(サガ/プロセスマネージャ)**: 複数集約・コンテキストをまたぎ、途中状態が長く残る、独立した失敗・再試行・タイムアウト・補償(キャンセル・返金等)を追跡する必要がある業務プロセスは、暗黙のイベント連鎖にせず明示的にモデル化する。イベント数だけを採用閾値にしない。イベントハンドラは冪等に(イベントIDで重複排除)。補償は技術的ロールバックではなく業務操作(「予約を取り消す」)として定義する。
6. **ドメインサービス**: 複数集約にまたがるロジック、どのエンティティにも自然に属さないロジックのみ。安易にサービスへ逃げてドメインモデル貧血症を起こさない。
7. **リポジトリ**: 集約単位で1つ。集約ルート以外にリポジトリを作らない。インターフェースはドメイン層、実装はインフラ層。
8. **ファクトリ**: 生成が複雑な場合のみ。新規生成と永続化からの再構成を区別する。

設計結果は以下の形式でユーザーに提示する。設計だけの依頼ならここで止める。実装まで明示的に依頼され、重大な未決事項がなければ提示と並行して実装へ進めてよい。境界・不変条件・データ移行など結果を大きく変える未決事項がある場合だけ、承認または修正指示を得てから実装する:

```
## ドメインモデル設計
### 集約: 注文 (Order)
- ルート: Order (エンティティ)
- 構成: OrderLine (エンティティ), Money, Quantity, OrderStatus (VO)
- 不変条件: 確定後の明細変更不可 / 合計金額 = Σ明細金額
- 発行イベント: OrderPlaced, OrderCancelled
- 他集約への参照: CustomerId, ProductId (IDのみ)
```

実例: `references/worked-example.md`

### フェーズ4: アーキテクチャ配置

- **依存の方向は常にドメインへ向ける**(ヘキサゴナル/オニオン/クリーンアーキテクチャのいずれでも本質は同じ)。
- レイヤー責務:
  - **ドメイン層**: エンティティ、VO、集約、ドメインイベント、ドメインサービス、リポジトリIF。**フレームワーク・DB・外部APIへの依存は原則ゼロ**。例外はORMのマッピング注釈のみ — 永続化モデル分離のコストが見合わない場合の明示的な妥協として許容し、その判断を記録する。
  - **アプリケーション層**: ユースケース単位のオーケストレーション。トランザクション制御、リポジトリ呼び出し、イベント発行。**ビジネスルールを書かない**(ルールがここに漏れたらドメイン層へ押し込む)。
  - **インフラ層**: リポジトリ実装、外部API、メッセージング。
  - **プレゼンテーション層**: DTO変換、入力の形式検証。ドメインオブジェクトをそのままAPIレスポンスにしない。
- 既存プロジェクトの場合は**既存のアーキテクチャ規約に従う**。理想形の押し付けより一貫性を優先し、乖離があれば指摘だけする。

### フェーズ5: 実装

- 実装順序: **VO → エンティティ・集約 → ドメインサービス → リポジトリIF → ユースケース → インフラ実装 → プレゼンテーション**(ドメインの内側から外側へ)。
- ドメインロジックは**振る舞いとしてモデルに置く**。`order.cancel()` であって `orderService.cancelOrder(order)` ではない(ロジックは情報の持ち主へ — Information Expert)。状態を訊き出して外側で判断しない(Tell, Don't Ask)。
- ユースケースは「コマンド → 純粋なドメインロジック → イベント/結果」のパイプラインとして構成し、I/Oは端に寄せる(functional core, imperative shell — Gary Bernhardt。Wlaschin『DMMF』の「I/Oをワークフローの端に寄せる」パイプライン設計も同旨)。業務上ありうる失敗はドメインエラーとして型で表現する(`references/tactical-patterns.md` のドメインエラー節)。
- setterを公開しない。状態変更は意図を表すメソッド経由(`confirm()`, `addLine()`)。
- 集約の更新は必ずルート経由。内部エンティティを外に漏らさない(必要なら読み取り専用で返す)。
- クエリ性能が問題になる画面表示系は、無理に集約を経由せず**CQRS的に読み取りモデルを分離**してよい(書き込みの整合性はドメインモデル、読み取りは最適化されたクエリ)。
- テスト: ドメイン層は**インフラ不要の高速単体テスト**で不変条件を網羅する。「不正な状態遷移が拒否されること」のテストを必ず書く。

### フェーズ6: セルフチェック

実装完了前に `references/anti-patterns.md` のアンチパターン一覧と照合し、該当がないか確認して結果を報告する。特に:

- [ ] ドメインモデル貧血症(getter/setterだけのクラス+ロジックはサービスに全部)になっていないか
- [ ] ドメイン層がフレームワーク・DBに依存していないか
- [ ] 1トランザクションで複数集約を更新していないか
- [ ] ユビキタス言語とコードの名前が一致しているか
- [ ] プリミティブ型が意味を持って引き回されていないか(String型の「メールアドレス」等)

設計成果物の作成・実装が依頼範囲に含まれる場合は、フェーズ1の用語集(対訳込み)とホットスポットの解消結果、フェーズ2のコンテキストマップ、フェーズ3の集約設計(不変条件・発行イベント)をリポジトリ内の文書として永続化する(配置と形式は `docs-for-ai` スキルの規約に従う。用語集は docs/glossary.md、不変条件・状態遷移などのドメイン知識は docs/domain/ が既定)。レビュー・相談だけの依頼では勝手に書き込まず、保存候補とドラフトを提示する。git commit は明示依頼がある場合だけ行う。

このチェックリストのうち構造規則(ドメイン層の依存ゼロ、集約間のID参照、レイヤー配置)は、一度きりの人手照合で終わらせず**アーキテクチャテストとして恒久化することを提案する**(ArchUnit / deptrac / dependency-cruiser 等。導入可否は既存のプロジェクト規約に従う)。型で表現した不変条件はコンパイラが、型で表現しきれない不変条件はドメイン単体テストが常時強制する。これらはエージェントが実装するときの決定論的ガードレールであり、どれをゲートにするかの設計は `harness-engineering` スキルへ接続する。
