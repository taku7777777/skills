# ddd-design 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの references/eval-template.md を参照。

追加の実行可能なraw fixtureとhidden rubricは [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離し、rubricをexecutorへ見せない。

## 発火テスト

| # | プロンプト | 理由 |

## 検証シナリオ

### S1: 予約ドメインの新規モデリング(中心ユースケース)
- **タスク**: 「貸し会議室サービスの予約機能を設計・実装して。ルール: 同じ会議室の同じ時間帯に予約は1件まで/開始24時間前までキャンセル無料、以降はキャンセル料50%/予約は15分単位・最長8時間。TypeScriptで新規リポジトリ」と依頼する。
- **スキルなしで予想される欠落**: ドメイン理解の整理(イベント・コマンド・不変条件)を提示せずコードから書き始める。金額・時間帯がプリミティブ型のまま(値オブジェクト抽出の欠落)。集約の不変条件と境界の明示の欠落。実装前の設計承認ステップの欠落。
- **合格条件**:
  - [ ] コードより先に、ドメインイベント(過去形)・コマンド・不変条件・用語集を含むドメイン整理が提示される
  - [ ] 時間帯・金額などが値オブジェクトとして設計され、不正値(15分単位でない、8時間超)をコンストラクタ等の型レベルで拒否している
  - [ ] 集約設計に不変条件(同一会議室・時間帯の重複不可等)と発行イベントが明記されている
  - [ ] 他の集約・エンティティ(会議室、会員)への参照がIDのみになっている
  - [ ] 設計をユーザーに提示し承認を得てから実装する段取りになっている(または省略判断が明示されている)

### S2: 集約境界をまたぐ整合性(境界・例外)
- **タスク**: 「イベントチケット販売の購入フローを設計して。座席在庫の引き当て→注文確定→外部決済の流れで、決済失敗時は座席を解放する。同時アクセスで同じ座席が二重に売れるのは絶対に防ぎたい」と依頼する。
- **スキルなしで予想される欠落**: 「絶対に防ぎたい」に引きずられて在庫・注文・決済を1つの巨大集約や1トランザクションにまとめる(Vernon 4原則の欠落)。集約間の結果整合性とドメインイベントの欠落。決済失敗時の処理を技術的ロールバックとして扱う(業務操作としての補償の欠落)。イベントハンドラの冪等性の欠落。
- **合格条件**:
  - [ ] 座席在庫と注文が別集約として設計され、相互参照がIDのみである
  - [ ] 「1トランザクション=1集約の更新」が守られ、集約間の整合性はドメインイベント経由の結果整合と明記されている
  - [ ] 真の不変条件(同一座席の二重引き当て禁止)がどの集約のトランザクション境界で守られるか特定されている
  - [ ] 決済失敗時の補償が「座席の引き当てを解放する」等の業務操作として定義されている(サガ/プロセスマネージャの明示)
  - [ ] イベントの重複配信に対する冪等化(イベントIDによる重複排除等)に言及がある

### S3: 肥大化したサービスクラスへの暗黙的介入(変化球)
- **タスク**: 「この `SubscriptionService` がテストしづらくて困ってる。なんとかして」と依頼し、コードを添付する: 状態が `"active"` / `"paused"` / `"canceled"` の文字列リテラル、public setter経由の状態変更、契約・請求・通知のロジックが1クラスに混在、金額は number 型(DDDの語は依頼に一切含めない)。
- **スキルなしで予想される欠落**: モック注入などテスト技法だけで対処する。ロジックをドメインオブジェクトの振る舞いへ移す発想(Tell, Don't Ask)の欠落。文字列状態・プリミティブ金額の型による保護の欠落。「不正な状態遷移が拒否されること」のテストの欠落。
- **合格条件**:
  - [ ] 契約のロジックをドメインオブジェクトの振る舞い(`subscription.pause()` 等)へ移す提案がある
  - [ ] 文字列の状態が型(enum/判別可能な型)に置き換えられ、setterではなく意図を表すメソッド経由の状態変更になっている
  - [ ] 金額の値オブジェクト化(プリミティブ型執着の解消)が提案されている
  - [ ] インフラ不要のドメイン単体テストで不変条件・不正な状態遷移の拒否を検証する方針が示されている
  - [ ] 既存プロジェクトの構造・命名の調査、または既存規約との整合への言及がある
