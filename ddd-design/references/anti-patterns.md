# DDD アンチパターン集(セルフチェック用)

実装完了前にこの一覧と照合する。

## モデリングのアンチパターン

| アンチパターン | 症状 | 対処 |
|---|---|---|
| ドメインモデル貧血症 (Anemic Domain Model) | エンティティが getter/setter のみ。ロジックが全部サービス層にある | ロジックをデータの持ち主へ移動。「このifはどのオブジェクトが知っているべき情報か」を問う |
| プリミティブ型執着 (Primitive Obsession) | String のメール、int の金額、生UUIDのIDが引き回される | VO・型付きIDを導入 |
| 巨大集約 (God Aggregate) | 1集約に数十のエンティティ。更新競合・ロック待ち・ロード遅延 | 真の不変条件で再分割。結果整合性へ |
| 汎用語モデリング | `Manager`, `Processor`, `Info`, `Data` などドメインに存在しない語 | ドメインエキスパートの語彙に戻る。名前が付けられない = 理解が足りない |
| 一つの巨大モデル | 全社で単一の「商品」クラスを共有し、属性が肥大化 | 境界づけられたコンテキストで分割。販売の商品と在庫の商品は別モデル |
| CRUDイベント | `XxxUpdated` / `XxxChanged` 等、業務的意味のないイベント名 | EventStormingに戻り「業務として何が起きたのか」を過去形で命名し直す(`ShippingAddressCorrected` 等)。1つのUpdatedを複数の意味あるイベントに分割 |
| CRUDドメインへの過剰DDD | 単純なマスタメンテに集約・イベント・レイヤーをフル装備 | Generic/SupportingサブドメインはシンプルなCRUDでよい。判断を明示 |

## 実装のアンチパターン

| アンチパターン | 症状 | 対処 |
|---|---|---|
| ドメイン層の汚染 | ドメインクラスがDB・HTTP・フレームワークへ依存(許容はORMマッピング注釈まで — それも明示的な妥協として記録) | 依存逆転。IFをドメイン層、実装をインフラ層へ。または永続化モデル分離 |
| ロジックの漏出 | アプリケーションサービスやコントローラにビジネスルールのif文 | ドメイン層へ押し込む。ユースケースは「取得→委譲→保存」の定型に保つ |
| 集約の複数同時更新 | 1トランザクションで2つ以上の集約を save | 境界見直し or ドメインイベント+結果整合性 |
| オブジェクト参照による集約間結合 | `order.getCustomer().getRank()` | IDで参照し、必要なデータはユースケースで取得して渡す |
| setter公開 | `setStatus()`, `setAmount()` が public | 意図を表すメソッド(`cancel()`, `applyDiscount()`)に置換 |
| ドメインオブジェクトのAPI直接露出 | エンティティをそのままJSONレスポンスに | DTO変換を挟む。内部構造変更がAPI破壊にならないように |
| リポジトリの肥大化 | 画面ごとの検索メソッドが数十個 | 読み取りはQueryService/ReadModelへ分離(CQRS) |
| テストのインフラ依存 | ドメインロジックのテストにDBが必要 | ドメイン層を純粋に保てていない兆候。依存を引き剥がす |
| なんちゃってVO | VOにsetterがある / 等価性未実装 | 不変+値等価を徹底 |
| トランザクションスクリプトの偽装 | レイヤーは分かれているがドメイン層が素通し | コアドメインなら振る舞いをモデルへ。単純ドメインならDDDをやめる判断も正しい |

## プロセスのアンチパターン

- **ドメインエキスパート不在のモデリング**: 開発者の想像で用語を作る → 必ず用語の定義をユーザー/ドキュメントで確認する
- **一発完成主義**: モデルは反復で洗練される。実装中に得た洞察(「この2つは実は別概念だ」)をモデルへ還元する — Evansのいう Deep Model への Breakthrough は継続的なリファクタリングと知識の噛み砕きの過程で(実装中にも)起きる
- **パターン先行**: 「まずCQRSとイベントソーシングを入れる」— 複雑さの実態より先に道具を選ばない
