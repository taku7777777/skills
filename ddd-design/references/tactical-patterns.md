# 戦術的パターン詳細ガイド

- 値オブジェクトとエンティティ

## 値オブジェクト (Value Object)

**判定基準**: 同一性ではなく「値そのもの」に意味がある → VO。

| 特性 | 実装ルール |
|---|---|
| 不変 | 全フィールド final/readonly。変更は新インスタンス生成(`money.add(other)` は新しい `Money` を返す) |
| 自己検証 | コンストラクタ/ファクトリメソッドで不正値を拒否。生成できた時点で常に正しい |
| 等価性 | 全属性の値で equals/hashCode を実装 |
| 閉じた演算 | 関連する振る舞いをVO自身に持たせる(`Money.add`, `Period.overlaps`, `Quantity.isGreaterThan`) |

**VO化すべき典型例**: 金額(通貨とセット)、数量、割合、メールアドレス、電話番号、住所、氏名、期間(開始・終了のペア)、ID(`OrderId` 等の型付きID)、ステータス(enumまたは代数的データ型)。

**型付きIDの効用**: `CustomerId` と `OrderId` を取り違えるバグをコンパイル時に排除する。全エンティティのIDを型付きにするのが原則。

**状態はできるだけ型で表現する**(Wlaschin):
- 「確定済み注文には確定日時がある」→ nullable な `confirmedAt` ではなく `UnconfirmedOrder` / `ConfirmedOrder` の型分割、または判別可能なユニオン型・sealed class を検討
- ただし言語・チーム慣習で過剰になる場合は、状態enumと状態遷移メソッド内のガードで妥協してよい

## エンティティ

- IDによる同一性。ライフサイクル(生成→変更→削除)を追跡する対象。
- **属性の変更はビジネス上の意図を表すメソッドで行う**。`setStatus(CANCELLED)` は禁止。`cancel(reason)` を作り、その中で遷移可能性の検証・イベント発行を行う。
- コンストラクタ or ファクトリで「生成時点の不変条件」を保証。フレームワーク都合の空コンストラクタが必要な場合はスコープを最小化する。正攻法としてORM用の永続化モデルとドメインモデルを分離しマッピング層を挟む選択肢もある(コストと相談し、選択を明示)。

## 集約 (Aggregate)

### 境界の決め方(Vernonの思考手順)

1. まず「1エンティティ = 1集約」から始める
2. **真の不変条件**(トランザクション内で必ず守るべき整合性)を洗い出す
3. 不変条件を共有するオブジェクトだけを同じ集約にまとめる
4. 「同時に更新されるか?」「片方だけ更新されて一瞬ズレても業務上許容できるか?」を問う — 許容できるなら別集約+結果整合性

### よくある誤り

- **「関連があるから」で集約に含める**: 関連と整合性は別。注文と顧客は関連するが、同一トランザクションで更新する不変条件はない → 別集約
- **コレクションの無限成長**: 「顧客とその全注文」を1集約にすると数千件ロードになる → 注文は独立集約にして `CustomerId` で参照
- **画面都合で集約を決める**: 表示に必要なデータの塊 ≠ 整合性の塊。表示は読み取りモデルで解決する

### 実装ルール

- リポジトリは集約ルート単位でのみ定義する(ドメインオブジェクトへのリポジトリ注入はしない)
- 集約内部のエンティティへの外部からの直接操作を禁止(ルートのメソッド経由)
- 楽観的ロック(バージョン番号)を集約単位で持つ
- 1ユースケース(1トランザクション)で更新する集約は1つ。複数更新が必要に見えたら: (a) 集約境界を見直す (b) ドメインイベント+結果整合性にする (c) それでもダメなら意識的な例外として文書化する。判断基準は「**その整合性を保つのは誰の仕事か**」(ユーザー/業務の仕事なら同一トランザクションも正当 — Vernon)

## ドメインイベント

- 命名は**過去形**(`OrderPlaced`, `PaymentCompleted`)。ユビキタス言語に含める。
- ペイロードは「何が起きたか」を再構成できる最小限(集約ID、発生日時、関連する値)。集約全体を詰め込まない。
- 用途: 集約間の結果整合性 / 別コンテキストへの通知 / 監査ログ / 非同期処理の起点。
- 発行タイミング: 集約のメソッド内で記録し、永続化成功後にディスパッチ(トランザクショナルアウトボックスパターンを検討)。

**Event Sourcing の採用判断**: 監査証跡が業務要件そのもの / 時点復元・時系列分析が必要 / 状態遷移自体がドメインの中心 — のいずれかに該当する場合のみ検討する(デフォルトは状態永続化)。採用する場合:
- (a) イベントは状態再構成に十分な情報を持たせる(上記「最小限」ルールは通知・統合イベント側の話)
- (b) 楽観的ロックはストリームバージョンで行う
- (c) イベントスキーマの進化戦略(upcasting等)を最初に決める
- (d) 読み取りはプロジェクションで構築し、CQRSが事実上前提になる
- (e) 個人情報・削除義務のあるデータはイベントペイロードに直接入れない(イベントは追記専用で後から消せない)。削除要求(GDPR・個人情報保護法等)への対応方針 — 参照IDのみ記録し実体は可変ストアで管理する、または暗号鍵の破棄で読取不能にする(crypto-shredding)— を採用判断の時点で決める

## ドメインエラー

業務上ありうる失敗(在庫不足、与信超過、期限切れ)は**ドメインエラーとして型で表現し、ユビキタス言語に含める**。インフラ障害(接続断等)と同じ例外機構に混ぜない。

- 代数的データ型を持つ言語では Result/Either で返す(Wlaschinのワークフロー型: `Command -> Result<Event列, DomainError>`)
- 例外ベースの言語ではドメイン例外の基底型を1つ定め、アプリケーション層で捕捉して結果DTOへ写像する
- **プロジェクト内でどちらか一方に統一し、混在させない**

## ファクトリ

生成ロジックが複雑な場合(複数VOの組み立て、生成時の採番・ポリシー適用)のみ導入する。**新規生成(create)と永続化からの再構成(reconstitute)を区別**し、再構成では生成時バリデーションやイベント発行を再実行しない。

## ドメインサービス

置いてよいもの:
- 複数集約を跨る判定(例: 口座間送金の可否判定 — ただし更新は各集約で)
- 外部の知識が必要な純ドメインロジック(例: 重複チェック)。ただし**集約を跨ぐ一意性はチェックでは保証できない**(チェックと保存の間に競合が入る check-then-act)。`exists` による事前チェックはUX向上のための早期検出と割り切り、**最終防衛線は必ずDBの一意制約(またはリザベーションパターン)**に置く。制約違反はインフラ例外のまま漏らさず、ドメインエラー(例: `DuplicateEmailError`)へ変換して返す

**危険信号**: サービスのメソッドが特定エンティティの属性を読んで判定しているだけ → そのロジックはエンティティ自身へ移す。

## リポジトリ

- インターフェースはドメイン層に置き、**ドメインの言葉で定義**する(`findOverdueOrders()` であって `findByStatusAndDueDateLessThan()` ではない)
- 集約を丸ごと保存・復元する。部分更新のメソッドを増やさない
- 検索条件が複雑化したら仕様パターン(Specification)を検討
- **読み取り専用のクエリ(一覧画面等)はリポジトリの責務外にしてよい** → QueryService / ReadModel として分離(CQRS)

## アプリケーションサービス(ユースケース)

1メソッド = 1ユースケース。定型フロー:

```
1. 入力DTOを受け取り、VO/IDに変換
2. リポジトリから集約を取得
3. 集約のメソッドを呼ぶ(ビジネスロジックはここに書かない)
4. リポジトリで保存
5. イベントのディスパッチ(アウトボックス採用時は保存トランザクション内で記録済み) / 結果DTOを返す
```

このフローに収まらないif文が増えてきたら、ドメイン層に置くべきロジックが漏れている兆候。
