# ddd-design 適用例

題材: 架空の中堅企業向け経費精算SaaS「ケイヒフロー」(TypeScriptのモジュラーモノリス + PostgreSQL)への「多段階承認ワークフロー」追加。`requirements-definition` の要件定義書(US-02: 金額閾値による2段階承認・職務分掌、FR-01: エスカレーション)と `arch-security-design` のADR-01(承認ルートは申請時に確定)が入力になっている前提だが、本例は単体でも読める。

## 入力(ユーザーの依頼と文脈)

> 「多段階承認ワークフローをDDDで設計して。要件定義書と設計書はdocs/にある」

## 各フェーズでの判断

- **フェーズ0(既存調査)**: 既存の`ExpenseReport`はgetter/setterだけのアクティブレコード風で、状態はstring。既存用語(申請・承認・差し戻し)を用語集の初期値とし、レイヤー構造は既存の`domain/application/infra`規約に従う。
- **フェーズ1(ドメイン理解)**:
  - ドメインイベント: 経費が申請された / 承認ステップが承認された / 申請が差し戻された / 申請が最終承認された / 承認がエスカレーションされた
  - コマンド: 申請を提出する / 承認する / 差し戻す(アクター: 申請者、承認者。エスカレーションはポリシー起点)
  - 不変条件: 申請合計 = Σ明細金額 / 承認はルート順にのみ進む / 申請者は自分の申請の承認者になれない / 最終承認後の申請は変更不可
  - ポリシー: 承認待ちが7営業日を超えたら代理承認者へ権限を付与する
  - リードモデル: 承認者の「自分宛の承認待ち一覧」(滞留日数つき)
  - 外部システム: 組織マスタ(上長・部門長の解決)、メール通知
  - ホットスポット: 申請者=部門長のときの2段階目(要件のTBD-01)→ 解消まで実装は可変長ルートで両案を吸収
- **フェーズ2(戦略的設計)**: 承認ワークフローは監査対応の中核 → Core。通知はGeneric(既存基盤を利用)。組織マスタは別コンテキスト(組織コンテキスト)であり、Customer-Supplier + ACL(`ApproverDirectory`ポート)経由で参照し、組織マスタの内部モデルを経費精算コンテキストへ持ち込まない。コンテキストが2つで関係も1本のため、コンテキストマップは箇条書きで代替。
- **フェーズ3(戦術的設計)**: 下記成果物のとおり。「承認ルート順・自己承認禁止・最終承認後変更不可」という真の不変条件が申請と承認ステップに跨るため、両者を1集約にまとめる(Vernon原則①)。組織はIDのみ参照(原則③)。通知は結果整合(原則④、イベント経由)。
- **フェーズ4(配置)**: 既存規約に従う。既存コードはORMエンティティをドメイン層に置いておりマッピング注釈が混在 → 乖離を作らずこれを踏襲し、妥協として記録。
- **フェーズ5(実装)**: VO(`Money`, `ExpenseStatus`)→ 集約 → リポジトリIF → ユースケースの順。承認は`report.approve(approverId)`の振る舞いとして実装し、業務上ありうる失敗(担当外承認・自己承認・確定済み申請への操作)はドメインエラー型で表現。コードは(略)。
- **フェーズ6(セルフチェック)**: 貧血症なし(状態遷移はすべて集約のメソッド)/ ドメイン層のDB依存は既存踏襲の注釈のみ(記録済み)/ 1トランザクション1集約を確認。集約間ID参照とレイヤー依存はdependency-cruiserでのアーキテクチャテスト化を提案。

## 成果物

```markdown
## ドメインモデル設計

### 集約: 経費申請 (ExpenseReport)
- ルート: ExpenseReport (エンティティ)
- 構成: ExpenseLine (エンティティ), ApprovalStep (エンティティ), Money, ExpenseStatus, ApprovalRoute (VO)
- 不変条件:
  - 申請合計 = Σ明細金額
  - 承認はApprovalRouteの順序どおりにのみ進む(先のステップを飛ばせない)
  - 承認者 ≠ 申請者(職務分掌)、かつ現在ステップの担当承認者のみが承認・差し戻し可能
  - 提出後は明細変更不可(差し戻し後の再申請でのみ編集可)、最終承認後は一切変更不可
- 発行イベント: ExpenseSubmitted, ApprovalStepApproved, ExpenseReturned, ExpenseApproved, ApprovalEscalated
- 他集約・他コンテキストへの参照: ApplicantId, ApproverId (IDのみ。承認者の解決は ApproverDirectory ポート経由)
- 補足: ApprovalRouteは申請時に確定するスナップショット(ADR-01)。ステップ数は可変長(TBD-01を吸収)

### 状態遷移 (ExpenseStatus)
draft → submitted → (approving) → approved / returned → (編集後) submitted
不正遷移(approved後の編集等)はドメインエラー ExpenseAlreadyFinalizedError で拒否する

### 読み取りモデル(CQRS)
「承認待ち一覧」(滞留日数・金額での絞り込み)は集約を経由せず専用クエリで実装

### 用語集(対訳)
| ドメイン用語 | 定義 | コード上の名前 |
|---|---|---|
| 経費申請 | 明細の束と承認状態を持つ申請単位 | ExpenseReport |
| 承認ステップ | ルート上の1承認単位(担当承認者を持つ) | ApprovalStep |
| 承認ルート | 申請時に確定した承認ステップの列 | ApprovalRoute |
| 差し戻し | 理由付きで申請者へ返し再編集可能にする操作 | return / ExpenseReturned |
| エスカレーション | 滞留時に代理承認者へ権限を付与する反応 | escalate / ApprovalEscalated |
```

用語集と不変条件は`docs/glossary.md`・`docs/domain/expense-approval.md`へ永続化し、ホットスポット(TBD-01)は解消されるまで文書に残す。
