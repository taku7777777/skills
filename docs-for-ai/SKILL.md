# AI時代のドキュメント設計スキル

Diátaxis(Daniele Procida)、Docs as Code、『Software Engineering at Google』(ドキュメント章)、David Parnas(情報隠蔽)、Anthropic のコンテキストエンジニアリング知見を統合した、人間とエージェント双方のためのドキュメント体系設計プロセス。

## 大原則

1. **ドキュメントはAI時代に価値が逆転した。** かつて「書いても読まれない」だった文書は、いまエージェントが**毎回読む入力**になった。ドキュメントの品質がエージェントの出力品質を直接決める。ドメイン知識の泥くさい整理は最もレバレッジの高いAI投資。
2. **信頼できないドキュメントは無いより悪い。** 古い記述はエージェントを自信満々に間違わせる。「すべてを書く」より「書いたものが正しくあり続ける仕組み」を優先する。コードから導出できるものは書かない(生成する)。
3. **読者ごとに文書の種類は違う**(Diátaxis)。チュートリアル(学習)/ ハウツー(作業)/ リファレンス(情報)/ 説明(理解)を混ぜない。エージェント向けにはさらに「制約・規約(してはいけないこと)」という種類が加わる。
4. **配置は情報隠蔽に従う。** 知識はそれが必要になる場所の近くに置く(モジュールの規約はそのディレクトリに)。グローバルな文書に全部書くのはコンテキストの浪費であり、更新も追いつかない。
5. **1文書1目的、自己完結。** エージェントは文書を断片で読む。「前章を参照」ではなく、各文書が単独で誤解なく読める単位にする。
6. **段階的開示(progressive disclosure)で層化する。** エージェントのコンテキストは有限の予算。どの文書も「1〜2行の要約(何が書いてあり、いつ読むべきか)→ 本文 → 詳細参照」の層を持たせ、読むか読まないかの判断を要約だけで下せるようにする。スキルの description、CLAUDE.md の `@` インポート、docs/ の目次はいずれもこの要約層にあたる。要約層の質が悪いと、本文がどれだけ正しくても読まれない(または全部読まれてコンテキストを浪費する)。

## ワークフロー

### ステップ1: 現状監査

詳細な検査項目は `references/claude-md-and-audit.md` の監査チェックリストに従う:

- 既存文書の棚卸し: 種類(Diátaxis分類)、最終更新、正確性の抜き取り検査(3〜5箇所をコードと突き合わせ)
- **エージェントの失敗ログから逆引き**: エージェント(や新人)が繰り返し間違えること・毎回質問されることを列挙 — これが書くべきものの優先順位リスト
- 暗黙知の所在: 特定個人の頭の中にしかない知識(ドメインルール、障害対応、歴史的経緯)を特定

### ステップ2: 体系の設計

知識の供給経路と標準配置を決める:

| 文書 | 内容 | 配置 |
|---|---|---|
| CLAUDE.md(ルート) | ビルド・テストコマンド、横断規約、頻出の落とし穴。**簡潔さが命**(毎回読み込まれる=常駐コスト)。目安は数百行以内。超えたら `@docs/xxx.md` インポートか下位のCLAUDE.mdへ分割 | リポジトリルート |
| CLAUDE.md(モジュール別) | そのモジュール固有の規約・制約・設計意図。該当ディレクトリの作業時のみ読み込まれる(=常駐しない)ため、ルートより詳しく書いてよい | 各ディレクトリ |
| スキル(.claude/skills/) | オンデマンドで呼び出す手順知識(リリース手順、定型調査、レビュー観点)。**CLAUDE.mdから追い出した低頻度知識の受け皿**。エージェントに実行させたい定型作業はスキル化が第一候補 | .claude/skills/\<name\>/SKILL.md |
| 用語集(ユビキタス言語) | ドメイン用語 → 定義 → コード上の名前。同音異義語の区別 | docs/glossary.md |
| ADR | 決定の理由と見直し条件(`arch-security-design` 参照) | docs/adr/ |
| ドメイン知識 | ビジネスルール、状態遷移、外部制約(法令・業界慣行)。**コードから読み取れない「なぜ」**を中心に | docs/domain/ |
| ハウツー/ランブック | 定型作業の手順(リリース、障害対応、データ修正)。エージェントが実行できる粒度で | docs/runbooks/ |
| オンボーディング | 学習順序のガイド(既存文書へのリンク集+読む順) | docs/onboarding.md |
| エージェントメモリ | エージェントがセッション中に自律追記する暫定知識。**正式文書ではなく受信箱**として扱い、定期レビューで有効なものを CLAUDE.md・docs/ へ昇格し、残りは破棄する(自動追記されるファイルの汚染リスクは `agent-security-governance` 参照) | ツール規定のメモリファイル |
| MCPリソース | リポジトリ外・更新頻度の高い知識(組織横断規約、社内APIカタログ、Wiki)。リポジトリへコピーせず、オンデマンドの参照経路として設計する(コピーは陳腐化の起点) | MCPサーバー経由 |

使い分けの基準: **常駐(毎回読む)= CLAUDE.md / オンデマンドの手順 = スキル / オンデマンドの知識 = docs/ 配下+参照導線 / セッション由来の暫定知識 = メモリ(昇格待ちの受信箱)/ リポジトリ外・動的な知識 = MCPリソース**。CLAUDE.md の詳細な設計基準(載せる/載せないの判断、分量の目安、モノレポでの階層化、リンクの作法)は `references/claude-md-and-audit.md` の設計ガイドに従う。

複数のコーディングエージェントを併用するチームでは **AGENTS.md を正とし CLAUDE.md をシンボリックリンクにする**(ツール横断標準)。公開プロダクトのドキュメントには **llms.txt / llms-full.txt** を用意し、外部のAIからの参照経路も設計する。

### ステップ3: 執筆

- **頻出エラー駆動で書く**: 網羅を狙わず、ステップ1の「繰り返し間違えることリスト」から着手。CLAUDE.md は「間違えたら追記、間違えなくなったら削除」で育てる
- **エージェント向け記述の作法**:
  - 指示は具体的・検証可能に(「適切にテストする」→「`npm run test:unit` が通ること。新規ロジックには境界値テストを含める」)
  - 禁止事項には理由を添える(理由があるとエージェントは例外状況でも正しく判断する)
  - コード例は最新の実在コードから引用し、ファイルパスを添える(架空の例はドリフトの温床)
- **ドメイン知識の泥くさい整理**(3段階で進める):
  1. **収集**: ソースは有識者ヒアリング・過去の障害・問い合わせ対応ログ・レビューでの頻出指摘。ヒアリングは白紙の質問より「コードと既存文書から起こしたドラフトに有識者が赤を入れる」形が速く、抜け(暗黙の前提)も見つかりやすい
  2. **構造化**: ルール・条件・対応関係は表へ(状態×操作の可否マトリクス、条件分岐表)。散文より人間にもAIにも誤読が少ない。本則と例外・境界事例は分けて明記する。用語集は `references/claude-md-and-audit.md` のテンプレートを使う
  3. **実事例で検証**: 過去の実事例(問い合わせ・障害・判断に迷ったケース)をテストケースとして、文書だけを与えたエージェントが正しい判断に到達するか確かめる(ステップ5のコールドスタートテストのドメイン版)。誤答箇所が文書の穴であり、有識者に確認して埋める
- 書けない箇所(不明・係争中)は「不明」と明記する。曖昧に埋めるとエージェントが事実として扱う

### ステップ4: 陳腐化の防止(最重要)

仕組みの実装例(CIでのリンク切れ・コード参照検査、鮮度メタデータ、更新トリガーの定義)は `references/claude-md-and-audit.md` を参照:

- **コードと同一PRで更新**: 文書の更新をDefinition of Doneに含める。関連文書の場所をCLAUDE.mdに書いておけばエージェント自身が更新できる
- **導出可能なものは生成**: APIリファレンスはコードから生成、図はMermaid(テキスト=diff可能)、設定一覧は実ファイルから抽出
- **検証の機械化**: 文書内のコマンド・コード片をCIで実行して検証(doc test)。リンク切れチェック。「最終確認日」フロントマターと定期棚卸し
- **棚卸しをエージェントに委任**: 「文書の記述とコードの現状を突き合わせ、乖離を報告(または修正PR)する」定期ジョブをエージェントで回す。drift gate(PR単位のブロック)が拾えない、変更されなかった側の文書の腐敗を捕捉できる。人間の役割は乖離の裁定(文書が古いのか、コードが規約違反なのか)に絞る
- **drift検出**: 仕様文書とコードの乖離をレビュー観点に含める(`harness-engineering` のdrift gateと連携)
- 更新されない文書は削除するか「歴史的文書(現状を反映しない)」とラベルする — 静かに腐らせるのが最悪

### ステップ5: 効果測定

- **コールドスタートテスト**(最も直接的な検証): 履歴のない新規エージェントセッションに文書だけを与え、代表タスク(ビルド、典型的な機能追加)を実行させる。エージェントが文書にない質問をした箇所・誤読した箇所が、次に書くべきもの
- エージェントの同種の間違いが減ったか(CLAUDE.md追記の前後比較)
- 人間への質問・割り込みが減ったか
- 新規メンバー(人間・エージェント)の立ち上がり時間

## アンチパターン

- **CLAUDE.mdの百科事典化**: 常駐コンテキストに低頻度の知識を詰める → 配置を階層化し、詳細は参照先ファイルへ
- **書いた時点で完成扱い**: ドキュメントはコードと同じく保守対象。保守計画のない文書は書かない方がまし
- **散文への執着**: ルール・条件・対応関係は表に。手順は番号付きリストに
- **「ソースコードを読めば分かる」**: コードは What と How は語るが Why と Why not(検討して捨てた案)は語らない。そこが文書の本領
- **ツール導入で解決した気になる**: Wiki・Notion・生成AIツールの導入は体系設計の代替にならない。構造と保守の仕組みが本体

## 成果物フォーマット

整備の依頼では、散文の助言で終えず以下を表形式で出す:

1. **監査表**: 文書 × Diátaxis分類 × 最終更新 × 抜き取り検査の結果(正確/乖離あり/検証不能) × 処置(維持/更新/生成に置換/削除)
2. **体系設計表**: ステップ2の配置表を対象プロジェクトの実パスで具体化したもの(新設・移動・分割を明記)
3. **優先順位付き執筆バックログ**: 頻出エラー・質問(ステップ1の逆引きリスト) × 書く文書 × 期待効果
4. **陳腐化防止の仕組み一覧**: 対象文書 × 検証手段(doc test / 生成 / drift gate / 棚卸し) × 実行タイミング

執筆・CI設定・CLAUDE.md変更まで依頼されている場合は、提案で止めず実装する。
