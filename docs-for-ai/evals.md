# docs-for-ai 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

追加の実行可能なraw fixtureとhidden rubricは [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離し、rubricをexecutorへ見せない。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 新しく入った人が毎回同じ質問をしてくるし、エージェントも同じ場所でつまずく。何をどこに書き残すべきか決めたい | 失敗ログからの逆引き+体系設計 |
| F2 | Wiki から移してきた仕様メモが散らかっていて、Claude に読ませても的外れな回答しか返ってこない | 既存文書の監査と再編(AIに使える形への整理) |
| F3 | 来月退職するベテランしか知らない業務ルールがある。引き継げる形で残すのを手伝ってほしい | 暗黙知の収集・構造化(ドメイン知識の整理) |
| F4 | README のセットアップ手順が半年前から動かないと言われた。直すだけじゃなく二度と腐らないようにしたい | 陳腐化防止の仕組みづくり |
| F5 | CLAUDE.md と AGENTS.md と .cursorrules が乱立してる。複数エージェント併用前提でどう統一する? | コンテキストファイルの体系設計(ツール横断標準) |

## 検証シナリオ

### S1: 配送管理SaaSのドキュメント整備
- **タスク**: 「架空の配送管理 SaaS(チーム6人、モノレポ `apps/api` / `apps/batch`)。`docs/` に8文書とルート README があるが最終更新は不明。エージェントの失敗ログとして『配送料の距離区分を誤解(旧料率表を参照した)』『DB マイグレーションのコマンドを間違える』が繰り返し観測されている。ドキュメントを整備してほしい」と依頼する(worked-example の店舗予約 SaaS とは別題材)。
- **スキルなしで予想される欠落**: Diátaxis 分類による監査、失敗ログ逆引きの優先順位付け、常駐/オンデマンドの配置基準、陳腐化防止の仕組み。素のモデルは既存文書を書き直す・README を厚くするだけで、保守の仕組みに到達しない。
- **合格条件**:
  - [ ] 既存文書の監査表(種類の分類×鮮度×正確性検査×処置)がある
  - [ ] 失敗ログから逆引きした優先順位付き執筆バックログがある
  - [ ] 常駐(CLAUDE.md)とオンデマンド(docs/・スキル)を区別した配置の体系設計表がある
  - [ ] 陳腐化防止の仕組みが「対象文書×検証手段(doc test / 生成 / 棚卸し等)×実行タイミング」の形で定義されている
  - [ ] ドメインルール(距離区分×料率)が散文ではなく表に構造化されている

### S2: CLAUDE.md 百科事典化の指示への対応
- **タスク**: 「ドメイン知識・リリース手順・API 仕様・過去の障害の教訓、全部まとめてルートの CLAUDE.md に書き込んでほしい。1ファイルにあれば確実に読まれるでしょ」と指示される。
- **スキルなしで予想される欠落**: 常駐コスト(毎回読み込まれる)の概念と段階的開示。素のモデルは指示どおり巨大な CLAUDE.md を生成する。
- **合格条件**:
  - [ ] 指示どおり1ファイルに全部書くのではなく、常駐コストを理由に分割・階層化を提案している
  - [ ] 移設先の使い分け(モジュール別 CLAUDE.md / スキル / docs/ 配下 / MCP 等)が基準つきで示される
  - [ ] CLAUDE.md に残すもの(コマンド・横断規約・頻出の落とし穴・参照導線)の選定基準が明示される
  - [ ] 要約層(何が書いてあり、いつ読むべきか)の設計に言及している

### S3: 相談形式の暗黙的依頼
- **タスク**: 「AI エージェント用のドキュメントと人間用のドキュメントって、分けて書くべきなんですかね? うちは今 README 1本だけなんですけど」と相談される(明示的な整備依頼ではない)。
- **スキルなしで予想される欠落**: 「分けなくてよい」「場合による」の一般論で終わる。読者別ではなく文書の種類(Diátaxis+制約・規約)で分ける整理、ツール横断標準(AGENTS.md)、効果検証手段が出てこない。
- **合格条件**:
  - [ ] 「人間用/AI用の二重管理」ではなく、文書の種類(学習/作業/情報/理解+エージェント向けの制約・規約)で分ける整理が示される
  - [ ] 二重管理を避ける正本を定め、対象ツールとOSの対応に応じてシンボリックリンク・生成同期・短いadapterのいずれかを選んでいる
  - [ ] 現状(README 1本)から始める具体的な次の一歩が提示される
  - [ ] 整備後の効果検証手段(コールドスタートテスト等)に言及している
