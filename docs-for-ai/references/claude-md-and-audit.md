# CLAUDE.md 設計とドキュメント監査の実務ガイド

SKILL.md のステップ1(現状監査)・ステップ2(体系の設計)・ステップ3(用語集)・ステップ4(陳腐化の防止)を詳細展開する。SKILL.md が判断の原則、本ガイドが手を動かすときのチェックリストと実装例。

## 1. CLAUDE.md 設計ガイド

### 載せるもの / 載せないもの

判断基準はただ一つ、**「毎回のセッションで読む価値があるか」**。CLAUDE.md は全セッションに常駐する=1行ごとに恒久的なコンテキスト税がかかる。

| 載せる | 理由 |
|---|---|
| ビルド・テスト・lint の実行コマンド | ほぼ全セッションで使う |
| 横断規約のうち、エージェントが**実際に間違えた実績のあるもの** | 失敗駆動 — 効果が実証済み |
| してはいけないこと+理由 | 禁止は事後に指摘しても遅い。理由があると例外状況でも正しく判断される |
| リポジトリの地図(主要ディレクトリの役割を各1行) | 探索コストの削減 |
| 詳細文書への導線(パス+「いつ読むべきか」) | progressive disclosure の要約層 |

| 載せない | 行き先 |
|---|---|
| 低頻度の手順(リリース、障害対応、定型調査) | スキル / docs/runbooks/ |
| モジュール固有の規約 | 該当ディレクトリの CLAUDE.md |
| ドメイン知識の本文 | docs/domain/(CLAUDE.md には導線のみ) |
| コードから導出できる情報(API一覧、設定値、依存一覧) | 生成に置換(手書きしない) |
| 一般常識(「テストを書け」「読みやすいコードを」) | 削除 — モデルは既に知っている。トークンの純浪費 |
| 一時的な情報(進行中の移行の注意) | 期限と削除条件を明記して載せるか、進捗文書へ |

### 分量の目安と縮小の規律

- **ルート CLAUDE.md の目安は数百行以内**(SKILL.md ステップ2)。ただし行数の絶対値より**追記時の規律**が本体: 追記するたびに「同じ価値以下の既存行を消せないか」を問う。
- **削除候補の見つけ方**: 直近3ヶ月でそのルール由来の間違いが起きていない行、コードの現状と食い違う行、下位 CLAUDE.md やスキルに移せる行。
- 「間違えたら追記、間違えなくなったら削除」(SKILL.md ステップ3)を運用に落とすには、**追記の起点をふりかえり・PRレビューの定型項目にする**(「今週エージェントが繰り返した間違いは? CLAUDE.md に足すか?」)。

### モノレポでの階層化

- **ルートには全パッケージに適用される内容だけ**を置く。「このルールは全パッケージに当てはまるか?」— No なら該当パッケージの CLAUDE.md へ。
- 下位 CLAUDE.md は該当ディレクトリの作業時のみ読み込まれる(常駐しない)ため、**ルートより詳しく書いてよい**: そのモジュールの設計意図、内部規約、テストの流儀、触ってはいけない箇所。
- **同じ規約をルートと下位の両方に書かない。** 重複は片方だけ更新されて矛盾する未来を確定させる。共通部分をルートに昇格させるか、下位からルートの該当節を参照する。
- 階層の目安: ルート+パッケージ直下の2層で始める。3層以上は「その中間層に固有の規約が実在するか」を疑う。

### リンクによる progressive disclosure

- **`@docs/xxx.md` インポートは分割の道具であって節約の道具ではない。** インポート先は常に読み込まれ、常駐コストは本文と同じ。ファイル管理上の分割(生成物の差し込み、チーム別の管理)に使う。
- **コンテキストを節約するのは「パス+読む条件」の導線**: 「DBマイグレーションを触るときは先に `docs/migrations.md` を読む」。エージェントが必要時のみ読むため、常駐するのは1行だけ。
- 導線の質が全て: **「何が書いてあるか+いつ読むべきか」を1〜2行で**。「詳細はこちら」だけのリンクは読む/読まないの判断材料にならず、無視されるか全部読まれるかの二択になる(SKILL.md 大原則6)。

## 2. ドキュメント監査チェックリスト

### 監査の手順

1. **棚卸し**: リポジトリ内の全文書(`find . -name "*.md"` + Wiki・共有ドライブ等の外部)を列挙し、監査表(SKILL.md 成果物1)の行にする
2. 各文書に対して以下の分類・検査を実施
3. 処置を決定: 維持 / 更新 / 生成に置換 / 統合 / 削除(または「歴史的文書」ラベル)

### Diátaxis 4分類への仕分け

判定質問で仕分ける。**1文書に複数分類が混ざっていたら分割候補**(読者と目的が違うものは同居させない)。

| 分類 | 判定質問 | 混入の兆候(分割シグナル) |
|---|---|---|
| チュートリアル(学習) | 初学者が手を動かして「できた」を得るためのものか | 途中で選択肢・応用の説明が始まる(ハウツー混入) |
| ハウツー(作業) | 特定の目的を達成する手順の列か | 背景・理論の講義が長い(説明混入) |
| リファレンス(情報) | 事実の正確・網羅的な記述か(無味乾燥でよい) | 手順や推奨が混ざる |
| 説明(理解) | なぜそうなっているか・何を捨てたかの背景か | 手順が混ざる(ハウツーへ分離) |
| 制約・規約(エージェント向け追加分類) | 「してはいけない」「必ずこうする」のルールか | 推奨(should)と禁止(must not)が区別されていない |

### 鮮度・オーナー・重複の検査

文書ごとのチェックリスト:

- [ ] **鮮度**: 最終更新日を機械取得(`git log -1 --format=%cs -- <file>`)。対象コードの最終更新より古ければ乖離疑い
- [ ] **オーナー**: 更新責任者(個人ではなくチーム/ロール)を特定できるか。**オーナー不在の文書は必ず腐る** — 監査時点で割り当てるか削除候補にする
- [ ] **重複**: 同じトピックを扱う文書が他にないか。あれば正本を1つ決め、他は正本へのリンクに置換(重複は矛盾の温床)
- [ ] **正確性の抜き取り検査**(3〜5箇所): 記載コマンドを実際に実行 / コード参照(パス・関数名)の実在を確認 / 数値・設定値を実ファイルと突合。1箇所でも乖離があれば全体を「乖離あり」扱い
- [ ] **リンク切れ**: 内部リンク・外部リンクの検査(機械化は §4)
- [ ] **読まれているか**: 被リンク数、アクセスログ、エージェントのセッションログでの参照実績。読まれていない原因が導線(要約層)の不備か、内容の不要かを切り分ける

処置の判定マトリクス: **正確×読まれる=維持 / 不正確×読まれる=最優先で更新 / 正確×読まれない=導線改善か統合 / 不正確×読まれない=削除**(または歴史ラベル)。

## 3. 用語集(ユビキタス言語)テンプレート

**コード上の名前との対応列が本体**。定義だけの用語集は既存の辞書と変わらない。

```markdown
# 用語集(ユビキタス言語)

<!-- 追加・変更は PR で行い、定義はドメインエキスパートのレビューを必須とする -->

| 用語 | 定義(1〜2文) | コード上の名前 | 類語との区別・注意 | 出典/決定 |
|---|---|---|---|---|
| 契約 | 顧客と料金プランの紐付け。解約後も履歴として残る | `Subscription` | 法務文書の「契約書」(`Contract`)とは別概念 | ADR-012 |
| 予約枠 | 予約可能な時間帯の在庫単位 | `TimeSlot` | 「予約」(`Booking`)は枠に対する確保行為 | ドメインWS 2026-04 |
| 確定 | 予約が支払い完了により不可逆になった状態 | `Booking.status = CONFIRMED` | 「承認」(スタッフ操作)とは別の遷移 | docs/domain/booking.md |
```

運用ルール:

- **用語とコード名のズレは隠さず表に明記する**。理想はリネームで解消、当面できないなら表が橋渡し(エージェントは表を見て正しく変換できる)
- 同音異義語(コンテキストで意味が変わる語)は**行を分けてコンテキストを明記**する(境界づけられたコンテキストの検出器にもなる — `ddd-design` 参照)
- 廃止した用語は削除ではなく「廃止(→新用語)」と記録する(旧コード・旧文書の読解に必要)

## 4. 陳腐化防止の仕組みの実装例

### CI でのリンク切れ・コード参照検査

- **リンク切れ**: lychee 等のリンクチェッカーを CI に組み込む。内部リンクは毎 PR、外部リンクは定期ジョブに分離する(外部サイト起因の flaky で PR を止めない)
- **コード参照検査**: 文書中のファイルパス参照が実在するかを検査する。実装例:

```bash
# docs/ 配下の Markdown からコードパス参照(`src/...` 形式)を抽出し実在を検査
grep -rhoE '`[a-zA-Z0-9_./-]+\.(ts|tsx|py|go|java|rb)`' docs/ CLAUDE.md \
  | tr -d '`' | sort -u \
  | while read -r f; do [ -e "$f" ] || echo "missing: $f"; done
# 出力が空でなければ CI を fail させる
```

- **doc test**: 文書内のコマンド・コード片を CI で実行して検証する(実行可能マーカー付きコードブロック+実行ランナー)。少なくとも「セットアップ手順」と「ビルド・テストコマンド」は実行検証の対象にする — ここが古いと全エージェント・全新人が最初に躓く

### 鮮度メタデータ

frontmatter で「最後に人が中身を検証した日」を持つ。git の更新日(typo 修正でも動く)とは別に管理するのが要点。

```yaml
---
last_verified: 2026-07-01   # 内容をコード・現実と突き合わせた日
owner: platform-team        # 更新責任(ロール/チーム名)
review_cycle: 90d           # この日数を超えたら棚卸しキューへ
status: current             # current | draft | historical
---
```

- 定期ジョブで `last_verified + review_cycle < today` の文書を列挙し、棚卸しキュー(issue 化、またはエージェント突き合わせジョブの入力)に積む
- `status: historical` は「現状を反映しない」の宣言。CLAUDE.md に「`status: historical` の文書を現状の根拠にしない」と1行書いておくと、エージェントの誤読を防げる

### 更新トリガーの定義

「何が起きたらどの文書を更新するか」を表で先に決め、Definition of Done・PR テンプレートに組み込む。トリガーが定義されていない文書は、更新されない文書である。

| トリガー(変更イベント) | 更新すべき文書 | 検出・強制の手段 |
|---|---|---|
| API エンドポイントの追加・変更 | API リファレンス(生成) | 生成物の再生成を CI で強制(差分があれば fail) |
| 環境変数・設定項目の追加 | セットアップ手順、設定一覧(抽出) | 設定ファイル変更を検知する PR チェック |
| ドメイン用語の新設・変更 | 用語集 | PR テンプレートのチェック項目+レビュー観点 |
| 障害対応で手順が変わった | 該当ランブック | ポストモーテムの定型アクション項目に固定 |
| ディレクトリ構成の変更 | CLAUDE.md の地図、オンボーディング | コード参照検査(上記)が落ちて発覚する |
| エージェントの同種の間違いが再発 | CLAUDE.md(追記) | ふりかえりの定型項目 |

- CODEOWNERS で docs/ の各領域を関連コードと同じオーナーに割り当てる(コードレビューと同じ導線で文書レビューが回る)
- PR 単位の強制(drift gate)で拾えない「変更されなかった側の文書」の腐敗は、定期棚卸しジョブ(エージェント委任 — SKILL.md ステップ4)で捕捉する
