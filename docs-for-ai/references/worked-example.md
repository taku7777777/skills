# AI時代のドキュメント設計スキル 適用例

## 題材

架空の店舗予約SaaS「(仮)予約管理サービス」。開発チーム8人、モノレポ(`apps/web` / `apps/api` / `packages/domain`)、コーディングエージェントを日常的に併用。ドキュメントは初期に書いた README と Wiki からの移植文書が `docs/` に雑多に存在する。

## 入力(ユーザーの依頼と文脈)

> 「エージェントが毎回同じ間違いをする(テストコマンドを間違える、キャンセル料の仕様を誤解する)。CLAUDE.md はあるが1,200行あって誰も保守していない。ドキュメントを整備してほしい」

文脈: 直近の失敗ログとして「`npm test` を実行(正しくは `pnpm test:unit`)」「キャンセル料を予約日基準で計算(正しくは利用日基準)」「非推奨の旧APIクライアントを新規コードで使用」が繰り返し観測されている。

## 各ステップでの判断

- **ステップ1(現状監査)**: `docs/` 配下12文書+CLAUDE.md を棚卸し(`references/claude-md-and-audit.md` のチェックリストを使用)。抜き取り検査で `docs/setup.md` のコマンド3/5が実行不能、`docs/pricing.md` は昨年の料金改定を未反映と判明。失敗ログの逆引きから優先リストは「テストコマンド」「キャンセル料仕様」「新旧APIクライアントの使い分け」の3件。キャンセル料の例外規定はリーダーの頭の中にしかない暗黙知と特定。
- **ステップ2(体系の設計)**: CLAUDE.md 1,200行の内訳を分析 — 6割がリリース手順とドメイン知識の本文だった。**常駐すべきはコマンド・横断規約・地図・導線のみ**と判断し、リリース手順はスキルへ、ドメイン知識は `docs/domain/` へ、モジュール規約は各パッケージの CLAUDE.md へ分割(目標150行)。
- **ステップ3(執筆)**: 網羅を狙わず優先リスト3件から着手。キャンセル料は散文でなく「キャンセル時期×プラン→料率」の表に構造化し、有識者ヒアリングは白紙質問でなくコードから起こしたドラフトへの赤入れ形式で実施 — 例外規定(無断キャンセルの扱い)の抜けが発見された。過去の問い合わせ3件をテストケースに、文書だけを与えたエージェントが正答するか検証してから確定。
- **ステップ4(陳腐化の防止)**: `docs/setup.md` のコマンドは doc test 化。API リファレンスは手書きをやめ OpenAPI から生成に置換。全文書に鮮度メタデータ(`last_verified` / `owner`)を導入し、コード参照検査を CI に追加(実装は `references/claude-md-and-audit.md` §4)。
- **ステップ5(効果測定)**: 新規セッションのエージェントに「キャンセル料計算の修正」タスクでコールドスタートテストを実施。旧APIクライアント選択の誤りが再発 → CLAUDE.md の導線不足と判定し、規約1行を追記して再テストで解消を確認。
