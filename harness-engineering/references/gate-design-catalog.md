# ゲート設計カタログ(分類判断・検証レイヤー・エスカレーション・計測)

SKILL.md ステップ2(ゲート分類)・ステップ3(検証レイヤー)・ステップ5(計測)の詳細展開。本カタログはゲートマトリクスを実際に書き下ろすときの作業手引きであり、SKILL.md と乖離したら SKILL.md の原則を正とする。

## 目次

- HARD / SOFT / AUTOの判断基準
- 変更種別ごとのゲート配置
- 検証レイヤー設計
- 委任・エスカレーションテンプレート
- 計測指標

## 1. HARD / SOFT / AUTO の分類判断基準

**分類は変更の「見た目」ではなく、失敗時のコストと検証の機械化可能性で決める。** 同じ「1行の変更」でも、lint設定の1行とドキュメントの1行は別世界にある。

判断は次の3問を順に通す:

1. **失敗を機械が確実に検出できるか?** → No なら AUTO は不可。検出できる(型・テスト・スキーマ・lint)なら次へ
2. **失敗したときの回復は安価か?** → revert一発で戻り、外部への影響(公開API利用者、本番データ、課金)がないなら AUTO 候補。回復に人間の判断・調整・謝罪が要るなら HARD 候補
3. **その変更は検証システム自体に触れるか?** → Yes なら無条件で HARD(SKILL.md 大原則: 検証する側の資産は検証される側から分離する)

| 判定 | 機械検出 | 回復コスト | 分類 |
|---|---|---|---|
| 両方良好 | 可能 | 安価(revert可) | AUTO |
| 検出は可能だが影響が読みにくい | 可能 | 中(境界内) | SOFT |
| どちらかが欠ける | 不能 or 高価 | 高価・不可逆 | HARD |
| ハーネス自体への変更 | — | — | 常に HARD |

**迷ったら一段厳しい側に置き、計測データ(差し戻し率)を見て緩める。** 逆方向(緩く始めて事故ってから締める)は信頼の回復コストが高くつく。

## 2. ゲート配置カタログ(変更種別ごとの標準配置)

### AUTO に置くもの(機械検証のみで通過)

| 変更種別 | 通過条件 | 備考 |
|---|---|---|
| コードフォーマット | フォーマッタ実行結果が冪等 | 議論の余地なし。人間レビューに載せるだけ無駄 |
| lintエラーの修正 | lint green + テスト green | lint**ルールの変更**は別物(HARD) |
| 型エラーの修正 | 型チェック green + テスト green | 型の**緩和**(any相当・キャスト・非nullアサーション・公開シグネチャ変更)を含む差分は SOFT へ自動昇格 |
| ドキュメントとコードの同期 | drift gate green | ドキュメントの**内容変更**(仕様の書き換え)は SOFT 以上 |
| 依存の脆弱性パッチ(semverパッチ) | テスト green + lockfile差分のみ | メジャー更新・新規依存の追加は HARD |

### SOFT に置くもの(エージェントレビュー+サンプリング人間確認)

| 変更種別 | レビューエージェントの必須出力 | サンプリング初期値 |
|---|---|---|
| 境界内の実装変更 | エスカレーション要否+根拠 | 30% |
| テストの新規追加 | 検出力の評価(実装を壊したら落ちるか) | 30% |
| 内部リファクタリング(公開契約不変) | 契約不変の確認方法 | 30% |
| バグ修正(再現テスト付き) | 再現テストが修正前に red だった証跡 | 20% |
| 新規タスク種別の最初のN件 | — | 100%(全件確認) |

### HARD に置くもの(人間の承認必須)

| 変更種別 | 理由 |
|---|---|
| アーキテクチャ境界・モジュール構造の変更 | 機械検証不能+影響半径大 |
| 公開API・イベントスキーマ・DBスキーマ | 外部に契約相手がいる。互換性判断は文脈依存 |
| 認証認可・課金・データ削除に触れる変更 | 失敗が不可逆 or 信頼毀損 |
| 依存の新規追加・メジャー更新 | サプライチェーンリスク+捏造パッケージ(実在確認は `impl-review` のチェックリスト) |
| 本番設定・インフラ・デプロイパイプライン | 影響半径が最大 |
| **ハーネス自体**: CI設定・hooks・lint/型設定・fitness閾値・ゲート定義・CLAUDE.md | 検証の独立性。緩和も強化も等しく HARD(「強化だから安全」ではない — 強化を装った実質緩和がある) |
| テストの検出力を弱める変更(削除・skip・アサーション緩和・期待値書き換え・タイムアウト/リトライ拡大) | 報酬ハックの主経路。検知対象の全一覧は `ai-adaptable-testing` の `references/quality-gates-catalog.md` |

### 判断が分かれるケースの考え方

- **セキュリティスキャン(SAST・secret検出・依存監査)**: 検出そのものは AUTO(CIで機械実行)だが、**検出結果への対応は AUTO にしない。** 「スキャンを通すための抑止コメント追加(`# nosec` 等)」はハーネス弱体化と同型であり HARD。修正パッチ自体は SOFT でよい
- **契約テストの更新**: consumer側が自分の使うフィールドを増やす方向(契約の強化)は SOFT。provider側が契約を破る/緩める方向は HARD(外部に相手がいる変更)。「どちら向きの変更か」を diff のパス+内容で機械判別できるようCI側に実装する
- **人間レビュー(従来のコードレビュー)をゲートとしてどこに置くか**: HARD の承認行為そのものであり、SOFT ではサンプリングの中身になる。**「全PRを人間が全文レビュー」はゲート設計の放棄**であり、レビュー負荷の計測値が改善しない最大要因。人間の目は HARD 一覧とサンプリングに集中投下する
- **生成されたマイグレーション**: スキーマ変更なので原則 HARD だが、「追加のみ・後方互換・ロールバック手順同梱」を機械検証できるなら SOFT に降格してよい。降格条件をCIチェックとして書けないなら HARD のまま
- **E2Eテストの変更**: テスト弱体化検知の対象だが、UI変更に伴うセレクタ修正など正当な追随が多い。一律 HARD にすると形骸化するため、「アサーション・待機条件・スキップに触れる差分」だけを HARD に絞り、セレクタのみの差分は SOFT に落とす

## 3. 検証レイヤー設計チェックリスト

SKILL.md ステップ3の7レイヤーを設計・監査するときの確認項目。**内側(高速)のレイヤーで捕まえられる失敗を外側(低速)に漏らしていないか**が全体を貫く問いになる。

### 静的検査(型・lint・スキーマ)

- [ ] エージェントがローカル/ワンコマンドで実行できるか(CIでしか動かない静的検査は自己修正ループに入らない)
- [ ] エラーメッセージに「何が・どこで・どう直すか」が含まれるか(ルール名だけのlintエラーは修正精度を下げる)
- [ ] 「レビューで毎回指摘すること」のうちルール化済みの割合を把握しているか(未ルール化の常連指摘がlint候補リスト)
- [ ] 禁止APIリスト・依存方向の強制など、アーキテクチャ規約が機械化されているか
- [ ] 抑止コメント(`eslint-disable` 等)の追加がdiff検査で検出されるか

### テスト

- [ ] 失敗メッセージだけで原因が特定できるか(期待/実際の差分表示、仕様文としてのテスト名)
- [ ] 内側のループ(単体)が秒単位か。テストサイズの制約が機械検証されているか
- [ ] flakyの隔離が運用されているか(非決定的なテストをブロッキングゲートに使っていないか)
- [ ] テスト品質そのものの検証(ミューテーション・弱体化検知)があるか → 詳細は `ai-adaptable-testing` の `references/quality-gates-catalog.md`

### ランタイム検証(実行時)

- [ ] エージェントが自分で動かして確認できる手段(verify相当のスクリプト)が一発で通るか
- [ ] E2E・スモークの対象がクリティカルパスに絞られているか(全部E2Eは速度と決定性を失う)
- [ ] 本番検出(カナリア・SLO監視)に回すと決めたリスクの監視が実装されているか(決めただけで未実装は空白)
- [ ] fitness functions(性能予算・バンドルサイズ・循環依存数)の閾値と、閾値変更がHARDであることが明文化されているか

### 人間レビュー(HARD承認・サンプリング)

- [ ] HARD一覧が明文化され、機械判定できる条件はCIチェックとして実装されているか(文書だけのHARDは存在しないのと同じ)
- [ ] レビューアが「何を見るべきか」の観点がゲートごとに定義されているか(全文通読の再演になっていないか)
- [ ] センサー(LLM-as-judge)の出力がブロックではなくフラグとして人間レビューに接続されているか
- [ ] サンプリング率と昇降条件が数値で決まっているか(「たまに見る」は計測不能)

## 4. 委任判断・エスカレーション基準のテンプレート

### 委任判断シート(タスク種別ごとに1行)

```markdown
| タスク種別 | 検証可能性(高/中/低) | 影響半径(狭/中/広) | 頻度・規模 | 委任形態 | 前提整備 |
|---|---|---|---|---|---|
| 例: lint修正 | 高(lint+テスト) | 狭 | 高頻度 | 自律(AUTO) | — |
| 例: バグ修正 | 中(再現テスト前提) | 中 | 高頻度 | 監督付き(SOFT) | 再現テスト先置きを必須化 |
| 例: 新規API設計 | 低 | 広 | 低頻度 | 委任しない(人間主導+補助) | 受け入れ基準の明文化後に再評価 |
```

**「委任しない」には再評価条件を書く。** 検証可能性は固定値ではなく、テスト・受け入れ基準の整備で上がる — 「委任しない」の行は前提整備のバックログでもある。

### エスカレーション基準(SOFT → HARD 昇格条件)

機械判定できる条件はCIで強制し、判断を要する条件はレビューエージェントのルーブリックに載せる。**どちらにも実装されない条件は書かない**(存在しないのと同じ — SKILL.md ステップ2)。

```markdown
## 機械判定(CIチェックとして実装)
- 差分サイズ: 変更 ___ 行超 / ___ ファイル超
- パス種別: 次のパスに触れる差分(migrations/, .github/, hooks/, CLAUDE.md, lint設定, ___)
- テスト弱体化: skip/only追加、テスト削除、スナップショット一括更新、カバレッジ除外プラグマ
- 型の緩和: any相当・キャスト・非nullアサーションの追加、公開シグネチャ変更

## 判断を要する(レビューエージェントのルーブリック必須出力)
- 新規パターンの導入(既存コードベースに前例のない構造・ライブラリ用法)
- 既存規約からの逸脱(CLAUDE.md・lint化されていない暗黙規約との乖離)
- 出力形式: エスカレーション要否(要/不要)+根拠(該当条件の引用)を必須とする

## 運用パラメータ
- サンプリング初期値: ___%(目安30%)。差し戻し率が ___%(目安5%)を下回る状態が ___ 週続いたら ___% ずつ下げる
- 差し戻し率が悪化したら1段階戻す(下げるより上げる方を即断する)
- 新タスク種別・新パターン: 最初の ___ 件は全件確認
```

## 5. 計測指標の定義例

SKILL.md ステップ5の指標を、データソースと式まで固定した定義例。**前提はエージェント作業の機械識別**(コミットトレーラー・PRラベル)— 識別できなければどれも算出できない。

| 指標 | 定義(式) | データソース | 初期目標の目安 | レビュー周期 |
|---|---|---|---|---|
| 委任成功率 | 人間の手直しなしでマージされたエージェントPR ÷ エージェントPR総数(タスク種別ごと) | PRメタデータ+マージ後N日以内の同一箇所修正/revert追跡 | 種別ごとにベースライン取得から | 週次 |
| 介入率 | 人間が差分修正・指示追加で介入したセッション ÷ 全セッション | セッションログ/PRのforce-push・追加コミット | 低下傾向の維持 | 週次 |
| ゲート通過率 | ゲート初回通過PR ÷ 全PR(ゲート別: 型/テスト/lint/契約) | CIログ(ゲート別の失敗率・再実行率) | ゲート別に把握(低いゲートが改善投資先) | 週次 |
| 手戻り率(差し戻し率) | レビューで差し戻されたPR ÷ レビュー対象PR | PRメタデータ(レビュー往復回数) | 5%未満でサンプリング減の判断材料 | 週次 |
| エスカレーション率 | SOFT→HARD昇格 ÷ SOFT総数 | CIチェック結果+レビューエージェント出力 | 過少(検知漏れ)と過多(基準が緩い)の両方を疑う | 隔週 |
| レビュー負荷 | 人間レビュー時間の合計/中央値 | レビュー所要時間(PRメタデータ) | ハーネス投資前より減っているか | 月次 |
| 品質負債 | 静的解析指標(重複・複雑度・違反数)の推移、AI起因コードの残存問題率 | 静的解析の定点観測 | 悪化していないこと(改善は副次目標) | 月次 |

**目標値は最初に固定しない。** 最初の2〜4週はベースライン取得に充て、目標は「傾向」(改善方向の維持)で持つ。絶対値目標は指標のゲーミング(簡単なタスクだけ委任して成功率を上げる等)を招く — 必ずタスク種別で層別する。

**失敗事例の分類は指標より先に整備する。** 仕様の曖昧さ/検証の欠落/コンテキスト不足/モデル能力の限界 の4分類(SKILL.md ステップ5)を差し戻し・失敗のたびに記録する。分類の分布が「次にハーネスのどこへ投資するか」を直接教える — 前3分類が多いうちは指標の絶対値を追うより穴を塞ぐ方が速い。
