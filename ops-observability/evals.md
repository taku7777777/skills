# ops-observability 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 深夜のオンコールで呼ばれても結局「様子見」で終わることが多い。呼び出しを本当に必要なものだけにしたい | アラート棚卸し・アラート設計(「アラートを整理して」と語彙が異なる) |
| F2 | 先月の障害、ユーザーからの問い合わせで初めて気づいた。次は自分たちが先に気づける状態にしたい | 検知の仕組み=SLI/SLO・アラート設計(監視という語を含まない) |
| F3 | 新しく切り出したサービスを本番に出す前に、ログやメトリクスに何を仕込んでおくべきか決めておきたい | 計装設計(運用への引き渡し・透明性) |
| F4 | 障害のたびに誰が指揮を執るのか曖昧で、Slackが大混乱になる。当番と動き方をちゃんと決めたい | インシデント対応プロセス構築(役割分担・コミュニケーション規約) |
| F5 | 昨日のDB全断の振り返り会をやることになったので、資料の叩き台を作ってほしい。犯人探しにはしたくない | ポストモーテム作成(blameless) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 本番のAPIが今朝から断続的に5xxを返している。どこが悪いのか突き止めてほしい | `systematic-debugging` が発火する(障害の原因調査であり、監視の整備ではない) |
| N2 | デプロイのたびに障害が起きるので、一部トラフィックだけに新版を当てて段階的に広げる仕組みを入れたい | `cicd-release` が発火する(デプロイ時の安全性の仕組み) |
| N3 | 外部決済サービスの障害に引きずられて全体が落ちないよう、サーキットブレーカーとリトライを設計して | `arch-security-design` が発火する(安定性パターンはアーキテクチャ設計の領分) |
| N4 | このダッシュボード用のグラフをきれいに整形して | スキルなしで応答する(可視化作業であり運用設計ではない) |

## 検証シナリオ

### S1: 動画学習サービスの監視・運用設計
- **タスク**: 「架空のオンライン動画学習サービス(受講者が講座を検索し、動画を再生し、進捗を保存する)の運用設計をして。体制は開発者5名の輪番オンコール、専任SREなし。現状のアラートは『CPU使用率80%超』『メモリ90%超』『5xxが100件/分超』の3本で、月20回ほど鳴るがほぼ対応不要。一方、先々月の動画配信障害(再生開始失敗が3時間継続)はSNSの苦情で発覚した。手元にはLBアクセスログ(直近28日の再生開始成功率の実測は99.7%)とホストメトリクスがある」
- **スキルなしで予想される欠落**: CUJ起点でなくシステム指標の羅列から始める。SLO目標を実測(99.7%)でなく理想(99.99%等)から置く。バーンレートアラート(マルチウィンドウ)でなく生のエラー率閾値のまま。エラーバジェットポリシー(枯渇時の行動合意)の欠落。アラートとランブックの1:1対応の欠落。
- **合格条件**:
  - [ ] クリティカルユーザージャーニーが明示され、SLIが「良いイベント数÷有効イベント数」の比率と測定点つきで定義されている
  - [ ] SLO目標値が実測値(99.7%)を根拠に設定され、エラーバジェットポリシー(残量に応じた行動)が含まれる
  - [ ] ページするアラートにSLOバーンレートベース(マルチウィンドウ・マルチバーンレート)の設定が含まれる
  - [ ] 既存のCPU/メモリアラートをページから外す(ダッシュボード・チケット降格または削除)判断が明記されている
  - [ ] ページするアラートすべてにランブックへの対応づけがある

### S2: 鳴りすぎるアラートの棚卸し
- **タスク**: 「社内向け経費精算システムのアラートを見直したい。直近90日の発報履歴: ①ディスク使用率85%超 → 42回発報、対応したのは2回(ログローテ設定変更) ②ジョブキュー滞留1000件超 → 18回発報、うち15回は月次締め日の正常なピーク ③レスポンスタイム悪化(p95>2秒が5分継続)→ 6回発報、うち4回で実際にユーザー影響あり ④夜間バッチ失敗 → 9回発報、全件で翌朝の再実行が必要だった。どう整理すべきか」
- **スキルなしで予想される欠落**: 「閾値を調整しましょう」という個別最適で終わる。「行動につながったか」による分類と、ページ/チケット/ダッシュボード(または削除)への3分法の欠落。「緊急かつ行動可能なものだけページする」という原則の欠落。symptom(ユーザー影響)ベースへの組み替えの欠落。
- **合格条件**:
  - [ ] 4アラートそれぞれについて「行動につながったか」の観点での分類・判定がある
  - [ ] 処遇が「ページ継続/チケット化/ダッシュボード化(または削除)」の区別で提示されている(全部を閾値調整で残さない)
  - [ ] 夜間バッチ失敗(緊急ではないが行動必要)がページでなくチケット系へ振り分けられている
  - [ ] ページに残すアラートに severity とランブック(または症状ベースの文言)への言及がある
  - [ ] 月次締め日の正常ピーク(②)への対処が単なる閾値引き上げ以外の観点(既知の周期・行動不能通知の扱い)で論じられている(人間判定)

### S3: 既存ランブックのエージェント対応レビュー(途中介入)
- **タスク**: 「うちのランブックを来月からAIエージェントの一次対応に使いたい。今のままで大丈夫か見てほしい」と依頼し、次の既存ランブックを渡す —「## API応答悪化時の対応: まずログを見て状況を把握する。おかしそうならアプリケーションを再起動する。それでもダメなら直近のデプロイを戻す。判断に迷ったら詳しい人に聞くこと」
- **スキルなしで予想される欠落**: 文言の曖昧さ指摘(「具体的に書きましょう」)で終わる。読み取り専用/変更系の区別と、変更系操作への人間承認ゲートの欠落。実行可能なコマンド・判断分岐の条件化・期待出力の明記という「エージェント実行可能形式」の要件の欠落。
- **合格条件**:
  - [ ] 再起動・ロールバック等の変更系操作に人間の承認を挟む(エージェントは読み取り専用から始める)方針が明記されている
  - [ ] 各手順に「読み取り専用/変更系」の区別を付ける改善が提案されている
  - [ ] 「おかしそうなら」等の判断を、条件(メトリクス値・ログの内容)で分岐として明文化する改善が提案されている
  - [ ] 実行可能なコマンドと期待出力を記載する形式への書き換え例が含まれる
  - [ ] 改善後のランブック案が元の手順の意図を保っている(人間判定)
