# SLO・アラート・ランブック・ポストモーテムのテンプレート集

SKILL.md の各ステップから参照される実務テンプレート。値はすべて出発点の目安であり、サービスの実測とユーザー期待で調整する。

## 目次

- SLI/SLOワークシート
- バーンレートアラート
- severity定義
- ランブック
- ポストモーテム

## 1. SLI/SLO定義ワークシート(ステップ1)

CUJ 1つにつき1シートを埋める。**埋められない欄が設計の宿題**(特に測定点と有効イベントの定義)。

```markdown
### CUJ: <ユーザーが達成したいこと(動詞で)>
- ビジネス影響: <損なわれたとき何が起きるか(売上、解約、信用)>
- 対象システム範囲: <関与するサービス・依存>

#### SLI(CUJごとに1〜3個)
| 項目 | 内容 |
|---|---|
| 種別 | 可用性 / レイテンシ / 品質 / 鮮度 |
| 良いイベントの定義 | 例: ステータス5xx以外 かつ 応答 < 500ms のリクエスト |
| 有効イベントの定義 | 例: 対象エンドポイントへの全リクエスト(ヘルスチェック・bot除外) |
| 測定点 | LBログ / サーバー側メトリクス / クライアント計測 / 合成監視 |
| 測定点の盲点 | 例: LBログではDNS障害・クライアント側の失敗が見えない |

#### SLO
| 項目 | 内容 |
|---|---|
| 目標値 | 例: 99.9%(現状実測 99.93% を根拠に設定) |
| ウィンドウ | 例: 28日ローリング |
| エラーバジェット | (1 − 目標値) × 有効イベント数。例: 99.9% なら 0.1% |
```

**SLO設定の注意**:
- 目標値は理想値ではなく**直近の実測値から出発**する。実測より高い目標は設定した瞬間に違反する
- ウィンドウは28日(4週間の倍数)にすると曜日効果が均される
- 依存サービスのSLOより高い可用性は、冗長化なしには約束できない(直列依存の可用性は掛け算で下がる)

### エラーバジェットポリシー(雛形)

| バジェット残量 | 行動 |
|---|---|
| 50%以上 | 通常運転。機能開発優先 |
| 50%未満 | 信頼性に関わるリスクの高いリリースを延期。消費要因の調査を開始 |
| 枯渇(0%) | 機能リリース凍結。信頼性作業を最優先。解除条件(バジェット回復水準)を明記 |
| 大量瞬間消費(単一障害で25%超) | ポストモーテム必須 + SLO/アーキテクチャの見直し検討 |

ポリシーは経営・プロダクト・開発の三者で**平時に**合意する。障害後に初めて交渉するポリシーは機能しない。

## 2. バーンレートアラート設定目安表(ステップ3)

バーンレート = 実際のエラー率 ÷ (1 − SLO)。バーンレート1は「ウィンドウ全期間かけてバジェットをちょうど使い切る」速度。

**30日ウィンドウのSLOに対する標準構成**(SRE Workbook 方式):

| バジェット消費 | 長ウィンドウ | 短ウィンドウ | バーンレート閾値 | 通知 |
|---|---|---|---|---|
| 2% を1時間で | 1時間 | 5分 | 14.4 | ページ(即時対応) |
| 5% を6時間で | 6時間 | 30分 | 6 | ページ(即時対応) |
| 10% を3日で | 3日 | 6時間 | 1 | チケット(数日内対応) |

- **マルチウィンドウ**: 長短**両方**のウィンドウが閾値を超えたときのみ発報する。長ウィンドウだけだと復旧後も鳴り続け、短ウィンドウだけだと瞬間ノイズで鳴る。短ウィンドウは長ウィンドウの 1/12 が目安
- 検証方法: 発報条件を過去数ヶ月のデータに適用し、「実際のインシデントを全部捕まえたか / 行動不要の発報が何件出たか」を確認してから本番化する
- トラフィックが少ないサービスでは短ウィンドウ内の母数が足りず不安定になる — ウィンドウを伸ばすか、複数サービスの合成SLIを検討する

## 3. severity定義表(ステップ3・5)

「誰を・いつ・どう動かすか」を機械的に決めるための表。組織に合わせて行動列を書き換えて使う。

| severity | 定義(ユーザー影響) | 対応 | 体制 | 連絡 |
|---|---|---|---|---|
| SEV-1 | CUJの全面停止、データ損失・漏洩の恐れ | 即時ページ、全員招集可 | IC + スクライブ + コミュ担当を必ず設置 | 社内全体 + 顧客向けステータス更新 |
| SEV-2 | CUJの重大な劣化、多数ユーザーに影響 | 即時ページ | IC設置 | 社内関係部署 + 必要に応じ顧客向け |
| SEV-3 | 部分的な劣化、回避策あり | 営業時間内に対応 | 担当チームで対応 | チーム内 + サポート部門へ共有 |
| SEV-4 | 軽微。ユーザー影響ほぼなし | チケット化し計画対応 | 通常の作業フロー | 不要 |

- 判断に迷ったら**高い方に倒して宣言**し、後で下げる(下げる操作は安い。上げ遅れは高い)
- severityは「現在の影響」で決める。「すぐ直りそう」という見込みで下げない
- SEV-1/2 はポストモーテム必須(ステップ6の実施基準と連動させる)

## 4. ランブックテンプレート(ステップ4)

### エージェント実行可能なランブックの要件

1. **コマンドはコピペ実行可能な形で書く**。プレースホルダは `<POD_NAME>` のように明示し、取得方法(コマンド)を併記する
2. **判断分岐を条件で書く**: 「様子を見る」ではなく「直近15分のp99が1秒超なら手順3へ、そうでなければ手順5へ」
3. **各手順に期待出力を書く**: 何が返れば正常/異常かを判定できる形で
4. **各手順に副作用の有無を明記**: `[読み取り専用]` / `[変更系・承認必要]` のタグをつける。エージェントには読み取り専用手順のみ自律実行を許可し、変更系は人間の承認を待つ
5. **前提(必要な権限・環境・ツール)を先頭にまとめる**: 手順の途中で権限がないと判明するのが最悪
6. **エスカレーション先と「これ以上進まない条件」を書く**: 15分ルール(15分で進展がなければエスカレーション)等

### テンプレート

```markdown
# ランブック: <アラート名(発報するアラートと1:1対応)>

- 症状: <ユーザーに何が起きているか>
- 影響するSLO/CUJ: <リンク>
- ダッシュボード: <リンク>
- 前提: <必要な権限・ツール・接続先>

## 診断 [読み取り専用]
1. <コマンド>
   - 期待出力: <正常/異常の判定基準>
   - 異常なら → 手順2へ / 正常なら → 手順3へ
2. ...

## 緩和 [変更系・人間の承認必要]
3. <コマンド(ロールバック、フラグ切替、再起動等)>
   - 実行前確認: <blast radiusと戻し方>
   - 期待される回復の確認方法: <SLIの回復をどこで見るか>

## エスカレーション
- 15分で進展がない場合: <誰に・どうやって>
- 過去の関連ポストモーテム: <リンク>
```

## 5. ポストモーテムテンプレート(ステップ6)

```markdown
# ポストモーテム: <障害の短い記述>
- 日付 / 作成者 / レビュー者 / ステータス(草稿・レビュー済・アクション完了)

## サマリ(3行以内)
## 影響
- 期間(開始〜検知〜緩和〜完全復旧の各時刻)
- ユーザー影響(影響ユーザー数・失敗リクエスト数・ビジネス影響)
- エラーバジェット消費(SLOごとの消費率)
## 根本原因と誘因(contributing factors)
<単一の「犯人」ではなく、重なった要因を列挙。個人名は役割名に置換(blameless)>
## 検知
<どうやって気づいたか(アラート/ユーザー問い合わせ/偶然)。検知までの時間。
 ユーザー問い合わせで発覚した場合は「なぜ監視で捕まらなかったか」を必ず書く>
## タイムライン
<時刻つきの事実の列挙。判断とその時点で見えていた情報も書く>
## うまくいったこと / 運が良かったこと / うまくいかなかったこと
## アクションアイテム
| # | 内容 | 種別(予防/検知/緩和/プロセス) | 担当 | 期限 | チケット |
## 教訓
```

**運用の注意**:
- 「うまくいったこと」「運が良かったこと」は省略しない — 運頼みだった箇所こそ次の障害の芽
- アクションアイテムは「気をつける」「周知する」を禁止語にする。仕組み(自動化・ガードレール・アラート・テスト)に落ちない項目は再発防止にならない
- 完成したポストモーテムは組織に公開し、定期レビューで横断パターンを拾う
