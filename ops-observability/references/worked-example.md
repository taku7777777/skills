# ops-observability 適用例

## 題材

架空の中規模ECサイト「(仮)マーケット」のチェックアウトサービス。モノリスから切り出されたAPIサービスで、決済ゲートウェイ(外部)と在庫サービス(内部)に依存する。運用の現状: CPU使用率とディスク使用率のアラートが週数回深夜に鳴るが対応はほぼ「様子見」。一方、先月の決済障害はユーザー問い合わせで発覚し、検知まで40分かかった。

## 入力

- 依頼: 「アラートがうるさいのに障害は拾えていない。監視を作り直したい」
- 手元にあるもの: LBアクセスログ、ホストメトリクス、断片的なアプリログ(非構造化)。トレースなし
- 体制: 開発者6名がオンコール輪番。専任SREなし
- 制約: 監視基盤の刷新予算はない(既存スタックの上で設計する)。決済ゲートウェイのSLAは99.95%

## 各ステップでの判断

**ステップ1(CUJとSLI/SLO)**: CUJを「商品をカートに入れて決済を完了できる」に絞る(閲覧系は別サービスの領分)。SLIは2つ — 可用性(チェックアウトAPIへのリクエストのうち5xx以外の割合)とレイテンシ(決済確定APIの応答2秒未満の割合)。測定点はLBログ(クライアント計測は未整備のため盲点として記録)。LBログ4週間の実測が99.94%だったため、SLOは99.9%/28日から開始 — 直列依存する決済ゲートウェイのSLA(99.95%)より高い目標は冗長化なしに約束できないことも99.9%の根拠にした。CPU使用率はSLIに採用しない — ユーザー体験を直接測らないため(大原則1)。エラーバジェットポリシーはプロダクトマネージャーを交えて平時のこの時点で合意した。

**ステップ2(計装)**: 最大の欠落は「決済のどの段階で失敗したか」を事後に問えないこと(unknown unknowns に答えられない)。OpenTelemetry の自動計装でHTTP境界のトレースを導入し、チェックアウト処理はリクエスト1件につき1つのワイドイベント(注文ID、決済手段、ゲートウェイ応答コード、在庫確認の所要時間、フィーチャーフラグ)を構造化ログに出す。注文IDやユーザーIDはメトリクスラベルに入れず(カーディナリティ爆発)、イベント側に置く。

**ステップ3(アラート)**: ページするアラートをSLOバーンレートベースの2本(14.4×[1h/5m]、6×[6h/30m])+ ゲートウェイ死活の計3本に削減。バーンレート1×[3d/6h]はチケット化。CPU・ディスクアラートはページから外しダッシュボードとチケット閾値に降格。過去3ヶ月のログで再生テストし、先月の決済障害なら6分で発報していたことを確認してから切替。

**ステップ4(ランブック)**: ページする3アラートに1:1でランブックを作成。診断手順(LBログ集計、ワイドイベントのゲートウェイ応答コード別集計、直近デプロイ確認)はすべて `[読み取り専用]` のコマンドつき。緩和手順(直近デプロイのロールバック、決済ゲートウェイのフェイルオーバーフラグ)は `[変更系・承認必要]`。オンコールのエージェント利用は読み取り専用の診断までを許可。

**ステップ5(インシデント対応)**: severity表を導入し、「決済成功率の急落=SEV-2以上、疑わしければ宣言」と明文化。6名体制なのでICとスクライブは兼任可、ただし**ICは手を動かさない**ことだけ死守。連絡は専用チャンネル1本 + 30分ごと更新と決めた。

**ステップ6(ポストモーテム)**: 実施基準を「SEV-2以上 / バジェット25%超の瞬間消費 / ユーザー問い合わせで発覚した全障害」と定義。先月の決済障害を新テンプレートで遡及作成し、「検知40分」を検知系アクションアイテム(ステップ3のバーンレートアラート)に接続して改善ループの起点とした。

## 成果物(抜粋)

```markdown
# チェックアウトサービス 運用設計書
## 1. CUJとSLI/SLO
- CUJ: カート投入〜決済完了(ビジネス影響: 直接売上)
- SLI-1 可用性: 5xx以外 ÷ 全有効リクエスト(LBログ、bot除外)。SLO 99.9%/28日
- SLI-2 レイテンシ: 決済確定APIの2秒未満応答 ÷ 全有効リクエスト。SLO 99%/28日
- エラーバジェットポリシー: 残50%未満でリスクの高いリリース延期、枯渇で機能リリース凍結
- 既知の盲点: クライアント側障害はLBログに現れない(次四半期に合成監視を追加)
## 2. 計装設計
- OpenTelemetry自動計装(HTTP境界)+ チェックアウト1リクエスト=1ワイドイベント
- 高カーディナリティ属性(注文ID等)はイベントへ、メトリクスラベルは低カーディナリティ維持
## 3. アラート一覧
| アラート | 条件 | severity | ランブック |
|---|---|---|---|
| バジェット高速消費 | BR>14.4 [1h/5m] 両方超過 | SEV-2 | RB-01 |
| バジェット中速消費 | BR>6 [6h/30m] 両方超過 | SEV-3 | RB-02 |
| 決済GW死活 | 合成監視3回連続失敗 | SEV-2 | RB-03 |
| バジェット低速消費 | BR>1 [3d/6h] | チケット | — |
## 4. ランブック
- RB-01〜03(診断=読み取り専用コマンド+期待出力、緩和=変更系・承認必要)
- エージェントの一次診断は読み取り専用手順のみ許可。15分進展なしでエスカレーション
## 5. インシデント対応プロセス
- 宣言基準: 決済成功率の急落はSEV-2以上。疑わしければ宣言(誤宣言を責めない)
- 役割: IC・スクライブ兼任可。ICは手を動かさない。連絡は専用チャンネル1本・30分ごと更新
## 6. ポストモーテム運用
- 実施基準: SEV-2以上 / バジェット25%超の瞬間消費 / 問い合わせ発覚の全障害
- 先月の決済障害を遡及作成済み。四半期の横断レビュー枠を設定
```

## この適用で変わったこと

- ページするアラート: 週数回の「様子見」→ 3本(すべて行動可能・ランブックつき)
- 検知: ユーザー問い合わせ(40分)→ バーンレートアラート(再生テストで6分)
- 「なぜ決済が失敗したか」にワイドイベントで事後から答えられるようになった(未知の未知への質問能力)
