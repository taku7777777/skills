### 指摘事項

#### 🔴 Blocker — マージ前に修正必須

1. `src/coupons/CouponService.ts:14` — `userId`が使われず、`coupon.ownerId`との所有権検証がありません。

   **失敗シナリオ**: ユーザーBがユーザーAの未使用クーポンIDを指定すると、そのまま適用・使用済み保存され、Bが割引を取得してAのクーポンが失われます。

   **修正案**: `coupon.ownerId === userId`を必須にし、できればRepositoryの原子的な引換操作でも所有者を条件に含めてください。

2. `src/coupons/CouponService.ts:19` — 未使用確認から保存までがcheck-then-actになっており、二重使用を防げません。

   **失敗シナリオ**: 同じクーポンへの2リクエストが並行して未使用状態を読み取ると、両方が検査を通過し、両方に割引成功を返します。

   **修正案**: `used_at IS NULL`、所有者、有効期限を条件とする条件付き更新をトランザクション内で実行し、更新件数が1件の場合だけ成功させてください。

3. `src/http/CouponController.ts:11` — 商品価格の信頼元がリクエスト本文で、商品・注文との対応をサーバー側で検証できません。

   **失敗シナリオ**: 実際には1,000円の商品でも、呼び出し側が`productPrice: 500`を送ると、APIはその偽の価格を基準に割引結果を生成してクーポンを消費します。

   **修正案**: 商品IDまたは注文IDを受け取り、価格をサーバー側の信頼できる商品・注文データから取得してください。

4. `src/http/CouponController.ts:11`, `src/coupons/CouponService.ts:30` — 金額の有限性、範囲、精度、割引上限を検証せず、無効な計算でもクーポンを消費します。

   **失敗シナリオ**: `{}`では`Number(undefined)`が`NaN`となり、レスポンスの金額はJSON上`null`になりますが、`usedAt`は保存されます。`productPrice: null`は0に変換され、300円クーポンなら`finalPrice: -300`になります。価格100・割引300でも`-200`です。

   **修正案**: 使用済み更新より前に、信頼済み価格とクーポン額について有限値・正値・通貨精度を検証し、割引が価格を超える場合の上限または拒否ルールを明示してください。

#### 🟠 Major — 修正を強く推奨

1. `src/coupons/CouponService.ts:30` — 浮動小数点による丸めが十進金額を正確に処理しません。

   **失敗シナリオ**: `productPrice = 1.015`、`coupon.amount = 0.01`では数学上の差は`1.005`です。小数第2位へのhalf-upなら期待値は`1.01`ですが、実値は`1.00`になります。

   **修正案**: 最小通貨単位の整数、またはDecimal型を使用し、丸め規則を明文化してください。

2. `src/coupons/CouponService.test.ts:3` — テストが使用済み更新と保存を検証していません。

   **失敗シナリオ**: `coupon.usedAt = now`と`save`を削除しても現在のテストは成功し、クーポンを再利用できる退行を検出できません。

   **修正案**: 保存された値を捕捉して`usedAt`を検証し、他人のクーポン、並行引換、無効価格、割引超過、使用済み、有効期限切れを追加してください。二重使用は実Repositoryを使った統合テストも必要です。

#### 💬 質問

1. `src/coupons/CouponService.ts:26` — `expiresAt === now`は有効として扱われます。`expiresAt`が無効化される瞬間を表す契約なら、比較は`<=`である必要があります。

2. `src/http/CouponController.ts:7` — async例外の伝播と、not-found・used・expiredのHTTPステータス変換は添付diffから確認できません。グローバルハンドラーがなければ、業務エラーが500または未処理のPromise rejectionになります。

### 確認したが問題なしと判断した点

- 正常系の価格1,000・割引300では、期待どおり700を返します。
- `now`を1回だけ注入しており、時刻比較とテストを決定的にできます。
- `save`をawaitしているため、保存失敗後に成功レスポンスを返す順序にはなっていません。
