## レビュー結果: クーポン適用・使用済み更新機能

**判定**: escalate — **現状はマージ不可**。認可・金額計算・単一使用保証に Blocker があり、認可・課金・公開 API に触れるため、修正後も人間レビューが必要です。

### 総評

Controller・Service・Repository の責務分離と、時刻を注入可能にした設計はテストしやすく良い判断です。一方、`userId` が認可に使われず、クーポン使用処理にも原子性がないため、他人のクーポン利用と二重使用が可能です。また、クライアント申告価格を無検証で使用しており、金額計算の信頼境界が成立していません。

### 指摘事項

#### 🔴 Blocker — マージ前に修正必須

1. `src/coupons/CouponService.ts:19` — クーポン所有者の認可がなく、`userId` が一度も使用されていません。

   **失敗シナリオ**: `user-2` が `user-1` 所有の `coupon-1` を指定すると、所有者不一致でも割引が成功し、さらにそのクーポンが使用済みになります。攻撃者が割引を得るだけでなく、正規所有者も利用不能になります。

   **修正案**: `coupon.ownerId === userId` を状態変更前に必ず検証してください。より安全には、Repository 側で所有者を条件に含めた原子的な利用処理にします。所有者不一致時にクーポンの存在を漏らさないエラー設計も必要です。

2. `src/coupons/CouponService.ts:19-32` / `src/coupons/CouponRepository.ts:10-11` — 未使用確認と使用済み更新が分離され、単一使用が保証されません。

   **失敗シナリオ**: 同じクーポンへの2リクエストが同時に `findById` を実行すると、両方が `usedAt` 未設定の状態を読み、両方が割引成功として返せます。後続の `save` はこの競合を検出する契約になっていません。

   **修正案**: `UPDATE ... WHERE used_at IS NULL AND owner_id = ... AND expires_at > ...` のような条件付き更新、行ロック、または楽観ロックで「検証と使用済み化」を1つの原子的操作にしてください。更新件数が0なら競合として失敗させます。

3. `src/http/CouponController.ts:9-13` / `src/coupons/CouponService.ts:30` — クライアント入力の価格を無検証で金額計算に使用しています。

   **失敗シナリオ**:

   - 実価格1,000の商品について、呼び出し側が `productPrice: 300` と300円クーポンを送ると、APIは `finalPrice: 0` を返します。サーバーは実価格を照合できません。
   - `productPrice: "abc"` は `NaN` になり、それでもクーポンは使用済みとして保存されます。JSONレスポンスでは非有限値が `null` となり得ます。
   - 価格100、割引300では `finalPrice: -200` となります。

   **修正案**: 商品または注文IDを受け取り、価格はサーバー側の正規データから取得してください。金額は有限・非負・許可された精度で検証し、整数の最小通貨単位またはDecimal型で扱います。割引超過時は0に丸めるか拒否するか、業務ルールを明示してください。

4. `src/coupons/CouponService.ts:31-32` — 使用処理に冪等性がなく、レスポンス喪失でクーポンが失われます。

   **失敗シナリオ**: `save` 完了後、レスポンス到着前に通信が切れると、クライアントは成功を確認できません。再送すると `"coupon already used"` になり、割引結果を取得できないままクーポンだけが消費されます。

   **修正案**: 注文IDまたは冪等キーと利用結果を関連付け、同じ操作の再送には同じ結果を返してください。実際の購入を伴うなら、注文確定とクーポン消費を同じトランザクション境界に置きます。

#### 🟠 Major — 修正を強く推奨

1. `src/coupons/CouponService.ts:26` — `expiresAt === now` のクーポンが利用可能です。

   **失敗シナリオ**: `expiresAt` と `now` がともに `2030-01-01T00:00:00Z` の場合、失効時刻に達しているのに検査を通過します。

   **修正案**: `expiresAt` を失効開始時刻とするなら `coupon.expiresAt <= now` 相当で判定し、等値境界をテストしてください。異なる業務定義なら契約として明記します。

2. `src/coupons/CouponService.test.ts:3-25` — テストが正常系1件だけで、重要な状態変更も検証していません。

   **失敗シナリオ**: 所有者チェックがない現在の実装でもテストは成功します。また、`coupon.usedAt = now` や `save` を削除してクーポンが無制限に再利用可能になっても、このテストは成功します。

   **修正案**: 所有者不一致、使用済み、期限切れ・期限等値、不正価格、割引超過、保存呼び出しと`usedAt`を検証してください。二重利用については、実際のRepository実装を使った並行テストが必要です。

### 確認したが問題なしと判断した点

- `now` を引数で固定できるため、時刻依存テストを決定的に実行できます。
- ControllerがHTTP入力、Serviceが業務判定、Repositoryが永続化契約を担当する基本的な責務分離は妥当です。
- 添付差分には、新規外部依存、機密情報、危険な文字列評価や正規表現はありません。
