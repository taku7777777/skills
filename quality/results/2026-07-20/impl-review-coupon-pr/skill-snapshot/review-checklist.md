# 正当性レビュー詳細チェックリスト

差分の性質に応じて該当セクションを適用する。全項目を機械的に舐めるのではなく、「この変更でこの欠陥クラスが起きうるか」を問う。

## 入力・境界値

- [ ] null / undefined / 空文字 / 空配列 / 空マップ
- [ ] 0, -1, 最小値, 最大値, オーバーフロー
- [ ] コレクションの 0件 / 1件 / 大量件数(ページネーション欠落、N+1、メモリ)
- [ ] 文字列: 絵文字・サロゲートペア・全角/半角・前後空白・改行混入・巨大入力
- [ ] 重複要素、順序が保証されない入力
- [ ] 型強制の罠(`"0"` は truthy、`parseInt("12abc")` 等、言語固有の落とし穴)

## エラーハンドリング

- [ ] 例外発生時のリソース解放(ファイル・接続・ロック)— finally / defer / using の欠落
- [ ] 部分的失敗: ループ内で3件目が失敗したら1-2件目はどうなる? 全体の状態は?
- [ ] 例外の握りつぶし(catchして無視/ログのみで処理続行が正当か)
- [ ] リトライ: 対象操作は冪等か。リトライ回数・バックオフ・リトライ嵐
- [ ] タイムアウトの設定漏れ(外部API・DB・ロック取得)
- [ ] エラーメッセージに内部情報(スタックトレース、SQL、パス)が漏れていないか

## 並行性・状態

- [ ] 共有可変状態への非同期アクセス(キャッシュ、シングルトン、staticフィールド)
- [ ] check-then-act の競合(存在確認→作成、残高確認→引き落とし)
- [ ] トランザクション境界: 一緒にコミットされるべき更新が分かれていないか。逆に長すぎるトランザクションでロック保持していないか
- [ ] 二重送信・二重実行(同じリクエストが2回来たら?)— 冪等キー、一意制約
- [ ] 楽観ロック/悲観ロックの選択と、競合時のユーザー体験
- [ ] 非同期処理の順序保証・at-least-once配信での重複処理

## 時間

- [ ] タイムゾーン: 保存はUTC? 表示はユーザーTZ? 「日付」の境界はどのTZ基準?
- [ ] 月末処理(1/31 の1ヶ月後は?)、うるう年、年度またぎ
- [ ] 処理実行中の日付変化(23:59に開始したバッチ)
- [ ] `now()` の複数回呼び出しによる不整合(処理の頭で1回取得して引き回すべき)

## データ・永続化

- [ ] マイグレーションの後方互換(旧コードが新スキーマで動く期間があるか、ロールバック可能か)
- [ ] 大テーブルへのインデックスなしクエリ、ロックを伴うALTER
- [ ] 論理削除データの考慮漏れ(一意制約、集計、JOIN)
- [ ] 文字数制限・精度(金額のfloat使用は原則Blocker)
- [ ] キャッシュ無効化のタイミングと整合性

## 契約・互換性

- [ ] 公開APIのレスポンス形式変更・フィールド削除・enum値追加(クライアントは未知の値を処理できるか)
- [ ] イベント/メッセージスキーマの変更とコンシューマ互換性
- [ ] 設定・フィーチャーフラグ: フラグOFF時の経路は引き続き動くか。フラグの消し忘れ
- [ ] 依存ライブラリ更新の破壊的変更
- [ ] 新規依存の導入: パッケージは実在するか(typo/捏造でないか)、メンテナンス状況、ライセンス、既存依存で代替できないか

## セキュリティ(実装レベル)

- [ ] SQL/NoSQL: 文字列結合クエリ(プレースホルダ必須)
- [ ] XSS: ユーザー入力のHTML出力時エスケープ、`dangerouslySetInnerHTML` 等
- [ ] パス操作: ユーザー入力からのファイルパス構築(`../` 対策)
- [ ] コマンド実行: ユーザー入力のシェル文字列への混入
- [ ] SSRF: ユーザー指定URLへのサーバーサイドfetch
- [ ] 認可: エンドポイント/クエリレベルで「誰のデータか」を検証しているか(IDOR)。ログインチェックだけでは不十分
- [ ] CSRF: 状態変更エンドポイントの対策(トークン、SameSite)
- [ ] オープンリダイレクト: ユーザー入力によるリダイレクト先の検証
- [ ] レートリミット: 認証・高コスト操作に制限があるか
- [ ] 機密情報: ログ・レスポンス・リポジトリへの混入(トークン、パスワード、個人情報)
- [ ] 乱数: セキュリティ用途に非暗号論的乱数を使っていないか
- [ ] 比較: トークン比較のタイミング攻撃(定数時間比較)
- [ ] LLM呼び出し: 信頼できない入力(ユーザー入力・外部取得コンテンツ)をプロンプトへ連結する箇所の分離・制約(プロンプトインジェクション)。LLM出力を検証なしに特権的シンク(SQL・シェル・HTML描画・URL・ファイルパス・ツール実行)へ渡していないか(insecure output handling — OWASP Top 10 for LLM Applications)

## パフォーマンス(計測可能な懸念のみ指摘)

- [ ] ループ内のDB/APIコール(N+1)
- [ ] 全件ロードしてアプリ側でフィルタ/ソート
- [ ] ホットパスでの同期I/O・重い計算
- [ ] 無制限の並列実行(セマフォなしの `Promise.all(全件)`)
- ※「速くなりそう」レベルのマイクロ最適化提案はしない。データ量の見積もりとセットで指摘する
