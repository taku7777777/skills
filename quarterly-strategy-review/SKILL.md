# 四半期戦略レビュースキル

Philip Tetlock『Superforecasting』(ベイズ更新・カリブレーション)、Annie Duke『Thinking in Bets』、レッドチーム思考(反証から始める)の知見に基づく、個人戦略文書の定期更新プロセス。

対象文書: 起動時の指示またはプロジェクトの CLAUDE.md 等で対象文書・対象ディレクトリの指定があればそれを使う。指定がない場合や、複数候補があってどれが対象か判断がつかない場合はユーザーに確認する。

## 大原則

1. **「どの前提が崩れたか」から読み始める。** 自己分析文書は全肯定に向かう構造的バイアス(motivated reasoning)を持つ。レビューの第一の仕事は自説の補強ではなく反証の探索。
2. **エビデンスの半減期は数ヶ月。** モデル世代・実証研究・市場データは急速に陳腐化する。前回「確度・高」だった知見も再検証対象。
3. **予測は捨てずに採点する。** 過去の予測と実際の結果を突き合わせ、外れ方のパターン(楽観/悲観、時期の前後)を自分のカリブレーション情報として蓄積する。
4. **本文は最新の判断、履歴は変化の軌跡。** 現在値(確率・確度・定量値)はin-place更新し、何がどう変わったかはレビュー履歴に追記で残す。変化の軌跡自体が資産。

## ワークフロー

### ステップ1: 前回からの差分確認

- 対象文書を読み、前回レビュー日・ウォッチリスト・反証条件・予測台帳の未判定予測(特に期限到来分)・意思決定論点(未決のもの)を抽出する
- ユーザーに確認: 前回からの自身の活動変化(新しい取り組み、登壇、組織の変化、意思決定論点への仮決め)

### ステップ2: ウォッチリストの現状調査

文書内のウォッチリスト各項目について、WebSearch/WebFetchで最新状況を調査する:

- 各シグナルの定量値の更新(例: METRホライズン、capexガイダンス、労働市場指数、国内導入事例数)
- **トリガー条件に達したものを判定**し、対応アクション(文書に記載済み)を発動するか評価
- 一次ソース優先。重要な数値は独立した2ソース以上で裏取り。ベンダー予測にはバイアスラベルを付ける
- 更新した定量値には**出典URLとアクセス日**を残す(次回レビューで「前回の根拠」を再検証可能にする)
- ウォッチリスト項目が4件以上ある場合は、独立に調査できる項目(またはテーマの近いグループ)ごとに並列サブエージェントを起動することを既定とする。各サブエージェントには「定量値 / 出典URL / アクセス日 / 一次・二次ソースの区分 / トリガー条件への接近度」の固定フォーマットで返させ、親側で突合してから文書に反映する。ステップ3の反例・撤回・追試失敗の検索も同様に並列化してよい。単一テーマの網羅的な深掘りが必要な場合のみ deep-research を使う

### ステップ3: 反証チェック(最重要)

`references/review-checklist.md` の質問リストと検索規律(反対の証拠を先に探す)に従って実施する。

- 各時間軸予測の「反証条件」を1つずつ照合し、**成立/不成立/グレー** を判定。グレー判定には「次回までに白黒をつけるために観測すべきもの」を必ず添える。2四半期連続グレーの反証条件は、条件自体が曖昧すぎるとみなして書き直す
- 前回「確度・高」とした知見への反例・撤回・追試の失敗を能動的に検索する(「<主張> criticism / replication / debunked」)
- 自分の強み・ポジション評価(無価値化条件)に対して: プラットフォームの新機能・OSS・競合発信者の動向を確認し、無価値化条件への接近度を評価

### ステップ4: 更新の反映

- シナリオ確率(強気/基本/弱気)をベイズ更新し、変更理由を明記(事前確率の固定→証拠の方向づけ→小刻みな更新、の記録手順は `references/review-checklist.md` に従う)。更新前に**外側の視点**を必ず通す: 類似の技術移行(クラウド化、モバイル化等)の歴史的ベースレートと照らし、内側の視点との乖離が大きい場合は理由を書く(内側だけの確率操作はmotivated reasoningの温床)
- 知見の確度ラベルを昇降格(高→中への降格を躊躇しない)
- 新たに登場した重要トレンド・標準・研究を知見として追加(採用基準は既存文書と同じ: 一次ソース+裏取り)
- ウォッチリスト自体の見直し: 決着したシグナルの除去、新しい先行指標の追加
- 戦略への含意: 短期/中期/長期の打ち手のうち、前提が変わったものを具体的に指摘(「この打ち手は前提#Nが崩れたため優先度を下げるべき」)
- 優先度を上げる打ち手には**プレモーテム**(Klein)を実施: 「1年後、この打ち手は失敗した。何が原因か」を3つ列挙し、ウォッチリストに監視シグナルとして追加する
- 確定前に**独立レッドチームパス**を通す: 調査ログや更新の経緯を渡さないフレッシュなサブエージェントに、更新後のシナリオ確率と戦略への含意だけを渡し、「この確率と打ち手が誤っている最強の論拠」を挙げさせる。反論に答えられない項目は確定せずグレーに留め、ウォッチリストに監視シグナルを追加する(更新した本人による反証チェックはアンカリングを免れないため、独立コンテキストで補完する)

### ステップ5: 予測の採点とレビュー履歴

- **予測台帳で管理する**: 対象文書に「予測台帳」セクションがなければレビュー履歴の直前に作成する。各予測は1行で「登録日 / 予測文 / 確率 / 期限 / 判定(的中・外れ・無効)/ Brier」を持つ。期限到来分は**全件**を判定する(黙って消さない — 除外は選択的採点となりカリブレーションを歪める)。判定は登録時の文言のみに基づき、採点前の文言修正は禁止。文言では白黒つけられない場合は「無効」として理由を記録し、無効が続くなら登録時の観測条件の書き方自体を改善する
- 期限が到来した予測(3ヶ月予測など)を実績と突き合わせて採点する。確率付き予測はBrierスコア((予測確率−結果)²)を算出して蓄積する(計算例・解釈の基準線・採点の規律は `references/review-checklist.md` を参照)。確率帯(60-70%, 70-80%…)ごとの実績的中率は**全履歴累積**で集計する(年12件前後では単年・帯別の的中率は統計的に意味を持たない)。各帯のサンプルがN<10のうちは「参考値」と明記し、主指標は平均Brierスコアの四半期推移とする。集計は毎年最初のレビューで実施する(過信/過小自信の検出)
- 外れの原因分類: 情報不足 / 推論の誤り / 構造変化 / 単なる分散 — 次回の予測手法へフィードバック
- **新規予測の登録**(予測ループを閉じる): 次回レビューまでに決着する予測を最低3つ、「YYYY-MM-DDまでにXが起きる: NN%」の形式で予測台帳に登録する。曖昧語(「普及する」「主流になる」)は判定可能な観測条件に置き換える。これがないと採点対象が枯渇し、カリブレーションの蓄積(大原則3)が機能停止する
- レビュー履歴セクションに追記: 日付、主な変更、判定したトリガー、次回レビュー予定日(`references/review-checklist.md` の議事テンプレートを使う)
- 意思決定論点の棚卸し: 決めたこと・先送りしたこと(先送りの期限つき)

### ステップ6: サマリーの提示

ユーザーへの最終報告は以下の形式:

```markdown
## 四半期レビュー結果(YYYY-MM-DD)
### 🔴 崩れた前提・発動したトリガー(あれば最初に)
### 🟡 接近中のシグナル(次四半期の注視点)
### 🟢 維持された前提
### シナリオ確率の変更: 55/15/30 → XX/XX/XX(理由)
### 戦略への含意(優先順位の変更提案)
### 新規登録した予測(期限・確率つき)
### あなたの意思決定が必要な点
```

実例: `references/worked-example.md`(架空の予測文書に対するレビュー一巡の適用例)

調査中に発信ネタになりうる発見(意外な事実、良い一次データ)があれば、`tech-writing` スキルのネタ帳形式(一文の学び+証拠)で併記する。

## 行動規範

- 調査結果が戦略に都合が悪いときこそ強調して報告する(このスキルの存在意義)
- 「変化なし」も明確な成果として報告する(調査した上での「変化なし」と、調査していない「たぶん変化なし」を区別)
- 文書の編集は二層で行う: (1) **現在値フィールド**(シナリオ確率、確度ラベル、ウォッチリストの定量値、予測台帳の判定欄)は本文をin-place更新し、旧値を本文に残さない。(2) **知見とレビュー履歴**は既存の構造(知見番号、セクション構成)を尊重して追記し、旧値と変更理由は履歴側に記録する。本文は常に最新の判断を、履歴は変化の軌跡を表す
- レビューの所要時間を抑えたい場合向けに、ウォッチリストの定量シグナルだけ確認する「軽量版」も提案できるようにする

## 知見の出典

Philip Tetlock & Dan Gardner『超予測力』(ベイズ更新、外側の視点、採点の習慣)/ Annie Duke『Thinking in Bets』(結果と意思決定の質の分離)/ Gary Klein(プレモーテム)/ レッドチーム思考 / 元文書自身のレビュー設計(反証条件・ウォッチリスト・確率の四半期更新)
