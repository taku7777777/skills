# レビュー実務チェックリスト(反証・ベイズ更新・採点・議事)

SKILL.md のワークフローのうち、判断の質を左右する4工程(反証チェック、ベイズ更新、予測の採点、議事の記録)の実務手順。**このチェックリストの目的は「自説の補強」を機械的に防ぐこと** — 手順を飛ばしたくなったときこそ motivated reasoning が働いている。

## 1. 反証チェックの質問リスト(ステップ3で使用)

### 前提が崩れるシグナルの探し方

各予測・知見・戦略前提に対して、以下を順に問う:

- [ ] **この前提が誤っていたら、今ごろ世界で何が観測されているはずか?** それを探したか(「反証が見つからない」と「反証を探していない」を区別する)
- [ ] この前提の**最も弱い依存関係**は何か(前提Aは前提Bが真であることに依存していないか。Bが崩れると連鎖的に崩れる前提群を特定する)
- [ ] 前回レビュー以降に出た情報のうち、**この前提と整合しない**ものを最低1つ挙げられるか。挙げられない場合、探し方が確証側に偏っていないか
- [ ] この前提を信じることで**自分が得をする**構造はないか(希望と予測の混同 — 望ましいシナリオほど反証チェックを厚くする)
- [ ] 定量シグナルは**水準だけでなく変化率**を見たか(閾値未達でも接近速度が速ければ「接近中」として報告する)

### 確証バイアス対策の検索規律

- **反対の証拠を先に探す。** 支持する証拠の検索は後回しにする(先に支持証拠を集めると、その後の反証が「例外」に見えるアンカリングが起きる)
- 検索語を否定形で設計する: 「<主張> criticism」「<主張> replication failure」「<主張> overestimated / debunked」「<技術名> limitations / 失敗事例」。中立語(「<主張> 最新動向」)だけの検索は肯定的な記事を返しやすい
- **立場の異なる情報源を最低1つ含める**(推進側ベンダー ⇄ 懐疑側の研究者・実務者)。片側しか見つからない場合はその旨を記録する
- 見出しではなく本文・データまで確認する(見出しは主張を誇張する。「〜の終わり」という見出しの本文が「一部条件で〜」であることは多い)
- 反証を1つ見つけて満足しない: 「この反証自体が誤っている可能性」も1手だけ確認する(反証の反証)。二重に確認して残ったものだけを判定に使う

### 判定の記録

反証条件ごとに次の形式で残す:

```
反証条件: <文書記載の条件そのまま>
判定: 成立 / 不成立 / グレー
根拠: <観測事実+出典URL+アクセス日>
グレーの場合 → 次回までに観測すべきもの: <具体的な観測対象と判定基準>
```

## 2. ベイズ更新の実務手順(ステップ4で使用)

**厳密なベイズ計算(尤度比の数値化)は不要。ベイズの規律だけを借りる** — すなわち「事前確率を明示してから証拠を見る」「証拠1つごとに小さく動かす」「動かした理由を書く」の3点。

### 手順

1. **事前確率の固定**: 更新前のシナリオ確率(例: 強気/基本/弱気 = 55/15/30)を先に書き写す。証拠を読んでから事前確率を思い出すと、事後に引きずられる(ヒンドサイトバイアス)
2. **外側の視点から始める**: 類似の技術移行・市場変化の歴史的ベースレート(普及に要した年数、途中で失速した比率)を先に確認し、内側の視点(自分の観察)との乖離を見る。乖離が大きい場合、どちらを信じるかの理由を明記する
3. **証拠の列挙と方向づけ**: 今回のレビューで得た新証拠を1つずつ、「この証拠は、シナリオXが真の世界と偽の世界の**どちらでより観測されやすいか**」で方向づける。どちらの世界でも同程度に観測される証拠(=診断力がない証拠)は更新に使わない — ニュースの大半はこれに該当する
4. **小刻みに更新する**: 証拠1つあたりの変動は原則±10ポイント以内。1回のレビューで±20ポイントを超える更新をする場合は、構造変化(前提そのものの崩壊)として扱い、確率操作ではなくシナリオの再定義を検討する。**0%・100%にはしない**(それ以降の更新が数学的に不能になる)
5. **記録**: 次の形式で残す

```
シナリオ確率: 55/15/30 → 50/20/30
動かした証拠: <証拠の要約+出典>
方向づけの理由: <なぜこの証拠が強気シナリオを弱めるのか>
外側の視点との照合: <ベースレートと整合か乖離か。乖離なら理由>
```

### カリブレーションの考え方

- **カリブレーションとは「70%と言ったことが実際に70%の頻度で起きる」性質**であり、的中率の高さとは別物。全予測を99%か1%にすれば的中率は上がるが、情報量のない予測になる
- 予測の目的は当てることではなく、**確率という言葉を自分の中で較正された道具にする**こと。較正された70%と、願望込みの70%は別の言葉である
- 過信(予測確率>実績的中率)と過小自信(予測確率<実績的中率)は帯別集計で検出する(§3)。ただしサンプルが少ないうちは平均Brierの推移を主指標とする(SKILL.md ステップ5)
- 更新をサボることもカリブレーションを壊す: 「前回から動かさない」判断にも「動かさない理由(診断力のある新証拠がなかった)」を書く

## 3. 予測の採点手順 — Brierスコア(ステップ5で使用)

### 計算

二値の予測(起きる/起きない)について:

```
Brier = (予測確率 − 結果)²   結果: 起きた=1, 起きなかった=0
```

- 例1: 「70%で起きる」と予測し、起きた → (0.7 − 1)² = **0.09**
- 例2: 「70%で起きる」と予測し、起きなかった → (0.7 − 0)² = **0.49**
- 例3: 「50%」と予測 → 結果によらず (0.5 − 結果)² = **0.25**

### 解釈の基準線

- **0 = 完璧**(100%と言って起き、0%と言って起きない)
- **0.25 = 情報ゼロ**(常に50%と答えるのと同じ)。平均Brierが0.25を下回っていれば、予測に「コイン投げ以上の情報」がある
- **1.00 = 最悪**(確信を持って外す)。高確信の外れ1件は低確信の外れ数件分スコアを悪化させる — これは仕様であり、**「確信の押し売り」へのペナルティ**として機能する

### 採点の規律

1. 期限到来分は**全件**採点する(外れそうな予測を黙って消すのは選択的採点 — カリブレーション記録全体を汚染する)
2. 判定は登録時の文言のみに基づく。採点前の文言修正・再解釈は禁止(「実質的には当たっていた」は禁句)
3. 文言で白黒つけられない場合は「無効」とし、理由を記録。無効が続く場合は登録時の観測条件の書き方を改善する(採点手法ではなく登録手法の問題)
4. 各予測のBrierを台帳に記入し、**平均Brierの四半期推移**を主指標とする
5. 外れの原因を分類する: 情報不足 / 推論の誤り / 構造変化 / 単なる分散。**「単なる分散」は正当な分類である** — 30%と予測したことは10回に3回起きる。外れ=誤りではない(結果と意思決定の質の分離、Annie Duke)
6. 年最初のレビューで帯別(60-70%、70-80%…)の実績的中率を全履歴累積で集計する。N<10の帯は「参考値」と明記

## 4. 四半期レビュー議事テンプレート(ステップ5〜6で使用)

レビュー履歴セクションへの追記と、ユーザーへのサマリー提示(SKILL.md ステップ6)の中間成果物。調査の生ログではなく**判断の記録**を残す — 次回のレビューで「前回なぜそう判断したか」を再検証できる粒度で書く。

```markdown
## レビュー議事(YYYY-MM-DD)

### 実施範囲
- 通常版 / 軽量版(定量シグナルのみ)の別、調査に使ったサブエージェント数

### ウォッチリスト判定
| シグナル | 前回値 | 今回値 | 出典(URL+アクセス日) | トリガー判定 |
|---|---|---|---|---|

### 反証チェック結果
- 成立: <条件と根拠>(→ 発動したアクション)
- グレー: <条件>(→ 次回までの観測対象)
- 不成立: <件数と特記事項>

### シナリオ確率の更新
- XX/XX/XX → XX/XX/XX(§2の記録形式で理由を添付)
- レッドチームパスの反論と応答: <反論の要旨 / 反映した点 / 反映しなかった理由>

### 予測の採点
- 判定した予測: N件(的中 n / 外れ n / 無効 n)、平均Brier: X.XX(前回: X.XX)
- 外れの原因分類と手法へのフィードバック

### 新規登録した予測(最低3件)
- YYYY-MM-DDまでに<観測条件>: NN%

### 意思決定の棚卸し
- 決めたこと / 先送りしたこと(期限つき)

### 次回レビュー予定日: YYYY-MM-DD
```

**「変化なし」も判断として記録する。** 調査した上での「変化なし」(出典つき)と、調査していない「たぶん変化なし」を議事の上で区別できない書き方をしない。
