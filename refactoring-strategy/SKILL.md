# リファクタリング戦略スキル

Kent Beck『Tidy First?』、Martin Fowler『リファクタリング 第2版』および martinfowler.com(Strangler Fig, Parallel Change)、Paul Hammant(Branch by Abstraction)、Michael Feathers『レガシーコード改善ガイド』、Adam Tornhill『Software Design X-Rays』、Mikado Method の知見を統合した、安全で経済合理的なリファクタリングプロセス。

## 大原則

1. **リファクタリングは経済活動である**(Beck)。構造改善はオプション(将来の変更を安くする権利)の購入。「その変更は近々来るのか」「整頓のコストは変更で回収できるのか」を常に問う。美化のためのリファクタは負債より高くつくことがある。エージェント時代の補正: 変更の実装コストはエージェントで急落するが、レビューと検証のコストは落ちない。回収見込みは「書く時間の削減」ではなく「人間のレビュー・検証コストがどれだけ減るか」で見積もる。また乱雑な構造はエージェントの成功率とトークン消費にも課税する(コンテキストに載る重複・嘘はモデルを誤らせる)ため、エージェントが頻繁に触るホットスポットの整頓は従来より回収が速い。
2. **構造の変更と振る舞いの変更を絶対に混ぜない。** 別コミット・別PRに分離する(Tidy Firstの核心)。レビュー可能性が桁違いになり、AIエージェントへの委任も安全になる(構造変更のみのPRはレビューを軽量化できる — `harness-engineering` スキルのゲート分類でいうSOFT/AUTO)。
3. **常にグリーンのまま進む。** どの中間状態でもテストが通り、デプロイ可能。「一度全部壊して作り直す」大爆発方式(Big Bang)は原則禁止。例外: 界面が安定し挙動を機械的に比較できる小さな単位(1モジュール・純関数群)は抽象の裏で丸ごと書き直してよい(Branch by Abstractionの一工程にすぎない)。
4. **安全網なしに触らない。** テストのないコードのリファクタは、characterization testの整備から始める(`ai-adaptable-testing` 参照)。
5. **負債はデータで狙う。** 感覚ではなく「変更頻度 × 複雑度」のホットスポット(Tornhill)で優先順位を決める。触らないコードの負債は返さなくてよい。

## ワークフロー

### ステップ1: 目的と経済性の確認

着手前に必ず言語化する(答えられなければ着手しない):

- **なぜ今か**: この構造が妨げている具体的な変更・障害は何か(「近々この機能追加があるが、現状では3倍かかる」)
- **回収見込み**: 整頓後に安くなる変更の頻度・規模
- **撤退条件**: どこまでコストが膨らんだら中断するか。中間状態で止まっても価値が残る計画か
- **負債の実測**(Tornhill): `git log --since="12 months ago" --format= --name-only | sort | uniq -c | sort -rn | head -30` で変更頻度上位を取り、複雑度(行数・インデント深度で近似可)と掛け合わせてホットスポットを特定する。同一コミットで一緒に変わるファイル対(**変更カップリング**)はモジュール境界の嘘を教えてくれる
- Tidy First の分類: **先に整頓**(これから触る場所を安くする)/ **後で整頓**(変更のついでに)/ **後回し**(TODO化して別PR)/ **整頓しない**(触らない場所)— デフォルトは「これから変更する場所だけ先に整頓」。先の整頓が1時間を超えそうなら、それは整頓ではなく独立した仕事 — 経済性判断からやり直す

### ステップ2: 安全網の確認・構築

- 対象範囲のテスト充足を確認。不足なら characterization test / 承認テストで現挙動を固定
- テストが書けない場合: 依存を差し替えられる縫い目(**seam**)を最小の機械的変更で導入する(手順と失敗モードは `references/pattern-catalog.md` の seam 技法の節)。新機能追加が絡むなら **Sprout**(新コードを別関数/クラスに書き、既存からは呼ぶだけ)でテスト済みの新規部分を増やしながら進む(Feathers)
- 大規模移行では**差分テスト**(新旧実装を並走させ結果を比較 — GitHubのScientistパターン)や本番トラフィックのシャドー実行を検討
- 観測可能性の確認: 移行中の異常を検知できるメトリクス・アラートはあるか

### ステップ3: 戦略の選択

規模と性質でパターンを選ぶ。各パターンの詳細(適用条件 / 手順 / 失敗モード / 撤去までの完了条件)とディシジョンツリーは `references/pattern-catalog.md` に従う(以下は要約):

| 状況 | 戦略 |
|---|---|
| 小さな整頓(数分〜数時間) | **Tidy Firstカタログ**: ガード節化、デッドコード削除、説明変数・説明ヘルパー抽出、宣言と初期化の隣接、対称性の統一 — 振る舞い変更の直前に別コミットで |
| モジュール内の再設計 | Fowlerのリファクタリングカタログを小ステップで適用。IDE/型システムの機械的リファクタを最大活用 |
| 実装を丸ごと差し替え(ライブラリ脱却等) | **Branch by Abstraction**: ①抽象(自前IF)を挟む ②呼び出し側を抽象経由に移行 ③抽象の裏で新実装を構築 ④切り替え(フラグで段階的に) ⑤旧実装と抽象を削除 |
| システム・サービスの置き換え | **Strangler Fig**: 入口(ルーティング)で新旧を振り分け、機能単位で段階移行。新規機能は新側にのみ追加 |
| DBスキーマ・公開契約(公開API/イベントスキーマ)の変更 | **Expand-Contract(Parallel Change)**: 追加(両対応)→ 利用側の移行 → 旧の削除。後方非互換なインターフェース変更全般に適用。DBでは各段階で**1つ前のバージョンのアプリと現スキーマが共存できる**こと(ロールバックはコードだけ戻る)を、公開契約では非推奨期間の明示と利用側の移行完了の計測(残利用数)を確認 |
| 依存の追加・脱却の判断 | 依存は「借金」: コア機能に近いほど自前化・抽象化の価値が上がる。脱却判断はADR化(`arch-security-design` 参照) |

**脱却対象が「ライブラリ」か「フレームワーク」かで戦略が変わる**(呼ぶのがライブラリ、呼ばれるのがフレームワーク): 呼び出す側に立てる依存はBranch by Abstractionで抽象の裏に隠せる。しかし型と制御フローがコード全体に浸潤する依存(エフェクトシステム、リアクティブ基盤、ORMエンティティ、UIフレームワーク)は、全体を覆う抽象を挟むこと自体が非現実的。この場合は**端から剥がす**: 依存グラフの葉に近いモジュールから縦のスライス単位で丸ごと新形式に変換し、新旧の境界はモジュール継ぎ目の薄い変換アダプタで保ち、差分テストで挙動を固定しながら境界線を内側へ進める(コードベース内のStrangler Fig)。着手順は「依存グラフの葉 × ホットスポット」の交点で決める。

### ステップ4: 分割計画

- 各ステップを「独立してマージ可能・常にグリーン・戻せる」単位に分解し、順序を計画として提示する
- 手順が見通せない大変更は **Mikado Method**(詳細は `references/pattern-catalog.md`)で依存グラフを発見する: 目標を書く → 素朴に変更を試す → 壊れたら「先に必要な前提」をグラフに追記して**変更をrevert**(壊れた状態を手元に溜めない)→ 前提のない葉から順に独立コミットで実施。revertを恐れないことが探索コストを下げる。この探索ループはエージェント委任に向く: 使い捨てワークツリーで変更を試行させ、壊れた箇所のリスト(=グラフの前提ノード)だけを持ち帰らせて作業ツリーは破棄する。複数の目標ノードを並列に探索させれば、依存グラフの構築コストはほぼゼロになる
- **進捗の可視化**: 移行系は残件数を機械的に数えられるようにする(旧APIの参照数、ESLintルールでの新規使用禁止+既存の許容リスト方式)。新規流入を止めてから在庫を返すのが鉄則
- 長期並走の上限を決める(新旧併存は認知負荷という利息を生む。「6ヶ月で旧を消す」等の期限)

### ステップ5: 実行

- 構造変更コミットと振る舞い変更コミットを分離(コミットメッセージに [tidy] / [behavior] 等の区別も有効)
- **エージェント並列移行の運用**(呼び出し側の一括移行など機械的・反復的なステップが対象):
  - **手本PRを先に固定する**: 最初の1〜2件は人間(または監督付きエージェント)が変換して丁寧にレビューし(HARD相当)、その差分を「このパターンのbefore/after仕様」として以降の全委任プロンプトに含める。手本なしの一斉展開は逸脱を量産する
  - **決定論で書けるならcodemodを先に検討**: 純粋に機械的な変換は、エージェントに各所を手編集させるのではなくcodemod(ast-grep / jscodeshift / OpenRewrite 等)を書かせ、**変換ルール+サンプル差分をレビューする** — 500ファイルの差分ではなく1つの変換規則がレビュー対象になる。判断が要る変換だけをエージェントの手編集に回す
  - **互いに素なシャードに分割して並列化**: 残件インベントリをディレクトリ/モジュール/オーナー境界で重複なく分割し、1シャード=1エージェント=1PR。複数シャードが共有ファイル(許容リスト、re-export 等)に触れる設計はコンフリクトで直列化するため、避けるかマージ順を先に決める
  - **1PR=1パターン+サイズ上限**: 1PRには単一の変換パターンのみ。レビュー可能な上限(例: 20〜30ファイル)を超えるならシャードをさらに割る。パターンに合わない箇所は**その場で創意工夫させず**例外リストに記録させ、例外リストを次のパターン設計の入力にする
  - **許容リストの縮小を同一PRに含める**: 変換PRは対応する許容リストのエントリ削除も含める(進捗メトリクスが単調減少し、逆行はCIが弾く)
  - レビューは「手本PRとの一致確認+サンプリング」に軽量化できる(`harness-engineering` のSOFTゲート運用)
- 各ステップ後に検証(テスト+実挙動)。メトリクスの逸脱があれば即ロールバック

### ステップ6: 完了の定義

- 旧コード・旧依存・移行フラグ・許容リストの**削除まで**が完了(「両対応のまま放置」は負債が倍になった状態)
- 旧パターンは**エージェントが模倣する文脈**にも残る: CLAUDE.md・ドキュメント・手本として参照される代表ファイルに旧APIの用例が残っていれば、エージェントが旧パターンを再生産し続ける(lintが届かない箇所で特に)。コードの削除と同時にこれらの用例を新パターンへ更新することを完了条件に含める(CLAUDE.md自体の保守は `docs-for-ai` 参照)
- ADRに結果を記録: 予実(想定コストとの差)、学び、残した意図的な妥協

## 成果物フォーマット(計画提示時)

```markdown
# <対象> リファクタリング計画
## 1. 動機と経済性(妨げている変更 / 回収見込み / 撤退条件)
## 2. 現状分析(ホットスポット実測値 / テスト充足 / 依存関係)
## 3. 戦略(採用パターンと理由)
## 4. ステップ表(番号 / 内容 / 種別[tidy|behavior] / 検証方法 / 戻し方)
## 5. 進捗メトリクス(残件の数え方、蛇口を締めるlintルール)
## 6. 完了条件(削除対象: 旧コード・フラグ・許容リスト。文書・CLAUDE.md内の旧用例の更新)
```

実例: `references/worked-example.md`

## アンチパターン

- **リファクタの旅の途中で機能追加**: 混ぜた瞬間にレビュー不能・ロールバック不能になる
- **完璧主義の全面改修**: 触らないコードまで理想形に。回収不能な投資
- **「ついで直し」の無限連鎖**: 開けたファイルの気になる箇所を全部直す → 差分爆発。気になったらTODOリスト化して別PRへ
- **移行の新規流入を止めない**: 返済より借入が速い状態。lintで蛇口を締めるのが先
- **フラグ・許容リストの放置**: 切り替え完了後の削除タスクを最初から計画に含める
