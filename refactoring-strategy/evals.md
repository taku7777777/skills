# refactoring-strategy 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

追加の実行可能なraw fixtureとhidden rubricは [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離し、rubricをexecutorへ見せない。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | この2,000行の God クラス、触るたびに何か壊れる。来月の機能追加の前にどうにかしておきたい | 段階的リファクタリングの計画(先に整頓の経済性判断) |
| F2 | 自前で書いた ORM もどきを標準的なライブラリに乗り換えたい。参照箇所が多すぎて手の付け方が分からない | 依存の脱却・大規模移行(Branch by Abstraction 等) |
| F3 | 負債返済にスプリントの2割を使っていいことになった。どこから手を付けるのが一番得か決めたい | 負債返済の優先順位づけ(ホットスポット実測) |
| F4 | 稼働中のサービスを止めずに、DB のカラム名をドメイン用語に合わせて直したい。可能? | 公開契約・スキーマ変更(Expand-Contract) |
| F5 | エージェントに呼び出し側の書き換えを一括でやらせたら差分が巨大でレビュー不能だった。進め方から考え直したい | エージェント並列移行の運用(シャード分割・codemod) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 昨日から集計バッチの結果がズレてる。原因を突き止めて直してほしい | `systematic-debugging` が発火する(壊れているものを直すのはデバッグ。構造改善ではない) |
| N2 | 注文と在庫の集約境界がしっくりこない。ドメインモデルから設計し直したい | `ddd-design` が発火する(モデルの設計判断そのもの) |
| N3 | 大きめのリファクタを控えてるので、先にテストを固めて安全網を作っておきたい | `ai-adaptable-testing` が発火する(安全網構築はテスト戦略層。本スキルはその後段) |
| N4 | この関数の命名がイケてないので良い名前を提案して | スキルなしで応答する(単発の命名相談は計画不要) |

## 検証シナリオ

### S1: 成績計算モジュールの負債返済計画
- **タスク**: 「学習管理システム(Django、約6万行)の成績計算モジュールが限界。素点・出席・課題の重み付けが1ファイル1,500行に絡み合い、テストは通っているが誰も触りたがらない。半年後に『再試験の点数上書き』機能が入る予定。リファクタ計画を立てて」と依頼する(リポジトリは git 管理、CI あり、という文脈を添える)。
- **スキルなしで予想される欠落**: 経済性の言語化(なぜ今か・回収見込み・撤退条件)を計画の冒頭に置くこと。ホットスポットの実測手順(git log による変更頻度×複雑度)の提示。構造変更と振る舞い変更のコミット/PR 分離。各ステップの「常にグリーン・独立マージ可能・戻せる」性。理想形への全面改修案(触らない箇所まで再設計)に流れる。
- **合格条件**:
  - [ ] 動機と経済性(妨げている変更=再試験機能 / 回収見込み / 撤退条件)が明記される
  - [ ] ホットスポット実測の具体的な手順またはコマンドが含まれる
  - [ ] ステップ表の各項目に種別(tidy/behavior 相当の区別)と検証方法・戻し方がある
  - [ ] 対象がこれから触る場所に絞られ、触らない箇所の負債を対象外とする判断が示される
  - [ ] どの中間状態でもマージ可能・グリーンである計画になっている(人間判定)

### S2: 浸潤したフレームワークからの脱却
- **タスク**: 「社内 CRM のフロントエンドが EOL 間近のレガシー UI フレームワークで書かれている。コンポーネント定義・状態管理・ルーティングまで全部そのフレームワークの流儀で、画面数は約90。標準的な構成に移行したい。抽象レイヤを挟んで一気に差し替える案をチームに出したら不安の声が出た。計画を見てほしい」と依頼する。
- **スキルなしで予想される欠落**: ライブラリ(呼ぶ側)とフレームワーク(呼ばれる側)の区別 — 浸潤型依存には全体を覆う抽象(Branch by Abstraction)が非現実的で、「端から剥がす」(依存グラフの葉から縦スライスで変換、境界は薄いアダプタ)が適することの指摘。着手順を「葉 × ホットスポット」で決めること。新規流入を止める lint+許容リスト。長期並走の期限設定。
- **合格条件**:
  - [ ] 提示された「全体に抽象を挟む」案への修正(浸潤型依存には不適)が理由付きで述べられる
  - [ ] 端(依存グラフの葉)からのスライス単位の段階移行と、新旧境界のアダプタ方針が示される
  - [ ] 進捗メトリクス(旧フレームワーク参照の残数等)と新規流入を止める仕組みが計画に含まれる
  - [ ] 新旧併存の期限または上限の設定がある
  - [ ] 移行中の挙動固定(差分テスト・characterization 等)への言及がある

### S3: 膨らんだ「ついで直し」への途中介入
- **タスク**: 「経費精算 API のバグ修正 PR を書いてたら、周辺のコードが汚くて気になって直し始めたら差分が3,000行になった。バグ修正・変数名の整理・重複ヘルパーの統合・ついでに見つけた別の軽微バグ修正が混ざってる。このままレビューに出していい?どう整理すべき?」と依頼する(「リファクタ」という語は使わない)。
- **スキルなしで予想される欠落**: 構造変更と振る舞い変更を混ぜない原則に基づく分割指示(当初バグ修正 / tidy 群 / 別バグ修正をそれぞれ別 PR に)。「ついで直しの無限連鎖」をアンチパターンとして指摘し、残りは TODO 化して別 PR へ回すこと。分割の実行手順(どの変更から切り出すか)。「大きいけど内容は良いのでこのままで可」と流される。
- **合格条件**:
  - [ ] このままのレビュー提出を不可とし、PR の分割が指示される
  - [ ] 分割の軸が構造変更(tidy)と振る舞い変更(バグ修正)の分離として説明される
  - [ ] 当初のバグ修正と「ついでに見つけた別バグ」も別の振る舞い変更として分けられる
  - [ ] 未着手の気になる箇所は直さず TODO リスト化して別 PR に回す方針が示される
