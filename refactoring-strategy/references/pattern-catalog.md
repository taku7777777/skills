# リファクタリング戦略パターンカタログ(適用条件・手順・失敗モード・完了条件)

SKILL.md ステップ3(戦略の選択)の詳細展開。各パターンは「適用条件 / 手順 / 失敗モード / 完了条件」で構成する。**完了条件は常に「撤去まで」** — 新旧併存は目的地ではなく通過点であり、旧側とフラグの削除で初めて投資が回収される(SKILL.md ステップ6)。

## 0. パターン選択のディシジョンツリー

3つの軸で絞り込む: **対象の規模 / テスト充足 / 止められるか**(一斉切替が許されるか)。

```
対象の規模は?
├─ 関数〜モジュール内 ─────────── Fowlerカタログ+Tidy First(戦略パターン不要。小ステップ+テスト)
├─ 依存(ライブラリ・サブシステム)の差し替え
│   ├─ 呼び出す側に立てる(=こちらが呼ぶ)── Branch by Abstraction(§2)
│   └─ 型・制御フローが全体に浸潤(=呼ばれる)─ 端から剥がすStrangler(SKILL.md ステップ3)
├─ 公開契約(API・イベント・DBスキーマ)の変更 ─ Parallel Change / Expand-Contract(§3)
└─ システム・サービス丸ごとの置換 ──── Strangler Fig(§1)

横断チェック(どの枝でも先に確認):
- テストが無い ────── どのパターンより先に seam 技法(§5)で安全網を作る
- 手順・波及が読めない ─ 先に Mikado Method(§4)で依存グラフを発見してから各前提にパターンを割り当てる
- 止められない(利用側を一斉変更できない/無停止要件)─ 段階切替(フラグ・振り分け)と
  差分テスト・シャドー実行を必須化する。逆に、利用側が全て手元にあり一括変更が
  レビュー可能な規模なら、並走パターンは過剰 — codemod での一斉置換が最安のことも多い
```

## 1. Strangler Fig(システム・サービスの置換)

**適用条件**: 置換対象がシステム/サービス丸ごとで、Big Bang の書き直しはリスク許容外。入口(HTTPルーティング、メッセージ購読、バッチ起点)に新旧の振り分け点を差し込める。機能単位の縦スライスで切り出せる。移行中もサービスと機能追加を止められない。

**手順**:
1. **入口を1つに束ねる**: リバースプロキシ・APIゲートウェイ・イベントの横取り層を置き、全トラフィックが振り分け点を通る状態を作る(ここが作れないなら本パターンは選べない)
2. 移行単位(機能・エンドポイント・ドメインの縦スライス)を列挙し、価値×リスクで順序付けする(最初の1本は「小さいが本物」— 学習用に低リスクで、しかし本番トラフィックが流れるもの)
3. 1スライスずつ新実装へ振り分ける。**データの所有権も一緒に移すか、共有DBを経由するかを毎スライスで明示的に決める**(データ移行が本丸。同期の方向と正本をスライスごとに1つに定める)
4. **新規機能は新側にのみ追加する**(旧側凍結)。凍結はチーム合意として文書化する
5. 全スライス移行後、旧システム・振り分けロジックを撤去する

**失敗モード**:
- **入口が実は1つでない**: DB直結のバッチ、裏口の管理画面、直接参照する他システム — 振り分けを通らない経路が旧側を延命させる。最初の棚卸しで全経路を列挙する
- 新旧のデータ二重管理で正本が曖昧になり、整合性が壊れる(双方向同期は原則避ける)
- **「残り20%」で移行が止まり永遠の併存になる**: 難しい機能を全部最後に回した結果。難物は中盤に混ぜ、撤退条件(SKILL.md ステップ1)を先に決めておく
- 旧側凍結の合意がなく、移行中も旧側が成長する(返済より借入が速い状態)

**完了条件**: 旧システムの停止・撤去、振り分けロジックと移行用同期の削除、旧側の監視・運用手順・デプロイパイプラインの廃止。撤去前に**「旧側へのトラフィックがゼロ」を計測で確認する**(ログ・メトリクスで一定期間観察 — 思い込みで消さない)。

## 2. Branch by Abstraction(依存の差し替え)

**適用条件**: 呼び出す側に立てる依存(ライブラリ、サブシステム、自前モジュール)の実装を丸ごと差し替えたい。呼び出し箇所が多く一括置換の PR がレビュー不能。長命の移行ブランチを作らず、trunk 上で常にグリーンのまま進めたい。

**手順**(SKILL.md ステップ3 の5段階を詳細化):
1. 依存の**現利用面**を調査し(全呼び出し箇所と使用API の棚卸し)、**自分のドメインの言葉で必要最小限の抽象IFを定義する** — 旧ライブラリAPIの写しにしない。実際に使っている機能だけを抽象化する
2. 呼び出し側を1箇所ずつ抽象経由に移行する(この間、抽象の裏は旧実装のみ。各コミットは独立にグリーン&マージ可能)。機械的な置換なので codemod・エージェント並列移行の好対象(SKILL.md ステップ5)
3. 抽象の裏に新実装を構築する。**テストは抽象IFに対して書く**(新旧共通で流せる=そのまま差分テストになる)
4. フラグ・設定で新実装へ段階的に切り替える(リスクに応じてカナリア/環境別)。挙動差の疑いがあれば新旧並走で結果比較(Scientist パターン)
5. 旧実装と旧依存を削除し、フラグを撤去する。実装が1つに戻った抽象は**残す価値を再判定する**(将来も差し替え点として残すなら設計判断として ADR 化、不要なら抽象も薄めて畳む)

**失敗モード**:
- **抽象が旧APIの写経になる**: 新実装が旧の癖(引数の形、例外の粒度)を引き継ぎ、差し替えの自由を得られない。IFレビューの観点は「このIFから旧ライブラリ名を当てられるか」— 当てられたら写経
- 手順2(呼び出し側の移行)が終わる前に手順3(新実装)を始め、IFの設計変更が新実装と呼び出し側の両方に波及して二重コストになる
- 使っていない機能まで抽象化して移行コストが膨張する(YAGNI)
- 切替フラグが恒久化する(削除タスクを最初から計画に入れていない)

**完了条件**: 旧実装コードと旧依存パッケージの削除(依存グラフ・lockfile から消えたことを確認)、切替フラグの削除、抽象IFの残置/撤去判断の記録、CLAUDE.md・文書内の旧APIの用例更新(SKILL.md ステップ6)。

## 3. Parallel Change / Expand-Contract(インタフェース・スキーマ変更)

**適用条件**: 後方非互換な契約変更(メソッドシグネチャ、公開API、イベントスキーマ、DBスキーマ)を、利用側を一斉変更せずに行いたい。利用側が別チーム・別デプロイ・外部にあり、提供側と同時に変更できない。

**手順**:
1. **Expand(拡張)**: 新形式を**追加**し、旧形式と両対応にする — 新カラム追加+新旧への二重書き込み、新エンドポイント/新フィールドの追加(オプショナルで)。この段階では何も壊れない
2. **Migrate(移行)**: 利用側を新形式へ順次移行する。既存データはバックフィルする。**残利用数を機械的に計測する**(アクセスログ、deprecation メトリクス、旧カラム参照のクエリ監査)— 計測できない移行は終わらない
3. **Contract(収縮)**: 残利用ゼロを**計測で確認してから**旧形式を削除する — 旧カラムの DROP、旧エンドポイントの廃止、二重書き込みと変換コードの削除

**DB特有の確認**(SKILL.md ステップ3): 各段階で「**1つ前のバージョンのアプリ+現スキーマ**」が動くこと(ロールバックはコードだけ戻ればよい状態を保つ)。破壊的操作(DROP・NOT NULL 化)は即時実行せず、無効化→観察期間→削除の2段にすると事故時に戻れる。

**公開契約特有の確認**: 非推奨の明示(ドキュメント・レスポンスヘッダ・チェンジログ)と移行期限の通知。外部利用者は移行を強制できないため、期限とサポートポリシーを最初に宣言する。

**失敗モード**:
- **Expand のまま放置**: 両対応コード・二重書き込みが恒久化し、負債が2倍になった状態で固定される(最頻の失敗)
- 二重書き込みの片系エラーを握りつぶし、新旧データが静かに乖離する(突合ジョブを Expand と同時に置く)
- Migrate 完了を確認せず Contract して、残っていた利用者を壊す
- **Expand と Contract を同一リリースに同居させる**: それは実質 Big Bang であり、ロールバック不能

**完了条件**: 旧カラム/旧API/旧イベントバージョンの削除、二重書き込み・変換・フォールバックコードの削除、突合ジョブの停止、契約定義(OpenAPI・スキーマレジストリ)からの旧定義削除、利用者向け文書の更新。

## 4. Mikado Method(依存が読めない大改造の計画)

**適用条件**: 変更の波及が事前に読めない大改造(中核の型変更、モジュール分割、フレームワーク更新)。「やってみたら芋づる式に壊れる」ことが予想され、着手前に分割計画(SKILL.md ステップ4)を立てる材料がない。

**手順**:
1. 目標を1文で書く(グラフのルートノード)
2. 素朴に変更を試す
3. 壊れたら、壊れを解消するために**先に必要な前提**を子ノードとしてグラフに追記する
4. **変更を revert する**(核心 — 壊れた状態を手元に溜めない。学びはグラフに残り、コードは常にグリーン)
5. 各前提ノードに 2〜4 を再帰適用する。前提のない**葉**に到達したら、葉から順に独立コミット/PR で実施し、グラフを上に戻っていく

グラフはテキスト(Markdown のチェックリスト、Mermaid)で管理し、PR・設計文書から参照できるようにする。探索(手順2〜4)は使い捨てワークツリーのエージェントに並列委任できる(SKILL.md ステップ4 — 持ち帰るのは壊れた箇所のリストだけ、作業ツリーは破棄)。

**失敗モード**:
- **revert を惜しむ**: 壊れた差分の上に作業を積んで WIP が巨大化 — Mikado の意味が消える。「revert は失敗ではなく計測」と心得る
- グラフを頭の中だけで持つ(共有・中断・再開ができない)
- 葉が大きすぎる: 1つの葉に数日かかるなら、その葉自体を目標にして再分解する
- **完璧な地図を待つ**: グラフは実施中も成長する。探索で全容が判明してから実行、ではなく葉が見つかり次第着手してよい

**完了条件**: ルート目標の達成とグラフ全ノードのクローズ。探索時に入れた一時コード(TODO・仮の分岐)の削除。グラフ自体は「なぜこの順序だったか」の記録として ADR/PR に残す。

## 5. Feathers の seam 技法(テストがないコードへの安全網)

**適用条件**: テストのないコードを変更する必要があるが、依存がハードコードされていて(コンストラクタでのI/O、グローバル状態、静的呼び出し)そのままではテストを書けない。どの戦略パターンよりも**先に**適用する(SKILL.md 大原則4)。

**seam** = その場所を編集せずに振る舞いを差し替えられる点。優先順に:
- **オブジェクト seam**: 依存を引数・コンストラクタ注入に変える(Extract Interface / Parameterize Constructor 等の機械的操作)。第一選択
- **リンク seam**: ビルド・リンク・DI設定での差し替え(テスト用モジュール置換)。コードを触れないときの次善
- **プリプロセッサ seam**: C系のマクロ置換。最後の手段

**手順**:
1. 変更対象と、テストで挟みたい範囲を決める(変更点そのものではなく、**変更の影響を観測できる最小の範囲**)
2. seam を**最小の機械的変更**で導入する — この時点ではテストがないため、IDE の自動リファクタ・型検査で守られる操作だけを使い、判断を要する編集をしない
3. **characterization test** を書く: 現在の出力・副作用をそのまま期待値として記録する。仕様の正しさの検証ではなく**現挙動の固定**が目的(バグも今は固定する — テスト名やコメントに「現挙動の記録」と明記)
4. 安全網ができてから、本来のリファクタ・変更に着手する
5. 新機能追加が絡む場合は **Sprout**(新コードを別関数・クラスとして書いてテストし、既存からは1行で呼ぶ)/ **Wrap**(既存呼び出しをラップして前後に処理を挟む)で、テスト済みの新規部分を増やしながら進む

**失敗モード**:
- **seam 導入のつもりで振る舞いを変える**: テストなしの変更は機械的操作に限定する原則の違反。「この操作は IDE/型が保証するか」を毎回問う
- characterization test に「あるべき値」を書いてしまう: 後で現バグを直したときにテストの意図が判別できず混乱する
- モックだらけのテストにする: seam は注入点に留め、検証は観測可能な出力・状態で行う(内部呼び出しの検証はリファクタで壊れるテストを量産する)
- 対象全体にテストを書こうとして本来の変更に着手できない(変更する場所の周辺だけでよい)

**完了条件**: 本来の変更の完了後に**足場を裁定する** — characterization test は意味のある期待値・名前を持つ仕様テストへ昇格させるか、恒久テストが揃った時点で削除する(「現挙動の記録」のまま放置しない)。設計改善として価値のある seam(依存注入)は残してよいが、テスト専用フック・プリプロセッサ seam は撤去する。
