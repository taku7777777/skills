# リファクタリング戦略スキル 適用例

## 題材

架空のECバックエンド(TypeScript、約18万行)。メール・SMS 送信に EOL が宣言された通知ライブラリ「(仮)notify-legacy」を使用しており、コード全体の約120箇所から直接呼ばれている。通知まわりはテストが薄く、送信結果のハンドリングが箇所ごとにバラバラ。

## 入力(ユーザーの依頼と文脈)

> 「notify-legacy が1年後にサポート終了する。後継の notify-next へ移行したい。ついでに通知コードが汚いので整理したい。移行計画を立てて」

文脈: 3ヶ月後に「通知チャネル追加(アプリ内プッシュ)」の機能開発が控えている。通知送信は課金に絡まない非同期処理で、数分の遅延は業務上許容される。

## 各ステップでの判断

- **ステップ1(目的と経済性)**: なぜ今か=EOL 期限+3ヶ月後のチャネル追加が現構造では二重実装になる。回収見込み=チャネル追加が抽象IF への実装1つで済む。撤退条件=抽象化と呼び出し側移行が完了していれば新実装が遅れても価値が残る(中間状態で止まれる計画)。ホットスポット実測で通知ヘルパー2ファイルが変更頻度上位と確認。**「ついでに整理」は分離** — 送信結果ハンドリングの統一は振る舞い変更を含むため、移行と混ぜず別トラックの TODO へ。
- **ステップ2(安全網)**: 通知呼び出し元のテスト充足は約3割。ライブラリ境界に seam がない(静的 import 直呼び)ため、抽象導入自体が seam になる。送信結果は正解が定義しやすいので、切替期は新旧並走の差分テスト(送信ペイロードの比較、実送信は旧のみ)を採用。
- **ステップ3(戦略の選択)**: `references/pattern-catalog.md` のディシジョンツリーで判定 — 対象は「呼び出す側に立てるライブラリの差し替え」→ **Branch by Abstraction**。120箇所の一括置換 PR はレビュー不能なので段階移行が必須。抽象IF は現利用面の棚卸しから `send(notification)` と `sendBulk` の2メソッドのみ(notify-legacy の7メソッドを写経しない)。
- **ステップ4(分割計画)**: 呼び出し側の移行は機械的なので codemod(ast-grep)+パッケージ単位のシャードで並列化。進捗メトリクスは `import 'notify-legacy'` の残数。ESLint の import 禁止ルール+既存120箇所の許容リストで新規流入を停止。
- **ステップ5(実行)**: 手本PR(2箇所の手動変換)を先に固定し、codemod の変換ルールをレビュー対象に。1PR=1パッケージ・20ファイル上限。パターンに合わない5箇所(添付ファイル付き送信)は例外リストに記録し、手動移行の別PRへ。
- **ステップ6(完了の定義)**: 旧ライブラリ・切替フラグ・許容リストの削除に加え、CLAUDE.md と docs/onboarding.md に残る notify-legacy の用例を新IFの用例へ更新。ADR に予実(想定6週→実績8週、差分は例外5箇所の設計判断)を記録。

## 成果物(SKILL.md の成果物フォーマットに準拠)

# 通知ライブラリ移行(notify-legacy → notify-next)リファクタリング計画

## 1. 動機と経済性

- **妨げている変更**: notify-legacy の EOL(12ヶ月後)。3ヶ月後のプッシュ通知追加が現構造では全呼び出し箇所の分岐追加になる
- **回収見込み**: チャネル追加が抽象IF への実装追加1つで完結。以後のライブラリ更新の影響範囲が抽象の裏に閉じる
- **撤退条件**: フェーズ2(呼び出し側移行)完了時点で中断しても抽象化の価値は残る。フェーズ2が6週を超過したら計画を見直す

## 2. 現状分析

- ホットスポット: `src/shared/notify-helper.ts`(12ヶ月で41コミット)ほか1ファイル
- テスト充足: 呼び出し元の約3割。ライブラリ境界に seam なし
- 依存関係: 直接 import 120箇所(23パッケージ)、うち添付送信の特殊利用5箇所

## 3. 戦略

**Branch by Abstraction**(呼び出す側に立てるライブラリの差し替え、一括置換はレビュー不能なため)。切替は環境フラグで段階実施、切替期は送信ペイロードの差分テストで挙動固定。

## 4. ステップ表

| # | 内容 | 種別 | 検証方法 | 戻し方 |
|---|---|---|---|---|
| 1 | 抽象IF `Notifier` 定義+旧実装アダプタ | tidy | 既存テスト+IFの単体テスト | revert |
| 2 | ESLint で notify-legacy の新規 import 禁止(既存は許容リスト) | tidy | CI で逆行が落ちること | ルール無効化 |
| 3 | 呼び出し側を codemod でシャード移行(1PR=1パッケージ、許容リスト縮小込み) | tidy | 差分テスト+パッケージ単位テスト | シャード単位 revert |
| 4 | 例外5箇所(添付送信)の手動移行 | tidy | characterization test を先に追加 | revert |
| 5 | notify-next 実装を抽象の裏に構築 | tidy | IF共通テストを新旧両実装に適用 | フラグ未切替のまま |
| 6 | 環境フラグで段階切替(dev → staging → 本番10%→100%) | behavior | 送信成功率メトリクス+差分テスト | フラグ戻し |
| 7 | 旧実装・旧依存・フラグ・許容リスト・差分テストの削除 | tidy | 依存グラフに notify-legacy が無いこと | revert |

## 5. 進捗メトリクス

- 残件: `import 'notify-legacy'` の箇所数(週次で単調減少を確認)
- 蛇口: ESLint `no-restricted-imports` + 許容リスト(縮小のみ可、CI が逆行を弾く)

## 6. 完了条件

- 削除対象: 旧ライブラリ(package.json / lockfile)、切替フラグ、ESLint 許容リスト、差分テスト基盤
- 文書更新: CLAUDE.md・onboarding 内の notify-legacy 用例を `Notifier` の用例へ差し替え
- ADR 記録: 予実差・例外5箇所の設計判断・抽象IF を残す判断の根拠
