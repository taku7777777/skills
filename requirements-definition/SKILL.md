# 要件整理・定義スキル

Karl Wiegers『Software Requirements』、Gojko Adzic(Impact Mapping / Specification by Example)、Jeff Patton(User Story Mapping)、Alistair Cockburn(ユースケース)、IPA非機能要求グレードの知見を統合した要件定義プロセス。

## 大原則

1. **解決策ではなく問題から始める。** ユーザーが持ち込むのはたいてい「解決策の要望」(〜画面が欲しい)。その裏の問題・目的(Why)まで遡ってから要件にする。
2. **要件は「書く」ものではなく「発見して検証する」もの。** 対話・具体例・反例で認識のズレを潰す。
3. **曖昧語を残さない。**「高速」「使いやすい」「適切に」「原則として」「など」— 検証可能な条件に変換するか、TBDとして明示的に残す。
4. **やらないことを決めるのが要件定義の半分。** スコープ外リストは要件リストと同格の成果物。
5. **完璧な要件文書より、認識が揃った薄い文書。** 文書は対話の記録であって代替ではない(Adzic)。

## ワークフロー

### ステップ0: 文脈の調査

リポジトリ・既存資料がある場合、**質問の前に**関連コード・既存の類似機能・ドメイン用語・README/ADRを調査し、「わかっていること / 推測 / 不明」を仕分けてから質問する。既存システムの制約(認証方式、DB、外部連携)は聞かずに確認できることが多い — 調べればわかることを聞くのは尋問である。

### ステップ1: Why の確定(Impact Mapping)

最初に4つを埋める。ユーザーの説明から埋まらない箇所は**推測で埋めずに質問する**(AskUserQuestion があれば選択肢つきで使う):

- **Goal(なぜ)**: 達成したいビジネス目標。可能なら測定可能に(「問い合わせ対応工数を30%削減」)
- **Actor(誰が)**: 目標達成に影響を与える人・システム。主要アクターと副次アクター。外部API、自動連携、利用者側アシスタントなどの兆候がある場合に限り、**AIエージェントがアクターになるか**を確認する。エージェントがアクターなら、機械可読なインターフェースの要否・操作を許す範囲・エージェントの識別と監査を要件に入れる。通常の人間向け機能で根拠なくAI利用を尋ね、スコープを広げない
- **Impact(どう行動が変わる)**: アクターの行動変化(「顧客が自己解決できるようになる」)
- **Deliverable(何を作る)**: 行動変化を起こす機能。**ここが最後**— 機能から始めない

Impact Mapは穴埋めフォームではなく**代替案の地図**: 各Impactに対してDeliverable候補を複数挙げ、行動変化を最も安く起こせる/検証できるものを選ぶ。Deliverableは仮説であり、「本当にその行動変化が起きるか」の測定方法までセットで書く。ユーザーが持ち込んだ機能案も候補の1つとして地図に置き、より安い代替がないか一度は問う。

このとき同時に確認する前提: 期限・予算などの制約、既存システムとの関係、成功の測定方法、ステークホルダー(決定権者・反対しうる人)。

### ステップ2: スコープの輪郭

- **ユーザーストーリーマップ**(Patton)で全体像を作る: 横軸にユーザーの活動の流れ(バックボーン)、縦軸に各活動のタスクを**上ほど必須・下ほど任意**の順で並べる(この順序がリリーススライスの前提)
- **リリーススライス**: マップに横線を引き、Walking Skeleton(端から端まで通る最小限)→ 以降の増分を定義。MoSCoW(Must/Should/Could/Won't)で優先度づけ
- **スコープ外リスト**: 「今回やらないこと」を明文化し、理由を1行添える(誤解が最も高くつく項目から)

### ステップ3: 機能要件の詳細化

主要ストーリーごとに:

1. **ユーザーストーリー形式**: 「<役割>として、<目的>のために、<機能>がしたい」— INVEST基準(独立・交渉可能・価値がある・見積可能・小さい・テスト可能)でチェック。大きすぎるストーリーは次の軸で分割する(Cohn の SPIDR + Lawrence の分割パターン): ワークフローの段階(まず幹線だけ)/ ビジネスルールのバリエーション / データの種類 / 操作(CRUDを分ける)/ インターフェース(1チャネルから)/ 不確実なら調査スパイクを切り出す。**「設計/実装/テスト」のフェーズ分割は価値が出ないため禁じ手**
2. **受け入れ基準を Given/When/Then の具体例で書く**(Specification by Example)。ルールを箇条書きするだけでなく、**境界の具体例**を必ず含める:
   ```gherkin
   ルール: 注文金額5,000円以上で送料無料
   例1: Given 商品合計 5,000円 When 注文確定 Then 送料 0円
   例2: Given 商品合計 4,999円 When 注文確定 Then 送料 500円
   例3: Given 商品合計 5,000円でクーポン500円適用 When 注文確定 Then 送料は? ← 曖昧点として質問
   ```
   具体例を作る過程で見つかる「決まっていないケース」こそが要件定義の成果物(Example Mapping: ルール/例/疑問を分けて可視化し、疑問が多いストーリーは着手不可と判定する)。
3. **例外・代替フロー**(Cockburn): ハッピーパスの各ステップについて「ここで失敗したら?」を問う(入力不正、権限なし、外部システム停止、同時実行、途中離脱)
4. 使い分け: ユーザー価値の検証例は Given/When/Then、**システム横断の恒常ルール・例外方針・状態依存の制約**(タイムアウト、監査ログ等)は **EARS記法**で書く(`references/ears-notation.md`)。同じ内容を両方に書かない
5. **出力が非決定的な機能(LLM・レコメンド等)の受け入れ基準**: 単発の Given/When/Then では合否を定義できない。次の3点セットで書く:
   - **統計的基準**: 代表入力セットに対する合格率で定義する(例:「想定質問100件に対しルーブリック合格90%以上、かつ既存基準からの劣化なし」)。「何を合格とするか」の判定基準(ルーブリック)への合意はこの段階で取り、評価セットの設計・運用は `test-analysis` / `ai-adaptable-testing` へ引き継ぐ
   - **ゼロ許容の制約**: 確率で妥協しない条件は EARS の望ましくない挙動型で決定的に書く(例:「もし応答に個人情報が含まれるならば、システムは応答を破棄して定型文を返さなければならない」)。ガードレールとして実装可能な形にする
   - **失敗時のふるまい**: 品質基準を満たせない場合の代替動作(フォールバック応答、人間へのエスカレーション、機能の縮退)を通常の機能要件として定義する

### ステップ4: 非機能要件

`references/nfr-checklist.md`(IPA非機能要求グレード準拠の簡易版)を使い、少なくとも以下は数値または明示的な「対象外」を決める:

- 性能(応答時間、同時ユーザー、データ量とその成長)
- 可用性(稼働率、計画停止可否、RPO/RTO)
- セキュリティ(認証、権限、データ分類、監査 — 詳細設計は `arch-security-design` へ)
- 運用(監視、バックアップ、サポート時間帯)
- 移行(既存データ、並行稼働)
- 法令・規制(個人情報、業界規制、アクセシビリティ)

### ステップ5: 曖昧さの検査

文書全体を Wiegers の曖昧語リストで走査する:

- **曖昧な形容詞・副詞**: 高速に、柔軟に、簡単に、最大限、適切に、通常は、原則として
- **未定義の集合**: など、その他、〜等、すべての(本当に全部か?)
- **主語のない受動態**: 「通知される」(誰から誰に? どの経路で?)
- **量化の欠落**: 「大量データに対応」(何件? 何GB?)
- **and/or の曖昧**: 「AとBの場合」(かつ? または?)

検出した曖昧箇所は (a) その場で確定 (b) **未決事項リスト**に担当者・期限つきで登録、のどちらかに必ず振り分ける。

### ステップ6: 検証とレビュー

- 各要件が **Goal に遡れるか**(トレーサビリティ)。どのGoalにも繋がらない要件は削除候補
- 各要件が**テスト可能か**。「テストケースが書けない要件」は要件ではない
- **前提とリスクの棚卸し**: 要件の裏の前提(「既存APIが〜を返す」「月次データは1万件以下」)を列挙し、外れたら影響が大きい順にリスク化する(成果物8章)。技術的前提は可能ならこの段階でスパイクで潰す
- ステークホルダー別レビュー観点を提示: 決定権者にはスコープとコスト、開発者には実現可能性と曖昧さ、運用者には運用要件
- **反例で認識を試す**: 「この要件だと、〜の場合は〜という動作になりますが、それで正しいですか?」という確認質問を3〜5個作ってユーザーに投げる。ここでズレが見つかれば成功

## 成果物フォーマット

```markdown
# <プロジェクト/機能名> 要件定義
## 1. 背景と目標(Impact Map: Goal / Actor / Impact)
## 2. スコープ
### 2.1 対象範囲(ストーリーマップ or 機能一覧 + MoSCoW)
### 2.2 スコープ外(理由つき)
## 3. 機能要件(ストーリー + 受け入れ基準(Given/When/Then) + 例外フロー + 横断ルール(EARS))
## 4. 非機能要件(数値つき)
## 5. 用語集(ユビキタス言語の種 — `ddd-design` スキルへ引き継ぐ)
## 6. 制約・前提
## 7. 未決事項(TBDリスト: 内容 / 決定に必要な情報 / 誰が / いつまで)
## 8. リスク
```

すべての要件(ストーリー、EARS文、非機能要件)に安定した一意ID(US-01 / FR-01 / NFR-01 等)を振る。IDは要件を削除しても再利用しない。下流の成果物(テスト条件・テストコード・コミットメッセージ)はこのIDで要件にトレースする — `test-analysis` のトレーサビリティ検査と、テスト変更時に「要件のどの変更に基づくか」を示す運用(`ai-adaptable-testing`)はこのIDを前提とする。

小さな機能追加では 1・2・3・7 だけの軽量版でよい(省略を明示)。

実例: `references/worked-example.md`

## ふるまいの規範

- 質問は**一度に3〜5個まで**、優先度順に。尋問にしない。回答で次の質問が変わるものから聞く
- ユーザーの用語をそのまま使う(勝手に言い換えない)。同じものを指す用語が揺れていたら指摘して統一する
- 「わからないことがわかっている」状態を成果として堂々と提示する。TBDを隠して埋めた要件定義が最も危険
- ユーザーが要件文書の作成・実装まで依頼しており、書き込みが依頼範囲内の場合は、会話ログで終わらせずリポジトリ内(例: `docs/requirements/<機能名>.md`。既存のspec配置規約があればそれに従う)へ保存する。レビュー・相談だけの依頼では、配置案またはドラフトを提示し、勝手にファイルを変更しない。**git commit は明示的に依頼された場合だけ**行う。正本化した後は要件変更をコードと同一PRで文書へ反映する(仕様とコードの乖離検出は `harness-engineering` のdrift gate、文書体系の設計は `docs-for-ai` の領分)
- 要件が固まったら、次工程(`arch-security-design` / `ddd-design`)への引き継ぎ情報(用語集、品質特性の優先順位)を明示する
