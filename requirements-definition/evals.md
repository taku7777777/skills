# requirements-definition 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの references/eval-template.md を参照。

追加の実行可能なraw fixtureとhidden rubricは [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離し、rubricをexecutorへ見せない。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 営業から「顧客が自分で見積もりを出せるページが欲しい」と言われた。開発に入る前に、何をどこまで作るのか固めるのを手伝って | 解決策の要望からのWhy遡及・要求の聞き出し |
| F2 | 新しい機能のアイデアが頭の中に散らかってる状態。質問しながら開発に渡せる文書に落としてもらえる? | 曖昧なアイデアからの要件整理・文書化 |
| F3 | このIssueにある「使いやすい検索」って、結局何ができたら完成なの? 合格ラインを決めたい | 受け入れ基準の定義(「受け入れ基準」の語を使わない言い回し) |
| F4 | 通知機能をやるかどうかで揉めてる。やる範囲とやらない範囲を、関係者に説明できる形で線引きしたい | スコープ整理(スコープ外リストが本領) |
| F5 | 粗い機能リストはあるので、開発チームが着手できる粒度のバックログに割ってほしい | ユーザーストーリー化・ストーリー分割(INVEST/SPIDR) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | チーム内で「契約」の意味が揺れてるので、用語を整理してモデルとコードに反映したい | `ddd-design` が発火する(ユビキタス言語とモデリングの領分) |
| N2 | 要件は固まってる。これを実現するシステム構成と技術スタックを決めたい | `arch-security-design` が発火する(構造・技術選定は下流工程) |
| N3 | この要件定義書、文章が硬くて読みにくいので誰でも読める形に書き直して | `tech-writing` が発火する(内容の発見・検証ではなく文書表現の改善) |

## 検証シナリオ

### S1: 解決策の要望からの要件定義(中心ユースケース)
- **タスク**: 「会員制フィットネスジムの予約システムに『キャンセル待ち機能』を作ってほしい。人気クラスがすぐ埋まって会員から不満が出ている。要件をまとめて」と依頼する(リポジトリなし、対話相手はプロダクト担当者という設定)。
- **スキルなしで予想される欠落**: 機能案を所与としてそのまま詳細化する(Goal/Impactへの遡及と代替Deliverableの検討の欠落)。スコープ外リストの欠落。受け入れ基準が箇条書きルールのみで境界の具体例(Given/When/Then)がない。要件への一意ID付与の欠落。
- **合格条件**:
  - [ ] Goal / Actor / Impact / Deliverable の整理があり、キャンセル待ち機能が仮説(Deliverable候補の1つ)として位置づけられている
  - [ ] キャンセル待ち以外の代替案(例: クラス増枠、ノーショー対策)が少なくとも1つ検討されている
  - [ ] 受け入れ基準に Given/When/Then 形式の具体例があり、境界値の例(繰り上げ期限ちょうど等)を含む
  - [ ] 「やらないこと」のリストが理由つきで明記されている
  - [ ] ストーリー・要件に一意ID(US-xx / FR-xx 等)が振られている

### S2: 非決定的出力を含む機能の受け入れ基準(境界・例外)
- **タスク**: 「社内規程(就業規則・経費規程など)への質問にLLMで答える社内チャットボットを作る。人事部の問い合わせ対応を減らしたい。要件を定義して」と依頼する。
- **スキルなしで予想される欠落**: 受け入れ基準を単発の Given/When/Then だけで書こうとして合否を定義できない。統計的基準(代表入力セット+合格率)・ゼロ許容制約のEARSによる決定的な記述・品質未達時のフォールバックの3点セットの欠落。
- **合格条件**:
  - [ ] 回答品質の受け入れ基準が、代表質問セットに対する合格率などの統計的基準で定義されている
  - [ ] 確率で妥協しない制約(例: 個人の給与情報を含む応答の禁止)が「もし〜ならば、システムは〜しなければならない」の決定的な形式(EARS)で書かれている
  - [ ] 回答できない・品質を満たせない場合のふるまい(定型応答、人事部への誘導等)が機能要件として定義されている
  - [ ] 合格判定基準(ルーブリック)への合意、または評価セットの下流スキルへの引き継ぎに言及がある

### S3: 書きかけメモへの途中介入(変化球)
- **タスク**: 「中古自転車の出張買取サービスの査定依頼フォームについて、この要件メモで開発に渡して大丈夫か見てほしい」と依頼し、曖昧語を含むメモを添付する(例: 「写真などをアップロードできる」「査定結果は速やかに通知される」「大量の依頼にも対応」「原則として翌日までに担当者が対応」)。
- **スキルなしで予想される欠落**: 体裁を整えるだけで通してしまう。曖昧語の系統的な走査(未定義の集合「など」、主語のない受動態「通知される」、量化の欠落「大量」)の欠落。未決事項をTBDリストとして構造化せず口頭指摘で終わる。反例による認識確認の欠落。
- **合格条件**:
  - [ ] メモ中の曖昧語(「速やかに」「など」「大量」「原則として」)のうち3つ以上が具体的に指摘され、検証可能な条件への変換またはTBD化が提案されている
  - [ ] 「通知される」について通知の主体・宛先・経路の未定義が指摘されている
  - [ ] 未決事項が、決定に必要な情報・誰が・いつまでを持つTBDリストの形で整理されている
  - [ ] 「この要件だと〜の場合どうなるか」形式の反例確認質問が2つ以上提示されている
