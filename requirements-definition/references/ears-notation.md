# EARS記法(Easy Approach to Requirements Syntax)

Alistair Mavin(Rolls-Royce)考案。自然言語の要件文を5つの定型パターンに制限することで、曖昧さ・複合文・抜け漏れを構造的に防ぐ。仕様書調の要件記述にはこれを使う。

## 5パターン

| パターン | テンプレート | 例 |
|---|---|---|
| 遍在型(Ubiquitous) | システムは<応答>しなければならない | システムはパスワードをArgon2idでハッシュ化して保存しなければならない |
| イベント駆動型(Event-driven) | **<トリガー>したとき**、システムは<応答>しなければならない | ユーザーが注文を確定したとき、システムは確認メールを送信しなければならない |
| 状態駆動型(State-driven) | **<状態>の間**、システムは<応答>しなければならない | メンテナンスモードの間、システムは管理者以外のログインを拒否しなければならない |
| オプション型(Optional feature) | **<機能>を備えている場合**、システムは<応答>しなければならない | 生体認証デバイスを備えている場合、システムは生体認証によるロック解除を提供しなければならない |
| 望ましくない挙動型(Unwanted behaviour) | **もし<望ましくない条件>ならば**、システムは<応答>しなければならない | もし決済APIが10秒以内に応答しなければ、システムは注文を保留状態にして利用者に通知しなければならない |

複合パターン: 「<状態>の間、<トリガー>したとき、システムは〜」のように組み合わせ可能。

## 運用ルール

- 1文に応答は1つ。「AしてBする」は2要件に分割
- 主語は常に**具体的なシステム名**(「注文管理システムは」)。複数システム構成では、どのシステムの責務かの曖昧さこそEARSが防ぎたい事故
- オプション型(Where)は製品構成・ビルド/デプロイ単位の変種に使う。実行時にオン/オフが変わるユーザー設定は状態駆動型(「二要素認証が有効な間」)で書く
- 「〜できる」「〜が望ましい」を避け、「しなければならない(shall)」で書く。任意なら優先度ラベル(Should/Could)を別途付ける
- **望ましくない挙動型が最も価値が高い**: 正常系は誰でも書ける。「もし〜ならば」を各要件に対して問うことで例外系の抜けを検出する
