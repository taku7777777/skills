# 非機能要件チェックリスト(IPA非機能要求グレード準拠・簡易版)

各項目は「数値で確定」「明示的に対象外」「TBD(未決事項リストへ)」のいずれかにする。
空欄のまま進めるのが最悪。小規模案件は★のみでよい。
数値は「測定尺度・測定方法・必達値/目標値」のセットで書く(例: 尺度=検索応答時間p95、測定=APMの本番計測、必達=3秒、目標=1秒 — Gilb の Planguage 簡易版)。

## 1. 性能・拡張性

- [ ] ★応答時間: 通常画面 / 重い処理(検索・帳票)それぞれの目標(例: p95で2秒以内)
- [ ] ★同時利用者数・リクエスト数: 通常時 / ピーク時(ピークの発生条件: 月初、セール等)
- [ ] ★データ量: 初期 / 年間増加量 / 保持期間(アーカイブ・削除方針)
- [ ] バッチ処理: 処理時間の上限(業務開始までに終わる等)、リラン可否
- [ ] 将来拡張: ユーザー数・拠点・多言語の見込み(1〜3年)

## 2. 可用性

- [ ] ★稼働率目標と、換算した許容停止時間(99.9% = 月43分)— 数字の意味を確認してから合意する
- [ ] ★サービス提供時間帯(24/365? 平日日中のみ?)、計画停止の可否
- [ ] ★RPO(失ってよいデータは何分ぶんか)/ RTO(何時間で復旧するか)
- [ ] 障害時の縮退: 全停止を避けたい機能はどれか(参照だけでも生かす等)
- [ ] 大規模災害時の要件(DRサイトの要否)

## 3. セキュリティ(要求レベルの確定 — 設計は arch-security-design で)

- [ ] ★扱うデータの分類: 個人情報 / 決済情報 / 機密情報の有無と件数規模
- [ ] ★利用者の種別と認証要件(社内のみ? 一般公開? SSO要否? MFA要否?)
- [ ] ★権限: 役割の種類と、役割によって見えてはいけないデータ
- [ ] 監査: 操作ログの要否、保持期間、監査対応(誰がいつ何を見たか)
- [ ] 準拠すべき規格・法令: 個人情報保護法、GDPR、PCI DSS、業界ガイドライン
- [ ] セキュリティテスト: 脆弱性診断の要否・時期

## 4. 運用・保守

- [ ] ★監視: 何を検知したら誰にどう通知するか。対応時間帯(24h? 営業時間?)
- [ ] ★バックアップ: 対象、頻度、世代数、リストア試験
- [ ] リリース頻度と方式: 停止リリース可否、切り戻し要件
- [ ] サポート体制: 問い合わせ窓口、SLA
- [ ] ログ: 保持期間、調査要件(障害調査に必要な情報)

## 5. 移行

- [ ] ★既存システムからのデータ移行: 対象データ、量、クレンジング要否
- [ ] 並行稼働期間の有無、切り替え方式(一括 / 段階)
- [ ] 移行リハーサル・切り戻し計画の要否

## 6. システム環境・制約

- [ ] インフラ制約: 指定クラウド / オンプレ要件 / リージョン制約(データ国内保持等)
- [ ] 対応環境: ブラウザ、OS、モバイル対応、画面サイズ
- [ ] アクセシビリティ: 準拠レベル(WCAG 2.2 AA / JIS X 8341-3 等)の要否
- [ ] 外部接続: 連携システム一覧、接続方式、相手側の制約・調整窓口

## 7. コスト・体制の制約(要件と同時に確認)

- [ ] ランニングコストの上限感
- [ ] 運用する人・スキル(専任運用者はいるか)— 運用能力を超える構成は要件不成立

## 8. AI/LLM機能を含む場合

- [ ] ★品質基準: 評価セットに対する合格率の必達値/目標値(受け入れ基準としての定義はSKILL.mdステップ3。ここでは継続監視の尺度・測定方法として確定)
- [ ] ★推論コスト: リクエスト単価の想定と月次上限、暴走時の停止条件(上限アラート・レートリミット)
- [ ] ★データ送信: 外部モデルAPIへ送ってよいデータの範囲(3章のデータ分類と突合 — 個人情報・機密情報を送るか、マスキング要否、プロバイダ側の学習利用可否)
- [ ] 応答時間: LLM呼び出し込みの目標(ストリーミング表示で体感を担保するか)
- [ ] プロバイダ障害・モデル廃止: フォールバック(縮退・代替モデル)の要否、モデル更新への追随方針(更新は挙動変更として回帰評価 — 運用は `ai-adaptable-testing`)
