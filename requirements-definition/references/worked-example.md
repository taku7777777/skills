# requirements-definition 適用例

題材: 架空の中堅企業向け経費精算SaaS「ケイヒフロー」への「多段階承認ワークフロー」追加。既存は直属上長による1段階承認のみで、TypeScriptのモジュラーモノリス + PostgreSQLで動いている。

## 入力(ユーザーの依頼と文脈)

> 「経費の承認を多段階にしたい。高額な経費は部門長も承認する形にして。要件をまとめてほしい」

リポジトリには既存の承認機能(1段階)と社員・組織マスタがある。

## 各ステップでの判断

- **ステップ0(文脈の調査)**: 質問の前に既存コードを調査。承認処理・組織マスタの上長/部門長参照・認証方式は確認できたので聞かない。「代理承認の仕組みが存在しない」ことだけ不明点として質問リストへ。
- **ステップ1(Why)**: 「多段階承認」は解決策の要望。遡るとGoalは内部統制監査の指摘(高額経費の単独承認)への対応だった。代替案として「高額経費の事後サンプリング監査」もImpact Mapに置いたが、監査要件が事前承認を求めるため不採用。AIエージェントがアクターになるかを確認 → 申請の自動起票は構想段階のみでスコープ外と判断。
- **ステップ2(スコープ)**: Walking Skeleton =「金額閾値による2段階承認が申請から最終承認まで通る」。承認ルートのGUI編集・3段階以上はWon'tへ。
- **ステップ3(機能要件)**: 境界の具体例を作る過程で「閾値は明細ごとか申請合計か」が曖昧と判明 → 質問し「申請合計」で確定。エスカレーションと監査ログは恒常ルールなのでEARSで記述(Given/When/Thenと重複させない)。
- **ステップ4(非機能)**: 性能と監査ログ保持のみ数値化。他は既存SLO継承として「対象外」を明示。
- **ステップ5(曖昧さ検査)**: 初稿の「不在時は適切にエスカレーション」を曖昧語として検出し、「7営業日」+ 代理承認者に確定。
- **ステップ6(検証)**: 反例確認「申請者が部門長本人のとき、2段階目は誰が承認するのか?」でズレが発覚 → TBD-01として登録。

## 成果物

小さな機能追加ではないが1機能なので、8章構成のうち6・8章は骨子のみの軽量版とした(省略を明示)。

```markdown
# 多段階承認ワークフロー 要件定義

## 1. 背景と目標(Impact Map)
- Goal: 内部統制監査の指摘「高額経費の単独承認」を次回監査(Q4)までに解消する。承認リードタイム平均3営業日以内は維持する
- Actor: 申請者(社員)/ 承認者(上長・部門長)/ 経理担当。AIエージェント経由の操作は今回対象外(スコープ外参照)
- Impact: 高額経費が必ず2名の承認を経る。承認者が承認待ちを7営業日以上放置しなくなる
- Deliverable: 金額閾値による承認ルートの自動決定 + エスカレーション通知(測定: 監査指摘のクローズ、リードタイムのダッシュボード)

## 2. スコープ
### 2.1 対象範囲(MoSCoW)
- Must: 閾値による2段階承認(US-01〜US-03)、エスカレーション(FR-01)、監査ログ(FR-02)
- Should: 承認待ち一覧の金額・滞留日数での絞り込み
### 2.2 スコープ外
- 承認ルートのGUI編集(閾値変更は設定値の変更で足りる。監査対応の納期を優先)
- 3段階以上の承認・代理承認者の自動選定(利用実態が不明なため次期判断)
- AIエージェントによる申請自動起票(構想段階。機械可読IFの要否は要求が具体化してから)

## 3. 機能要件
- US-01: 申請者として、経費の払い戻しを受けるために、明細を添えて経費申請を提出したい(受け入れ基準は略)
- US-02: 承認者として、不正な経費支出を防ぐために、自分宛の承認待ち申請を承認または差し戻ししたい
  - ルール: 申請合計5万円以上は上長承認の後に部門長承認が必要。申請者自身は自分の申請の承認者になれない(職務分掌)
  - 例1: Given 申請合計 49,999円 When 上長が承認 Then 申請は「承認済み」になる
  - 例2: Given 申請合計 50,000円 When 上長が承認 Then 申請は「部門長承認待ち」になる
  - 例3: Given 部門長本人による申請 60,000円 When 上長が承認 Then 2段階目の承認者は? ← TBD-01
  - 例外: 承認操作時に申請が取り下げ済み → 操作を拒否し、状態が変わった旨を表示する
- US-03: 承認者として、不備のある申請を修正させるために、理由を付けて差し戻したい(受け入れ基準は略)
- FR-01(EARS): もし承認待ちが7営業日を超えたならば、システムは代理承認者に通知し、承認権限を付与しなければならない
- FR-02(EARS): システムは常に、すべての承認・差し戻し・エスカレーション操作を操作者と日時つきで監査ログに記録しなければならない

## 4. 非機能要件
- NFR-01: 承認待ち一覧の応答時間 p95 1秒以内(同時50ユーザー)
- NFR-02: 監査ログの保持期間7年、改変不可
- 可用性・運用・移行: 既存SLO・運用体制を継承(本機能固有の追加要求なし = 対象外)

## 5. 用語集(ユビキタス言語の種 — `ddd-design` へ引き継ぐ)
- 経費申請(ExpenseReport)/ 承認ステップ(ApprovalStep)/ 承認ルート(ApprovalRoute)/ 差し戻し(Return)/ エスカレーション(Escalation)

## 6. 制約・前提
- 組織マスタの上長・部門長情報が最新であること(外れた場合のリスクは8章)。他は(略)

## 7. 未決事項
- TBD-01: 申請者=部門長のときの2段階目の承認者。決定に必要な情報: 内部統制ルールの該当条項。担当: 経理部長。期限: 次回定例

## 8. リスク
- 組織マスタの陳腐化により誤った承認者へルートされる(発生時は監査指摘の再発に直結)。他は(略)
```

次工程への引き継ぎ: 用語集(5章)と品質特性の優先順位(セキュリティ・監査性 > 性能)を `arch-security-design` / `ddd-design` へ渡す。
