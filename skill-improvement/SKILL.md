# スキル評価・改善スキル

Anthropic のスキル作成ベストプラクティス(評価を先に作る反復プロセス、skill-creator)、スキル自動最適化研究(実行トラジェクトリからの蒸留、検証信号の設計、厳密改善時のみ採用)、公開スキルの効果実測調査の知見を統合した、スキルを「書きっぱなし」にしないための評価・改善運用プロセス。

対象は Claude Code のスキル(SKILL.md + references)。エージェント委任全般の検証ゲート設計は `harness-engineering`、ドキュメント体系そのものは `docs-for-ai` の領分。本スキルは「スキルという1つの成果物」の効果測定と改善に特化する。

## 大原則

1. **評価が先、本文が後。** Anthropic の公式プロセスは「ギャップ特定 → 検証シナリオ作成 → ベースライン測定 → 最小限の記述 → 反復」であり、広範な本文を書く前に評価を作れと明言している。既存スキルへの後付け(retrofit)でも順序は同じ — まず評価、それから編集。
2. **測定していないスキルは、効いていない可能性の方が高い。** 公開スキルの実測調査(SWE-Skills-Bench)では49本中39本に性能向上がなく(平均+1.2%)、明確に改善したのは7本、3本はむしろ劣化した。「丁寧に書いた」ことは効果の根拠にならない。
3. **スキルは、凍結されたモデルに対する唯一の学習パラメータ。** スキル改善は深層学習の訓練ループと同型である — 実行記録が訓練データ、検証シナリオが損失関数、テキスト編集が勾配、採否判定が過学習監視。この対応を意識すると「評価なしの編集=損失関数なしの学習」の無意味さが明確になる。
4. **発火しないスキルは、存在しないのと同じ。** 起動時にコンテキストへ載るのは name と description のみで、発火判断はそこだけで決まる。本文の充実より先に、発火テストが通ることを確認する。
5. **厳密改善時のみ採用する。** 編集後に評価を再実行し、改善した場合だけ取り込む。悪化・同等なら差し戻す。また**1回きりの失敗への対症編集は過適合** — 繰り返し観測された失敗パターンだけを本文に反映する。
6. **蒸留は使った直後が最も安い。** スキルを使ったセッションの終了前に、成功したアプローチとつまずいた箇所をそのセッション自身に要約させ、編集候補として記録する。後日ゼロから思い出すより桁違いに安く、正確である。
7. **機械判定に落とせない評価は、人間の仕事として明示的に残す。** 成果物の専門的な質の判断を無理に自動化すると損失関数が壊れ、評価全体が無意味になる。「(人間判定)」と書いて残すことは失敗ではなく設計である。

## 評価セットの規約

各スキルはディレクトリ直下に `evals.md` を持つ。構成は3部: **発火テスト**(発火すべきプロンプト5+発火すべきでないプロンプト3)、**検証シナリオ3つ**(タスク / スキルなしで予想される欠落 / 合格条件チェックリスト)、**実行記録**(日付・モデル・条件・判定の追記型ログ)。テンプレートと書き方・実行方法の詳細は `references/eval-template.md` に従う。

## ワークフロー

### ステップ0: モードの判定

依頼を4モードに分類する: **(A) 新規スキル作成**(ステップ1から順に)/ **(B) 既存スキルへの評価整備**(ステップ2から)/ **(C) 改善反復**(ステップ5から。評価がなければ先にBを実施)/ **(D) 発火診断**(ステップ2の発火テストのみ整備・実行し、description を修正)。

### ステップ1: ギャップの特定(新規作成時)

スキルなしの素のモデルに代表タスクを実行させ、**具体的にどこで失敗するか**を記録する。失敗が観測できないなら、そのスキルは作らない(一般知識で足りている)。失敗の記録がそのまま検証シナリオの「スキルなしで予想される欠落」になる。

### ステップ2: 評価セットの作成

`references/eval-template.md` に従い `evals.md` を書く。要点: 発火すべきプロンプトは description のトリガー例の言い換えではなくユーザーが実際に打ちそうな文にする / 発火すべきでないプロンプトは隣接スキルとの境界を突く / 検証シナリオは worked-example と別題材にする / 合格条件は第三者が Yes/No 判定できる形に寄せる。

### ステップ3: ベースライン測定

スキルなしの新セッションで検証シナリオを実行し、成果物を保存・採点して実行記録に残す。**baseline が全条件を満たすシナリオは検出力ゼロ**なので書き直す。ここを飛ばすと「スキルの効果」が永遠に測れない。

### ステップ4: 最小限の記述・編集

ギャップに対処する**最小限**の内容を書く。網羅的な教科書ではなく、素のモデルが自発的にやらないことだけを足す。分量が増えるほど発火後のコンテキスト税が上がる — SKILL.md は簡潔に保ち、詳細は references へ(progressive disclosure)。

### ステップ5: 評価実行と採否判定

with-skill 条件で検証シナリオを再実行し、baseline と比較採点する(採点は実行と別セッションで)。**厳密に改善した場合のみ編集を採用**し、実行記録に追記する。改善しない場合は差し戻し、ギャップ仮説(欠落候補・シナリオ設計)を疑う。

### ステップ6: 継続運用(蒸留と定期見直し)

スキルを使ったセッションから失敗パターンを蒸留して編集候補に積み、**繰り返し現れたものだけ**ステップ4〜5の反復にかける。四半期ごとに実行記録を見直し、(a) 常に合格するシナリオの難化、(b) 発火テストの実績確認、(c) 出典の陳腐化チェックを行う。

## 成果物フォーマット

- モードA/B: 対象スキルの `evals.md`(テンプレート準拠)+ ベースライン測定の実行記録。
- モードC: 編集diff + 編集前後の評価結果比較表 + 採否判定(採用/差し戻し+理由)。
- モードD: 発火テスト結果表 + description の修正案(修正理由: どのプロンプトが不発/誤発火だったか)。

実例: `references/worked-example.md`

## 知見の出典

- Anthropic「Skill authoring best practices」および skill-creator(評価を先に作る反復プロセス、progressive disclosure、最小限の記述)
- スキル自動最適化の複数研究(2026年上半期。例: arXiv 2606.11435, 2605.27760, 2606.26790): 実行トラジェクトリからのスキル蒸留、繰り返し現れる編集のみ保持する過適合防止、検証信号の設計が反復性能を決めるという知見、テキスト編集を「厳密改善時のみ採用」する更新則
- SWE-Skills-Bench(arXiv 2603.15401。公開スキル49本の効果実測: 大半が効果なし、測定の不可欠性)
- 深層学習の訓練ループとの構造対応(パラメータ=SKILL.md、訓練データ=実行記録、損失関数=検証シナリオ、過学習監視=採否判定)
