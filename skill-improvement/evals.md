# skill-improvement 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの references/eval-template.md を参照(本スキル自身への自己適用)。

S1の実行可能な架空スキルfixtureとhidden rubricはリポジトリレベルの [task manifest](../quality/tasks/cases.jsonl) に登録済み。実行時は `scripts/eval_tasks.py prepare` で分離パケットを生成する。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | うちの CLAUDE.md 配下にあるスキル、ちゃんと役に立ってるのか確認したい | 効果測定の依頼(語彙は「評価」を含まない) |
| F2 | このスキル、頼んでも全然使われないんだけど原因を見てほしい | 発火診断(モードD) |
| F3 | 昨日のセッションでスキルの手順どおりに動かなくて困った。スキル側を直したい | 失敗の反映・改善反復(モードC) |
| F4 | チーム用に PR レビューのスキルを新しく作りたい | 新規作成(モードA、eval-first で進めるべきケース) |
| F5 | スキルを書き換えたんだけど、この変更を取り込んでいいか判断して | 編集の採否判定 |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | エージェントに安全にタスクを任せるための検証ゲートを設計して | `harness-engineering` が発火する(スキル単体でなく委任全般) |
| N2 | CLAUDE.md を整備してドキュメント体系を作りたい | `docs-for-ai` が発火する(スキルでなくドキュメント全般) |
| N3 | この Python 関数のスキルフルなリファクタ案を出して | スキルなし(または `refactoring-strategy`)。「スキル」の語に釣られない |

## 検証シナリオ

### S1: 既存スキルへの評価整備(モードB)
- **タスク**: `quality/fixtures/skill-improvement/commit-message-skill.md` をraw artifactとして添付し、「このスキルに評価を整備して」と依頼する。期待する欠落・rubricはexecutorへ渡さない。
- **スキルなしで予想される欠落**: 発火テストという概念自体の欠落(検証シナリオのみ作る)、baseline 測定の手順の欠落、トリガー例をそのまま評価プロンプトに流用する自己言及的な評価。
- **合格条件**:
  - [ ] evals.md がテンプレート3部構成(発火テスト/検証シナリオ/実行記録)で出力される
  - [ ] 発火すべきプロンプトが description のトリガー例の言い換えになっていない
  - [ ] 発火すべきでないプロンプトに隣接領域との境界ケースが含まれる
  - [ ] 各シナリオに「スキルなしで予想される欠落」が明記される
  - [ ] baseline 測定を実施(または手順として明示)してから編集に進む段取りになっている

### S2: 編集の採否判定(モードC)
- **タスク**: 「スキルに手順を3節追記した。評価は編集前 4/5・編集後 4/5、ただし発火テストで新たに誤発火が1件増えた」という状況を与え、採否を求める。
- **スキルなしで予想される欠落**: 「せっかく書いたので取り込む」方向のバイアス。厳密改善の原則(同等+悪化項目ありは差し戻し)の欠落。
- **合格条件**:
  - [ ] 判定が「差し戻し」である
  - [ ] 理由に「厳密改善していない(同等かつ誤発火が悪化)」に相当する説明がある
  - [ ] 追記内容を活かすための次の反復(分割検証・description の見直し等)が提案される

### S3: 暗黙的な新規作成依頼(モードA)
- **タスク**: 「チームでよくやる障害報告書の作成を Claude に任せたいので、いい感じの手順書スキルを一気に書いて」と依頼する(評価には言及しない)。
- **スキルなしで予想される欠落**: いきなり網羅的な本文を書き始める(eval-first の欠落)。ギャップ特定(素のモデルの失敗観測)を飛ばす。
- **合格条件**:
  - [ ] 本文執筆の前に、ギャップ特定または評価セット作成を先行させる進め方になっている
  - [ ] 「素のモデルで失敗が観測できなければ作らない」に相当する判断基準への言及がある
  - [ ] 本文を書く場合も「最小限の記述」の方針が明示される
