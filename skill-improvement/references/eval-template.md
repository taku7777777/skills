# 評価セット(evals.md)テンプレートと運用ガイド

各スキルのディレクトリ直下に `evals.md` を置く。SKILL.md や references と違い、**エージェントがスキル実行時に読むためのファイルではなく、スキルそのものを測定するためのファイル**である。スキル利用時のコンテキストには載らない。

## テンプレート

```markdown
# <skill-name> 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | <ユーザーが実際に打ちそうな依頼文> | <descriptionのどの用途に該当するか> |
| F2〜F5 | ... | ... |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | <隣接スキルの領分の依頼> | <代替スキル名> が発火する |
| N2 | <一見似ているが対象外の依頼> | スキルなしで応答する |
| N3 | ... | ... |

## 検証シナリオ

### S1: <シナリオ名>
- **タスク**: <エージェントに与える依頼文と文脈。必要な入力データを含めて自己完結させる>
- **スキルなしで予想される欠落**: <素のモデルが落とすはずの、このスキル固有の価値>
- **合格条件**:
  - [ ] <出力に○○が含まれる / ○○の形式である、など判定可能な条件>
  - [ ] <3〜5項目>
  - [ ] <機械判定に落とせない項目は「(人間判定)」と明記>

### S2: <シナリオ名>
(同上)

### S3: <シナリオ名>
(同上)

## 実行記録

| 日付 | モデル | 条件 | シナリオ | 判定 | メモ |
|---|---|---|---|---|---|

(未実施)
```

## 書き方ガイド

### 発火テスト

- **発火すべきプロンプトに、description のトリガー例をそのまま使わない。** トリガー例との一致確認は「訓練データでの評価」であり検出力がない。ユーザーが実際に打ちそうな、トリガー例と語彙の異なる言い回し・文脈つきの依頼文にする(例: トリガー例が「テスト分析して」なら、評価プロンプトは「この PR、どこを重点的に確認すべきか洗い出してほしい」)。
- **発火すべきでないプロンプトは、隣接スキルとの境界を突く。** 最も紛らわしい依頼(テスト3層、設計3スキルの境界など)を選ぶ。全スキル共通の「明らかに無関係な依頼」は1件で十分。
- 5件 / 3件は下限ではなく目安。境界が複雑なスキルは増やしてよい。

### 検証シナリオ

- **worked-example と別題材にする。** 同じ題材では「スキル本文の再現能力」を測ってしまい、汎化性能(過適合の有無)が見えない。
- **「スキルなしで予想される欠落」から逆算する。** シナリオの目的は「素のモデルでも書ける一般論」ではなく「このスキルだけが足す価値」の検出。スキルの大原則・ワークフローのうち、素のモデルが自発的にやらないことを欠落候補にする(例: 反証チェック、EARS記法、二層防御、仮説ログ)。
- **合格条件は機械判定に寄せる。** 「○○の表が含まれる」「各項目に△△列がある」「□□への言及がある」など、第三者(別セッションの Claude や人間)が Yes/No で判定できる形にする。専門的な質の判断が必要な条件は無理に機械化せず「(人間判定)」と明記して残す — ごまかした自動判定は損失関数の崩壊であり、評価全体を無意味にする。
- 3シナリオの役割分担の目安: S1=中心ユースケース、S2=境界・例外(スキルが特に効くはずの難所)、S3=発火文脈の変化球(暗黙的な依頼、途中からの介入など)。

## 実行方法

### 発火テスト

1. スキルを配置した環境で**新しいセッション**を開始し、プロンプトを1件ずつ投げる(明示呼び出しはしない)。
2. スキルが起動したか(Skill ツールの呼び出し、またはスキル本文に沿った挙動)を記録する。
3. 全件の再実行が重い場合の簡易法: 別セッションの Claude に「このユーザープロンプト一覧と全スキルの description を渡すので、各プロンプトでどのスキルを選ぶか判定せよ」と依頼する。ただし実セッションでの確認を正とする。

### 検証シナリオ

1. **baseline**: スキルを配置しない環境の新セッションに S1〜S3 のタスクを与え、成果物を保存する。
2. **with-skill**: スキルを配置した環境で同じタスクを実行し、成果物を保存する。
3. 両成果物を合格条件で採点する。採点者はタスク実行とは**別のセッション**の Claude(合格条件のチェックリストのみ渡す)または人間。実行したセッション自身に採点させない。

### 記録

- 実行記録表に「日付 / モデル / 条件(baseline・with-skill) / シナリオ / 判定(合格条件の充足数) / メモ」を追記する。
- **baseline が全条件を満たすシナリオは評価として機能していない。** そのシナリオは「素のモデルとの差分」を検出できていないので、欠落候補を見直して書き直す。

## 採否判定(スキル編集時)

- 編集後に検証シナリオを再実行し、**厳密に改善した場合のみ**編集を採用する(合格条件の充足数が増える、または同数で発火テストが改善)。悪化・同等なら差し戻す。
- **1回きりの失敗への対症編集は採用しない。** 実行記録やセッションで繰り返し観測された失敗パターンのみ本文に反映する(1回きりの修正は過適合)。
- 編集で評価も同時に書き換えない。評価の更新(シナリオ追加・合格条件の見直し)と本文の更新は別のコミット・別の判断として行う。
