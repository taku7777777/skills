# 評価セット(evals.md)テンプレートと運用ガイド

各スキルのディレクトリ直下に `evals.md` を置く。SKILL.md や references と違い、**通常のスキル発火時に自動ロードするファイルではなく、スキルそのものを測定するための定義**である。ただし同じファイルシステムへ置けば実行エージェントが探索できるため、`evals.md` をhidden rubricとして信用しない。raw fixtureとgrader rubricはリポジトリレベルの評価領域で分離し、実行時にはtask packetだけを別workspaceへ渡す。

## テンプレート

```markdown
# <skill-name> 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | <ユーザーが実際に打ちそうな依頼文> | <descriptionのどの用途に該当するか> |
| F2〜F5 | ... | ... |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | <隣接スキルの領分の依頼> | <代替スキル名> が発火する |
| N2 | <一見似ているが対象外の依頼> | スキルなしで応答する |
| N3 | ... | ... |

## 検証シナリオ

### S1: <シナリオ名>
- **タスク**: <エージェントに与える依頼文と文脈。必要な入力データを含めて自己完結させる>
- **スキルなしで予想される欠落**: <素のモデルが落とすはずの、このスキル固有の価値>
- **合格条件**:
  - [ ] <出力に○○が含まれる / ○○の形式である、など判定可能な条件。安全性・データ損失等はcriticalと明記>
  - [ ] <3〜5項目>
  - [ ] <機械判定に落とせない項目は「(人間判定)」と明記>

### S2: <シナリオ名>
(同上)

### S3: <シナリオ名>
(同上)

## 実行記録

| 日付 | モデル | 条件 | シナリオ | 判定 | メモ |
|---|---|---|---|---|---|

(未実施)
```

## 書き方ガイド

### 発火テスト

- **発火すべきプロンプトに、description のトリガー例をそのまま使わない。** トリガー例との一致確認は「訓練データでの評価」であり検出力がない。ユーザーが実際に打ちそうな、トリガー例と語彙の異なる言い回し・文脈つきの依頼文にする(例: トリガー例が「テスト分析して」なら、評価プロンプトは「この PR、どこを重点的に確認すべきか洗い出してほしい」)。
- **発火すべきでないプロンプトは、隣接スキルとの境界を突く。** 最も紛らわしい依頼(テスト3層、設計3スキルの境界など)を選ぶ。全スキル共通の「明らかに無関係な依頼」は1件で十分。
- 5件 / 3件は下限ではなく目安。境界が複雑なスキルは増やしてよい。

### 検証シナリオ

- **worked-example と別題材にする。** 同じ題材では「スキル本文の再現能力」を測ってしまい、汎化性能(過適合の有無)が見えない。
- **「スキルなしで予想される欠落」から逆算する。** シナリオの目的は「素のモデルでも書ける一般論」ではなく「このスキルだけが足す価値」の検出。スキルの大原則・ワークフローのうち、素のモデルが自発的にやらないことを欠落候補にする(例: 反証チェック、EARS記法、二層防御、仮説ログ)。
- **合格条件は判定可能にするが、本文への従順さだけを測らない。** 「○○の表が含まれる」だけでなく、コンパイル、既知攻撃の遮断、契約互換性、正しい計算結果など成果の妥当性を優先する。第三者が Yes/No で判定できる形にし、専門的な質の判断が必要な条件は無理に機械化せず「(人間判定)」と明記する。セキュリティ、データ損失、権限逸脱、誤発火等は **critical** とし、他項目の加点で相殺させない。
- 3シナリオの役割分担の目安: S1=中心ユースケース、S2=境界・例外(スキルが特に効くはずの難所)、S3=発火文脈の変化球(暗黙的な依頼、途中からの介入など)。

## 実行方法

### 発火テスト

1. スキルを配置した環境で**新しいセッション**を開始し、プロンプトを1件ずつ投げる(明示呼び出しはしない)。実行側には期待スキル名・正解・過去結果を見せない。
2. スキルが起動したか(Skill ツールの呼び出し、またはスキル本文に沿った挙動)を記録する。
3. 全件の再実行が重い場合の簡易法: 別セッションの Claude に「このユーザープロンプト一覧と全スキルの description を渡すので、各プロンプトでどのスキルを選ぶか判定せよ」と依頼する。ただし実セッションでの確認を正とする。

### 検証シナリオ

1. raw taskとfixture、grader rubric、期待答えを別ファイルにする。実行エージェントへ渡すworkspaceにはtask packetだけを置き、スキルの開発リポジトリ・grader・過去出力を読ませない。
2. **baseline**: スキルを配置しない新セッションへtask packetを与え、同一モデル・tool policyで最低3回実行する。
3. **with-skill**: 対象スキルだけを配置した新セッションへ同じtask packetを与え、baselineと同じ回数実行する。
4. 両成果物をblind採点する。採点者はタスク実行とは**別のセッション**または人間とし、条件名を隠せる場合は隠す。実行したセッション自身に採点させない。
5. 得点だけでなくcritical失敗数、発火、token、latency、費用、tool errorを記録する。providerが実額を公開しない場合、費用は推測や0埋めをせず `null` とする。3回は大きな回帰検知の下限であり、有意差を主張する場合は分散に応じて反復数を増やす。

本リポジトリでは `scripts/eval_tasks.py prepare` が実行用とgrader用のパケットを別ディレクトリへ生成し、`scripts/eval_tasks.py score` がpaired結果を集計する。

### 記録

- `evals.md` の実行記録表には日付・モデル・条件・シナリオ・集約判定を追記し、反復ごとのcriterion、token、latency、cost、tool errorはリポジトリレベルの結果ファイルに保存する。
- **baseline が全条件を満たすシナリオは評価として機能していない。** そのシナリオは「素のモデルとの差分」を検出できていないので、欠落候補を見直して書き直す。

## 採否判定(スキル編集時)

- 編集後に全検証シナリオを再実行し、**critical失敗とtool errorが増えず、反復平均が改善した場合のみ**編集を採用する。部分実行は診断には使えても採用判定には使わない。合計点の増加でcritical悪化を相殺しない。発火改善とtask品質改善は別指標として報告し、悪化・同等なら差し戻す。
- **1回きりの失敗への対症編集は採用しない。** 実行記録やセッションで繰り返し観測された失敗パターンのみ本文に反映する(1回きりの修正は過適合)。
- 編集で評価も同時に書き換えない。評価の更新(シナリオ追加・合格条件の見直し)と本文の更新は別のコミット・別の判断として行う。
