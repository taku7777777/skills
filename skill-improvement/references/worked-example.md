# skill-improvement 適用例

題材: あるチームが自作した架空のスキル `sql-review`(SQL変更のレビュー観点を提供する)。「書いたが効いているのか分からない」という状態から、評価整備と改善を1周回す。

## 入力

ユーザーの依頼:「`sql-review` というスキルを作って半年使っているが、正直効いているのか分からない。評価して、必要なら改善したい。」

## 各ステップでの判断

- **ステップ0(モード判定)**: 既存スキルで評価が存在しない → モードB(評価整備)から入り、結果次第でモードC(改善反復)へ続ける。
- **ステップ2(評価セット作成)**: SKILL.md を読み、固有の価値を「インデックス影響の確認」「ロック時間の見積もり」「ロールバック可否の判定」と特定。worked-example の題材(ECの注文テーブル)を避け、検証シナリオは勤怠管理システムのマイグレーションSQLを題材に作成。発火すべきでないプロンプトには、隣接する一般コードレビュー(`impl-review` 相当)との境界「このPRをレビューして(SQL含まず)」を含めた。
- **ステップ3(ベースライン測定)**: スキルなしの新セッションで3シナリオを実行。S1(ALTER TABLE のレビュー)は baseline が合格条件5つ中4つを満たしてしまった → 検出力不足と判断し、S1 を「オンラインDDLが使えない古いDBバージョン制約つき」に難化して再測定(baseline 2/5 に低下、検出力を確認)。
- **ステップ5(評価実行と採否判定)**: with-skill 条件で実行。S1: 5/5、S2: 4/5、S3: 3/5。S3(暗黙の依頼「このマイグレーション大丈夫?」)で**発火せず**素の応答になっていたことが判明。
- **モードD(発火診断)へ分岐**: description が「SQLをレビューして」系の明示依頼しかカバーしていなかった。トリガー例に「このマイグレーション大丈夫?」「この ALTER、本番に流して平気?」を追加する修正案を作成。
- **ステップ5(再判定)**: description 修正後、S3 で発火し 5/5 に改善。厳密改善(発火テスト 4/5→5/5、S3 3/5→5/5、他は同等)のため**採用**。
- **ステップ6(継続運用)**: 過去半年の利用セッションから「ロック見積もりを毎回人間が補足していた」という繰り返しの失敗を蒸留し、編集候補として記録(次の反復で検証予定)。1回だけ起きた「特定ORMの方言の見落とし」は過適合リスクとして本文反映を見送り、実行記録のメモに留めた。

## 成果物(モードC/Dの出力例・抜粋)

### 評価結果比較表

| シナリオ | baseline | 編集前 with-skill | 編集後 with-skill |
|---|---|---|---|
| S1: 制約付き ALTER | 2/5 | 5/5 | 5/5 |
| S2: ロック時間見積もり | 1/5 | 4/5 | 4/5 |
| S3: 暗黙の依頼 | 2/5 | 3/5(不発火) | 5/5 |
| 発火テスト | — | 発火4/5・誤発火0/3 | 発火5/5・誤発火0/3 |

### 採否判定

**採用。** description へのトリガー例2件追加により S3 の不発火が解消し、他シナリオに悪化なし(厳密改善)。SKILL.md 本文は変更していないため、本文への編集候補「ロック見積もり手順の明文化」は次回反復で別途検証する。
