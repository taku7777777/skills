# 体系的デバッグスキル

David Agans『Debugging』(9つのルール)、Andreas Zeller『Why Programs Fail』(科学的デバッグ、デルタデバッギング)、git bisect(時間軸の二分探索)、Julia Evans(デバッグの心構えの言語化)、Brendan Gregg(USEメソッドによる性能分析)の知見を統合した、仮説駆動のデバッグ・障害調査プロセス。

修正後の回帰テストの設計は `test-design`、再発防止の仕組み化・監視は `ops-observability`、修正コードのレビューは `impl-review` の領分。本スキルの成果物は「根本原因のメカニズム説明を含む調査記録」である。

## 大原則

1. **推測より観測。** コードを睨んで犯人を当てるのではなく、証拠を集めて容疑者を絞る(Agans「考えるのをやめて見ろ」)。「たぶんここだろう」と感じた瞬間こそ、その感覚を仮説として書き出し、観測で検証する対象にする。読んだ行数ではなく、集めた観測事実の数が調査の進捗である。
2. **再現が最優先。** 再現できないバグは、直せたことも確認できない(Agans「失敗させろ」)。再現手順が得られたら次は最小化する — 入力・手順・環境を系統的に削るデルタデバッギング(Zeller)により、最小再現そのものが原因の候補を絞り、回帰テストの種になる。
3. **二分探索が最強の武器。** 容疑範囲を毎回半分にすれば、100万行でも20回で1行に到達する。空間軸(層・コンポーネントの境界で入出力を観測し、正常と異常の境目を探す)と時間軸(git bisect、「いつから壊れたか」— デプロイ・設定変更・データ変化の履歴)の両方に使える。
4. **一度に一つだけ変える。** 複数を同時に変えると、どれが効いたのか因果が消える(Agans)。実験は一時環境・一時worktree・明示したタスク所有の差分で行い、変更を1つに限定する。効かなかった変更は実験環境で元に戻すが、共有作業ツリーにあるユーザー所有の変更をstash・revert・破棄しない。
5. **仮説と実験結果を記録する。** 記録がないと同じ袋小路を何度も回る(Agans「監査証跡を残せ」)。仮説ログ(仮説/予測/実験/結果/判定)は調査の作業メモリであり、中断・交代・委任をまたいで調査を継続可能にする。棄却された仮説も消さない — 「何を否定済みか」が探索空間を狭める。
6. **直った理由を説明できないなら、直っていない**(Agans)。偶然消えたバグは必ず戻ってくる。症状の消失ではなく、「このメカニズムでこの症状が出ていた。だからこの修正で消える」という説明と、修正前に失敗し修正後に成功する回帰テストが「直った」の定義である。
7. **AI時代の補正: 観測は委任し、修正は説明でゲートする。** エージェントは大量のログ・コード・履歴を読む探索が得意なので、観測の収集(層別のログ突合、環境差分の列挙、bisect の実行)は並列委任し、人間は仮説の判断に集中する。委任時は仮説ログを成果物として必ず要求し、**「原因のメカニズム説明」を修正着手のゲートにする** — 説明なしに手当たり次第へ変更して「動いたから直った」とする shotgun 修正を許さない。

## ワークフロー

### ステップ1: 症状の固定

調査を始める前に、追うべき症状を動かぬ形で記録する:

- **期待と実際**: 何が起きるべきで、実際に何が起きたか。エラーメッセージ・スタックトレースは要約せず原文のまま記録する
- **発生条件**: いつから / どの環境で / どの頻度で / どの操作・ユーザー・データで。「常に」か「ときどき」かは調査方針を分ける
- **影響範囲**: 誰に・どの機能に・どの程度。本番障害では**復旧が調査に優先** — 切り戻し・フラグ無効化を先に検討し、ただし復旧操作の前に証拠(ログ、メトリクス、状態のスナップショット)を保全する
- **最近変わったもの**: デプロイ、設定変更、依存の更新、データ量・トラフィックの変化を列挙する(時間軸の切り分けの入力になる)

### ステップ2: 再現の確立と最小化

- 再現手順を確立し、**再現率を数える**(10回中10回か、100回中3回か)。間欠的なら、以降の実験の判定に反復回数を明記する(「N回中0回なら消えたと判定」)
- 再現手順はスクリプト化する。以降の全実験(切り分け・bisect・修正の検証)がこのスクリプトを判定器として回るため、再現の自動化は調査全体の速度を決める
- **最小化する**: 入力データ・操作手順・環境構成を半分ずつ削り、再現が維持される最小形を探す(デルタデバッギングの手動適用 → [references/debugging-heuristics.md](references/debugging-heuristics.md) §2.4)
- 再現しない場合は、再現しないバグへの対処カタログ([references/debugging-heuristics.md](references/debugging-heuristics.md) §4)へ — Heisenbug・タイミング依存・環境依存・状態依存のそれぞれに定石がある

### ステップ3: 切り分け

二分探索で容疑範囲を半分ずつ潰す(ヒューリスティック詳細: [references/debugging-heuristics.md](references/debugging-heuristics.md) §2):

- **層別(空間軸)**: フロント / API / ドメインロジック / DB / 外部依存の境界ごとに入出力を観測し、「正しいデータが入り、間違ったデータが出る」最初の層を特定する
- **時間軸**: 動いていた時点が特定できるなら git bisect(`git bisect run` で自動化)。コード以外の時間軸 — デプロイ履歴・設定変更・データ増加 — も同じ二分探索が効く
- **環境差分**: 再現する環境としない環境の差(バージョン・設定・データ・負荷)を表にし、一つずつ揃えて再現の有無を見る
- **データ差分**: 失敗する入力と成功する入力の差を最小化する
- **性能劣化**: USE メソッド(リソース×使用率/飽和/エラー)と RED メソッドで系統的に切り分ける([references/debugging-heuristics.md](references/debugging-heuristics.md) §5)

軸は排他ではなく、最も速く範囲を半分にできる軸を選ぶ。切り分けの目的は犯人の断定ではなく容疑範囲の縮小であり、十分に絞れたらステップ4の仮説検証に移る。独立な軸の調査(層別のログ突合と環境差分の列挙など)はエージェントに並列委任してよい — ただし持ち帰らせるのは観測事実と仮説ログであって「結論」ではない。

### ステップ4: 仮説検証ループ

- **仮説 → 予測 → 実験 → 観察**のループを回す(Zeller の科学的デバッグ)。仮説は反証可能な形で書き、**実験の前に予測を書く** — 結果を見てから「そう思っていた」は検証ではない
- すべてを仮説ログに記録する(テンプレート: [references/debugging-heuristics.md](references/debugging-heuristics.md) §3)。エージェントに調査を委任する場合も同じ形式で提出させる
- 実験は安価で判別力の高いものから。一度に一つだけ変える
- **仮説が3回連続で外れたら、前提を疑う**(Agans「電源プラグを確認しろ」): 見ているログは本当にそのリクエストのものか、動いているのは本当にそのコードか、その設定は本当に読まれているか。行き詰まったら人に説明する(説明の途中で矛盾に気づくことが多い)か、新鮮な目を入れる

### ステップ5: 根本原因の特定と修正

- **説明ゲート**: 「このメカニズムでこの症状が出る」を、発生条件・再現率・観測事実のすべてと整合する形で説明できて初めて修正に着手する。説明できない観測事実が残っているなら、原因は別にあるか複数ある
- **修正前に回帰テストを書き、失敗を確認する**。修正によってそのテストが成功に変わることが「直った」の証明(テスト設計の詳細は `test-design`)
- 修正は最小に。原因周辺の「ついで直し」は分離する(`refactoring-strategy` 大原則2)

### ステップ6: 再発防止

- **水平展開**: 同型のバグが他にないかを検索する — 同じ誤用パターン、同じ API の他の呼び出し箇所、同じ開発者・同じ時期のコード。原因のメカニズムが分かった今が、同型バグを最も安く見つけられる瞬間
- **検出の仕組み化**: 「なぜ今まで検出されなかったか」に答え、テスト・lint・監視・アラートのどこに検出点を足すかを決める(監視・アラートの設計と運用は `ops-observability` の領分)
- 調査記録(仮説ログ+根本原因の説明)を残す。次の同型調査で「何を疑い、何を否定したか」が再利用できる

## 成果物フォーマット

```markdown
# <症状> 調査記録
## 1. 症状(期待/実際/発生条件/影響範囲/最近の変更)
## 2. 再現手順(最小化済み、再現率)
## 3. 仮説ログ(番号 / 仮説 / 予測 / 実験 / 結果 / 判定)
## 4. 根本原因(メカニズムの説明 — 全観測事実との整合を確認)
## 5. 修正(回帰テスト / 修正内容 / 検証結果)
## 6. 再発防止(水平展開の結果 / 追加した検出の仕組み)
```

軽微なバグでは 1・3・4 の軽量版でよい(省略を明示)。実例: [references/worked-example.md](references/worked-example.md)
