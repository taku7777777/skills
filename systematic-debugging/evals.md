# systematic-debugging 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | CIでこのテストが3回に1回くらい落ちる。ローカルでは一度も落ちたことがない。何が起きてるのか見てほしい | 再現しない(間欠的な)不具合の調査 |
| F2 | 昨日の昼から「検索結果に古い価格が出る」という報告が複数来てる。どこがおかしいのか特定したい | 本番の不具合の原因特定(「バグの原因を調べて」と語彙が異なる) |
| F3 | 先週のリリース以降、一覧画面の表示が体感で3倍遅くなった。どの変更のせいか突き止めたい | 性能劣化の切り分け+時間軸の調査(いつから壊れたか) |
| F4 | このスタックトレースを貼るので、なんでここでNullになるのか追いかけてもらえる? | エラーの原因調査(仮説駆動で進めるべきケース) |
| F5 | ごく稀に同じ注文が2件登録されることがあるらしい。手がかりが少ないけど調査を始めたい | 低頻度事象の調査(再現の確立から始めるケース) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 先週みたいな障害が起きたら今度はすぐ気づけるように、検知の仕組みを整えたい | `ops-observability` が発火する(原因調査ではなく監視・検知の整備) |
| N2 | 障害のたびに切り戻しで1時間かかる。すぐ安全に戻せる仕組みを作っておきたい | `cicd-release` が発火する(復旧の仕組み=ロールバック設計) |
| N3 | このPRのコード、バグが入ってないか観点を持ってレビューして | `impl-review` が発火する(症状のない静的レビュー) |
| N4 | 場当たり修正が積み重なったこのモジュールを安全に整理し直したい | `refactoring-strategy` が発火する(挙動維持の構造改善) |

## 検証シナリオ

### S1: 承認ワークフローの二重承認バグ調査
- **タスク**: 「社内の稟議承認システムで、承認ボタンを押すと稀に承認レコードが2件できる。分かっていること: 直近30日で7件発生、全件が部長職以上のユーザー、うち5件は月曜朝に集中。アプリログには同一稟議IDに対する `POST /approvals` が約1.2秒差で2回記録されている(同一セッションID)。エラーログなし。3週間前に承認ボタンのUI刷新(モーダル化)と、承認APIのDB接続プール拡大を同日にリリースしている。原因を調査して修正方針まで出してほしい」
- **スキルなしで予想される欠落**: いきなり「二重送信防止を入れましょう」と対症修正に飛ぶ。仮説ログ(仮説/予測/実験/結果/判定、実験前に予測を書く)の欠落。再現の確立・再現率の計数を経ない断定。2つの同日リリース(UI刷新と接続プール)を切り分けずに扱う。根本原因のメカニズム説明が観測事実(部長職のみ・月曜朝集中)と整合するかの確認の欠落。
- **合格条件**:
  - [ ] 仮説ログ(仮説/予測/実験/結果/判定の列を持つ表またはそれに準ずる記録)が出力に含まれる
  - [ ] 各仮説に、実験より先に書かれた反証可能な予測がある
  - [ ] 再現の確立(再現手順と再現率の計数)が修正方針より前の手順として含まれる
  - [ ] 同日リリースされた2つの変更(UI刷新/接続プール)を切り分ける実験が設計されている
  - [ ] 提示された全観測事実(部長職のみ・月曜朝集中・1.2秒差・同一セッション)を説明できているか、説明できない事実を残課題として明示しているか(人間判定)

### S2: ログを足すと消える間欠的なクラッシュ
- **タスク**: 「組み込み向けデータ収集エージェント(マルチスレッドのデーモン)が、長時間稼働でごく稀にクラッシュする(直近1ヶ月で100台中3台、稼働3日目以降のみ)。調査のためにデバッグログを大量に足したビルドでは2週間再現していない。コアダンプは取得設定が漏れていて無い。どう調査を進めるべきか計画を立ててほしい」
- **スキルなしで予想される欠落**: 「ログを足して様子を見ましょう」という、観測が挙動を変える事実を無視した提案。Heisenbug・タイミング依存への定石(観測の軽量化、再現条件の増幅)の欠落。「N回中0回なら消えたと判定する」という間欠的バグの判定基準の欠落。証拠保全(コアダンプ設定)を先に仕込む発想の欠落。
- **合格条件**:
  - [ ] デバッグログ追加がタイミングを変えて再現を消している可能性(観測が挙動に影響する問題)への言及がある
  - [ ] 再現率を上げる・待ち時間を縮める工夫(負荷増幅、時間加速、対象条件の絞り込み等)が計画に含まれる
  - [ ] 「消えた/直った」の判定基準が反復回数つきで定義されている(例: N台×N日で0件なら)
  - [ ] 次回発生に備えた証拠保全(コアダンプ取得設定、状態スナップショット等)が先行タスクとして含まれる
  - [ ] 調査の記録(仮説ログ等)を残す段取りがある

### S3: 当たりをつけた修正への途中介入
- **タスク**: 「レポート画面がタイムアウトする件、たぶんキャッシュが効いてないのが原因だと思ったので、キャッシュTTLを10分から60分に延ばして、ついでにDBのコネクション数も倍にする修正を書いた。これで出していい?」と依頼する。あわせて事実として「タイムアウトは全ユーザーでなく、データ量の多い上位1割のテナントのみで発生」「キャッシュヒット率のメトリクスは取得していない」を伝える。
- **スキルなしで予想される欠落**: 修正内容の妥当性レビューだけして通す。「原因のメカニズム説明」を修正のゲートにする規律の欠落。複数の変更(TTLとコネクション数)を同時に入れると因果が消える問題の指摘漏れ。「特定テナントのみ」という観測事実がキャッシュ仮説で説明されていないことの指摘漏れ。
- **合格条件**:
  - [ ] このまま出すことを承認せず、原因のメカニズム説明(またはそれを裏づける観測)を先に求めている
  - [ ] 「上位1割のテナントのみ」という観測事実が現仮説で説明できていないことを指摘している
  - [ ] キャッシュ仮説を検証する具体的な観測手段(ヒット率計測、該当テナントのクエリ実測等)を提案している
  - [ ] 2つの変更を同時に適用すると効果の因果が分からなくなることを指摘している
  - [ ] 「変えたら直った」ではなく、修正前に失敗し修正後に成功する検証(回帰テストまたは再現手順での確認)を求めている
