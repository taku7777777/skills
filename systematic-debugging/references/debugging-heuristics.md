# デバッグヒューリスティック集

SKILL.md のワークフローから参照される実務詳細。「切り分けの手筋」「仮説ログの型」「再現しないバグの定石」「性能問題の系統的切り分け」を収録する。

## 1. Agans 9つのルールの実務展開

| ルール | 実務での意味 | よくある違反 |
|---|---|---|
| 1. システムを理解する | 触る前にドキュメント・API契約・アーキ図を読む。「そもそもどう動くはずか」を知らないと、異常を異常と認識できない | マニュアルを読まずに「たぶんこう動く」で調査を始める |
| 2. 失敗させる | 意図的に・繰り返し再現できる状態を作る。再現手順は自動化する(スクリプト化すれば実験の反復が安くなる) | 「一度起きたが再現しない」を放置して修正に進む |
| 3. 考えるのをやめて見る | 推理する前に観測する。仮説はログ・デバッガ・メトリクスで確認してから信じる | コードリーディングだけで犯人を「確信」し、確認せず修正する |
| 4. 分割統治 | 正常と異常の境目を二分探索で挟み込む。範囲を半分にする一手を常に探す | 端から順に全部読む(線形探索) |
| 5. 一度に一つ変える | 実験ごとに変更は1つ。効かなかった変更は戻す | 「怪しい箇所を3つ直したら消えた」— どれが原因か永遠に不明 |
| 6. 監査証跡を残す | 何を試し、何を観測したかを時系列で記録する(§3 の仮説ログ) | 記憶頼みで同じ実験を繰り返す。「昨日何を試したっけ」 |
| 7. 電源プラグを確認する | 前提そのものを疑う: 正しい環境か、正しいビルドか、設定は読まれているか、ログは本当にそのプロセスのものか | 高度な仮説を積み上げた末に「デプロイされていなかった」 |
| 8. 新鮮な目を入れる | 人に説明する(説明の途中で自分の矛盾に気づく)。先入観のない人・専門家に見せる | 一人で同じ仮説の周りを何時間も回る |
| 9. 直していないなら直っていない | 修正の前後で同じ再現手順を実行し、失敗→成功の変化を確認する。説明できない消失を「直った」と呼ばない | 「最近見なくなったのでクローズ」 |

## 2. 切り分けヒューリスティック

### 2.1 層別(空間軸の二分探索)

1. データの通り道を層として書き出す(例: ブラウザ → API → アプリケーションロジック → ORM/SQL → DB → 外部API)
2. 中間点の層境界で入出力を観測する: 「この層に入るデータは正しいか。出るデータは正しいか」
3. 「正しく入って間違って出る」最初の層が容疑者。以降はその層の内部で同じ二分探索を繰り返す
4. 観測点がない層境界は、まず観測手段(ログ・デバッガ・プロキシ・クエリログ)を確保する — 観測手段の追加はデバッグ作業の一部であり、回り道ではない

補助手筋: 疑わしい要素を**引き算**する(キャッシュを無効化、並列度を1に、対象プラグインを無効化)。引いて症状が消えれば、その要素が関与している。

### 2.2 時間差分(git bisect の実務)

「昔は動いていた」なら、履歴の二分探索が層別より速いことが多い:

```bash
git bisect start
git bisect bad HEAD                # 壊れている点
git bisect good v1.4.0             # 動いていたと確認済みの点
git bisect run ./repro.sh          # 自動判定(exit 0=good, 1〜127=bad, 125=判定不能スキップ)
```

- **good 側は「動いていたはず」でなく「動いていたと確認した」点にする**。good が誤っていると bisect 全体が嘘をつく
- 判定スクリプトは再現の自動化(§1 ルール2)そのもの。間欠バグでは判定内で N 回反復しないと誤判定が混入する
- ビルドできないコミットは exit 125 でスキップ
- 特定したコミットは「原因」ではなく「引き金」のことがある(潜在バグを露出させた無実の変更)。コミット特定後も §4 の説明ゲートは省略しない
- **コード以外の時間軸にも同じ発想を使う**: デプロイ履歴、設定変更履歴、依存バージョン、データ量。「最後に正常だった時刻」と「最初に異常だった時刻」の間にある変更を全て列挙し、二分できるなら二分する

### 2.3 環境差分

再現する環境と再現しない環境があるなら、差分そのものが手がかり:

1. 差分を表に列挙する: OS / ランタイムバージョン / 依存バージョン / 設定値 / 環境変数 / データ / 負荷・並行度 / ネットワーク構成 / タイムゾーン・ロケール
2. 一度に一つずつ揃えて(または入れ替えて)再現の有無を観測する — 再現状態が反転した差分が容疑者
3. 「環境は同じはず」は仮説であって事実ではない。実測で比較する(バージョンは表示コマンドの出力で、設定は実効値で)

### 2.4 データ差分(デルタデバッギングの手動適用)

失敗する入力から、失敗に必要な最小部分を系統的に炙り出す(Zeller の ddmin の実務簡略版):

1. 失敗する入力を半分に割り、各半分で再現を試す
2. 再現する側を新たな全体として繰り返す
3. どちらの半分でも再現しなければ、削る単位を細かくする(1/4 ずつ削る)。それでも割れなければ複数要素の組み合わせが原因 — 片側を固定してもう片側を削る
4. 入力だけでなく、操作手順・設定・データベースの行に対しても同じ手順が使える

最小化の産物は (a) 原因の候補の絞り込み(残った要素が本質)、(b) 高速な再現手順(実験のループが速くなる)、(c) 回帰テストの入力、の三役を兼ねる。

## 3. 仮説ログのテンプレート

```markdown
| # | 仮説(反証可能な形で) | 予測(実験前に書く) | 実験 | 結果 | 判定 |
|---|---|---|---|---|---|
| 1 | 6/30 のデプロイが原因 | 直前コミットに戻せば再現しない | v2.3.0 で再現手順を実行 | 再現した | 棄却 |
| 2 | キャッシュの不整合 | キャッシュ削除で症状が消える | 対象キーを削除して再実行 | 症状消えず | 棄却 |
| 3 | ... | ... | ... | ... | 支持(追加検証へ) |
```

運用ルール:

- **予測を実験の前に書く**。結果を見てから予測を書くと、どんな結果も「想定どおり」になる
- 判定は「支持/棄却/判定不能」の3値。支持は確定ではない — 同じ仮説から別の予測を立てて追試する
- 棄却された仮説も削除しない。否定済み空間の記録が、堂々巡りと再委任時の重複調査を防ぐ
- エージェントへの委任時は、このテンプレートでの提出を成果物条件にする。「結論だけ」の報告は、根拠の検証も引き継ぎもできない

## 4. 再現しないバグへの対処カタログ

| 型 | 兆候 | 定石 |
|---|---|---|
| Heisenbug(観測が挙動を変える) | デバッガを繋ぐ・ログを足すと消える | 観測の侵襲を下げる: ブレークポイントではなくカウンタ・リングバッファへの記録、最適化設定を本番と揃える。実行の記録リプレイ(record & replay)ツールが使える環境なら、一度の捕獲で何度でも再生できる |
| タイミング依存(競合・非同期順序) | 負荷が高いときだけ、まれに、CI でだけ失敗する | 発生確率を上げる: 負荷・並列度を上げる、疑わしい区間に人工遅延を注入して競合ウィンドウを拡大する、N 回反復ループで統計的に再現する。データ競合はサニタイザ(ThreadSanitizer 等)で決定的に検出できることがある。決定的再現の設計は `test-design` の同時実行テストを参照 |
| 環境依存 | 特定のマシン・OS・リージョンでだけ発生 | §2.3 の環境差分表。再現する環境を保全する(「とりあえず再起動」は証拠隠滅)。コンテナ化して環境ごと手元に持ってくる |
| データ依存 | 特定のユーザー・レコードでだけ発生 | 該当データを(マスキングして)取り寄せ §2.4 で最小化。現実データの定番の特異値を疑う: NULL・空文字、絵文字・サロゲートペア、タイムゾーン境界、うるう日、極端な桁数、移行由来の不整合データ |
| 状態依存・履歴依存 | 長時間稼働後・特定の操作順の後にだけ発生 | 蓄積を疑う: メモリリーク、コネクション・ファイルディスクリプタの枯渇、キャッシュ汚染、カウンタ溢れ。稼働時間を圧縮する実験(リソース上限を意図的に小さくして枯渇を早送りする)で再現を引き寄せる |
| 発生率が低すぎて待てない | 本番で月に数回だけ | **待ち構える**: 次の発生で原因が確定するだけの観測(コンテキスト付きログ、当該経路のトレース、失敗時のダンプ)を仕込んでリリースし、発生を捕獲する。仕込みの前に「次に起きたら何が知りたいか」を仮説ログから逆算する |

どうしても再現できない場合の次善策: 観測事実と整合する原因候補を仮説ログ上で絞り込み、最有力候補への防御的修正+**修正が効いたことを検出するログ**を同時に入れる(発生率の変化で修正の当否を事後判定する)。これは説明ゲートの免除ではなく、判定の後払いである。

## 5. 性能問題の切り分け(USE / RED)

性能劣化は「遅い」という症状のまま調査しない。どこの・何の資源が・いつから、に分解する。

### 手順

1. **いつから**: メトリクスの時系列で劣化開始点を特定し、§2.2 の時間軸(デプロイ・設定・データ量・トラフィックの変化)と突き合わせる
2. **どこで**: リクエストの経路上のどの区間が遅いか(分散トレース、各層のレイテンシ内訳)。§2.1 の層別と同じ二分探索
3. **何の資源か**: 特定した区間のホスト・依存に USE メソッドを適用する

### USE メソッド(Brendan Gregg)

すべてのリソースについて、**Utilization(使用率)/ Saturation(飽和 — キュー待ちの発生)/ Errors(エラー)** を順に確認するチェックリスト方式。勘の良い箇所からつまみ食いせず、リソース一覧を上から機械的に潰すのが要点:

- ハードウェア資源: CPU、メモリ、ストレージ I/O、ネットワーク
- ソフトウェア資源(見落としやすい): コネクションプール、スレッドプール、ロック、キュー、ファイルディスクリプタ、セマフォ
- 使用率が低くても飽和は起こる(バーストがキューを作る)。平均使用率だけで無罪にしない

### RED メソッド(サービス視点)

リクエスト駆動のサービス・エンドポイントごとに **Rate(流量)/ Errors(エラー率)/ Duration(所要時間の分布)** を見る。USE が「資源の側」からの視点、RED が「利用者の側」からの視点で、両方から挟むと齟齬(利用者は遅いのに資源は暇 = 待ち合わせ・ロック・外部依存)が見える。

### よくある犯人と落とし穴

- 定番の犯人: N+1 クエリ、インデックスが効かなくなったクエリ(データ量閾値超えで実行計画が変わる)、キャッシュヒット率の低下、GC 停止、ロック競合、リトライストーム、コネクションプール枯渇
- **平均ではなくパーセンタイル(p95/p99)で見る**。平均は少数の激遅リクエストを隠す
- 測定自体のオーバーヘッド(詳細プロファイラを本番に常時)で症状が変わることがある — Heisenbug と同じ扱いで侵襲を管理する
- ベンチマーク環境での再現は負荷の「形」(同時実行数、データ分布、キャッシュの温まり)まで揃えないと別の問題を測ることになる
