# systematic-debugging 適用例

## 題材

架空の SaaS の月次レポート機能。障害報告:「月初の朝、先月分のレポートを開くと 0 件で表示されることがある。昼には直っている」。

## 入力

- 障害チケット: 7/1 の 08:12 に複数テナントから「6月分レポートが空」。サポートが 09:30 に確認した時点では正常表示
- 過去チケット検索で、6/1 にも同内容の問い合わせが 1 件(当時は「再現せず」でクローズ)

## ステップ1: 症状の固定

- 期待: 7/1 に 6月分レポートを開くと 6月の集計が表示される。実際: 0 件表示(エラーなし、正常応答で空)
- 発生条件をログで確認: 7/1 の 00:30〜09:00(JST)の該当 API 応答がすべて 0 件。**09:00 を境に正常化** — この境界値が最重要の観測事実として記録される
- 影響範囲: 全テナント。過去の月初にも同様のログパターンあり(= 今月のデプロイ起因ではない疑い)
- 復旧判断: すでに自然回復しており切り戻し不要。代わりに 7/1 未明のバッチログとキャッシュテーブルのスナップショットを証拠として保全
- 最近の変更: 6/28 に集計サービスのデプロイあり(容疑者として記録)

## ステップ2: 再現の確立と最小化

- 「月初の 09:00 前」という発生条件から、時刻依存を疑って時刻の固定から試す(発生条件の記録が再現手段の候補を直接教えてくれる)
- テスト環境でシステム時刻を「7/1 08:00 JST」に固定 → UI から再現(10/10)。時刻を「7/1 09:30 JST」にすると再現しない(0/10)— 決定的に再現でき、以降の実験は反復回数の考慮が不要になった
- 最小化: UI を外し `GET /reports/monthly?tenant=t1` の直接呼び出しで再現 — フロントは無関係。再現スクリプト化して以降の実験を自動化

## ステップ3: 切り分け(層別)

- API → レポートサービス → レポートキャッシュテーブル → 集計バッチ、の経路を層として観測
- API はキャッシュテーブルをそのまま返している(入: 0 行、出: 0 件)→ API は無罪
- 生データのテーブルには 6月分のレコードが存在する → データ欠損ではなく、**キャッシュテーブルに 6月分の行がない**ことが問題。容疑はキャッシュを生成する側(深夜バッチ)に絞られた

## ステップ4: 仮説検証ループ(仮説ログの実例)

| # | 仮説 | 予測(実験前に記述) | 実験 | 結果 | 判定 |
|---|---|---|---|---|---|
| 1 | 6/28 デプロイが原因 | デプロイ前のバージョンなら再現しない | 前バージョンで時刻固定再現 | 再現した。過去月初のログとも整合 | 棄却 |
| 2 | バッチが失敗している | 7/1 未明のバッチ実行ログにエラーがある | バッチログ確認 | 正常終了。ただし出力対象月が「2026-05」 | 棄却(ただし新事実) |
| 3 | バッチの対象月計算が UTC 依存(JST 7/1 00:30 = UTC 6/30 のため「先月」が 5月になる) | (a) コードの対象月計算が UTC 現在日時を使っている (b) 時刻を UTC 7/1 00:30 相当にすれば正しく 6月分を生成 | コード確認+時刻を変えてバッチ実行 | (a) 該当箇所を確認 (b) 予測どおり | 支持 |

- 仮説3 は「09:00 に自然回復する」観測事実とも整合する: JST 09:00 = UTC 00:00 で日付が 7/1 になり、毎時のリカバリジョブが正しい対象月(6月)を計算し直すため
- 全観測事実(月初のみ / 00:30〜09:00 のみ / 全テナント / 何もしなくても回復 / 今月のデプロイと無関係)がこの一つのメカニズムで説明できる → 説明ゲート通過

## ステップ5: 根本原因の特定と修正

- 根本原因: 深夜バッチが「先月」を UTC の現在日時から計算している。JST の月初 00:00〜09:00 は UTC ではまだ前月末のため、対象月が 2 か月前にずれ、当月に見るべきキャッシュが生成されない
- 回帰テストを先に作成: 時刻を注入可能にし、「JST 7/1 00:30 相当で実行したとき対象月が 2026-06 になる」を検証 → 修正前に失敗することを確認
- 修正: 対象月の計算をテナントのタイムゾーン基準に変更。回帰テストが成功に変わり、時刻固定の再現スクリプトも 0/10 に

## ステップ6: 再発防止

- 水平展開: 「UTC 現在日時から日付境界を計算している箇所」をコードベース全体で検索 → 週次レポートに同型バグを発見し、同じ修正と回帰テストを適用
- 検出の仕組み化: 「月初バッチの生成行数が 0 のとき警報」の監視を追加項目として `ops-observability` へ引き継ぎ。時刻注入をテスト標準にする件は `test-design` へ
