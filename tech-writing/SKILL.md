# 技術発信(登壇・記事)スキル

Nancy Duarte『Resonate』、Garr Reynolds『プレゼンテーションZen』、Google Technical Writing Course、Simon Willison(技術ブログの実践)、Julia Evans(伝わる技術解説)の知見を統合した、技術発信の構成・執筆プロセス。

## 大原則

1. **1発表・1記事 = 1メッセージ。** 聴衆が翌日他人に一文で語れるもの(ビッグアイデア)を最初に確定する。それに貢献しないコンテンツは(どんなに面白くても)捨てる。
2. **経験と検証がすべての差別化。** 一般論の要約はAIで誰でも書ける時代、価値は「実際にやった・測った・失敗した」一次情報にのみ宿る。主張には必ず実例・実測・実コードを添え、留保(条件、再現範囲、反証可能性)を正直に書く — 正直な留保は信頼を毀損せず、むしろ蓄積する。
3. **聴衆は「何を知っているか」ではなく「何に困っているか」で分析する。** 聴衆の現在地(Duarteの"What is")から理想("What could be")への往復がストーリーの骨格。
4. **構造が先、スライド/文章は後。** いきなりスライドツールや本文を書き始めない。メッセージ→構成→肉付けの順。
5. **発信は資産である。** 一回の登壇を、記事・英語転用・シリーズ化・次の登壇へ再利用できる形で設計する(検証済みコンテンツの多重利用)。

## ワークフロー(登壇)

### ステップ1: 戦略の確定

- **ビッグアイデア**を1文で: 「<聴衆>が<行動/認識変化>すべきだ、なぜなら<自分だけが提供できる根拠>」
- 聴衆分析: 経験レベル、既に信じていること、抱えている痛み、この発表への期待
- 自分だけが語れる角度は何か(実組織での適用、実測データ、失敗経験、独自の統合視点)。**他の人が話せる内容なら角度を変える**
- 構成に入る前に、素材となる一次情報をユーザーから聞き出す: 実際に何をやったか / 何を測りどんな数字が出たか / 何が失敗・想定外だったか / 聴衆の常識に反する発見は何か。回答は「一文の学び+証拠」の形で棚卸しし、以降の構成はこの棚卸しにある素材だけで組む

### ステップ2: 構成設計(スライドを作る前に)

Duarteのスパークライン(現状⇄理想の往復)をベースに:

```
1. フック: 聴衆の痛みの言語化(「あるある」で信頼獲得)or 意外な事実
2. 現状(What is): 問題の構造化 — なぜ既存のやり方では苦しいのか
3. 転換(What could be): 中心アイデアの提示
4. 本体: 実例・実測・実装で証明(3点まで。ここが一次情報の見せ場)
   - 各点に「うまくいったこと」と「うまくいかなかったこと」を対で入れる
5. 現実的な適用: 聴衆が明日試せる最初の一歩(制約・向き不向きも正直に)
6. New Bliss: 採用後の世界像+行動喚起
```

- 構成段階でユーザーとレビューし、合意してからスライド化する
- 時間配分: 1スライド1分を目安に上限枚数を決め、本体(実例)に50%以上を配分
- パートごとの時間配分の目安と、時間尺(LT/20分/40分)による構成の違いは `references/structure-templates.md` の登壇テンプレートに従う

### ステップ3: スライド化

- 成果物はスライドドラフト(Markdown): スライドごとに「主張文タイトル / 視覚的証拠の指定(図の構造・表の中身・抜粋コードをテキストで具体化)/ 発表者ノート」の3点セットで書く。ユーザーの使用ツール(Marp、Google Slides、Keynote等)を最初に確認し、直接取り込める形式があればそれで出力する
- **1スライド1主張**。スライドタイトルは体言止めのラベルではなく主張文にし、本体はそれを支える視覚的証拠にする(「テスト戦略」→「テストはAIへの報酬関数である」— assertion-evidence構造、Michael Alley)。タイトルだけ拾い読みしてもストーリーが通り、各スライドは3秒で要旨が取れること(グランステスト、Duarte)
- 図解優先: 構造・流れ・比較は図・表に。コードは見せたい行だけ抜粋しハイライト
- 発表者ノートに口頭で補う内容を書き、スライドは信号(記憶の錨)に徹する
- 自己紹介は30秒以内。実績列挙より「なぜ自分がこのテーマを話す資格があるか」1点
- 実例: `references/worked-example.md`(戦略確定からスライドドラフトまでの適用例)

### ステップ4: リハーサル観点でのセルフレビュー

- [ ] ビッグアイデアに貢献しないスライドはないか(愛着があっても削る)
- [ ] 聴衆が置いていかれる箇所はないか(前提知識のジャンプ、初出の略語)
- [ ] 「で、明日何をすればいい?」に答えているか
- [ ] 数字・事実は出典を確認したか(発表は記録に残る — 誤りは信頼の負債)
- [ ] 声に出した通しで時間超過していないか(超過時に削るスライドを事前に指名しておく)
- [ ] 想定される反論・質問トップ3に回答を用意したか
- [ ] ライブデモには録画/スクリーンショットのフォールバックがあるか

## ワークフロー(CFP・プロポーザル)

- **読者はレビュアー**: 数百件を数分ずつ読む人に、「この発表でしか聞けないものは何か」が30秒で伝わるか
- 構成: 聴衆の痛み → 自分の一次情報(実測・実適用・失敗)→ 持ち帰れるもの3点 → なぜ自分が話せるか(実績1点)
- タイトルは検索結果・タイムテーブル上で単独で戦う: 具体的な成果・数字・意外性を1つ入れる
- 落選に備え、同じ素材の記事化パスを最初から設計しておく(発信資産の運用に接続)
- 記入テンプレートと採択されやすい構成の要点は `references/structure-templates.md` のCFP節を使う

## ワークフロー(記事)

### ステップ1: 読者と約束の確定

- 想定読者1人(ペルソナ)と、読了後に得られるもの(約束)を決める
- リードで「その問題を放置すると読者に何のコストがあるか」を明示する(McEnerney: 文章の価値は新規性ではなく読者にとっての重要性が決める)
- タイトルは約束の明文化: 内容を過大にも過小にも表現しない。釣りタイトルは一回の閲覧と引き換えに信頼を失う
- 本文で使う一次情報(実測値、動くコード、失敗談)をこの段階でユーザーから聞き出して棚卸しする。棚卸しにない素材は本文に書かない

### ステップ2: 構成

```
1. リード(3〜5文): 誰向けか / どんな問題を扱うか / 読むと何が得られるか / 結論の要約
2. 背景・問題の構造化(最小限 — 読者の現在地との接続)
3. 本体: 実例・手順・実測(コード/設定は動く完全形を。断片は再現できない)
4. うまくいかなかったこと・制約・適用条件(このセクションが信頼を作る)
5. まとめ: 要点の再掲 + 次の一歩 + 関連リソース
```

- 記事タイプ(経験談・技術解説・チュートリアル・調査まとめ)ごとのアウトラインと冒頭の約束の書き方は `references/structure-templates.md` から選ぶ
- 見出しだけ読んでも流れが分かるように(スキャン読者が大半)
- 読者には人間だけでなくAI(検索・要約エージェント)が含まれる。要点の主張は、段落から一文だけ切り出されても誤読されない自己完結の文にする(主語・適用条件・時点を文内に含める)。固有の主張・実測値には自分の名前・日付・検証条件を近接して書く — 引用・再配布されたときに帰属と留保が一緒に運ばれる
- アウトラインをユーザーとレビューし、合意してから執筆に入る(登壇ワークフローと同じゲート)

### ステップ3: 執筆

- セクション単位で書き、各セクションがステップ1の棚卸しのどの一次情報に対応するかを添えて提示する。対応する素材なしに書けてしまったセクションは一般論の疑いがあるため、その旨を明示してユーザーの判断を仰ぐ
- リードは本文完成後に書き直す(書く前の想定と書き上がった結論はしばしばずれる)
- 分量の目安をユーザーと確認する(読了時間の想定を超えるなら分割・シリーズ化を提案)
- Google Tech Writing 原則で推敲: 1文1アイデア、能動態、具体的な動詞、用語の一貫性(同じものを同じ名前で呼ぶ)

### ステップ4: 公開前チェック

確認手順つきの詳細版は `references/structure-templates.md` の公開前チェックリストを使う。要点:

- [ ] コード・コマンドは記事の記載どおりに実行して再現するか(バージョン明記)
- [ ] 機密情報・社内情報の混入はないか(スクリーンショット内も)
- [ ] 主張と根拠の対応: 「〜と思う」(意見)と「〜である」(検証済み)が区別されているか
- [ ] 3ヶ月後に古くなる箇所に日付・バージョンの目印があるか
- [ ] AIが下書きした箇所の事実(数値・引用・手順)を著者自身で検証したか
- [ ] 掲載先のAI利用ポリシー(開示要否)を確認したか
- [ ] (Zennの場合)topicsは検索される正式名称か。slugは公開後に変えない

## AIを執筆補助に使う際の規律(このスキル自身への制約)

このスキルの実行者(Claude)は執筆補助であり、一次情報の生成者ではない。以下を厳守する:

- **体験・数値・引用を創作しない。** ユーザーから提供されていない実測値・事例・経緯・判断理由で本文を埋めない。不足箇所は `[要確認: 実測値]` `[要確認: 当時の判断理由]` のプレースホルダとして明示し、埋めるための質問をユーザーに返す
- **もっともらしい一般論で行間を埋めない。** 一次情報が薄いセクションは「薄い」と指摘するのが仕事であり、AIの背景知識で厚く見せることではない
- **出典はすべて実在確認。** 書名・記事・仕様・統計の引用は、ユーザー提供の情報か、その場で実物を確認できたものに限る。記憶ベースの引用には `[出典未確認]` を付けて残す
- **公開責任は著者にある。** AI補助の有無にかかわらず、公開物の全主張は著者本人が検証し全責任を負う(Simon Willisonの公開原則)。公開前チェックのAI関連項目(AI下書き箇所の事実検証、掲載先のAI利用ポリシー確認)で担保する
- **著者の声を保持する。** 執筆前にユーザーの過去記事・登壇資料を1〜2本読み、語彙・文のリズム・一人称・断定の強さを把握して合わせる。推敲時に個性的な言い回しや思考の癖を「無難な標準文」に均さない。迷いや判断の過程が表れている文は著者性の証拠なので残す
- **AI定型文体を避ける。** 過剰な箇条書き化、「〜ではなく〜である」構文の反復、意味の薄い前置きとまとめの水増し、全段落が同じ長さになる均質なリズムは、内容以前に「AI量産記事」の信号として読者に弾かれる。推敲の最終パスでこれらを能動的に除去する

## 発信資産の運用

- **ネタ帳の構造化**: 日々の開発での発見(実験結果、失敗、意外な挙動)を「一文の学び+証拠」の形で記録 → 溜まったらクラスタリングして記事・登壇テーマに昇格
- **多重利用の設計**: 登壇 → 補足つき記事化(登壇では話せなかった詳細)→ データ部分の英語転用、の変換パイプラインを前提に一次情報を整理しておく
- 反応(質問・誤解された点)は次のコンテンツの種として記録する
