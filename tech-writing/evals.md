# tech-writing 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 来月の社内勉強会で30分もらった。DB マイグレーション地獄を乗り越えた話をしたいが、何から手をつければいい? | 登壇の構成設計(戦略の確定から) |
| F2 | 半年かけたパフォーマンス改善の取り組み、社外に出せる形で残しておきたいんだよね | 記事化(経験の一次情報の棚卸し) |
| F3 | カンファレンスの応募締切が来週。ネタはあるのに応募文がまとまらない | CFP・プロポーザル作成 |
| F4 | ブログの下書きがあるんだけど、読み返すとなんかダラダラしてる。締まった文章にしてほしい | 記事の推敲(レビュー) |
| F5 | 開発中の気づきをメモに溜めてるのに、発信につながった試しがない。整理の仕方を変えたい | 発信ネタの構造化(ネタ帳運用) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | 新メンバー向けのオンボーディング文書を作りたい | `docs-for-ai` が発火する(社内・エージェント向け文書体系) |
| N2 | 自分のキャリア戦略メモ、四半期ぶんの動向を反映して見直したい | `quarterly-strategy-review` が発火する(予測文書の定期更新) |
| N3 | 新機能リリースのプレスリリース文面を考えて | スキルなしで応答する(広報文書は技術登壇・技術記事の対象外) |

## 検証シナリオ

### S1: N+1 解消経験の記事化
- **タスク**: 「社内 API の N+1 問題を潰してレスポンスを改善した経験を Zenn 記事にしたい。実測値は p95 レイテンシ 1.8秒→0.4秒 だけ手元にある。ほかの詳細は聞かれれば答える。書き始めて」と依頼する(worked-example の flaky テスト登壇とは別題材)。
- **スキルなしで予想される欠落**: 想定読者と約束の確定を飛ばした本文執筆、一次情報の棚卸しなしの執筆開始、提供されていない数値・経緯のもっともらしい創作、アウトライン合意ゲートの省略、失敗・制約セクションの欠如。
- **合格条件**:
  - [ ] 執筆前に想定読者(ペルソナ)と読了後に得られるもの(約束)を確定するプロセスがある
  - [ ] 一次情報の棚卸し(追加ヒアリング)を行い、提供されていない数値・経緯を創作していない(不足箇所は `[要確認]` 等で明示)
  - [ ] アウトラインをユーザーと合意してから本文執筆に進む段取りになっている
  - [ ] 構成に「うまくいかなかったこと・制約・適用条件」に相当するセクションが含まれる

### S2: 一次情報ゼロの依頼(スキルの規律が試される境界)
- **タスク**: 「最近話題の(架空の)新ビルドツール、まだ自分では触ったことがないんだけど、注目度が高いうちに入門記事を書いてバズらせたい。中身は AI でいい感じに埋めて」と依頼する。
- **スキルなしで予想される欠落**: 「経験と検証がすべての差別化」の原則と体験創作の禁止。素のモデルは背景知識でそれらしい入門記事を丸ごと生成する。
- **合格条件**:
  - [ ] 一次情報がないまま書くことへの問題提起(差別化できない・信頼リスク)が最初にある
  - [ ] 体験談・検証結果・実測値を創作していない
  - [ ] 書く前にまず素材を作る提案(実際に触って検証する、検証ログを取る等)がある
  - [ ] それでも書く場合の誠実な代替(調査まとめであることの明示等)が提示される

### S3: 構造を飛ばしたスライド即製依頼(変化球)
- **タスク**: 「明後日 LT なんだけど時間がない。『テストを書く文化の作り方』ってタイトルだけ決まってる。Marp で10枚、いい感じに作っといて」と依頼する。
- **スキルなしで予想される欠落**: ビッグアイデア未確定のままのスライド生成、体言止めのラベルだけのスライドタイトル、ユーザーから聞いていない社内事例・数値による肉付け。
- **合格条件**:
  - [ ] スライド生成の前にビッグアイデア(聴衆が一文で語れるメッセージ)と一次情報の確認を挟んでいる
  - [ ] ユーザーから提供されていない社内事例・数値を創作していない
  - [ ] 生成したスライドのタイトルが体言止めのラベルではなく主張文になっている
  - [ ] LT という時間尺に応じた構成(1メッセージへの絞り込み・枚数の根拠)が示される
  - [ ] タイトルだけ拾い読みしてもストーリーが通る(人間判定)
