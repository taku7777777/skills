# 構成テンプレート集(登壇・記事・CFP・公開前チェック)

SKILL.md のワークフローを詳細展開したテンプレート集。使い方: 該当する型を選び、穴埋めの前に必ずステップ1(ビッグアイデア・一次情報の棚卸し)を終えていること。**棚卸しなしにテンプレートを埋めると、形だけ整った一般論が量産される** — テンプレートは構成の道具であって内容の代用ではない。

## 目次

- 登壇構成テンプレート
- 記事タイプ別アウトライン
- CFPテンプレート
- 公開前チェックリスト

## 1. 登壇構成テンプレート(スパークライン)

**Duarte のスパークラインは「現状(What is)」と「あるべき姿(What could be)」の往復運動である。** 聴衆は現状の描写で「自分の話だ」と感じ、あるべき姿の提示で緊張(ギャップ)を感じる。この往復を本体の実例ごとに繰り返し、最後に「New Bliss(採用後の世界像)」で着地する。一方向に説明を積み上げる構成(背景→手法→結果→まとめ)との最大の違いは、**聴衆の感情が平坦にならない**こと。

### 基本テンプレート(20分登壇)

| # | パート | 時間目安 | 中身 |
|---|---|---|---|
| 1 | フック | 1〜2分 | 聴衆の痛みの言語化(「あるある」)or 意外な事実。自己紹介は30秒以内 |
| 2 | 現状(What is) | 3分 | 問題の構造化。なぜ既存のやり方では苦しいのかを聴衆の言葉で |
| 3 | 転換(What could be) | 1分 | 中心アイデアの提示。ビッグアイデアを一文で言い切る |
| 4 | 本体(証明) | 10分 | 実例・実測・実装で2点(各5分)。**各点の内部でも「やる前の現状→やった後」の小さな往復を作る**。うまくいかなかったことを対で入れる |
| 5 | 現実的な適用 | 2分 | 聴衆が明日試せる最初の一歩。制約・向き不向きも正直に |
| 6 | New Bliss | 1分 | 採用後の世界像+行動喚起。「まとめ」スライドの箇条書き羅列で終わらない |

- 残り1〜2分はバッファ(転換の待ち時間、笑いの間、機材トラブル)。**質疑応答の時間が枠に含まれるかを最初に主催者へ確認する**
- 1スライド1分を目安に上限枚数を決める(20分なら15〜18枚)。本体に50%以上を配分できていなければ、フック・背景が肥大している

### 時間尺による構成の違い

| 尺 | 往復回数 | 本体の実例数 | 削るもの / 足すもの |
|---|---|---|---|
| LT(5分) | 1回 | 1点のみ | 背景説明・適用条件の詳細を削る。フック→現状→転換→実例1点→行動喚起で一直線。**LTは「続きを知りたくさせる」が成功条件** — 記事・リポジトリへの導線を最終スライドに必ず置く |
| 20分 | 2回 | 2点 | 上の基本テンプレート。3点入れたくなったら1点を「参考: 記事に書いた」に落とす |
| 40分 | 3回以上 | 3点 | 15分前後に**再フック**(会場への問いかけ、デモ、意外な実測値)を計画的に置く — 聴衆の集中は一本調子では持たない。適用パートを厚くし、「自分の環境ではどうか」の場合分けまで踏み込む |

### 構成セルフチェック(スライド化に入る前)

- [ ] 各パートがビッグアイデアの一文に貢献しているか
- [ ] 「現状」パートは聴衆の実感に根ざしているか(自分の組織固有の事情だけになっていないか)
- [ ] 本体の各実例に「一文の学び+証拠」が対応しているか(棚卸しにない素材が紛れていないか)
- [ ] New Bliss が具体的な行動喚起で終わっているか(「ご清聴ありがとうございました」は結論ではない)

## 2. 記事タイプ別アウトライン

**冒頭の約束(リード)が記事の型を決める。** 読者は冒頭3〜5文で「読み続けるか」を判断するため、タイプごとに約束の書き方が異なる。共通ルール: 誰向けか / どんな問題を扱うか / 読むと何が得られるか / 結論の要約、をリードに含める(SKILL.md 記事ステップ2)。

### 2.1 経験談(やってみた・失敗談)

- **約束の型**: 「<状況>で<やったこと>をした結果と、そこで踏んだ罠を書く。同じ状況の人が同じ罠を避けられる」
- 構成: 状況設定(制約・前提を数値で)→ やったこと → 結果(実測)→ 失敗と原因分析 → 今ならこうする → 適用条件(どんな環境なら再現するか)
- **時系列の垂れ流しにしない。** 作業ログの清書ではなく、学びから逆算して再構成する。読者が知りたいのは「何が起きたか」ではなく「自分にも起きるか、どう避けるか」
- 失敗談は原因の分析まで書けて初めて価値になる。「うまくいかなかった」で終わる失敗談は読者に何も渡せない

### 2.2 技術解説

- **約束の型**: 「<概念/技術>が何を解決するものかを説明する。読み終えると、自分のケースで使うべきかを判断できる」
- 構成: 読者の困りごとから入る(概念の定義から入らない)→ 概念の最小説明 → 動く最小例 → 仕組みの解説(読者の判断に必要な深さまで)→ 使いどころと限界 → 公式ドキュメントへの導線
- **網羅に走らない。** 全機能の紹介は公式ドキュメントの仕事。解説記事の価値は「どの順で・どの深さで理解すればよいか」という道筋の提供
- 用語の初出には一言の定義を添える。読者の前提知識はリードで宣言した想定読者に固定する

### 2.3 チュートリアル

- **約束の型**: 「この手順どおりに進めると<完成物>が動く。所要<N>分、前提は<環境・バージョン>」
- 構成: 完成物を最初に見せる(スクリーンショット/出力例)→ 前提条件(バージョン明記)→ 手順(**各ステップの末尾に「ここまでで何が動くか」の検証点**)→ よくあるつまずきと対処 → 次のステップ
- コードは動く完全形で載せる。省略(`...`)を使う場合は完全版へのリンク(リポジトリ)を添える
- **公開前にクリーン環境で全手順を通す。** 自分の環境の残留設定に依存した手順は読者の環境で必ず壊れる

### 2.4 調査まとめ(比較・サーベイ)

- **約束の型**: 「<選択肢群>を<基準>で比較した。読み終えると自分の条件での選定判断ができる」
- 構成: 調査の動機と選定基準(先に宣言)→ 比較表 → 各項の根拠(一次情報リンク付き)→ 条件別の推奨(「〜なら A、〜なら B」)→ 調査の限界(いつ時点か、何を実際には試していないか)
- **試していないものを試したように書かない。** 「ドキュメント調査のみ」「実際に動かした」を項目ごとに区別して明示する — この区別自体が記事の信頼を作る
- 調査時点の日付を目立つ位置に書く(この型の記事は最も陳腐化が速い)

## 3. CFP(プロポーザル)テンプレート

**レビュアーは数百件を数分ずつ読む。** 概要の冒頭2文で「この発表でしか聞けないもの」が伝わらなければ、残りは読まれない前提で書く。

```markdown
## タイトル
<具体的な成果・数字・意外性を1つ入れる。タイムテーブル上で単独で戦える形>

## 概要(Abstract)
1文目: 聴衆の痛み(この発表が誰のどんな問題を扱うか)
2文目: 自分の一次情報(実際に何をやり、何が分かったか — 数字があれば入れる)
3文目以降: 発表で話す範囲、話さない範囲

## 対象聴衆と前提知識
<経験レベル+「こういう状況の人」。「全員向け」は誰にも刺さらない>

## 聴衆が持ち帰れるもの(3点)
1. <明日試せること>
2. <判断基準・考え方>
3. <落とし穴の回避>

## アウトライン(時間配分つき)
<スパークライン構成の要約。レビュアーは「時間内に収まるか」も見ている>

## なぜ自分がこのテーマを話せるか
<実績1点に絞る。列挙は薄める>

## 補足資料
<関連する自分の記事・リポジトリのURL(一次情報の実在証明として効く)>
```

### 採択されやすい構成の要点

- **「〜について話します」ではなく「〜が分かります」で書く。** レビュアーが評価するのはテーマの紹介ではなく聴衆への提供価値
- 概要に数字を1つ入れる(規模、期間、改善幅)。数字は一次情報の存在証明として機能する
- カンファレンスのテーマ・過去の採択傾向を読み、「なぜこのカンファレンスでこの話か」を1文で言えるようにする
- 同種の一般論セッションとの差別化を自問する: 「このプロポーザルは自分にしか書けないか?」— Yes と言えない場合は角度を変える(SKILL.md ステップ1)
- 落選しても素材は死なない: 提出と同時に同素材の記事化パスを設計しておく(SKILL.md「発信資産の運用」)

## 4. 公開前チェックリスト(詳細版)

SKILL.md 記事ステップ4のチェックを、確認手順つきで具体化したもの。

### タイトル

- [ ] 内容の過大・過小表現がないか(本文を読み終えた読者がタイトルを見返して裏切られないか)
- [ ] 読者が検索しそうな語(正式名称)が入っているか
- [ ] 一覧表示で末尾が切れても意味が通るか(重要語を前半に置く)
- [ ] 数字・具体性が入れられるなら入れたか(「速くする」→「p95を40%削る」)

### 冒頭100字(リード)

- [ ] 誰向けか / 問題 / 得られるもの / 結論、の4点が3〜5文に入っているか
- [ ] 一覧のプレビュー(冒頭が自動抜粋される)で読む価値が伝わるか
- [ ] 本文完成後にリードを書き直したか(書く前の想定と結論のずれを反映)

### コード・手順の検証

- [ ] 記事に記載のとおりにクリーン環境で実行して再現したか
- [ ] 言語・ライブラリ・ツールのバージョンを明記したか
- [ ] コピー&ペーストを妨げる要素(行番号混入、全角空白、プロンプト記号)がないか
- [ ] 出力例は実際の出力か(想像で書いた出力例は微妙に間違う)

### 一次情報・出典

- [ ] 事実の主張それぞれに出典(一次ソース優先)へのリンクがあるか
- [ ] 引用は実物を確認したか(`[出典未確認]` が残っていないか)
- [ ] 自分の実測値に測定条件・日付が近接して書かれているか(切り出されても帰属が運ばれる形)
- [ ] 「〜と思う」(意見)と「〜である」(検証済み)の区別が文面で分かるか

### 読了時間・分量

- [ ] 想定読了時間がリードの約束と釣り合っているか(5分の約束で30分の記事にしない)
- [ ] 想定を超えるなら分割・シリーズ化を検討したか
- [ ] 見出しだけ読んで流れが分かるか(スキャン読者のパス)

### 最終ゲート

- [ ] 機密情報・社内情報の混入はないか(スクリーンショットの背景・タブ名まで)
- [ ] AIが下書きした箇所の事実を著者自身で検証したか(SKILL.md「AIを執筆補助に使う際の規律」)
- [ ] AI定型文体(過剰な箇条書き、均質なリズム、水増しの前置きとまとめ)を最終パスで除去したか
