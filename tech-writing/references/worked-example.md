# tech-writing 適用例

架空の現実的シナリオに対して、登壇ワークフロー(ステップ1〜4)を一巡させた例。登場する組織・人物・数値はすべて架空。

## 題材

中規模Webサービスの開発チームに所属するエンジニアが、半年かけて取り組んだ「CIのflakyテスト削減」の経験を、テスト系カンファレンスの20分枠で発表したい。

## 入力(ユーザーの依頼と文脈)

- 依頼: 「flakyテスト対策の経験でCFPが通った。20分の登壇資料の構成を考えてほしい。ツールはMarp」
- 文脈ヒアリングで得た情報: 聴衆はテスト自動化の実務者中心。flakyテストの存在自体は全員が知っている(=「flakyとは何か」の説明は不要)

## 各ステップでの判断

### ステップ1: 戦略の確定

- 一次情報の棚卸しをまず実施。ユーザーへの質問で「一文の学び+証拠」を4つ抽出した:
  1. flaky率の計測を先に自動化したことで対策の優先順位が変わった(証拠: 体感で犯人扱いしていたE2Eより、unitテストの時刻依存が件数最多だった集計表)
  2. 「リトライで隠す」を6週間運用して撤回した(証拠: リトライ導入後にflaky新規発生率が上がった推移グラフ)
  3. 修理よりも検疫(隔離)を先にすると心理的抵抗が減った(証拠: 検疫レーン導入前後の修理着手リードタイム)
  4. 時刻固定の共通ヘルパー導入で再発が止まった(証拠: 導入後3ヶ月の再発ゼロ記録)
- **ビッグアイデア**: 「テスト自動化の実務者は、flakyテストを『直す』前に『測って隔離する』仕組みを作るべきだ。なぜなら我々の半年の実測で、直感による修理は優先順位を外し、リトライによる隠蔽は事態を悪化させたからだ」
- 聴衆の痛み: 「flakyを直す時間が取れず、リトライでしのいでいる」— 学び2がこの常識に反するため、発表の山場に置くと判断

### ステップ2: 構成設計(スパークライン)

`references/structure-templates.md` の20分テンプレートに割り付けた:

| パート | 時間 | 内容(対応する棚卸し) |
|---|---|---|
| フック | 1.5分 | 「再実行ボタンを押しながら祈った経験は?」+ 自チームの当時のCI再実行率 |
| 現状 | 3分 | なぜ「気づいた人が直す」が機能しないか(学び1の前半) |
| 転換 | 1分 | ビッグアイデア提示: 直すな、まず測って隔離しろ |
| 本体 | 10分 | 学び1(計測、5分)→ 学び2+3(リトライの失敗と検疫への転換、5分)。学び4は適用パートへ |
| 適用 | 2.5分 | 明日試せる一歩: flaky率の計測スクリプトから。学び4と「小規模チームでは検疫レーン不要」の留保 |
| New Bliss | 1分 | 「CIの赤が信号として機能する世界」+ 計測スクリプトのリポジトリ案内 |

- 3つ目の実例候補(検疫レーンの実装詳細)は面白いがビッグアイデアへの貢献が薄いため削り、「補足記事に書く」へ回した(大原則1・5)
- この構成表の段階でユーザーとレビューし、合意を得てからスライド化に進んだ

### ステップ3〜4: スライド化とセルフレビュー

- 上限18枚と決め、Marp形式でドラフト作成。セルフレビューで「リトライ導入の経緯スライドに初出の社内用語がある」「時間超過時に削るスライド未指名」の2点を検出し修正

## 成果物(スライドドラフト抜粋 — 本体パートの2枚)

```markdown
## 犯人はE2Eではなかった
<!-- 視覚的証拠: flaky発生源の棒グラフ。unit(時刻依存) 41件 / E2E 17件 / 結合 9件。
     unitの棒だけハイライト -->
<!-- ノート: 対策前は全員がE2Eを疑っていた。計測を自動化して初めて分かった。
     直感で修理していたら半年を最少数の犯人に使うところだった -->

## リトライは「解決」ではなく「培養」だった
<!-- 視覚的証拠: 折れ線グラフ2本(リトライ導入週を境に、CI成功率は横ばい、
     flaky新規発生率は上昇)。導入週に縦線 -->
<!-- ノート: 赤が出なくなると、flakyを生むコードパターンへの抑止が消える。
     6週間で撤回した判断の経緯を口頭で 30秒 -->
```

各スライドが「主張文タイトル / 視覚的証拠の指定 / 発表者ノート」の3点セット(ステップ3の成果物仕様)になっており、タイトルだけ拾い読みしてもストーリーが通ることを確認して納品した。
