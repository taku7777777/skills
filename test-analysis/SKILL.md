# テスト分析スキル

ISTQB CTAL-TA v4.0(テスト分析プロセス・リスクベースドテスト)、James Bach & Michael Bolton(Rapid Software Testing: HTSM・オラクルヒューリスティック)、Cem Kaner(BBST)、日本のテストコミュニティ(テスト観点の体系化)の知見を統合した分析プロセス。

テストケースの具体化は `test-design` スキル、テスト戦略・自動化・品質ゲートは `ai-adaptable-testing` スキルの領分。本スキルの成果物は「優先順位とオラクル付きのテスト条件一覧」であり、test-design への入力になる。

## 大原則

1. **テストは無限、資源は有限。** 全数テストは不可能なので、テスト分析の本質は「何をテストするか」ではなく「**何をテストしないと最も痛いか**」の選別(リスクベース)。テストしないと決めたことを明示するのも成果物。AIでケース生成コストが下がっても、実行時間・保守・人間のレビュー注意・オラクル整備は有限のままであり、選別の必要は消えない。
2. **カバレッジはモデルに対してしか定義できない**(Kaner)。「十分テストした」は常に「どのモデル(状態遷移、ルール、データ空間、シナリオ)を何%覆ったか」としてしか語れない。分析の中心作業はテスト対象のモデル化である。
3. **テスト分析は最初の欠陥検出活動。** テストベース(要件・仕様)の曖昧・矛盾・欠落は、モデル化しようとした瞬間に露見する。実装前に見つけた仕様欠陥は、テスト実行で見つけるバグより価値が高い(シフトレフト/フェーズコンテインメント)。
4. **オラクルなきテスト条件は条件ではない。** 「正しく動くこと」は条件ではない。何と比較して合否を判定するのか(仕様値、性質、関係、一貫性)を条件ごとに決める。
5. **チェッキングはテスティングの一部**(Bach/Bolton)。アルゴリズム化して機械に任せられる確認(checking)を切り出しても、学習と評価としてのテスティングは残る。分析段階で「checkingに落とせる条件」と「探索セッションで学ぶべき領域」を区別し、後者をチャーターとして計画する。
6. **人間の合意が要るのはモデルとリスクと対象外。** テスト条件・ケースはモデルから機械的に再導出できるが、モデルの正しさ・リスクの重み付け・「やらないこと」は合意が必要な判断。ユーザーレビューはそこへ集中させる(条件一覧の網羅読みを求めない)。

## ワークフロー

### ステップ0: インプット収集

以下を集め、無いものは「無い」と記録する(テストベースの欠如自体がリスク):

- テストベース: 要件・受け入れ基準(`requirements-definition` の成果物があれば最優先)、設計書・ADR、API仕様、UIモック
- コードベース: 実装、既存テスト、変更差分(テスト分析の対象が変更なら差分と影響範囲)
- 経験データ: 欠陥履歴(どこで・どんな欠陥が出てきたか)、問い合わせ・障害履歴、ホットスポット(変更頻度×複雑度)
- ドメイン知識: 用語集、業務ルール、規制要件

### ステップ1: テストベースのレビュー(欠陥検出パス)

モデル化に入る前に、テストベース自体を検査する:

- **曖昧**: 曖昧語(「適切に」「など」「原則」)、量化の欠落、主語のない受動態(`requirements-definition` ステップ5の検査と同じ観点)
- **矛盾**: 要件同士・要件と実装・本文と例の不一致
- **欠落**: 例外系の未定義(「失敗したらどうなるかが書かれていない」)、境界の未定義(以上/超の区別)、非機能の未言及
- **テスト容易性**: 観測手段(結果を確認できるログ・API・画面)と制御手段(状態を作り込めるか)があるか。なければ実装側への要求として起票

検出した問題は**質問リスト**としてユーザーに提示し、回答を得てから(または仮定を明記して)先へ進む。

**変更起点の分析(差分・PRが対象の場合)**: 全量分析ではなく、(1) 差分が触るモデル要素を特定する(どの遷移・どのルール・どの変数の定義域が変わったか) (2) 既存モデルを更新し、影響を受ける既存テスト条件を再導出する(`test-design` 大原則3) (3) 新規条件は変更部分にのみ追加し、blast radius(呼び出し元・共有データ・契約の利用者)を発生可能性の根拠に使う (4) 成果物は差分条件一覧(既存分析への追記)。CI上の自動テスト選択(TIAツール)は戦略層の領分 — ここで定義するのは影響分析の手順。

### ステップ2: テスト対象のモデル化

対象の性質に応じて1つ以上のモデルを作る(このモデルが test-design の技法選択を決める):

| 対象の性質 | モデル | 露見する欠陥 |
|---|---|---|
| 入力・パラメータがある | 変数と定義域の一覧(境界・型・依存関係) | 境界の未定義、変数間制約の欠落 |
| 条件分岐・業務ルール | デシジョンテーブル(条件×アクション) | ルールの矛盾・重複・未定義の組み合わせ |
| 状態を持つ | 状態遷移図/表 | 不到達状態、未定義イベント、デッドエンド |
| データのライフサイクル | CRUDマトリクス(エンティティ×機能) | 作りっぱなしのデータ、削除時の整合性未定義 |
| ユーザーの目的達成フロー | シナリオ一覧(メイン/代替/例外/エッジ) | 例外フローの未定義、途中離脱の扱い |
| 出力の正しさが定義しにくい | 性質・メタモルフィック関係の一覧 | 仕様の暗黙の前提 |
| 出力がAI生成・非決定的(LLM機能) | 評価セット(入力コーパス × 判定基準ルーブリック) | 合否基準の未定義、「良い出力」の暗黙の前提 |

モデル化の過程で見つかる「表の空欄」「図の行き止まり」がステップ1の質問リストに追加される。

**実装が存在するモデルは二重に作る**: 仕様から作ったモデル(あるべき)と、実装から機械的に抽出したモデル(現状 — 分岐条件、enum/状態フィールド、遷移を起こす関数、バリデーション、DBスキーマ制約をコードから読み取り同じ形式の表にする)を突き合わせ、差分をすべて列挙する。各差分は「実装の欠陥」「仕様の陳腐化」「暗黙仕様」のいずれかであり、ステップ1の質問リストに追加する(乖離検出の継続ゲート化は `harness-engineering` の drift gate)。**期待値を実装から写さないこと** — 実装モデルは「どこが疑わしいか」を見つける道具であり、正解の出所は常にテストベース。

### ステップ3: テスト観点の洗い出し

モデルを縦糸、観点を横糸にして広げる。`references/analysis-heuristics.md` の観点カタログ(HTSM/SFDIPOTベース)を使い、対象に関係する観点だけを選ぶ:

- 機能そのもの × データバリエーション × 状態 × タイミング(同時実行・中断) × 環境/プラットフォーム × ユーザー種別/権限 × 品質特性(全リストは ISO/IEC 25010 を基準に、関係するものだけ選ぶ)
- **過去の欠陥履歴から観点を逆引きする**(このプロダクト固有の弱点は一般カタログには載っていない)。繰り返し出る欠陥パターンはプロジェクト固有チェックリストとして蓄積する
- 品質特性の深掘りが必要なら該当スキルへ接続(セキュリティ → `arch-security-design` の脅威分析、性能 → 品質特性シナリオ)

### ステップ4: リスク分析と優先順位付け

各観点・機能領域に **影響度 × 発生可能性** でリスクレベルを付ける:

- **影響度**(ビジネス視点): 誤動作したときの被害 — お金・データ破壊・法令・信用・利用者数。ユーザー/ドメイン知識で判断し、根拠を書く
- **発生可能性**(技術視点): 複雑度、変更の新しさ・大きさ、過去の欠陥密度、関係者の理解度、外部依存
- スケールは両軸とも High/Med/Low の3段階とし、リスクレベルは3×3マトリクス表で決める(数値の掛け算は疑似精度を生むため避ける)。**各評価に根拠1行を必須**とする
- 出力はリスクマトリクス。**高リスク=複数技法+探索の重ね掛け、低リスク=薄いスモーク or 対象外**という濃淡が次工程への指示になる
- **リスク対応はリリース前テストだけではない。** 各高リスク項目に検出ポイントを決める: (a) 事前テストで防止 / (b) **本番で検出+高速復旧**(カナリア比較、SLO・メトリクス監視、synthetic monitoring、フラグによる即時無効化、実ユーザーフィードバックの還流 — 低評価・問い合わせ・修正操作を収集してテスト条件・評価セットへ戻す経路。LLM機能では監視やjudgeサンプリングが見逃す欠陥を拾う低コストのシグナル)。事前再現コストが極端に高い・本番データやスケールに依存する・発生しても復旧が容易 — のいずれかに該当するリスクは (b) が合理的なことが多い。(b) を選んだ項目もテスト条件一覧から消さず「検証ポイント=production」の監視条件として残す(復旧手段の実装は運用・戦略層の領分)

### ステップ5: テスト条件の定義

観点×モデルから、検証可能な粒度のテスト条件を書き出す:

- 形式: 「<対象>が<状況>のとき<期待される性質>」が検証できる粒度(具体値はまだ不要 — それは test-design の仕事)
- 各条件に付与: ID / トレース元(要件・リスク) / 優先度 / 検証ポイント(unit/integration/E2E — 配分原則は `ai-adaptable-testing` — または production: カナリア指標・synthetic・アラート) / 技法候補 / オラクル
- **やらないことリスト**: 分析したが対象外とした領域と理由(「カバレッジの空白」と「意図的な受容」を区別可能にする)

### ステップ6: オラクルの決定

条件ごとに合否判定の根拠を決める。仕様に期待値が書かれていない条件は、`references/analysis-heuristics.md` のオラクル一覧(FEW HICCUPPS / 疑似オラクル / 性質 / メタモルフィック関係)から選ぶ。**どのオラクルも使えない条件は「テスト不能」として実装・仕様側への要求に変換する**(観測手段の追加、仕様の明確化)。

**注: オラクルはすべてヒューリスティックであり誤りうる**(Bolton/Bach)。不一致は「欠陥の疑い」であって証明ではなく、一致は正しさを保証しない。高リスク条件には複数オラクルを併用し、オラクル間の矛盾(例: 意図的な仕様変更後は History と Claims が対立し、旧挙動との比較は偽陽性を出す)はどちらを優先するかを条件に明記する。

### ステップ7: セルフチェックと引き継ぎ

- [ ] 全テスト条件が要件またはリスクへトレースできるか(トレースできない条件は削除候補、トレースされない要件はカバレッジ漏れ)
- [ ] 高リスク領域に複数の観点・モデルが当たっているか
- [ ] 宣言した判断規則と個別判定が矛盾していないか: リスクマトリクスのマッピングを宣言したなら全セルの判定がそこから機械的に導出できるか、モデルの定義域(条件の値数・水準)は成果物内で一貫しているか(ここでの不一致は test-design の導出全体を歪める)
- [ ] 「正常系だけの領域」が残っていないか(例外・境界・同時実行は分析したか)
- [ ] 探索が必要な領域(仕様が薄い・新規性が高い)がチャーター候補として印付けされているか
- [ ] 質問リスト・テスト不能項目がユーザーに提示されているか

## 成果物フォーマット

```markdown
# <対象> テスト分析
## 1. テストベースの問題点(質問リスト: 曖昧/矛盾/欠落/テスト容易性)
## 2. モデル(該当するもの: 変数表 / デシジョンテーブル / 状態遷移 / CRUD / シナリオ / MR)
## 3. リスクマトリクス(領域 × 影響度 × 発生可能性 × 根拠)
## 4. テスト条件一覧(ID / 条件 / トレース元 / 優先度 / 検証ポイント / 技法候補 / オラクル)
## 5. 対象外とその理由
## 6. 探索チャーター候補(仕様の薄い領域)
```

小さな変更では 1・4(+必要なモデル1つ)の軽量版でよい(省略を明示)。
