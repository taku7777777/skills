# test-analysis 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 来週リリースの会員ランク改定、テストの時間が足りない。どこを削っても致命傷にならないか、判断材料を整理してほしい | リスクベースの優先順位付け(削る判断も分析の成果物) |
| F2 | この仕様書からテストを書こうとしたら、期待値がどこにも書いてなくて手が止まった。合否をどう決めればいい? | テストオラクルの決定 |
| F3 | 先月の帳票金額ズレ障害の再発防止として、帳票モジュールの確認事項にどんな穴があったのか棚卸ししたい | テスト観点の洗い出し・抜け漏れ分析(欠陥履歴の逆引き) |
| F4 | この差分、影響範囲が読みきれない。既存テストのどれを見直して、何を足すべきか教えて | 変更差分起点のテスト影響分析 |
| F5 | レコメンドの並び順って「正解」がないんだけど、リリース判定のための確認項目ってどう立てればいいの? | オラクル問題(性質・メタモルフィック関係のモデル化) |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | この機能のテストケースを作って。試す値の一覧まで具体的にほしい | `test-design` が発火する(条件の具体化=設計層) |
| N2 | テストが遅くて flaky で困ってる。テスト全体の方針から立て直したい | `ai-adaptable-testing` が発火する(戦略層) |
| N3 | この PR の実装コードをレビューして問題を指摘して | `impl-review` が発火する(レビュー自体が目的。テスト条件の導出ではない) |
| N4 | 探索的テストのチャーターを書いて、明日のセッションで使えるようにして | `test-design` が発火する(チャーター設計は設計層) |

## 検証シナリオ

### S1: キャンセル料ルールのテスト分析
- **タスク**: 「会議室予約システムのキャンセル料機能を実装する前にテスト分析してほしい」と依頼し、仕様抜粋を渡す: (1) 利用日の7日前まで無料 (2) 6日前〜前日は室料の30%、当日は100% (3) 法人契約プランはキャンセル料を半額にする (4) 月をまたぐ予約変更の場合は適切に処理する (5) キャンセル料は請求データに計上される。
- **スキルなしで予想される欠落**: テストベース自体の欠陥検出(「7日前まで」の境界時刻の未定義、「適切に処理」の曖昧語、半額の端数処理の欠落)を質問リストとして提示すること。デシジョンテーブル等のモデル化と未定義の組み合わせの列挙。条件ごとのオラクル決定。「やらないことリスト」。いきなりテストケースの箇条書きに飛ぶ。
- **合格条件**:
  - [ ] 仕様の曖昧・欠落(境界時刻、「適切に」、端数処理のいずれかを含む)が質問リストとして提示される
  - [ ] デシジョンテーブルまたは変数表などのモデルが作られ、未定義の組み合わせ・空欄が指摘される
  - [ ] テスト条件一覧の各条件にオラクル(合否判定の根拠)が付いている
  - [ ] リスクの濃淡(影響度×発生可能性と根拠)が示され、優先度に反映されている
  - [ ] 対象外とした領域が理由付きで明記される

### S2: 変更差分起点の影響分析
- **タスク**: 「勤怠管理システムの打刻丸め仕様を『15分単位切り捨て』から『1分単位(丸めなし)』に変える差分がある。丸め関数の変更と、それを呼ぶ日次集計・残業判定の2箇所の修正が diff に含まれる。何をテストすべきか整理して」と依頼する(既存テストは丸め関数の単体10件と日次集計の統合3件が存在する、と文脈で伝える)。
- **スキルなしで予想される欠落**: 全量分析でなく差分が触るモデル要素の特定から入ること。影響を受ける既存テスト条件の再導出(丸め前提の既存13件は期待値ごと見直し対象)。blast radius(残業判定・給与計算など下流)を発生可能性の根拠に使うこと。新旧仕様の切替日またぎのような変更固有の条件。
- **合格条件**:
  - [ ] 差分が変えるモデル要素(丸めルールの定義域・依存する集計/判定)が特定されている
  - [ ] 既存テスト13件への影響(期待値の見直しが必要なもの)が判定される
  - [ ] 新規条件が変更部分と下流影響(blast radius)に限定して追加され、根拠が書かれている
  - [ ] 成果物が既存分析への差分(追記)形式であることが明示されている
  - [ ] 切替タイミング・過去データの再計算有無など変更固有の欠落が質問として挙がる(人間判定)

### S3: 既存テストケース一覧の十分性レビュー(暗黙的依頼)
- **タスク**: 「サブスクの解約時日割り返金のテスト、これで足りてるか見て」とだけ依頼し、既存ケース一覧を渡す: (1) 月初に解約→ほぼ全額返金 (2) 月央に解約→半額返金 (3) 月末に解約→返金なし (4) 年払いプランの解約→月割り+日割り返金 (5) 無料トライアル中の解約→返金なし。
- **スキルなしで予想される欠落**: 「十分か」をモデルに対するカバレッジとして語ること(感覚的な「だいたい良さそう+少し追加」で終わる)。ケースの背後のモデル(プラン種別×解約タイミング×支払状態)を再構成して空欄を出すこと。正常系偏重の指摘(決済失敗中の解約、解約当日の再契約、返金額0円境界)。返金額のオラクル(端数・消費税の扱い)未定義の検出。
- **合格条件**:
  - [ ] ケース一覧の背後にあるモデル(変数表・デシジョンテーブル等)が再構成され、それに対する空欄・未カバー領域が列挙される
  - [ ] 「十分性はモデルへのカバレッジとしてしか語れない」に相当する枠組みで回答している(人間判定)
  - [ ] 例外系・境界(0円返金、決済失敗状態、日割り計算の端数のいずれかを含む)の欠落が具体的に指摘される
  - [ ] 返金額の合否判定根拠(オラクル)の未定義が仕様への質問として提示される
