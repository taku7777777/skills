## A. テスト観点カタログ(HTSM / SFDIPOT ベース)

James Bach の Heuristic Test Strategy Model のプロダクト要素(SFDIPOT)を軸に、実務観点を展開したもの(原義の Structure=コード・ファイル等の物理構成を、デプロイ・設定に翻案している)。**全部を適用するのではなく、対象に関係する行だけ選ぶ**。

### Structure(構造)
- デプロイ単位・モジュール構成: 部分的デプロイ・バージョン混在時の動作
- フィーチャーフラグ: (1) ロールアウト遷移中のフラグはON/OFF両系を必須条件化 — OFF系はフラグ削除まで回帰対象として維持(削除予定日を条件に記録) (2) 全組み合わせは不要 — 同じデータ・画面・フローに触るフラグ対のみ相互作用候補としてペアワイズへ (3) ロールアウト経路自体も条件に: 段階公開のユーザー割付の一貫性(同一ユーザーがON/OFFを行き来しないか)、切替瞬間をまたぐ進行中処理、kill switchで旧コードに戻したとき**新コードが書いたデータを旧コードが読めるか**(後方互換の逆方向)

### Function(機能)
- 主機能とその副作用(通知、ログ、キャッシュ更新、イベント発行)
- 機能間の相互作用: 同じデータ・リソースを触る機能の同時利用
- 冪等性: 同じ操作の2回実行、リトライ
- 権限: 各ロールで「できること」と「**できてはいけないこと**」(認可のネガティブ)

### Data(データ)
- 型・形式: 空 / null / 0 / 負数 / 最大最小 / 精度 / 全角半角 / 絵文字・サロゲートペア / 前後空白 / 巨大入力
- 一意性・重複、順序(ソート・タイブレーク)
- ライフサイクル: 作成→更新→削除→復元。削除済みデータへの参照
- 量: 0件 / 1件 / ページ境界 / 大量(性能とUIの両面)
- 整合性: 部分更新の失敗、二重送信、移行データ(旧形式)
- 時間依存データ: 期限切れ、未来日付、タイムゾーン、うるう年、月末、夏時間

### Interfaces(インターフェース)
- API: 契約(スキーマ・enum追加)、後方互換、バージョン混在
- 外部システム: 障害・遅延・不正応答時の挙動、リトライと重複
- UI: 入力デバイス、コピペ、ブラウザ戻る/リロード、複数タブ
- イベント/メッセージ: 順序逆転、重複配信、遅延到着

### Platform(プラットフォーム)
- OS / ブラウザ / デバイス / 画面サイズ / ネットワーク品質(遅い・切れる)
- 依存サービス・ライブラリのバージョン
- ロケール: 言語、通貨、日付形式、RTL
- アクセシビリティ: キーボードのみ操作、スクリーンリーダー、コントラスト(WCAG)— 法規制対象になりうる(Statutes オラクルに接続)

### Operations(運用・利用)
- 実際のユーザーの使い方: 典型ペルソナ、極端なユーザー(パワーユーザー・初心者)
- 運用操作: デプロイ中の利用、メンテナンスモード、データ修正
- 長時間運用: リソースリーク、ログ肥大、キャッシュ・セッションの経年

### Time(時間)
- 同時実行: 同一ユーザー2セッション、複数ユーザーの競合、check-then-act
- 中断: 処理途中のキャンセル、タイムアウト、電源断・再起動後の状態
- 順序: 想定外の操作順(スキップ、逆順、二度押し)
- 処理をまたぐ日付変化、バッチとオンラインの重なり

## B. オラクルヒューリスティック(FEW HICCUPPS — Bolton/Bach)

期待結果が仕様に明記されていないとき、「何との一貫性で判定するか」の選択肢:

| 一貫性の相手 | 問い |
|---|---|
| **F**amiliar problems | 既知の欠陥パターンと同じ問題を起こしていないか |
| **E**xplainability | 挙動を開発者/テスターが説明できるか(説明できない挙動は欠陥候補) |
| **W**orld | 現実世界の事実と整合するか(存在しない日付、負の在庫) |
| **H**istory | 過去バージョンの挙動と整合するか(意図しない変化=リグレッション) |
| **I**mage | ブランド・製品イメージと整合するか(恥ずかしい文言・見た目) |
| **C**omparable products | 類似製品・競合の挙動と整合するか |
| **C**laims | 仕様・ドキュメント・マーケティングの主張と整合するか |
| **U**sers' desires | 合理的なユーザーが望むことと整合するか |
| **P**roduct | 製品内部の一貫性(同じ操作が画面ごとに違う挙動をしないか) |
| **P**urpose | 製品の目的と整合するか |
| **S**tatutes & standards | 法令・業界標準・社内規約と整合するか |

**注: これらはすべてヒューリスティックであり、一致=正しい / 不一致=欠陥のどちらも保証しない。** 高リスク条件では複数を併用し、矛盾したとき(意図的変更後の History vs Claims 等)はどちらを優先したかと根拠を記録する。

### B-2. 技術的オラクル(自動判定向け)

| オラクル | 使いどころ |
|---|---|
| 仕様値(明示的期待値) | 仕様に期待値が書ける場合の第一候補 |
| 疑似オラクル | 同じ仕様の別実装(旧システム、簡易実装、移行前後)との結果比較(差分テスト) |
| 性質(プロパティ) | 出力が満たすべき不変条件(`ai-adaptable-testing` のPBT) |
| メタモルフィック関係 | 入力の変換と出力の変換の関係(例: `count(A) ≥ count(A AND B)`、`sort(X) == sort(shuffle(X))`)。正解が定義できないシステム(検索・ML・数値計算)の定石 |
| 統計的オラクル | 非決定的システムの分布・傾向での判定(閾値と試行回数を明記) |
| 評価セット+ルーブリック(LLM-as-judge) | LLM/生成系機能。golden set(固定の入力コーパス)にルーブリック判定を適用。judge自体を人間ラベル付きサンプルとの一致率で較正してから使い、統計的オラクルとして扱う(1回の判定を信用せず、N回・閾値を明記) |
| SLO・本番メトリクス | カナリア vs ベースラインの比較(本番における疑似オラクル)。エラー率・レイテンシ・ビジネスKPIの逸脱で判定。閾値と観測窓を明記 |

## C. リスク要因チェックリスト(発生可能性の根拠)

- [ ] 新規性: 新技術・新ドメイン・チーム初経験の領域
- [ ] 複雑度: 条件分岐の多さ、状態数、連携先の数
- [ ] 変更: 直近の変更量、ホットスポット(変更頻度×複雑度 — `refactoring-strategy` の実測手順)
- [ ] 欠陥履歴: 過去の欠陥密度が高いモジュール(欠陥クラスタは再発する — ISTQB原則)
- [ ] コミュニケーション: 仕様の伝言ゲーム段数、ドメインエキスパートとの距離
- [ ] 時間圧力: 締切に追われて書かれたコード
- [ ] 検証の薄い生成コード: 弱いハーネス(テスト・レビューゲート不在)の下で生成された領域、人間が差分を読まずに受け入れた領域(受け入れ時の検証記録の有無で判別。欠陥分布の傾向は `impl-review` ステップ2.5)

## D. 影響度の根拠(ビジネス視点)

- [ ] お金: 誤請求・決済・課金への影響
- [ ] データ: 破壊・漏洩・不可逆な誤り
- [ ] 法令・契約: 規制違反、SLA違反
- [ ] 利用規模: 影響するユーザー数・頻度(全員が毎日使う画面 vs 管理者が月1で使う画面)
- [ ] 回復可能性: 気づけるか、直せるか、回避策はあるか
- [ ] 信用: 公開されたら恥ずかしいか(Image oracle)

## E. 一般欠陥タキソノミーからの逆引き

自プロダクトの欠陥履歴が薄い新規領域では、公開の欠陥タキソノミー・攻撃リストを仮説源として使う:

- Kaner, Falk & Nguyen『Testing Computer Software』付録の bug taxonomy(約400の欠陥パターン)
- James Whittaker『How to Break Software』の攻撃リスト(入力攻撃・状態攻撃・環境攻撃)
- AI生成コードの欠陥分布(`impl-review` ステップ2.5)

## F. 探索候補の判定(checking vs exploring)

以下に該当する領域は、スクリプト化より探索チャーター(`test-design` ステップ6)に回す:

- 仕様が薄い・暗黙知が多い(触って学ぶ必要がある)
- アルゴリズム化したチェックに落とせない品質(ユーザビリティ・違和感。ルーブリック付きエージェント評価で近似できる場合もある — `harness-engineering` のセンサー)
- 新規性が高く、どんな欠陥が出るか予想がつかない
- 複雑な相互作用(スクリプトの組み合わせ爆発を探索者 — 人間またはエージェント — の判断で刈る)
