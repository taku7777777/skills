# テスト設計スキル

ISTQB CTAL-TA v4.0(テスト技法4分類)、Cem Kaner『Domain Testing Workbook』、NIST組み合わせテスト研究(Kuhn)、Elisabeth Hendrickson『Explore It!』、James Bach(セッションベースドテスト)の知見を統合した設計プロセス。

入力は `test-analysis` の成果物(優先順位・オラクル付きテスト条件)。無い場合は簡易分析(対象のモデル化とリスク確認)を先に行い、その判断を明示する。テストコードの実装規約・自動化戦略は `ai-adaptable-testing` の領分。

## 大原則

1. **技法は欠陥仮説の道具である。** 各技法は特定の欠陥タイプを狙う照準器(境界値→境界の実装ミス、デシジョンテーブル→ルールの誤解、状態遷移→不正な遷移の許容、ペアワイズ→因子間の相互作用)。「どの技法を使うか」は「どんな欠陥が潜んでいそうか」から逆算する。
2. **少数の強いテスト > 多数の弱いテスト**(Kaner)。強いテストとは、失敗する可能性が高い入力を選び、失敗が起きたら確実に検出できるテスト。ケース数はカバレッジの証明にならない — モデルに対するカバレッジ基準で語る。
3. **一次成果物はテストケースではなく設計モデル。** デシジョンテーブル・状態遷移表・ドメイン分析表を保守し、ケースはそこから導出する。仕様変更時はモデルを直してケースを再導出する(モデルなしのケース修正が抜け漏れの温床)。
4. **1テスト1目的、ただし1テストに複数の観察。** 検証目的は1つに絞り(失敗原因の特定容易性)、実行時の観察(ログ・副作用・違和感)は広く拾う。
5. **スクリプトと探索は補完。** ルール化できる確認は機械化し(checking)、仕様の隙間・違和感は探索(exploring)に割り当てる。両方を設計するのがテスト設計。
6. **AI時代の分業**: ケースの量産・表からの展開はAI/ツールに委ね、人間(と本スキル)は「モデルの正しさ」「カバレッジ基準の選択」「何を狙うか」に集中する。生成されたケースはモデルに対するカバレッジで検収する(件数で満足しない)。

## ワークフロー

### ステップ1: 入力の確認

- テスト条件・リスク・オラクル(`test-analysis` の成果物)を確認。無ければ対象のモデル化とリスク濃淡だけ最低限実施
- 条件ごとの**リスクレベルが設計の深さを決める**: 高=複数技法+ネガティブ+探索 / 中=主技法1つ+代表ネガティブ / 低=ハッピーパスのスモークのみ

### ステップ2: 技法の選択

**欠陥仮説の生成(実装が読める場合)**: 技法を選ぶ前に対象コード・差分を読み、壊れていそうな箇所の仮説を列挙する(浮動小数点の等値比較、文字列/数値比較の混在、check-then-act、例外の握りつぶし、境界の比較演算子)。**コードから取るのは仮説のみ。期待値は必ずテストベースとオラクルから取る** — 実装から期待値を写したテストは実装の写経であり、バグを仕様として固定する。

`references/technique-catalog.md` を正とする(以下は要約 — 乖離したらカタログ優先):

| テスト条件の性質 | 主技法 | 補助 |
|---|---|---|
| 数値・範囲・サイズの入力 | 境界値分析 / ドメイン分析(多変数) | ネガティブ値 |
| 複数条件の組み合わせでアクションが決まる | デシジョンテーブル | 境界値(条件の閾値に) |
| 設定・環境・パラメータが多い | ペアワイズ(2因子)/ 高リスクなら3因子 | ベースチョイス |
| 状態・モード・ライフサイクルを持つ | 状態遷移(遷移カバレッジ+不正遷移) | Nスイッチ(高リスク時) |
| データのCRUD | CRUDマトリクス | 参照整合性・同時実行 |
| ユーザーの目的達成フロー | シナリオ(メイン/代替/例外/エッジ) | ユースケース分岐網羅 |
| 正解が定義しにくい(検索・ML・計算) | メタモルフィック / プロパティ | 疑似オラクル(差分) |
| LLM出力・プロンプト | 評価セット+ルーブリック採点 | メタモルフィック(言い換え・無関係情報付加への安定性) |
| 信頼境界を越える入力の構文解析(パーサ・デコーダ・ファイル形式) | ファジング(coverage-guided) | 性質オラクル・差分ファジング |
| 外部依存の障害時挙動(タイムアウト・リトライ・フォールバック) | フォールトインジェクション | 状態遷移(サーキットブレーカ)・同時実行シナリオ |
| サービス間のAPI/イベント連携 | 契約ケース設計(相互作用×provider state) | スキーマ進化の互換ケース |
| データパイプライン・集計 | データ品質expectation+突合 | メタモルフィック・統計的オラクル |
| 仕様が薄い・新規性が高い | 探索(SBTチャーター) | チェックリスト |

**高リスク条件には技法を重ねる**(例: 決済ルール = デシジョンテーブル×境界値×同時実行シナリオ)。

### ステップ3: 技法の適用

`references/technique-catalog.md` の手順に従い、**モデル(表・図)を先に作り、ケースを導出する**。適用時の普遍ルール:

- カバレッジ基準を宣言する(例: 「デシジョンテーブル全ルール網羅」「遷移カバレッジ100%+不正遷移全数」「ペアワイズ2因子網羅」)
- 有効系だけでなく**無効系(ネガティブ)を必ず含める**: 不正入力の拒否、不正遷移の防止、権限外の遮断。「拒否されること」自体が期待結果
- 期待結果はオラクルから具体化する。**副作用も期待結果に含める**(DBの状態、発行イベント、ログ、通知 — 戻り値だけ見るテストはすり抜けを許す)

### ステップ4: テストデータ設計

- 各ケースのデータは**なぜその値か**を説明できる代表値にする(境界のオン/オフポイント、同値クラスの代表)。「テキトーな値」を使わない — 値の選定理由がないケースは弱いテスト
- データの作り込み: 前提状態を作る手段(ファクトリ、フィクスチャ、API)と後始末を設計。他ケースとの共有データは変更の連鎖を生むため原則ケースローカルに(`ai-adaptable-testing` の自己完結性)
- 個人情報・本番データは使わない(必要なら仮名化・合成 — 機密性の判断は `agent-security-governance`)
- 時間依存データ: 単体・統合テストでは**クロックを注入して時刻を固定**する(決定性優先 — `ai-adaptable-testing`)。境界日付(月末・うるう年・DST)は固定クロックで明示的にケース化する。クロック注入できない共有環境・E2Eのデータのみ「現在時刻からの相対」で定義してエージング(データの陳腐化)を防ぐ

### ステップ5: ハイレベル/ロウレベルの判断と整理

- **ロウレベル(具体値まで固定)**: 自動化するケース、規制・監査で再現性が必要なケース
- **ハイレベル(論理的条件のみ)**: 探索の指針にするケース、UIが流動的な領域(具体値の保守コストが回収できない)
- 実行順序の依存・前提条件を明示し、独立実行可能な設計を優先
- 自動化候補の判定と実装への引き継ぎは `ai-adaptable-testing`(failing テストとして先置きすればハーネスの完了定義になる)

### ステップ6: 探索チャーターの設計

分析で「探索候補」と印を付けた領域に、セッションベースドテストのチャーターを書く。

**実行者を先に決める**: エージェント実行(API/CLI/ブラウザ操作が可能で、判定が FEW HICCUPPS の適用として記述できる領域)/ 人間実行(実機の感触・ユーザー体験の違和感そのものが目的の領域)。エージェント実行では時間枠の代わりに**行動予算(操作数・試行数)と停止条件**を定め、セッションログ(試行・観察・欠陥仮説)を成果物にする。デブリーフは両形態で行う — エージェントのセッションレポートを人間がレビューし、次のチャーターを更新する。

```yaml
チャーター: <対象>を<リソース/ヒューリスティック>を使って探索し、<情報>を見つける
対象範囲: <機能・画面・API>
時間枠: 45〜120分(標準90分。不確実性が高いほど短く刻んでデブリーフ頻度を上げる)
使うヒューリスティック: <例: 中断攻撃(処理中にキャンセル/リロード/オフライン)、
  変数を1つずつ極端に、ペルソナ(せっかちな初心者)、CRUD一巡>
想定リスク: <この領域で出そうな欠陥>
記録: 発見事項 / 新たなテストアイデア / カバレッジ実績(対象範囲のうち実際に触れた部分と触れなかった部分)
```

### ステップ7: 設計レビュー(セルフチェック)

- [ ] 宣言したカバレッジ基準を**機械的に突合したか**: モデル(テーブルの列・遷移・ペア)をデータとして持ち、ケースIDとの対応をスクリプトで検査する。目視・自己申告の「網羅した」を成果物に書かない。**突合スクリプトのマッピング自体が自己申告の混入口** — ケース⇔モデル要素の対応には、成果物に文書化された前提・手順から導出できる要素だけを書く(文書に現れない遷移・値をマッピングに足した網羅主張は検証の偽装)
- [ ] **成果物内の宣言と本文が突合できるか**: 「全ケースで◯◯」型の一律申告(副作用検証など)・件数・条件の定義域(値数・水準が分析モデルの定義と一致するか)を本文から数え直して確認する。例外があるなら一律申告にせず例外を明記する(例: 「BV系unitケースは計算値のみ検証、副作用は確定系ケースで担保」)
- [ ] **このテスト群が見逃す欠陥は何か**を1つ以上言えるか → それは受容か、追加設計か(言えない=分析不足)
- [ ] ネガティブ系・同時実行・中断が「量産しやすいハッピーパス」に埋もれていないか
- [ ] 期待結果に副作用が含まれているか。オラクル不明のまま「エラーにならないこと」で誤魔化したケースはないか
- [ ] ケース間の重複(同じ欠陥しか捕まえないケース)はないか — 削る勇気(必要性基準)

## 成果物フォーマット

```markdown
# <対象> テスト設計
## 1. 設計モデル(技法ごと: ドメイン分析表 / デシジョンテーブル / 状態遷移表 / ペアワイズ表...)
## 2. カバレッジ基準の宣言と達成状況
## 3. テストケース一覧
   | ID | 条件(トレース元) | 技法 | 前提/データ | 手順(要点) | 期待結果(副作用含む) | レベル | 自動化 |
## 4. 探索チャーター
## 5. このテスト群の限界(見逃しうる欠陥と受容判断)
```
