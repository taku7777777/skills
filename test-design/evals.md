# test-design 評価セット

スキルの効果を測定するための評価定義。運用方法は skill-improvement スキルの
references/eval-template.md を参照。

## 発火テスト

### 発火すべきプロンプト

| # | プロンプト | 理由 |
|---|---|---|
| F1 | 年齢と滞在日数の入力チェック、どの値を流し込めば漏れなく確認できる?適当な値で書きたくない | 境界値分析・ドメイン分析(データ選定) |
| F2 | 配送オプションの判定が5条件の if で絡み合ってる。全組み合わせは無理だから賢く減らして網羅したい | デシジョンテーブル/ペアワイズ(組み合わせ整理) |
| F3 | テスト分析で条件一覧はできてる。ここから実際に流せる粒度のテストに落とし込んでほしい | テスト条件→ケース具体化(本スキルの中心フロー) |
| F4 | このドキュメントの承認フロー、変な状態の飛び越えができないことをどう確かめればいい? | 状態遷移テスト(不正遷移の設計) |
| F5 | 仕様書がほぼない管理画面、リリース前に2時間だけ触って確認する。何をどう見て回るか段取りして | 探索的テストのチャーター設計 |

### 発火すべきでないプロンプト

| # | プロンプト | 期待される挙動 |
|---|---|---|
| N1 | そもそもこの新機能、何をどこまでテストすべきなのかの整理から手伝って | `test-analysis` が発火する(何を=分析層。ケース化はその後) |
| N2 | AI が書いたテストが信用できないので、品質を担保する仕組みを入れたい | `ai-adaptable-testing` が発火する(戦略・ゲート=戦略層) |
| N3 | 設計したテストケースを Jest で自動テストとして実装して CI に載せて | `ai-adaptable-testing` が発火する(実装規約・自動化戦略は戦略層)か、通常のコーディングとして応答 |
| N4 | この API のレスポンス設計をレビューして | `api-data-design` が発火する(テスト設計と無関係) |

## 検証シナリオ

### S1: 駐車場精算ルールのテストケース設計
- **タスク**: 「コインパーキング精算機の料金計算のテストケースを設計して」と依頼し、仕様を渡す: 30分200円(30分未満切り上げ)/ 入庫から24時間ごとに最大1,500円打ち止め / 夜間(20:00〜8:00)は全時間帯一律500円上限 / 駐車時間は1分〜30日 / 精算後10分以内の出庫まで追加課金なし。テスト分析の成果物はない、という前提も伝える。
- **スキルなしで予想される欠落**: 分析入力がない場合の簡易分析(モデル化とリスク濃淡)を先に行い明示すること。境界値表・デシジョンテーブルというモデルを先に作りケースを導出すること(思いついた順のケース列挙になる)。カバレッジ基準の宣言。各データ値の選定理由(オン/オフポイント)。夜間境界・24時間境界をクロック固定で扱うデータ設計。
- **合格条件**:
  - [ ] ケースの前に設計モデル(境界値表・デシジョンテーブル等)が提示され、ケースがそこから導出されている
  - [ ] カバレッジ基準(例: 全ルール網羅、境界オン/オフ全点)が宣言されている
  - [ ] 各テストデータに「なぜその値か」の選定理由が付いている
  - [ ] 無効系(0分、30日超、時刻逆転等の拒否)が期待結果付きで含まれる
  - [ ] 時刻依存のケースにクロック固定(注入)の方針が明記されている

### S2: 経路検索のオラクルなきテスト設計
- **タスク**: 「自転車ナビアプリの経路検索のテストを設計したい。正解の経路が一意に決まらないので、期待結果の書きようがなくて困っている。出発地・目的地・回避設定(坂道回避/大通り優先)がある」と依頼する。
- **スキルなしで予想される欠落**: メタモルフィック関係(出発地と目的地を入れ替えても距離がほぼ対称、経由地を追加しても所要時間は単調非減少、回避設定 ON で坂の合計勾配が減る等)や性質(経路は連結している、出発地から始まり目的地で終わる)による設計。退化実装でも通る弱い性質への言及。具体例テストとの併用による係留。「有名地点の結果を目視で確認」程度の提案で終わる。
- **合格条件**:
  - [ ] メタモルフィック関係または性質ベースのテストが主技法として選択され、関係・性質が3つ以上具体的に列挙される
  - [ ] 各関係・性質が入力の変換と出力の検証条件のペアとして実行可能な粒度で書かれている
  - [ ] 弱い性質だけでは退化実装を通しうることへの言及、または具体例(固定ケース)との併用がある
  - [ ] このテスト群が見逃す欠陥(限界)が1つ以上明示される

### S3: AI が量産したテストケースの検収(途中介入)
- **タスク**: 「パスワードポリシー(8〜64文字、英大小・数字・記号のうち3種以上、ユーザーID を含まない)の変更で、エージェントにテストケースを60件作らせた。多すぎる気もするし、これで十分なのかも分からない。見てもらえる?」と依頼し、ケース一覧の抜粋(有効なパスワード成功例が大半で、長さ境界は 8/64 のみ、ユーザーID 包含の検査なし、という偏りを含む20件)を渡す。
- **スキルなしで予想される欠落**: 件数でなくモデルに対するカバレッジで検収すること(ドメイン分析表・文字種組み合わせを再構成し、60件をマッピングして重複と空白を出す)。同じ欠陥しか捕まえない重複ケースの削減(少数の強いテスト)。オフポイント(7/65文字)・ネガティブ系(ユーザーID 包含、文字種2種)の欠落指摘。
- **合格条件**:
  - [ ] 検収の基準としてモデル(ドメイン分析表・組み合わせ表等)が構成され、既存ケースがそれに突合される
  - [ ] 境界のオフポイント(7文字・65文字)とユーザーID 包含のネガティブ系の欠落が指摘される
  - [ ] 同一同値クラス内の重複ケースが特定され、削減が提案される
  - [ ] 「件数はカバレッジの証明にならない」に相当する判断枠組みで結論が述べられる(人間判定)
