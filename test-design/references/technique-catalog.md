# テスト技法カタログ(適用手順・例・落とし穴)

各技法は「狙う欠陥 / 手順 / カバレッジ基準 / 落とし穴」で構成。文献により用語に揺れがあるため、**オン/オフポイントの定義とスイッチカバレッジの用語は本カタログを正とする**。

## 目次

- 境界値・デシジョンテーブル・状態遷移
- 組み合わせ・CRUD・シナリオ
- メタモルフィック・探索・ファジング
- フォールトインジェクション・契約・データ品質
- 同時実行と技法の重ね掛け

## 1. 境界値分析・ドメイン分析(データベース技法)

**狙う欠陥**: `<` と `<=` の取り違え、境界の一つずれ(off-by-one)、型・精度の丸め、変数間制約の見落とし。

**手順**:
1. 変数ごとに定義域を特定し、同値クラス(有効/無効)に分割
2. 各境界に4種の点を取る: **オン**(境界式に現れる値そのもの。閉境界 `x ≤ 100` なら100は有効側、開境界 `x < 100` なら100は無効側に落ちる)/ **オフ**(オンの隣接値で、オンと**逆のクラス**に落ちる値)/ **イン**(有効内代表)/ **アウト**(無効代表)。**オン/オフがどちらのクラスに落ちるかは境界の開閉で決まる** — 仕様の「以上/超」の確認が最初の仕事(未定義なら test-analysis の質問リストへ)
3. 多変数が関係する場合はドメイン分析表にまとめ、**1ケースで検証する境界は1つ**(他の変数はインポイントに固定 — 複数境界を同時に動かすと失敗原因が特定できない)

**例**(振込金額 1〜500,000円 — 両端とも閉境界):

| 変数 | 有効クラス | オン | オフ | イン | アウト |
|---|---|---|---|---|---|
| 金額 | 1〜500,000 | 1, 500000(有効) | 0, 500001(無効) | 50000 | -1, 9999999 |

**落とし穴**: 仕様に「以上/超」が明記されていない境界(→ test-analysis の質問リストへ)。浮動小数点の境界(等値比較の罠)。文字列長の境界はバイト数か文字数か(サロゲートペアで割れる)。

## 2. デシジョンテーブル(ルールベース技法)

**狙う欠陥**: ルールの組み合わせ漏れ(仕様に書かれていない組み合わせ)、条件の評価順序ミス、矛盾したルールの実装。

**手順**:
1. 条件(Y/N or 多値)とアクションを抽出
2. 全組み合わせ(二値なら2^n、多値なら各条件の値数の積)を列挙し、各列のアクションを仕様から決定 — **決められない列が仕様の穴**(質問リストへ)。条件数が多く全列挙が非実務的なら、独立な条件群に表を分割するか組み合わせテスト(§4)へ切り替える(その場合「2因子までの保証」であることを明記)
3. 結果が同じで根拠も同じ列のみ don't care で統合(縮小)。**安易な縮小は依存関係の見落としを生む**ため、高リスクなら全列版を維持
4. 各列を1ケースに。条件の閾値には境界値分析を重ねる

**カバレッジ基準**: 全ルール(列)網羅が最低線。

**落とし穴**: 条件が独立でない(あり得ない組み合わせ)場合は明示的に「不能」とマークし、**不能のはずの組み合わせが実際に作れてしまわないか**を1ケース試す。

## 3. 状態遷移テスト(振る舞いベース技法)

**狙う欠陥**: 不正な遷移の許容(キャンセル済み注文の出荷)、未定義イベントでのクラッシュ、遷移時の副作用漏れ、履歴依存バグ。

**手順**:
1. 状態・イベント・**ガード条件**・遷移・遷移時アクションを図/表にする(ddd-design の集約の状態遷移と一致しているか確認)。ガード付き遷移はガード成立/不成立の両方をケース化する
2. 状態遷移**表**を作る — 図では見えない「空欄」(未定義のイベント×状態)が全て見える。各空欄の期待挙動(無視/エラー/不能)を決める — **決まっていない空欄は仕様の穴**
3. カバレッジ基準を選ぶ(Chowの定義: **0スイッチ=全有効遷移を1回以上**=遷移カバレッジ、**1スイッチ=連続2遷移のペアを網羅**。文献によって用語が揺れるため定義ごと明記する)。標準は遷移カバレッジ(0スイッチ)。高リスク(課金・ワークフロー)は**1スイッチ** + **不正遷移全数**
4. 遷移の道筋(パス)としてケース化し、各遷移後の状態と副作用を検証

**落とし穴**: 「戻る」遷移(キャンセル、差し戻し)後の再遷移は履歴依存バグの巣。並行イベント(2ユーザーが同時に別イベント)は状態遷移表の外にある — 高リスクなら同時実行テスト(§13)を別途設計。

## 4. 組み合わせテスト / ペアワイズ(データベース技法)

**狙う欠陥**: 特定の因子の組み合わせでだけ起きる欠陥(設定Aと環境Bの相互作用)。

**根拠**(NIST Kuhn らの観測研究): 調査対象となった複数領域では、障害の大半は1〜2因子の相互作用で発生し、観測された障害は概ね4〜6因子以下で発生した。ただし全システムへの保証ではなく、NIST自身も「全欠陥検出の保証ではない」としている。→ 2因子(ペアワイズ)を費用対効果の高い出発点にし、重大性・過去欠陥・相互作用の複雑さに応じて3〜6因子へ引き上げる。強度を上げてもオラクル不足やモデル漏れは補えない。

**手順**:
1. 因子(パラメータ)と水準(取りうる値)を列挙。**制約**(あり得ない組み合わせ)も列挙
2. ツールで生成: PICT(`pict model.txt`)等。無ければ小さなスクリプトで生成してよいが、**全対象ペアの出現を機械検証する**(自己生成配列の網羅主張を検証なしで通さない)。手作業で組むのは9ケース程度まで
3. ベースチョイス(最頻構成)を必ず含め、生成結果に無ければ追加
4. 高リスク因子ペアは明示的に固定ペアとして指定
5. 低リスク領域は **Each Choice**(各水準を最低1回だけ登場させる。ケース数=最大水準数)まで落とす選択肢もある — カバレッジ保証の弱さを明記して使う

**落とし穴**: 因子が独立でないのに制約を書かず、実行不能ケースが混ざる。水準の選び方が同値分割になっていない(水準内の代表性がない)。「ペアワイズで網羅した」は2因子までの保証であることを成果物に明記。

## 5. CRUDテスト(振る舞いベース技法)

**狙う欠陥**: 作りっぱなし(Deleteが無い)、削除時の参照整合性、Update競合、権限別のRead漏れ。

**手順**:
1. エンティティ×機能のCRUDマトリクスを作る(test-analysis のモデルを流用)
2. 各エンティティで C→R→U→D のライフサイクル一巡ケース
3. 行/列の偏りを検査: Cだけあるエンティティ(ゴミが溜まる)、Dが無い(削除要件の欠落?)、複数機能からUされる(競合テスト候補)
4. Deleteは重点的に: カスケード/参照整合性、論理削除後の一意制約・集計・検索への影響、復元

**カバレッジ基準**: 全エンティティ×該当CRUDセルの網羅。Dには整合性ケース必須。

**落とし穴**: 権限別のRead(誰が読めるか)をマトリクスに含め忘れる。UIからは消せないがAPIからは消せるエンティティ。

## 6. シナリオベーステスト(振る舞いベース技法)

**狙う欠陥**: 機能単体は正しいがフローとして繋がらない、例外フローからの復帰不能、途中離脱でのデータ残骸。

**手順**:
1. メインシナリオ(目的達成の最短路)を1本
2. 分岐点ごとに代替シナリオ(別経路の成功)と例外シナリオ(失敗と復帰)を列挙
3. エッジシナリオ: 最小/最大構成、同時実行(在庫1つに2人 — 決定的再現は§13)、途中離脱→再開
4. 各シナリオの検証点は「最終状態+途中の副作用」(メール送信、在庫引当、監査ログ)

**カバレッジ基準**: メインシナリオ+全例外分岐を最低1シナリオでカバー。

**落とし穴**: シナリオテストで単機能の網羅を兼ねようとしてケースが爆発する — 単機能の網羅は技法1〜5に任せ、シナリオは「繋がり」の検証に徹する。

## 7. メタモルフィックテスト / プロパティ(ルールベース技法)

**狙う欠陥**: 正解が定義できないロジック(検索・推薦・ML・数値計算・集計)の誤り。

**手順**:
1. メタモルフィック関係(MR)を設計する: 「入力をこう変えたら、出力はこう変わる/変わらないはず」
   - 絞り込み: `count(query A AND B) ≤ count(A)`
   - 不変: `sort(shuffle(X)) == sort(X)`、`translate(translate(x, en), ja) ≈ x`
   - 加法: `sum(A ++ B) == sum(A) + sum(B)`(リスト連結)
   - 単調: 入力を増やせば結果は減らない/増えない
2. MRごとに複数のソース入力を流し、関係の成立を検証(実装は `ai-adaptable-testing` のPBTツールと相性がよい)
3. 非決定的システムは統計的判定(N回中の成立率と閾値を明記)

**落とし穴**: 弱いMR(ほぼ何でも通る関係)で満足する。MR自体が仕様の誤解を含む(MRはドメインエキスパートとレビューする)。

**LLM機能への適用**: 評価セット(入力コーパス×ルーブリック)を固定資産として保守する。**プロンプト変更・基盤モデル更新・知識コーパスの更新(RAG構成の検索対象文書の追加・改訂)は全面回帰に相当** — 変更ごとに評価セットを再実行し、スコア低下をリグレッションとして扱う。コーパス更新は反映機構のテスト(CRUD・データ品質expectation)だけでは足りず、更新後の回答品質の回帰を別途実行する。MRは「言い換え・無関係情報の付加に対する出力の安定性」として設計できる。

## 8. 探索的テスト・SBT(経験ベース技法)

**狙う欠陥**: スクリプトの網の目にかからない欠陥 — 違和感、使いにくさ、想定外の使い方、仕様の隙間。

**チャーター設計**(テンプレは SKILL.md ステップ6)と代表ヒューリスティック(Hendrickson『Explore It!』):

- **変数を1つずつ極端に**: 1項目だけ最大/最小/空にして他は普通
- **中断攻撃**: 処理中にキャンセル / リロード / ネットワーク断 / 戻るボタン / タイムアウト待ち
- **CRUD一巡+逆順**: 作って消して作り直す、消してから参照する
- **ペルソナ**: せっかちな初心者(連打・未入力送信)、パワーユーザー(ショートカット・一括操作)、攻撃者気分(URL直叩き、他人のID)
- **悪夢の見出し**: 「この製品が新聞沙汰になるとしたら何が起きたか」から逆算(Hendrickson)
- **ゾンビとりんご**: 定番の値だけでなく、現実の汚いデータ(移行データ、途中で仕様変更されたデータ)

**運用**: タイムボックスは45〜120分(標準90分。対象の不確実性が高いほど短く刻む)。セッションシート(カバレッジ・発見・新アイデア)を残し、時間内訳(テスト実行/バグ調査/セットアップ)を概算記録すると探索の実効率が見える(SBTM)。セッション後に短い**デブリーフ**を行い、発見の共有と次チャーターの見直しをする。発見された欠陥パターンはチェックリスト/自動テストへ昇格(探索→checkingへの還流)。

## 9. ファジング(coverage-guided)

**狙う欠陥**: クラッシュ・メモリ安全性違反・無限ループ・未捕捉例外 — 人手の値選定では到達できない入力空間での欠陥。信頼境界を越える入力の構文解析(パーサ・デシリアライザ・ファイル/画像処理・プロトコル実装)が主対象。

**手順**:
1. 信頼境界を越える入力を受けるエントリポイントを列挙(`agent-security-governance` の信頼境界と同じ地図が使える)
2. ハーネス(バイト列→対象関数)を設計。深い状態(認証後等)はハーネス側で状態を作らないとfuzzerが到達しない
3. オラクルを設計: クラッシュ / サニタイザ(ASan・UBSan)/ 性質(`parse(serialize(parse(x))) == parse(x)` の恒等)/ 差分(別実装との比較)
4. 正常系の代表入力をシードコーパスにする

**落とし穴**: クラッシュしか見ないと「落ちないが間違った結果」を見逃す — 性質オラクルを足す。ツール(libFuzzer/AFL++/Jazzer/fast-checkのfuzzモード等)のCI常時実行・コーパス運用は戦略層の領分。

## 10. フォールトインジェクション(レジリエンス設計)

**狙う欠陥**: タイムアウト未設定、リトライによる重複実行・リトライストーム、フォールバック未実装、部分障害時の不整合、カスケード障害。

**手順**:
1. **依存×障害モードのマトリクス**を作る: 各外部依存 × (停止 / 遅延 / エラー応答 / 不正応答 / 途中切断)
2. 各セルの期待挙動を**定常状態仮説**として書く(「依存Xが落ちてもコア機能YのSLOは維持される」)— **決まっていないセルは仕様の穴**(test-analysis の質問リストへ)
3. 注入点(テストダブル、Toxiproxy等のネットワーク層)と中止条件を設計
4. 回復もケース化: 依存復旧後に自動で正常へ戻るか(サーキットブレーカの状態遷移テストと接続)

**落とし穴**: 単一障害のみ設計して同時障害・復旧過渡期を見ない。本番カオス実験(GameDay)の運用は戦略/運用の領分 — ここで設計するのは仮説・blast radius・中止条件まで。

## 11. 契約ケース設計(サービス間連携)

**狙う欠陥**: provider の変更による consumer の破壊、スキーマ進化の非互換、未知のenum値での誤動作。

**手順**:
1. consumer が**実際に使う**相互作用(エンドポイント×参照フィールド)を列挙する。使っていないフィールドは契約に入れない(過剰契約は進化を阻害)
2. 相互作用ごとに provider state(データ前提)を定義
3. スキーマ進化の互換ケースを設計: フィールド追加(consumerは無視できるか)/ enum値追加(unknown値の処理)/ 必須→任意化 / フィールド削除(どのconsumerがいつ壊れるか)
4. イベント連携では順序逆転・重複配信も契約の一部として扱う

**落とし穴**: 契約テストで振る舞い(ビジネスロジック)まで検証しようとする — 契約は「形と前提」の合意に徹する。ツール(Pact等)とCI配線は `ai-adaptable-testing` の領分。

## 12. データ品質・パイプラインテスト

**狙う欠陥**: silent data corruption(落ちずに間違ったデータを流し続ける)、スキーマドリフト、部分ロード、再実行による重複。

**手順**:
1. データセットごとに **expectation** を設計: 量(行数の許容範囲)/ 一意性(キー重複)/ 参照整合性 / 値域・分布 / 鮮度(最終更新からの経過)
2. **突合(reconciliation)**: ソースとターゲットの件数・合計値の一致(移行の差分テストと同型)
3. 再実行・バックフィルの冪等性をケース化(同じ期間を2回処理して結果が変わらないか)

**落とし穴**: パイプラインの「成功」ステータスを品質の証拠と誤認する(成功して間違ったデータを流すのが最頻の障害)。expectation の実行基盤(dbt test / Great Expectations 等)の運用は戦略層。

## 13. 同時実行テスト(競合の決定的再現)

**狙う欠陥**: check-then-act競合(確認と更新の間への割り込み)、二重送信の二重実行、ロストアップデート、一意制約をすり抜けた重複作成、デッドロック・ロック待ちタイムアウト。

**手順**:
1. 競合対象(共有リソース: 残高・在庫・状態フィールド・一意キー・利用回数カウンタ)と競合ウィンドウ(check〜actの区間)を特定する
2. **「ほぼ同時に実行」を手順に書かない** — 競合ウィンドウを決定的に踏む手段を選び、ケースの手順に明記する:
   - **同期開始**: N本の要求をバリア/ラッチで同一時点に発火させる(unit/integration)
   - **遅延・フック注入**: checkとactの間に人工遅延やテストフックを仕込み、その間に2本目を完走させる(競合ウィンドウの強制拡大)
   - **トランザクション一時停止**: 1本目をコミット直前で保留 → 2本目を実行 → 1本目を再開(DB分離レベル・ロックの検証)
3. 決定的再現ができない層(E2E・実環境)では**反復予算と判定基準**を明記する(例: 100回反復し1回でも不変条件が破れたら失敗)。「N回で再現しなかった」は不在の証明にならないことを「このテスト群の限界」に記録する
4. 期待挙動は排他方式で異なる — 悲観ロック=後着の待機/タイムアウト、楽観ロック=後着の競合エラー+再試行導線、一意制約=後着の重複拒否。**どの方式か仕様に無ければ test-analysis の質問リストへ**
5. 判定は個々の応答の成否ではなく**結果状態の不変条件**で行う(引き落としは合計1回、在庫は負にならない、成立する遷移は1つだけ)— 全応答が「成功」でも二重実行はあり得る

**カバレッジ基準**: リスク分析で特定した共有リソース × 競合操作ペア(同一操作×2、相反操作の交差)の網羅。全機能の直積は不要。

**落とし穴**: タイミング任せのケースは「通ったが競合を踏んでいない」状態になる — 競合ウィンドウを踏んだことをログ・フック通過回数で観測する。flakyとして無効化された同時実行テストは競合バグの検出手段そのものの喪失(無効化ではなく決定的再現への置き換えを検討)。フォールトインジェクション(§10)は外部依存の障害用 — 競合再現は本節を使う。

## 14. 技法の重ね掛けパターン(高リスク領域の定石)

| 領域 | 重ね掛け |
|---|---|
| 決済・課金ルール | デシジョンテーブル × 境界値 × 同時実行(§13) × 冪等性(二重送信) |
