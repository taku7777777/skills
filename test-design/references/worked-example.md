# テスト設計スキル 適用例

## 題材

架空のECサイト「ミドリ雑貨店」の注文確定時クーポン適用機能(注文金額1〜999,999円、適用下限3,000円、定率10%オフ・上限2,000円・端数切り捨て/定額500円オフ、期限日当日23:59:59まで有効)。`test-analysis` の適用例(`test-analysis/references/worked-example.md`)の成果物を入力とする。

## 入力(ユーザーの依頼と文脈)

依頼: 「テスト分析が終わった。条件一覧(TC-01〜TC-08)からテストケースを設計してほしい」

入力の要点: 優先度・オラクル付きテスト条件8件、デシジョンテーブル(全8列)、変数表、探索チャーター候補1件(送料無料キャンペーン併用)。

## 各ステップでの判断

- **ステップ1(入力の確認)**: 条件・リスク・オラクル揃いのため簡易分析は不要。高リスク(TC-01〜04)=複数技法+ネガティブ、中(TC-05〜07)=主技法1つ、TC-08は production 監視条件のためテストケース化せず運用へ引き継ぐ。
- **ステップ2(技法の選択)**: 実装差分を読み欠陥仮説を列挙 — 割引計算に浮動小数点使用(端数)、下限判定の `>` / `>=` 取り違え、使用済みフラグ更新が check-then-act。決済ルールの定石(カタログ§14)に従い、デシジョンテーブル×境界値×同時実行(§13)を重ね掛け。期待値はすべて仕様とQ1〜Q3の回答から取る(実装からは仮説のみ)。
- **ステップ3(技法の適用)**: 分析のデシジョンテーブル(8列)を設計モデルとして引き継ぎ、高リスクのため縮小せず全列維持。境界値表を作成(1ケース1境界、他変数はイン固定)。全ケースにネガティブ系(E01/E02)を含めた。
- **ステップ4(テストデータ設計)**: 時刻境界はクロック注入で 2026-07-31T23:59:59 / 08-01T00:00:00 に固定。各値の選定理由を境界値表に記録。データはケースローカルのファクトリで生成、本番データ不使用。
- **ステップ5(レベル判断)**: unit/integration は自動化前提のロウレベル。E2E(D-14)は UI が流動的なためハイレベル。
- **ステップ6(探索チャーター)**: 併用領域に1本設計。判定に実機の表示確認と体験の違和感評価を含むため人間実行とした。
- **ステップ7(設計レビュー)**: DT列⇔ケースIDの対応をスクリプトで突合(8/8列)。境界オン/オフ全点のケース化を確認。副作用検証は一律申告にせず例外を明記(成果物2章)。

## 4. 探索チャーター

```yaml
チャーター: 送料無料キャンペーンとクーポンの併用を、適用順序と表示の一貫性を軸に探索し、金額計算・文言の矛盾を見つける
対象範囲: カート〜注文確定〜注文履歴の画面とAPI
時間枠: 90分(人間実行 — 表示の違和感評価を含むため)
使うヒューリスティック: 変数を1つずつ極端に(キャンペーン閾値ぎりぎりの金額)、中断攻撃(適用後にカートへ戻る/リロード)、CRUD一巡(適用→取消→再適用)
想定リスク: 割引適用順による金額差、片方だけ取消したときの表示残骸
記録: 発見事項/新たなテストアイデア/触れた範囲・触れなかった範囲
```

## 5. このテスト群の限界(見逃しうる欠陥と受容判断)

- 大量アクセス時の性能劣化は捕捉できない(性能テストは対象外 — 戦略層で別途計画することを条件に受容)
- D-13は方式変更(悲観ロック化等)で期待結果が変わる — 排他方式の変更時は再設計
- 併用領域はチャーター1本のみ — 探索の発見をchecking(自動ケース)へ昇格して補う
